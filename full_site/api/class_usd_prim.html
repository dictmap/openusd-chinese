逐段双语理解 / Paragraph-Level Bilingual Coverage

官方说明把 `UsdPrim` 称为 `UsdStage` 上唯一持久的 scenegraph object。这里的 persistent 指客户端可以长期保存一个轻量 handle，并通过它访问 stage 中当前组合出的 prim；它不是 layer 文件中的原始 spec，也不是把所有 authored opinions 复制到一个对象里。`UsdPrim` 的意义在于把 USD composition 的结果暴露为稳定、可查询、可编辑的运行时入口。

The official description calls `UsdPrim` the sole persistent scenegraph object on a `UsdStage`. In this context, persistent means that client code can hold a lightweight handle for accessing the currently composed prim on a stage. It is neither the raw spec inside a layer nor a copied container of every authored opinion; it is the runtime entry point to the composed result.

官方还说 `UsdPrim` is the principal container of other types of scene description。这个 container 包含 attributes、relationships、variant sets、references、payloads、inherits、specializes、metadata 和 applied API schemas 等接口。中文阅读时应把这些看作 prim 上可见的 composition 结果和可 author 的场景描述入口，而不是一个简单的 C++ 成员字段列表。

The official text also says that `UsdPrim` is the principal container of other types of scene description. The contained interfaces include attributes, relationships, variant sets, references, payloads, inherits, specializes, metadata, and applied API schemas. Read these as visible composed interfaces and authoring entry points on a prim, not as a simple list of C++ member fields.

`UsdPrim` 与 `SdfPrimSpec` 的分界很关键：`SdfPrimSpec` 是某个 `SdfLayer` 中的底层 prim spec，保存具体 authored opinions；`UsdPrim` 则是 `UsdStage` 在 layer stack、references、payloads、variants、inherits 和 specializes 等规则作用后呈现给客户端的 composed prim handle。调试“为什么值会这样组成”时，需要从 `UsdPrim` 进入 `GetPrimIndex()`、layer stack 或 `Pcp` 相关信息继续追踪。

The boundary between `UsdPrim` and `SdfPrimSpec` is essential. `SdfPrimSpec` is the low-level prim spec inside an `SdfLayer` and stores concrete authored opinions. `UsdPrim` is the composed prim handle exposed by a `UsdStage` after layer stacks, references, payloads, variants, inherits, and specializes have been applied. To debug why a value composes a certain way, use `UsdPrim` as the entry point and inspect `GetPrimIndex()`, the layer stack, or related `Pcp` data.

属性和关系 API 是 `UsdPrim` 最常用的部分。`GetAttribute()` / `CreateAttribute()` 处理 `UsdAttribute`，`GetRelationship()` / `CreateRelationship()` 处理 `UsdRelationship`，`GetProperties()`、property predicate 和 property order 则用于枚举或排序 property。创建操作会 author scene description；查询操作读取的是 stage 上可见的 composed result，二者在读写语义上不能混淆。

Property APIs are among the most commonly used parts of `UsdPrim`. `GetAttribute()` / `CreateAttribute()` work with `UsdAttribute`, `GetRelationship()` / `CreateRelationship()` work with `UsdRelationship`, and `GetProperties()`, property predicates, and property order support enumeration and ordering. Creation authors scene description, while queries read the visible composed result on the stage; these read/write semantics should not be conflated.

composition arc 相关 API 让 `UsdPrim` 成为资产组装的核心入口。`GetReferences()`、`GetPayloads()`、`GetInherits()`、`GetSpecializes()` 和 `GetVariantSets()` 分别对应不同的 composition 机制；它们可能在 layer 中 author 新意见，也可能用于检查当前 prim 的组合结构。把这些 arc 当作普通 metadata 会导致错误判断，因为它们直接参与 scene description 的强弱、选择和延迟加载。

Composition-arc APIs make `UsdPrim` central to asset assembly. `GetReferences()`, `GetPayloads()`, `GetInherits()`, `GetSpecializes()`, and `GetVariantSets()` correspond to different composition mechanisms. They may author new opinions in layers or inspect the current prim's composed structure. Treating these arcs as ordinary metadata is misleading because they affect strength ordering, selection, and deferred loading of scene description.

lifetime 相关函数决定一个 `UsdPrim` handle 是否仍可使用。`IsValid()`、`operator bool`、`GetStage()`、`GetPath()` 与 `GetPrimIndex()` 帮助确认 handle 是否仍对应 stage 中的有效对象；stage mutation、payload loading、composition 重新计算或 prim 移除都可能改变句柄的有效性。生产代码不应把 `UsdPrim` 当成永久裸指针，而应在关键读写点重新验证。

Lifetime-related functions determine whether a `UsdPrim` handle can still be used. `IsValid()`, `operator bool`, `GetStage()`, `GetPath()`, and `GetPrimIndex()` help confirm that the handle still corresponds to a valid object on the stage. Stage mutation, payload loading, recomposition, or prim removal can change validity. Production code should not treat `UsdPrim` like a permanent raw pointer and should revalidate at important read/write points.

遍历 API 体现 `UsdPrim` 的 scenegraph 位置。`GetChildren()`、`GetFilteredChildren()`、`SiblingIterator`、`SiblingRange`、`SubtreeIterator` 和 `SubtreeRange` 可用于兄弟节点或子树遍历；遍历结果会受到 active、loaded、defined、abstract、instance、model kind 和 predicate 等条件影响。做工具或导出器时，需要明确是遍历所有 prim、可加载 prim、活动 prim，还是满足某个 schema/type 的 prim。

Traversal APIs express the position of `UsdPrim` in the scenegraph. `GetChildren()`, `GetFilteredChildren()`, `SiblingIterator`, `SiblingRange`, `SubtreeIterator`, and `SubtreeRange` support sibling and subtree traversal. Results are affected by active, loaded, defined, abstract, instance, model kind, and predicate conditions. Tools and exporters should state whether they traverse all prims, loadable prims, active prims, or prims satisfying a schema/type predicate.

applied API schema 让 `UsdPrim` 获得额外行为声明。`ApplyAPI()`、`AddAppliedSchema()`、`RemoveAppliedSchema()`、`HasAPI()` 和 `GetAppliedSchemas()` 操作 `apiSchemas` 相关 authored metadata，并让 `UsdSchemaRegistry` 能识别这个 prim 上附加的 API schema。应用 schema 不等于自动补全所有属性值；仍然需要根据 schema 文档 author 或检查对应 attributes、relationships 和 tokens。

Applied API schemas attach additional behavior declarations to a `UsdPrim`. `ApplyAPI()`, `AddAppliedSchema()`, `RemoveAppliedSchema()`, `HasAPI()`, and `GetAppliedSchemas()` operate on authored metadata related to `apiSchemas` and allow `UsdSchemaRegistry` to recognize API schemas attached to the prim. Applying a schema does not automatically author all property values; matching attributes, relationships, and tokens still need to be authored or checked according to the schema documentation.

instancing 是阅读 `UsdPrim` 时最容易误解的部分之一。页面结构中的 Instancing 区域通常涉及 instance、prototype、instance proxy 和相关遍历限制。实例化可以显著减少重复 scene description 的开销，但也限制了某些 authoring 操作；如果工具需要编辑实例内部内容，应先确认当前 prim 是 ordinary prim、instance、prototype 还是 instance proxy。

Instancing is one of the easiest parts of `UsdPrim` to misread. The Instancing section usually involves instances, prototypes, instance proxies, and related traversal restrictions. Instancing can reduce repeated scene-description cost, but it also restricts some authoring operations. A tool that edits contents inside an instance should first determine whether the current prim is an ordinary prim, an instance, a prototype, or an instance proxy.

`UsdPrim` 也是 typed schema class 的基础载体。`UsdGeomMesh`、`UsdShadeOutput`、`UsdLuxDiskLight`、`UsdPhysicsJoint` 等类通常包装、验证或创建某个 `UsdPrim`，从而提供领域特定 API；但是 typed schema 不替代 `UsdPrim` 的通用 composition、metadata、property、lifetime 和 traversal 语义。遇到 schema API 看不懂时，回到 `UsdPrim` 的通用接口通常能解释底层行为。

`UsdPrim` is also the base carrier for typed schema classes. Classes such as `UsdGeomMesh`, `UsdShadeOutput`, `UsdLuxDiskLight`, and `UsdPhysicsJoint` usually wrap, validate, or create a `UsdPrim` to expose domain-specific APIs. A typed schema does not replace the general `UsdPrim` semantics for composition, metadata, properties, lifetime, and traversal. When a schema API is confusing, returning to the generic `UsdPrim` interface often explains the underlying behavior.
