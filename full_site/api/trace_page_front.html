中文主阅读路径

Trace 是 OpenUSD 的 Performance tracking 模块，用于 counting、timing、measuring 和 recording events。它不是优化算法本身，也不是任务调度器；它提供的是一套在代码路径中标记事件、收集事件、再生成报告的性能观察机制。读这页时可以按三层理解：先用 instrumentation 标出要观察的 scope，再用 TraceCollector 决定何时收集，最后用 TraceReporter 把事件转成 ASCII report 或 Chrome tracing report。Trace defines utility classes for counting, timing, measuring, and recording events.

官方 Overview 中最重要的边界是 TraceCollector 和 TraceReporter：TraceCollector 记录 TraceEvent 对象，回答“事件有没有被采集”；TraceReporter 根据 collector 里的事件生成报告，回答“这些事件怎样被汇总、排序和展示”。如果报告为空，先检查 collector 是否启用、目标代码是否执行、宏是否被编译进来；如果报告很大或解释困难，再检查 report 类型和聚合方式，而不是立刻认为被测代码没有耗时。TraceCollector records TraceEvent objects, while TraceReporter generates reports on events gathered by the collector.

Trace 的价值在于给复杂系统提供可对齐的性能证据。OpenUSD 里很多慢路径跨越 Sdf、Pcp、Usd、Hydra、插件加载、资源解析和多线程调度；单靠日志通常只能看到“发生了什么”，而 Trace 可以补上“在哪里花了多久、调用了多少次、哪些 scope 包含了哪些子 scope”。因此它更适合作为性能排查和回归比较工具，而不是替代 profiler、benchmark 或业务日志。Trace is useful for performance investigation across USD subsystems, but it does not replace profilers, benchmarks, or logs.

Instrumentation：用 TRACE 宏和 Python API 插桩

C++ 插桩通过在代码中加入 TRACE macros 完成。典型用法是在函数入口放 TRACE_FUNCTION()，在需要单独计时的内部范围放 TRACE_SCOPE("Inner Scope")，如果希望给函数内特定路径命名，则可使用 TRACE_FUNCTION_SCOPE("Slow path")。这些宏会把执行范围变成 Trace 可识别的事件边界，后续由 collector 记录、reporter 汇总。Instrumentation is done by adding TRACE macros such as TRACE_FUNCTION() and TRACE_SCOPE("Inner Scope") to code.

Python tracing 也受支持。页面示例通过 from pxr import Trace 引入模块，用 @Trace.TraceFunction 装饰函数，用 with Trace.TraceScope("Inner Scope"): 标记局部范围。这个设计让 Python orchestration 和 C++ 热路径可以出现在同一类性能证据里，适合排查 usdview 启动、脚本批处理、插件初始化或工具层调用中的跨语言耗时。Python tracing is supported through Trace.TraceFunction and Trace.TraceScope.

插桩要有边界。Trace 宏适合包住“需要解释的工作单元”，比如一次层栈解析、一段 composition、一次插件查找、一组 Hydra 任务或一段并行调度。它不适合无差别地撒在每个微小循环上，因为过多事件会稀释报告可读性，也会放大启用 trace 时的运行时开销。排查时应先从粗粒度 scope 建立轮廓，再对可疑路径加更细的 scope。Instrumentation should start from meaningful scopes and become more granular only where the report shows a suspicious path.

官方示例提到 small overhead，这一点很关键：即使 tracing disabled，宏也不是完全免费的。对性能敏感函数，如果只有少数 slow path 需要观察，应把 TRACE_FUNCTION_SCOPE("Slow path") 放在慢路径附近；如果一个路径会被极高频调用，优先评估是否需要抽样、合并 scope，或改用更上层的测量点。Trace macros have a small overhead even when tracing is disabled; slow-path-only instrumentation can reduce overhead.

Recording and Reporting：采集和报告

Recording 通过 TraceCollector 完成。C++ 示例里调用 TraceCollector::GetInstance().SetEnabled(true) 开始记录事件，执行目标代码后再调用 SetEnabled(false) 停止记录。只有 collector enabled 时，前面的 TRACE macros 才会真正把事件写入采集器；所以“已经插桩”和“已经采集”是两个不同状态。Recording is done through TraceCollector; enabling the collector causes TRACE macros to record events.

Reports 由 TraceReporter 生成。C++ 中 TraceReporter::GetGlobalReporter().Report(reportFile) 可以输出 ASCII report，ReportChromeTracing(chromeReportFile) 可以输出 Chrome tracing format。Python 侧对应 Trace.Reporter.globalReporter.Report("report.trace") 和 Trace.Reporter.globalReporter.ReportChromeTracingToFile("report.json")。前者更适合在终端或文本报告中看聚合调用图，后者适合在 Chrome tracing viewer 中看时间线和并发关系。TraceReporter generates ASCII reports and Chrome tracing reports through Report and ReportChromeTracing APIs.

默认 report 是 aggregated call graph，显示每个 scope 的 inclusive time、exclusive time 和 call count。inclusive time 表示包含子调用的总时间，exclusive time 表示当前 scope 自身消耗，call count 表示样本或调用次数。读报告时要区分“这个节点包含慢路径”和“这个节点自身慢”；前者需要继续下钻到子节点，后者才更像直接优化目标。The default report is an aggregated call graph showing inclusive time, exclusive time, and call count for each scope.

Chrome tracing format 的优势是时间线视角。它能帮助判断不同线程上的事件是否重叠、是否串行化、是否被某个等待点阻塞。对于 Work 或 Hydra 相关问题，如果 ASCII report 显示总体耗时很高，而问题可能来自并发调度或任务等待，Chrome report 往往比纯聚合树更容易解释。Chrome tracing output is useful when timeline and cross-thread relationships matter.

全局启用与工具层采样

官方页给出的全局开关示例是 PXR_ENABLE_GLOBAL_TRACE=1 usdview HelloWorld.usda --quitAfterStartup。这类用法适合在不改源码的情况下观察工具启动、插件加载、场景打开或初始化路径，但它也可能生成大量事件。使用时应保存输入文件、命令行、环境变量和 OpenUSD 构建版本，否则不同机器或不同插件集下的 trace report 很难横向比较。The official page shows PXR_ENABLE_GLOBAL_TRACE=1 usdview HelloWorld.usda --quitAfterStartup as a global tracing example.

全局 trace 不应该被理解成“打开后所有性能答案都会自动出现”。它只是扩大采集范围，仍然需要读者结合场景规模、资产结构、插件状态、线程限制和缓存状态解释结果。对回归问题，建议固定输入资产和命令，先收集基线 report，再收集变更后的 report，用相同 scope 名称比较 call count 和 inclusive/exclusive time 的变化。Global tracing is a collection mode; it still needs controlled inputs and careful report comparison.

Details：事件树、聚合树和报告解释

TraceReporter::GetAggregateTreeRoot 和 TraceReporter::GetEventTree 分别对应不同的分析视角。aggregate tree 把相同 scope 聚合起来，适合看总体热点；event tree 保留事件顺序和嵌套关系，适合看一次执行中的真实路径。性能排查通常先看 aggregate tree 找热点，再用 event tree 或 Chrome tracing report 解释这个热点在时间线上如何出现。GetAggregateTreeRoot and GetEventTree expose different views of the collected trace data.

如果报告中某个 scope 的 call count 异常高，优先怀疑它被放在了过细的循环里，或某个上层逻辑重复触发了同一工作。若 inclusive time 高而 exclusive time 不高，慢点通常在子 scope；若 exclusive time 高，才更值得看当前函数或当前范围本身。这个区分能避免把优化工作投向错误层级。Inclusive and exclusive timings help decide whether the cost lives in a parent scope or inside its children.

Trace report 的 scope 名称也会影响可读性。静态宏产生的名字稳定，适合长期对比；动态名字可以表达更多上下文，但如果包含路径、prim 名或频繁变化的字符串，聚合效果会变差，报告也会变得难读。因此需要在“可定位”和“可聚合”之间做取舍。Stable scope names make reports easier to aggregate and compare over time.

Performance Overhead：插桩开销和编译边界

官方开销说明很具体：tracing disabled 时，TRACE_FUNCTION()、TRACE_FUNCTION_SCOPE() 和 TRACE_SCOPE() 会初始化 16 bytes of stack memory、读取一个 atomic int，并有 2 branches。这个成本通常很小，但并不等于零；在高频路径中，即使禁用采集，也要避免过度插桩。When tracing is disabled, the static trace macros initialize 16 bytes of stack memory, read an atomic int, and have 2 branches.

当 tracing enabled 时，官方说明运行时开销大约比 disabled 时大 100 times larger，并给出 .33ns vs 33ns 这样的数量级比较。这个差异解释了为什么 trace 更适合短时间采样和定向排查，而不是长期在生产路径上无控制启用。需要长期监控时，应优先考虑更低成本的计数器、日志采样或外部 profiling 方案。The official page notes that enabled tracing can be around 100 times larger than the disabled overhead, such as .33ns vs 33ns.

TRACE_FUNCTION_DYNAMIC() 和 TRACE_SCOPE_DYNAMIC() 的开销显著更高，因为 scope 名称要在 runtime construct strings at runtime；静态版本的名字可以作为 constexpr data 编译进程序。除非确实需要运行时动态名称，否则优先使用静态 TRACE_FUNCTION()、TRACE_SCOPE() 或 TRACE_FUNCTION_SCOPE()。Dynamic trace macros cost more because they construct strings at runtime, while static names can be compiled as constexpr data.

如果构建或部署环境要求完全移除 trace 宏代码，官方页提到可以用 TRACE_ENABLE=0 禁用宏代码生成。另一个重要结论是 TraceCollector 和 TRACE macros 是 thread-safe 的；这让 Trace 可以用于多线程路径，但 thread-safe 不代表报告一定容易解释，仍然要结合 Work、Hydra 或 OpenExec 的并发模型看时间线。TRACE_ENABLE=0 disables macro code generation, and TraceCollector plus TRACE macros are thread-safe.

相邻模块关系与调试路径

Trace 与 Work 的关系是“观察并发工作”而不是“负责并发调度”。Work 决定任务如何分发、并发限制如何生效，Trace 记录相关 scope 的耗时和调用次数。排查 Work 性能时，可在任务提交、任务执行、等待或合并结果的位置加 Trace scope，再用 Chrome tracing format 看线程间关系。Trace observes work; Work schedules work.

Trace 与 Tf 的关系偏基础设施。Tf 提供诊断、类型和运行时基础能力，Trace 提供性能事件采集。Trace report 解释的是时间和调用关系，Tf 诊断解释的是消息、错误和运行时状态，两者经常一起用于定位复杂问题。Trace complements Tf diagnostics rather than replacing them.

Trace 与 Hd、Hdx 和 OpenExec 的关系主要体现在渲染和执行路径排查。Hydra 场景更新、render index 同步、adapter 转换或任务执行可能包含多个线程和多个子系统；Trace 能帮助把这些阶段拆成可比较的 scope，但渲染结果正确性仍要回到 Hd/UsdRender/UsdGeom 等模块检查。Trace can help measure Hydra and OpenExec-related paths, but correctness remains in the consuming subsystem.

Trace 与 Sdf、Pcp、Plug 的关系常见于场景打开和 composition 变慢。Sdf 负责层和路径数据，Pcp 负责 composition 索引和 layer stack，Plug 影响插件发现与加载；Trace 可以把这些阶段的耗时串起来，帮助判断慢点来自数据规模、composition 规则、插件初始化还是下游消费。Trace can connect timing evidence across Sdf, Pcp, Plug, and downstream consumers.

常见误读和边界

  • 不要把 TraceCollector 当成报告器。collector 负责采集事件，reporter 才负责生成 ASCII report 或 Chrome tracing report。TraceCollector collects; TraceReporter reports.
  • 不要认为 disabled trace 完全没有成本。官方已经说明静态宏仍会有少量栈内存、atomic read 和 branch 成本。Disabled trace macros still have a small overhead.
  • 不要在高频循环里滥用 dynamic trace macros。TRACE_FUNCTION_DYNAMIC() 和 TRACE_SCOPE_DYNAMIC() 会运行时构造字符串，开销和报告碎片化风险都更高。Dynamic trace macros are higher overhead and can fragment reports.
  • 不要把高 inclusive time 直接等同于当前 scope 自身慢。先比较 exclusive time 和子 scope，再决定优化目标。High inclusive time may come from children rather than the current scope.
  • 不要把 Chrome tracing report 当成唯一输出。文本聚合报告适合快速找热点，Chrome report 适合解释时间线和并发关系，两者服务于不同问题。ASCII reports and Chrome tracing reports answer different questions.

建议的排查流程

第一步，确认问题类型：是启动慢、打开场景慢、composition 慢、渲染同步慢，还是某个工具脚本慢。第二步，选择粗粒度 scope，先包住阶段边界，避免一开始就插太细。第三步，用 TraceCollector::GetInstance().SetEnabled(true) 或 Python 的 Trace.Collector().enabled = True 收集一次受控样本，并同时记录输入资产、命令行、环境变量和构建版本。Start with the problem type, then choose coarse scopes and collect controlled samples.

第四步，先读 aggregated call graph，查看 call count、inclusive time 和 exclusive time。第五步，如果怀疑并发或等待问题，再导出 Chrome tracing report，并在时间线上看不同线程的事件关系。第六步，只对可疑 scope 加更细插桩；如果发现事件过多或名称过于动态，应先整理 scope 名称，而不是继续扩散插桩。Use aggregated reports for hotspots and Chrome tracing reports for timeline questions.

最后，Trace 结果应与其他证据合用。性能基准可以证明总体耗时是否回归，日志和 Tf 诊断可以说明错误或状态，Trace 负责解释“耗时分布在哪里”。只有这几类证据一致时，才适合把某个子系统、某段代码或某个插件加载路径确定为主要瓶颈。Trace evidence should be interpreted alongside benchmarks, logs, and diagnostics.
