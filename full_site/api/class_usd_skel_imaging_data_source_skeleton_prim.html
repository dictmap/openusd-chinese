逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdSkelImagingDataSourceSkeletonPrim` 是 UsdSkel `Skeleton` prim 的 imaging data source。官方说明 “A prim data source for UsdSkel's Skeleton” 的重点是 data-source 表达层，而不是 schema authoring 或动画求解层：它把一个 skeleton prim 包装成 Hydra 可以按名称查询的 `HdContainerDataSource` 风格对象，使 scene index / Hydra 查询路径能读取骨架 prim 的成像数据。

A prim data source for UsdSkel's Skeleton.

这个类位于 USD prim 和 Hydra data source 之间。`UsdPrim` 和 `SdfPath` 标识数据来自哪个 stage、哪个 prim；`UsdImagingDataSourceStageGlobals` 提供成像层共享上下文；`UsdImagingDataSourcePrim` / `UsdImagingDataSourceGprim` 提供通用 prim / gprim data-source 行为；`HdContainerDataSource` 定义按 token 名称暴露子数据源的容器协议。它不是直接遍历 stage 的工具，也不是替代 `UsdSkelSkeleton` 的 schema wrapper。

The class adapts a USD skeleton prim into the Hydra data source model.

`GetNames()` 返回该 container data source 当前可查询的 `TfTokenVector`。调试 skeleton 数据缺失时，应先看 `GetNames()` 是否包含预期字段，再用对应 `TfToken` 调 `Get()`；如果 name 不存在，通常说明 data source 层没有发布该字段，或者 locator / adapter 路径没有把相关数据接入，而不是 mesh deformation 计算失败。

GetNames() returns the names of child data sources exposed by the container.

`Get()` 根据 `TfToken` 名称返回对应的 `HdDataSourceBaseHandle`。它是按需查询入口，不意味着一次性展开所有 skeleton 数据，也不保证返回值一定非空。调用方需要按 Hydra data-source 约定解释返回 handle 的具体类型，并把空返回、类型不匹配和 locator 失效分别排查。

Get() retrieves a named child data source from the container.

因此，调用 `Get()` 的代码应把 token name 视为 data-source contract，而不是把它直接等同于 USD attribute name；这也是定位数据缺口的第一步。

The token name used by Get() is a data-source contract, not automatically a USD attribute name.

与 skinning 和 animation 的边界必须明确：`UsdSkelImagingDataSourceSkeletonPrim` 负责把 skeleton prim 的相关信息放到 imaging data-source 层；joint order、bind transforms、animation sampling、skinning matrices 和 mesh deformation 属于 UsdSkel schema、animation evaluation 或下游渲染/计算路径。看到角色姿态不对时，不应直接把问题归因于这个 data source；应先区分数据是否暴露、数据是否被 scene index 消费、以及变形计算是否正确。

This data source exposes skeleton prim data; it is not the skinning or animation solver.

与 Hydra scene index 的关系也要分清。`HdDataSourceLocator` 和 `HdDataSourceLocatorSet` 描述数据源路径和失效范围；当 skeleton prim 上的 USD 属性变化时，成像层需要把变化映射到对应 locator，使 Hydra 知道哪些数据需要重新读取。该类本身提供查询表面，失效传播和 dirty tracking 还依赖 adapter、scene index、stage globals 和 delegate 流程。

Data source locators describe where values live and what ranges are invalidated.

在实际排查中，`GetNames()` 与 locator 失效报告要一起看：前者说明当前 data source 公开了哪些字段，后者说明哪些字段因为 USD 变化而应被重新拉取。两者不一致时，常见原因是 adapter 没有把 skeleton-specific 字段加入 container，或者 dirty locator 没有精确映射到 skeleton data source 子路径。

GetNames() and data source locators should be checked together when diagnosing missing or stale skeleton data.

继承链中的 `UsdImagingDataSourceGprim` 提示 skeleton prim 在 imaging 侧仍享有部分 gprim / prim 通用数据源能力，例如 path、visibility、purpose、transform 或其他继承字段的处理方式。阅读时应把骨架专用字段与通用 prim 字段分开：骨架字段解释 UsdSkel skeleton 的成像信息，通用字段解释 prim 在 Hydra data source 树中的基础元数据。

Inherited data-source classes separate generic prim data from skeleton-specific data.

常见误读包括：把 `GetNames()` 当成 USD 属性列表；把 `Get()` 当成直接读取 `UsdAttribute`；把 `UsdSkelImagingDataSourceSkeletonPrim` 当成 `UsdSkelSkeleton` 的 authoring API；把 data source 层错误当成动画求解错误；或者忽略 `UsdImagingDataSourceStageGlobals` 中的成像上下文。正确排查顺序通常是：确认 prim 是有效 Skeleton，确认 `SdfPath` 和 stage globals 正确，检查 `GetNames()` 暴露的 token，按 token 调 `Get()`，再沿 locator / adapter / scene index 追踪数据是否被 Hydra 消费。

Common debugging separates USD attribute authoring, data-source exposure, locator invalidation, and downstream animation or rendering behavior.
