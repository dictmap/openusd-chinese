逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdValidationError` 是 validation task 返回给调用方的结构化错误实体。它不是 `UsdValidationValidator` 本身，也不是一段普通字符串；它把验证器发现的问题拆成可定位、可分类、可展示、可筛选、可关联 fixer 的字段，便于 UI、批处理工具和持续集成系统按同一套结构消费验证结果。

`UsdValidationError` is a structured entity returned by a validation task. It is associated with a `UsdValidationValidator`, but it is not the validator itself.

官方定义中特别强调 associated with a `UsdValidationValidator`。这意味着错误对象需要保留“由哪个 validator 产生”的上下文，调用方可以通过 `GetValidator()` 追溯规则来源。调试时不要只看 message 文本；validator 上下文通常决定了错误的规则名称、修复策略、严重程度解释和是否允许在管线中自动处理。

The validator association explains provenance: the error object carries the result, while the validator defines the rule that produced it.

`GetName()` 返回 validator writer 为该错误提供的名称，`GetIdentifier()` 则把名称转换成更稳定的 identifier。面向用户的界面可以显示 name，但自动化过滤、去重和统计通常应使用 identifier；如果只比较 message 字符串，换行、局部化文本或补充说明变化都会导致不稳定。

`GetName()` exposes the error name, while `GetIdentifier()` provides a stable identifier suitable for filtering, grouping, and diagnostics.

`UsdValidationErrorType` 表示错误类型或严重性。它回答的是“这条发现应该如何被对待”，例如是否只是提示、是否应阻断发布、是否代表验证执行状态。中文说明必须保留 `UsdValidationErrorType` 原名，因为它是 API 类型名，不应翻译成模糊的“级别对象”。

`UsdValidationErrorType` describes the severity or type of the error and should be consumed as structured data rather than prose.

`UsdValidationErrorSites` 是错误定位集合，通常指向 prim、property、layer 或其他可报告位置。`GetSites()` 的价值在于让工具能够高亮场景中的具体对象、批量过滤受影响范围、或者把错误落到某个 layer / property 上；如果只保留 message，调用方就失去了可导航的定位信息。

`UsdValidationErrorSites` identifies where the validation error was reported, enabling UI highlighting, filtering, and navigation.

`GetMessage()` 与 `GetErrorAsString()` 都提供可读输出，但用途不同。`GetMessage()` 是错误关联的说明文本，适合展示给用户；`GetErrorAsString()` 更像日志和调试格式化入口。结构化消费端不应只解析字符串，而应同时读取 identifier、type、sites、data 和 fixer 列表。

`GetMessage()` and `GetErrorAsString()` are useful for display and logs, but structured tools should read the individual fields directly.

`GetData()` 暴露附加数据，通常用于携带验证器想返回的上下文，例如原始值、期望范围、修复所需参数或更细的诊断信息。它的边界类似 payload：调用方可以利用它增强报告或 fixer 输入，但不能假设所有 validator 都提供同一种 data schema。

`GetData()` provides additional validation-specific payload; consumers should treat it as structured context whose schema depends on the producing validator.

fixer 相关 API 包括 `GetFixers()`、`GetFixerByName()`、`GetFixerByNameAndErrorName()`、`GetFixersByErrorName()` 与 `GetFixersByKeywords()`。这些函数说明 validation error 可以连接到可执行修复策略，但“存在 fixer”不等于应该无条件自动修改 USD；管线仍需要检查权限、目标 layer、变更风险和用户确认。

Fixer lookup APIs connect an error to possible remediation actions, but choosing and executing a fixer remains a pipeline or tool policy decision.

`HasNoError()` 表示无错误或空错误状态，常用于占位返回值或快速判断。它不是把错误降级成 warning，也不表示 validator 没有运行；调用方应把它看作“该对象不携带实际错误”的状态，并和空 sites、空 message 或默认构造的对象区别开来。

`HasNoError()` represents a no-error state for the error object; it is not a severity downgrade and does not by itself describe whether validation ran.

`operator==()` 与 `operator!=()` 服务于错误集合的比较、去重和测试断言。比较语义应围绕结构化字段理解，而不是只围绕用户可见字符串。对于 CI 报告或批量验证结果，稳定比较能减少重复报警，也能帮助检查同一 validator 是否在不同 stage 或 layer 上产生了相同问题。

The comparison operators support deduplication and tests over structured validation errors rather than ad hoc string comparison.

常见误读有四类：第一，把 `UsdValidationError` 当成 validator；第二，只记录 `GetErrorAsString()` 而丢弃 sites 和 type；第三，把 fixer 当成一定会执行的自动修复；第四，把 `HasNoError()` 当成验证未运行。正确理解是：本类是验证结果的数据模型，规则执行在 `UsdValidationValidator`，定位在 `UsdValidationErrorSites`，严重性在 `UsdValidationErrorType`，修复入口由 fixer 查询 API 暴露。

The class is the data model for a validation finding: rule provenance comes from `UsdValidationValidator`, location from `UsdValidationErrorSites`, severity from `UsdValidationErrorType`, and remediation from fixer lookup APIs.
