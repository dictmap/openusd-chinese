逐段双语理解 / Paragraph-Level Bilingual Coverage

官方说明把 `UsdGeomMesh` 定义为 encodes a mesh with optional subdivision properties and features。这里的 mesh 是一个 typed schema，包装的是 stage 上的 `UsdPrim`；它定义如何读取和创作 mesh 所需的属性，而不是单独持有几何内存。`Define()` 会在 stage 上 author 一个 typed prim，`Get()` 只是为已有 prim 提供 `UsdGeomMesh` 视图。

The official description defines `UsdGeomMesh` as encoding a mesh with optional subdivision properties and features. This mesh is a typed schema wrapping a `UsdPrim` on a stage; it defines how mesh-related attributes are read and authored rather than owning geometry memory. `Define()` authors a typed prim on the stage, while `Get()` returns a `UsdGeomMesh` view for an existing prim.

作为 point-based primitive，mesh 的几何基础来自 `points`，该属性通常继承自 `UsdGeomPointBased`。points 是几何点数组，mesh 拓扑通过索引引用这些点。调试几何错误时，先检查 points 数量、索引范围和 time-sampled 数据是否一致，再检查 subdivision 或 primvars，否则容易把拓扑问题误判为渲染问题。

As a point-based primitive, the geometric basis of a mesh comes from `points`, usually inherited from `UsdGeomPointBased`. Points form the geometric point array, and topology references those points through indices. When debugging geometry problems, first check point count, index ranges, and consistency of time-sampled data before blaming subdivision or primvars.

官方刻意避免混用 vertex 一词，因为不同资料会把 vertex 当作 point，也会当作 face-vertex。OpenUSD 在本页中用 points 表示几何点，用 face-vertices 表示面中引用 points 的位置。这个区分很重要：一个 point 可以被多个 face-vertices 引用，而 face-varying 数据通常绑定到 face-vertex 级别。

The official text intentionally avoids the ambiguous term vertex because some references use it for points while others use it for face-vertices. OpenUSD uses points for geometric points and face-vertices for positions inside faces that reference points. This distinction matters because one point can be referenced by many face-vertices, and face-varying data is usually bound at the face-vertex level.

mesh 的面拓扑由 `faceVertexCounts` 和 `faceVertexIndices` 共同编码。`faceVertexCounts` 给出每个 face 包含多少个 face-vertices；`faceVertexIndices` 按 face 顺序展开所有 face-vertices 并存储 point 索引。`sum(faceVertexCounts)` 必须匹配 `faceVertexIndices` 长度，且每个索引必须落在 points 数组范围内。

Mesh face topology is encoded by `faceVertexCounts` and `faceVertexIndices` together. `faceVertexCounts` stores the number of face-vertices in each face; `faceVertexIndices` flattens all face-vertices in face order and stores point indices. `sum(faceVertexCounts)` must match the length of `faceVertexIndices`, and every index must be within the points array.

`subdivisionScheme` 决定消费端把 mesh 解释为 polygonal mesh 还是 subdivision surface。`interpolateBoundary`、`faceVaryingLinearInterpolation` 和 `triangleSubdivisionRule` 进一步控制边界、face-varying 数据和三角面细分规则。这些属性影响解释和渲染语义，不等于直接改写 authored topology。

`subdivisionScheme` determines whether consumers interpret the mesh as a polygonal mesh or as a subdivision surface. `interpolateBoundary`, `faceVaryingLinearInterpolation`, and `triangleSubdivisionRule` further control boundary behavior, face-varying data, and triangle subdivision rules. These attributes affect interpretation and rendering semantics; they do not directly rewrite authored topology.

creases、corners 和 holes 是 subdivision features 的核心属性组。`creaseIndices`、`creaseLengths` 和 `creaseSharpnesses` 需要一起理解：indices 给出折痕经过的 point 序列，lengths 划分每条 crease，sharpnesses 给出锐度。`cornerIndices`/`cornerSharpnesses` 控制尖角，`holeIndices` 标记不参与显示或细分的面。

Creases, corners, and holes are central subdivision feature groups. `creaseIndices`, `creaseLengths`, and `creaseSharpnesses` must be interpreted together: indices provide point sequences, lengths divide those sequences into individual creases, and sharpness values control crease strength. `cornerIndices`/`cornerSharpnesses` describe sharp corners, and `holeIndices` mark faces excluded from display or subdivision.

interpolation 相关属性经常和 primvars 一起造成误读。`UsdGeomMesh` 的核心 schema 属性定义 topology、subdivision 和 feature behavior；UV、颜色、法线或其他 primvars 通常通过 `UsdGeomPrimvarsAPI` 或相关 API 表达。读取 mesh 时要同时检查 primvar interpolation、element size 和 face-varying 数据长度是否与 topology 匹配。

Interpolation properties are often misread together with primvars. Core `UsdGeomMesh` schema attributes define topology, subdivision, and feature behavior; UVs, colors, normals, and other primvars are usually expressed through `UsdGeomPrimvarsAPI` or related APIs. When reading a mesh, check primvar interpolation, element size, and face-varying data length against the topology.

属性创建方法如 `CreateFaceVertexCountsAttr()`、`CreateFaceVertexIndicesAttr()`、`CreateSubdivisionSchemeAttr()` 和 `CreateInterpolateBoundaryAttr()` 会 author 对应属性或返回可写的 `UsdAttribute`。这些方法不会替调用者验证完整 mesh 合法性；工具仍需检查数组长度、索引范围、time sample 对齐和 fallback token 是否符合使用场景。

Attribute creation methods such as `CreateFaceVertexCountsAttr()`, `CreateFaceVertexIndicesAttr()`, `CreateSubdivisionSchemeAttr()`, and `CreateInterpolateBoundaryAttr()` author or return writable `UsdAttribute` objects. They do not validate a complete mesh for the caller; tools still need to check array lengths, index ranges, time-sample alignment, and fallback tokens for the intended use case.

`UsdGeomMesh` 与 `UsdPrim` 的关系和前几轮晋级页直接相连。`UsdPrim` 提供 composition 后的 prim 句柄，`UsdGeomMesh` 在其上提供 mesh schema 视图；若 prim 的 type name 或 schema 兼容性不正确，typed schema wrapper 可能存在但语义上不可用。生产工具应先确认 prim 有效，再确认 schema 类型和关键属性存在。

The relationship between `UsdGeomMesh` and `UsdPrim` connects directly to the previously promoted pages. `UsdPrim` provides the composed prim handle, and `UsdGeomMesh` provides a mesh schema view over it. If the prim type name or schema compatibility is wrong, a typed schema wrapper may exist but not be semantically useful. Production tools should validate the prim first, then check schema type and required attributes.

在导出、导入和渲染链路中，mesh 质量问题通常来自三类边界：拓扑数组不一致、subdivision/crease/corner/hole 属性组不匹配、primvars 与 face-varying topology 不对齐。本页阅读重点就是把这三类问题分开，避免把 `UsdGeomMesh` 当成只包含 points 和 indices 的简单多边形容器。

In export, import, and rendering pipelines, mesh quality problems often fall into three boundary cases: inconsistent topology arrays, mismatched subdivision/crease/corner/hole attribute groups, and primvars that do not align with face-varying topology. The key reading task on this page is to separate those cases and not treat `UsdGeomMesh` as a simple polygon container containing only points and indices.
