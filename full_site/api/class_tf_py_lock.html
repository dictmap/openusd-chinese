逐段双语理解 / Paragraph-Level Bilingual Coverage

`TfPyLock` 是 Tf 模块提供给 Python 绑定层和 C++ 扩展代码的便利类，用来集中管理 Python Global Interpreter Lock，也就是通常说的 GIL。官方说明强调 Python API 本身不是 thread-safe；如果 C++ 线程不是在 Python execution context 中运行，却要访问 Python API，就必须显式处理 GIL 和 Python thread state，否则同一进程内的多个线程可能破坏解释器状态。

`TfPyLock` is a convenience class for accessing the Python Global Interpreter Lock. Python API access from outside Python execution needs extra care when multiple threads may be present.

本页解决的问题不是 USD scene graph 的并发写入，也不是 `UsdStage`、`SdfLayer` 或 Hydra 数据结构的互斥保护；它只处理“当前 C++ 代码是否安全持有或释放 Python GIL”这一层。把 `TfPyLock` 当成通用 mutex 会误导调试方向：它保护的是 Python interpreter interaction，而不是 OpenUSD 数据模型本身。

The class is about Python interpreter locking and thread state, not about locking USD scene data structures.

`Acquire()` 和 `Release()` 是最直接的锁状态操作。`Acquire()` 表示当前线程进入需要访问 Python API 的区域，需要拿到 GIL 并建立/恢复合适的 Python thread state；`Release()` 表示离开这段 Python API 访问区域，释放 GIL。调用顺序必须符合页面中的 `State Valid Transitions`，否则可能出现重复释放、未持锁访问 Python API 或 thread state 不匹配。

`Acquire()` and `Release()` manage direct acquisition and release of the GIL, and their order must obey the documented state transitions.

`BeginAllowThreads()` 和 `EndAllowThreads()` 对应 Python C API 中“暂时释放 GIL 让其他 Python 线程运行，然后重新获取 GIL”的模式。典型场景是：C++ 代码已经从 Python 进入并持有 GIL，但接下来要执行较长的纯 C++ 工作；这时可以用 `BeginAllowThreads()` 放开 GIL，避免阻塞其他 Python 线程，随后用 `EndAllowThreads()` 恢复原来的 Python thread state。

`BeginAllowThreads()` and `EndAllowThreads()` bracket the pattern of releasing the GIL around long-running C++ work and then restoring the Python thread state.

构造函数 `TfPyLock()` 和析构函数 `~TfPyLock()` 是 RAII 生命周期边界。阅读本页时要把对象作用域、异常路径和锁状态恢复放在一起考虑：如果对象在函数退出、异常抛出或早返回时析构，析构函数必须把锁状态恢复到合法状态。也正因为如此，本类比散落调用 Python C API 更适合放在 OpenUSD 的 Python/C++ 桥接代码中。

`TfPyLock()` and `~TfPyLock()` provide RAII boundaries so lock state can be restored when the object leaves scope.

`State Valid Transitions` 是本页的核心阅读区。它不是装饰性说明，而是告诉读者哪些状态机路径是合法的：例如先 acquire 再 release，或在已持锁上下文中 begin-allow-threads 再 end-allow-threads。调试 deadlock、崩溃或 Python finalization 相关问题时，应该优先对照这个状态转换表，而不是只看单个方法名。

The `State Valid Transitions` section describes the legal state-machine paths and is essential for debugging lock misuse.

`TfPyEnsureGILUnlockedObj` 出现在 friends/related function 区块，说明 `TfPyLock` 还与其他 RAII 辅助对象协同，形成一套 Python/C++ 边界锁协议。这里的 friend 关系不是业务层访问授权，而是为了让相关锁管理对象能共享内部状态，安全地表达“确保 GIL 已解锁”或“确保 GIL 已恢复”的场景。

`TfPyEnsureGILUnlockedObj` is related to the same RAII lock-management protocol at the Python/C++ boundary.

常见误读包括：把 GIL 当成 OpenUSD 对象锁；在没有检查当前 thread state 的情况下调用 Python API；把 `BeginAllowThreads()` 当成普通 unlock 而忘记配对 `EndAllowThreads()`；在异常路径上绕过 RAII 对象；以及以为 `TfPyLock` 能解决所有跨线程 USD 访问问题。正确用法是：在需要进入 Python API、离开 Python API、或临时让出 GIL 的边界上，用它表达清晰、可恢复的状态转换。

Common misuses include treating the GIL as a USD object lock, forgetting to pair `BeginAllowThreads()` with `EndAllowThreads()`, or bypassing RAII restoration paths.
