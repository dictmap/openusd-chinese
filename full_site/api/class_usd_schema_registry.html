逐段双语理解 / Paragraph-Level Bilingual Coverage

官方定义的第一层含义是“singleton registry”：`UsdSchemaRegistry` 是 USD schema 系统的集中查询入口，不是一个会在 stage 上写入意见的 authoring API。它回答的是“某个已注册 schema 声明了哪些类型信息、属性、metadata、fallback 和 prim definition”，而不是“某个 `UsdPrim` 当前在 composition 后持有什么 authored value”。调试时如果看到 prim 上没有预期属性，应先区分是 schema definition 没注册、API schema 没应用、还是 layer 中没有 authored opinion。

The official description starts with the idea of a singleton registry: `UsdSchemaRegistry` is the central lookup point for USD schema metadata, not an authoring API that writes opinions to a stage.

本页中的 `IsA` schema 指 concrete typed schema，例如一个 prim 的 primary type 所对应的 schema；applied API schema 则是通过 `apiSchemas` 或 `UsdPrim::AddAppliedSchema(const TfToken&) const` 这样的路径附加到 prim 上的行为或属性集合。`UsdSchemaRegistry` 同时覆盖这两类 schema，因为最终可见的 prim definition 往往来自 primary schema 与多个 applied API schema 的组合。

The page covers both registered Usd `IsA` schema types and applied API schema types because a composed prim definition can combine a primary schema with applied API schemas.

官方摘录提到的数据来源是每个 schema-defining module 里的 `schema.usda` 经 `usdGenSchema` 处理后生成的 `generatedSchema.usda`。这句话很关键：registry 不是临时扫描 scenegraph 得到属性，而是消费生成式 schema 数据。遇到新插件或自定义 schema 不生效时，应检查插件是否注册、schema 类型是否暴露、`generatedSchema.usda` 是否包含期望声明，而不是只检查当前 stage 文件。

The registry contains data generated from `schema.usda` into `generatedSchema.usda`; that generated schema data is what prim definitions use for properties and fallback values.

`UsdSchemaRegistry::SchemaInfo` 保存单个 schema 的注册信息。它把 `TfType`、schema type name、family、version、`UsdSchemaKind` 等元数据连接起来。`FindSchemaInfo()` 的多个重载用于从不同标识查找同一个 schema identity；`FindSchemaInfosInFamily()` 则面向 schema family 和 version 查询。读取这些接口时要保留 exact type name、family name、versioned schema 之间的区别，避免把版本化 schema 当作一个普通 token 字符串处理。

`SchemaInfo` records the metadata for a registered schema. `FindSchemaInfo()` and `FindSchemaInfosInFamily()` are lookup paths for schema identity, family, and version information.

`VersionPolicy` 是处理 schema family 时的版本选择策略。它不是 composition strength，也不是 layer stack 规则；它只约束 registry 在面对一个 schema family 的多个版本时如何选择或匹配 schema 信息。把它和 value resolution 或 layer strength 混在一起，是读本页最常见的误区之一。

`VersionPolicy` is about schema-family version lookup, not layer strength or value resolution.

`FindConcretePrimDefinition()`、`FindAbstractPrimDefinition()` 和 `FindAppliedAPIPrimDefinition()` 是 prim definition 查询的三条主要路径。concrete prim definition 面向可实例化的 typed prim；abstract prim definition 描述不可直接实例化但可作为 schema 继承基础的定义；applied API prim definition 描述 API schema 带来的属性、metadata 和 fallback。三者返回的对象都帮助解释“schema 声明了什么”，但用途和合法输入并不相同。

The `FindConcretePrimDefinition()`, `FindAbstractPrimDefinition()`, and `FindAppliedAPIPrimDefinition()` methods distinguish concrete typed schemas, abstract schemas, and applied API schemas.

`BuildComposedPrimDefinition()` 用于把 primary schema 与 applied API schemas 组合成一个综合的 prim definition。这个接口解释了为什么某个 prim 的“可用属性集合”可能不只来自 primary type：例如 typed schema 提供基础属性，多个 API schema 再叠加额外属性、metadata 或 fallback。它仍然是在构建定义视图，不是在当前 layer 上 author 属性值。

`BuildComposedPrimDefinition()` builds a composed prim definition from a prim type and applied API schemas, explaining how multiple schema sources can contribute properties and fallback metadata.

`GetAPISchemaTypeName()`、`GetAPISchemaCanOnlyApplyToTypeNames()` 和 `CollectAddtionalAutoApplyAPISchemasFromPlugins()` 这一组接口围绕 API schema 的名字映射、可应用范围和 auto-apply 规则展开。注意官方页面里 `CollectAddtionalAutoApplyAPISchemasFromPlugins()` 的拼写保留为原 API 名称；阅读或搜索时不要自行改成另一个拼法。

The API-schema helpers cover API schema type-name mapping, allowed target type names, and additional auto-apply API schemas contributed by plugins.

`TokenToTokenVectorMap`、`TfToken` 和 `TfHash` 说明这个 registry 大量使用 token 化名称和 token-vector 映射来保存 schema 规则。应用层代码应保留 `TfToken` 的精确 spelling，尤其是 schema type name、API schema instance name、property name 和 fallback prim type 名称；大小写或命名空间差异都可能导致查找失败。

The `TokenToTokenVectorMap` typedef and related links to `TfToken` indicate that schema names and rule maps are token based; exact token spelling matters.

本页与 `UsdPrimDefinition` 的边界是：`UsdSchemaRegistry` 负责找到或构建 definition，`UsdPrimDefinition` 才承载属性声明、metadata、fallback 等具体定义内容。与 `UsdSchemaBase`、`UsdTyped`、`UsdAPISchemaBase` 的边界是：这些类是用户拿来访问 prim 或 schema 行为的 C++ facade；registry 则是这些 facade 背后的 schema metadata 查询层。

`UsdSchemaRegistry` locates or builds prim definitions, while `UsdPrimDefinition` stores the concrete declared properties, metadata, and fallback information.

本页与 `UsdStage::WriteFallbackPrimTypes` 和 “Fallback Prim Types” 的链接表明 registry 也参与 fallback prim type 的解释链路。fallback 不是 authored prim type，也不是某个 layer 中显式写入的默认值；它是 schema 系统在缺少 authored type 或需要解释 schema 默认行为时可查询的定义信息。排查 fallback 行为时，应该同时检查 schema registry、stage 写出的 fallback prim types 和实际 layer opinions。

The links to `UsdStage::WriteFallbackPrimTypes` and fallback prim types connect registry metadata to fallback-type behavior without making fallback values authored opinions.

实际使用中，`UsdSchemaRegistry` 常用于工具、插件和 schema 调试：确认一个 schema type name 是否注册，判断某个 API schema 是否只能应用到特定 prim type，解释一个 composed prim definition 为什么包含某个属性，或者定位自定义 schema 生成文件与运行时 registry 不一致的问题。它的输出应与 `UsdPrim` 的实际 authored state 分开分析：前者是声明和 fallback，后者是场景数据和 composition 结果。

In practice, the registry is useful for tools and diagnostics: checking registered schema type names, API schema applicability, composed prim definitions, and mismatches between generated schema data and runtime behavior.

如果 `FindSchemaInfo()` 返回空、`FindAppliedAPIPrimDefinition()` 找不到定义，或 `BuildComposedPrimDefinition()` 没有包含预期属性，排查顺序通常是：先确认插件和 schema-defining module 是否被加载，再确认 `schema.usda` 到 `generatedSchema.usda` 的生成结果是否包含该 schema，接着检查 exact `TfToken` 名称、schema family/version 选择和 API schema instance name，最后再看 `UsdPrim` 上是否真的应用了对应 API schema。这个顺序能避免把注册失败误判成 composition 或 layer authoring 问题。

A practical debugging order is plugin/schema-module registration, generated schema contents, exact token names, schema family/version lookup, API schema instance names, and finally whether the prim actually has the relevant applied API schema.

对于工具作者，本页也说明了 registry 层适合做“能力发现”和“定义解释”：可以用它生成 UI 中的属性列表、验证 API schema 是否允许应用到某类 prim、解释 fallback value 从哪里来，或在导出器中把 schema 声明与实际 authored data 分开报告。不要用 registry 的 prim definition 代替对 stage 的读取；真正的运行时值仍要通过 `UsdPrim`、`UsdAttribute`、composition 和 time samples 查询。

For tool authors, the registry supports capability discovery and schema-definition explanation, but authored runtime values still come from stage and prim queries.

还要注意，registry 依赖插件和生成文件提供 schema metadata，但它本身不是插件发现流程的全部。插件未加载、类型未声明、`TfType` 未注册、生成文件内容不匹配，都会表现为 registry 查不到信息；而 schema facade 类可以编译通过，也不代表运行时注册表中一定存在完整 prim definition。

The registry depends on plugin-provided generated schema metadata, but a compiled schema facade alone does not guarantee a complete runtime prim definition.

结论：先查定义，再查场景。

Diagnostic summary: check definitions before scene data.

因此，读 `UsdSchemaRegistry Class Reference` 时的主线是：先理解 singleton registry 的角色，再看 `SchemaInfo` 和 `VersionPolicy` 如何描述 schema identity，接着区分 concrete/abstract/applied API 三类 prim definition 查询，最后用 `BuildComposedPrimDefinition()` 和 API schema 辅助接口解释最终 schema 定义如何组合。所有 class name、method name、token literal、header name 和 link 都应按英文原样保留。

The reading path is: singleton registry, schema identity via `SchemaInfo` and `VersionPolicy`, prim-definition lookup by schema kind, composed definitions, and API-schema application rules.

原文摘录 / English Source Excerpts

Singleton registry that provides access to schema type information and the prim definitions for registered Usd "IsA" and applied API schema types. It also contains the data from the generated schemas that is used by prim definitions to provide properties and fallbacks. The data contained herein comes from the generatedSchema.usda file (generated when a schema.usda file is processed by usdGenSchema ) of each schema-defining module.
