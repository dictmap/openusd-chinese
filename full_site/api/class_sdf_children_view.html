逐段双语理解 / Paragraph-Level Bilingual Coverage

官方定义说 `SdfChildrenView< _ChildPolicy, _Predicate, _Adapter >` “Provides a view onto an object's children”。这里的关键词是 view：它让调用方以统一的 STL-style container interface 观察 Sdf 对象的 children，但不表示这个对象拥有 children，也不表示它负责 author、删除或组合这些 child specs。实际 authoring 仍发生在 `SdfLayer`、`SdfPrimSpec`、`SdfPropertySpec` 等底层对象上；`SdfChildrenView` 只是为读取和遍历提供稳定的 C++ 视图。

The official definition says SdfChildrenView provides a view onto an object's children. In this bilingual edition, view means a C++ container-like observation layer over Sdf children, not an owning authoring container.

三个模板参数构成了本类的核心边界。`_ChildPolicy` 决定从 owner 中怎样取得 children、底层 child storage 是什么，以及 `ValueType`、key、iterator 等类型如何定义；`_Predicate` 决定哪些 child 在这个 view 中可见；`_Adapter` 决定底层 child 值如何转换为调用者看到的 `value_type`。保留 `_ChildPolicy`、`_Predicate`、`_Adapter` 原名很重要，因为它们不是普通中文名词，而是 C++ template contract 的一部分。

The three template parameters define the contract: _ChildPolicy supplies child access, _Predicate filters visibility, and _Adapter adapts stored child values into the view value_type.

`Public Types` 中的 `_Traits`、`Adapter`、`ChildPolicy`、`ChildrenType`、`Predicate`、`KeyPolicy`、`key_type`、`value_type`、`size_type`、`difference_type`、`const_iterator` 和 `const_reverse_iterator` 让本类可以像标准容器一样参与泛型算法。阅读这些 typedef 时不要只看名字，应把它们理解为“把不同 Sdf child 集合统一成同一种遍历接口”的桥接层。比如 root prims、property specs、name-ordered children 或经过 predicate 过滤的 children，都可以通过类似 view 模式被访问。

The typedefs make the view usable by STL-style generic code. They bridge different Sdf child collections into one traversal shape.

构造函数有多个重载，说明 `SdfChildrenView` 可从不同 owner、policy 或 adapter 状态构造出视图。析构函数 `~SdfChildrenView()` 只结束 view 对象本身的生命周期，并不意味着删除底层 children。这个边界在调试引用失效时很关键：如果 owner 的 Sdf 数据发生结构修改，旧 view、iterator 或返回值是否仍可用，取决于底层 child storage 和 policy，而不是取决于 view 是否“看起来像容器”。

Constructors create the view over existing Sdf child state, and ~SdfChildrenView() destroys the view object rather than deleting the underlying children.

`begin()` / `end()`、`rbegin()` / `rend()`、`front()` / `back()`、`size()`、`empty()` 等接口提供 STL-style 访问体验。它们适合用于只读遍历、查找、检查数量和取首尾元素；调用方不应因为这些接口熟悉，就假设可以像 `std::vector` 一样长期保存 iterator、按索引随意修改，或把 children 的顺序当作独立于 Sdf namespace 的永久事实。

begin()/end(), rbegin()/rend(), front()/back(), size(), and empty() provide familiar STL-style reading operations, but they do not turn the view into an owning std::vector-like container.

如果页面或成员列表中出现 `find()`、key lookup 或 key-related typedef，应理解为“根据 Sdf child 的 key/name 在当前 view 可见集合里定位元素”。这和 USD composition 的最终解析结果不同：`SdfChildrenView` 通常面对的是 layer/spec 层的 authored child 集合，而不是 `UsdStage` 上 composed prim 的最终可见视图。调试时要区分 Sdf 层是否有 child spec、predicate 是否把它过滤掉、adapter 是否改变了返回形态，以及 Usd 层 composition 是否另有更强或更弱意见。

Key lookup operates within the current Sdf view and should not be confused with composed UsdStage visibility or final composition results.

默认相关类型 `SdfChildrenViewTrivialPredicate` 和 `SdfChildrenViewTrivialAdapter` 说明最简单的使用场景是不做过滤、不做转换，直接暴露底层 child 值。但本模板真正的价值在于可替换策略：同一套 view 接口可以面对不同 child policy、predicate 和 adapter 组合，从而避免每个 Sdf child 容器都重新实现一套遍历 API。

SdfChildrenViewTrivialPredicate and SdfChildrenViewTrivialAdapter describe the simplest no-filter/no-conversion case, while the template design allows richer policies and adapters.

和相邻类型协作时，本页应同 `SdfLayer`、`SdfPrimSpec`、`SdfPath`、`TfToken`、`SdfChildrenViewTrivialAdapter` 一起读。`SdfLayer` 和 spec 类型提供具体数据与 namespace；`SdfPath` 与 `TfToken` 提供路径和名称标识；`SdfChildrenView` 则把这些 child 集合组织为可遍历、可过滤、可适配的接口。它不是 `UsdPrim` traversal，也不是 composition engine，也不是编辑命令本身。

Read this page with SdfLayer, SdfPrimSpec, SdfPath, TfToken, and SdfChildrenViewTrivialAdapter. SdfChildrenView organizes Sdf child collections; it is not UsdPrim traversal, composition, or an editing command.

实际排障时，`SdfChildrenView` 常帮助回答“这个 layer/spec 层对象到底有哪些 authored children”。如果某个 prim、property 或 variant child 在最终 `UsdStage` 上没有出现，不能只看本 view 的遍历结果就下结论。应分层检查：底层 Sdf spec 是否存在、当前 `Predicate` 是否把它过滤掉、`Adapter` 是否把返回值包装成另一种类型、owner 是否在遍历期间被修改、composition 是否被更强 layer 或 list-edit 操作覆盖。这样可以避免把 Sdf 层 authored data、view 层过滤、adapter 转换和 Usd 层 composition 混成一个问题。

For debugging, SdfChildrenView helps answer which authored children exist at the Sdf layer, but missing composed results may still come from filtering, adaptation, owner mutation, stronger layers, or list-edit composition.

使用场景上，本类适合写只读检查、诊断报告、泛型遍历工具和内部 helper，而不适合直接作为长期缓存。若工具需要跨多个操作保存 child 列表，通常应复制出稳定的 `SdfPath`、`TfToken` 或 spec handle 列表，并在修改 layer 后重新获取 view。若只是临时遍历，则 `begin()` / `end()` 与 typedef 提供的容器式语义能让代码保持简洁，同时仍尊重 Sdf 对象的真实所有权边界。

Use the view for short-lived read-only traversal and diagnostics. For durable caches, copy stable identifiers or handles and reacquire the view after layer mutations.

性能上也要把它当作视图而非快照：`size()`、`empty()`、迭代器移动和 key lookup 的成本可能由 policy、predicate 和底层 storage 决定。若 predicate 很复杂，遍历中看到的“容器式”接口并不保证和普通数组一样便宜。写工具时应避免在内层循环反复重建 view 或重复执行昂贵过滤，而应在明确的 Sdf 数据版本边界内组织读取。

Performance depends on policy, predicate, and storage; the container-like interface does not guarantee array-like cost.

API 分组阅读 / API Group Reading

  • `_Traits`、`ChildPolicy`、`KeyPolicy`、`Predicate`、`Adapter` 是模板策略层，决定 view 怎样取数、过滤和适配。The policy typedefs define access, filtering, and adaptation.
  • `const_iterator` 与 `const_reverse_iterator` 表示该 view 主要面向遍历读取；不要把它误读为拥有型可变容器。The iterator typedefs describe traversal, not ownership.
  • `key_type`、`value_type`、`size_type`、`difference_type` 支持泛型算法和容器式调用，但语义仍由 child policy 决定。Container-like typedefs support generic code while policy still owns semantics.
  • 构造函数重载用于建立不同策略组合下的 view；析构函数只释放 view 状态，不删除底层 Sdf children。Construction configures the view; destruction does not remove child specs.
  • `begin()`、`end()`、`rbegin()`、`rend()`、`front()`、`back()`、`size()`、`empty()` 是读取接口，适合短生命周期遍历和检查。STL-style methods are for reading, traversal, and inspection.

边界与常见误读 / Boundaries and Common Misreadings

  • 不是 authoring API：它不创建、不删除 child spec；修改应回到 `SdfLayer` 或具体 spec API。This is not the authoring API; edits belong to SdfLayer or concrete spec APIs.
  • 不是 composition 结果：它查看的是某个 Sdf 对象的 children view，不等同于 composed `UsdStage` traversal。It is not a composed UsdStage traversal result.
  • 不是拥有型容器：不要长期保存 iterator 或引用而不考虑 owner 生命周期和底层修改。It is not an owning container; iterator lifetime depends on owner and storage changes.
  • 不是单一 child 类型：`_ChildPolicy`、`_Predicate`、`_Adapter` 让同一 view 模板服务不同 children 集合。The template parameters let one view pattern serve multiple child collections.
