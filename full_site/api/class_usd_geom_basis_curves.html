逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdGeomBasisCurves` 表达的是一个 prim 内的批量曲线集合，而不是把每一根 hair、grass 或 strand 建成独立 prim。这个设计的核心动机是高密度 aggregate geometry：大量曲线共享一个 schema wrapper 和数组数据布局，消费端通过数组解释每条曲线的顶点、线段、插值数据和渲染形态。

`UsdGeomBasisCurves` is a batched curve representation, commonly used for dense aggregate geometry such as hair or grass.

官方把 BasisCurves 类比为 RIB 的 `Basis` 和 `Curves` 语句，重点是“基函数”和“曲线数据”一起决定曲线如何从控制点生成。阅读本页时不要只看 `points`；还必须同时看 `curveVertexCounts`、`type`、`basis`、`wrap`、segment indexing 和 interpolation 规则，否则很容易把控制点数量、线段数量和 primvar 样本数量混在一起。

The class is analogous to classic RIB `Basis` and `Curves` concepts: curve data and basis-related interpretation must be read together.

`curveVertexCounts` 是本页最重要的数据布局字段。它的数组长度隐式决定这个 prim 中有多少条曲线；每个元素给出对应曲线占用的连续 `points` 数量。因此第 N 条曲线并不是另一个 prim，而是 `points` 数组中的一个连续段。调试拓扑错误时，应先核对所有 `curveVertexCounts` 之和是否等于有效控制点数量。

The length of `curveVertexCounts` determines how many curves are stored in the prim; each entry tells how many consecutive points belong to that curve.

`type` 决定曲线是 linear 还是 cubic。linear BasisCurves 不使用 `basis` attribute；cubic BasisCurves 则通过与 basis 关联的 matrix 和 `vstep` 插值 vertices。也就是说，`basis` 不是对所有曲线类型都有效的通用开关，解释 basis 前必须先确认 `type`，否则会把 linear 数据错误地按 cubic 规则消费。

The `basis` attribute matters for cubic curves, where a basis matrix and `vstep` are used to interpolate vertices; it is unused for linear BasisCurves.

`wrap` 描述曲线是否周期闭合，并影响 segment count 的计算。对于 open 曲线，线段数量通常少于控制点数量；对于 periodic 或 pinned 等约定，segment indexing 会发生变化。消费端在计算 `ComputeSegmentCounts()` 或生成渲染数据前，必须把 `type`、`basis`、`wrap` 与每条曲线的 vertex count 放在一起解释。

`wrap` participates in segment-count interpretation, so segment indexing must be derived from curve type, basis, wrap, and per-curve vertex counts together.

`ComputeSegmentCounts()`、`ComputeUniformDataSize()`、`ComputeVaryingDataSize()`、`ComputeVertexDataSize()` 和相关 helper 是把拓扑解释映射到数据尺寸的入口。它们解决的问题不是 author 新曲线，而是帮助调用方判断 uniform、varying、vertex 或 face-varying 类数据需要多少样本，避免 primvar 数组长度与曲线拓扑不一致。

The `Compute*DataSize()` helpers translate curve topology and interpolation rules into expected data sizes for consumers and validation tools.

`UsdGeomBasisCurves` 继承自 `UsdGeomCurves` / `UsdGeomPointBased` 语义，因此 `points`、widths、normals、primvars 等数据仍遵循几何基础类的规则。它和 `UsdGeomMesh` 的边界是：Mesh 用 face topology 表达面片，而 BasisCurves 用 curve topology 表达线状集合；不要用 mesh 的 faceVertexCounts / faceVertexIndices 心智模型解释本页的 `curveVertexCounts`。

Basis curves follow point-based geometry conventions, but their topology is curve-based rather than mesh face topology.

`CreateBasisAttr()`、`CreateTypeAttr()`、`CreateWrapAttr()`、`CreateCurveVertexCountsAttr()` 等 schema API 负责 author 对应属性；它们不会自动证明数组长度一致，也不会替调用方推导所有 primvar 样本。写入这些属性后，仍需要验证 `points`、`curveVertexCounts`、interpolation 和渲染相关 primvars 是否相互匹配。

The `Create*Attr()` APIs author schema attributes; they do not by themselves validate consistency between points, curve counts, interpolation, and primvars.

`Segment Indexing`、`Cubic Vertex Interpolation`、`Linear Vertex Interpolation` 和 `Primvar Interpolation` 小节需要连在一起读。segment 是由曲线类型和控制点布局推导出的渲染/插值单位；vertex interpolation 关注曲线控制点或插值顶点；primvar interpolation 关注属性样本如何对应到 curve、segment 或 point。三者不是同一个索引空间。

Segment indexing, vertex interpolation, and primvar interpolation describe different index spaces and must not be collapsed into one concept.

`Tubes and Ribbons` 说明曲线可以按 tube 或 ribbon 方式渲染。这个选择影响宽度、法线、orientation、屏幕朝向和材质解释，但不改变 USD 中曲线拓扑的基本定义。调试外观问题时应区分“拓扑数据错误”和“渲染解释或 primvar 数据不足”。

Tubes and ribbons affect rendering interpretation, but they do not change the underlying curve topology stored by `UsdGeomBasisCurves`.

常见误读有四类：第一，把每条曲线误认为独立 prim；第二，把 `curveVertexCounts` 当成 mesh 的 face vertex counts；第三，在 linear curves 上解释 `basis`；第四，把 `Compute*DataSize()` 当成自动修复拓扑的函数。正确理解是：本页定义批量曲线 schema 的数据布局和解释规则，验证与修复仍需要上层工具根据数组长度、interpolation 和渲染需求执行。

Common mistakes include treating each curve as a separate prim, reading `curveVertexCounts` like mesh face counts, applying `basis` to linear curves, or expecting compute helpers to repair topology.
