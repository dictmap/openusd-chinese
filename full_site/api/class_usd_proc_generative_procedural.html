逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdProcGenerativeProcedural` 表示一个 abstract generative procedural prim。官方描述的重点不是“这个 prim 会自己生成几何”，而是“它在 USD scene description 中交付 procedural 的定义和输入参数”。因此它首先是一个 schema contract：把程序化生成所需的系统标识、输入属性和关系放到 stage 上，让后续 consumer、插件或 runtime 决定是否识别并执行。

The official page describes an abstract generative procedural prim that delivers a procedural definition and input parameters through scene description.

官方摘录强调输入参数通过 `primvars:` namespace 下的 properties，包括 attributes 和 relationships 来传递。这意味着 procedural 输入既可以是数值、token、asset path 等属性，也可以是指向其他 scene data 的 relationship。翻译和工具实现都必须保留 `primvars:` 前缀，因为它是输入参数命名空间的一部分，不是普通中文标签。

Inputs are delivered via properties, including relationships, within the `primvars:` namespace.

`proceduralSystem` 是本类最关键的自有属性。`CreateProceduralSystemAttr()` 用来 author 该属性，`GetProceduralSystemAttr()` 用来取得对应 `UsdAttribute`。它的值说明这个 procedural definition 对哪个系统有意义；该值可以来自 authored opinion，也可以来自 API schema fallback。它不是执行入口，也不是插件路径本身，而是让 consumer 判断应由哪个 procedural system 解释这段定义。

`proceduralSystem` identifies the procedural system for which the procedural definition is meaningful; it may be authored or provided by schema fallback.

`Define(stage, path)` 和 `Get(stage, path)` 仍遵循 typed schema 的常见语义。`Define()` 在指定 `UsdStage` / `SdfPath` 上创建或确保存在该类型 prim；`Get()` 只是把已有 prim/path 包装为 `UsdProcGenerativeProcedural` wrapper。两者都不会加载 generator、展开 geometry、启动 Hydra task，或缓存生成结果。

`Define()` and `Get()` create or retrieve typed-schema views; they do not execute the procedural.

官方说明明确说该 prim 本身不 awareness 或 participation in procedural execution。中文阅读时应把它与 renderer、Hydra task、asset resolver、DCC 插件和具体 generator runtime 分开：这些系统可以读取 `proceduralSystem` 和 `primvars:` 输入并做自己的处理，但 `UsdProcGenerativeProcedural` 只负责描述 USD 层的数据契约。

The prim itself does not participate in procedural execution; it serves as a delivery mechanism for definitions and inputs.

`GetSchemaAttributeNames()`、`schemaKind`、`_GetSchemaKind()` 和 `UsdSchemaRegistry` 条目说明该类由 schema registry 管理。它们适合做 schema introspection、UI 生成和属性清单校验；具体 stage 上的有效值仍需要通过 USD composition、layer strength、time samples、fallback 和 authored relationships 来解析。

Schema registry and schema-kind APIs describe declared schema structure; effective authored values still come from normal USD composition.

与 `UsdPrim`、`SdfPath`、`UsdAttribute`、`UsdRelationship`、`SdfValueTypeName` 和 `UsdProcTokens` 的关系也要分清。`UsdPrim` 是 stage 上的对象视图，`SdfPath` 定位 prim，`UsdAttribute` / `UsdRelationship` 承载 `primvars:` 输入，`UsdProcTokens` 提供 token 常量；本类把这些 USD 基础机制组织成“可被 procedural system 消费”的 schema 约定。

Related USD types provide path identity, prim views, attributes, relationships, type names, and tokens used by the procedural schema contract.

常见误读包括：把该 prim 当成会立即生成内容的 node；把 `proceduralSystem` 当成插件文件路径；把 `primvars:` 输入和 `UsdGeomPrimvar` 的渲染插值语义混为一谈；只调用 `Get()` 就以为 prim 已存在；或忽略 fallback 与 authored value 的差异。调试时建议按 path、schema type、`proceduralSystem`、`primvars:` properties / relationships、composition、consumer 支持和 runtime 日志的顺序排查。

Common debugging separates path validity, schema type, procedural system token, authored inputs, composition, consumer support, and runtime diagnostics.

在内容管线中，本类通常用于把“生成规则的声明”放进 USD 文件，而不是把生成后的 mesh、points 或 volumes 直接写入该 prim。若某个系统支持该 `proceduralSystem`，它可以读取 `primvars:` 输入并在加载、渲染或离线烘焙阶段生成实际结果；若系统不支持，它仍应能把该 prim 作为普通 USD 数据保存、传递或检查。

In pipelines, the schema stores a declaration and inputs for generation; compatible systems may consume it, while unsupported systems can still preserve it as USD data.

`primvars:` 在这里强调“作为 procedural input 的命名空间”，不必等同于 `UsdGeomPrimvarsAPI` 中面向几何插值的 primvar 工作流。某些输入可能只是普通属性值，某些输入可能是 relationship 指向的场景对象。排查输入缺失时应检查 property name、namespace、value type、relationship target、layer strength 和 consumer 期望的 schema，而不是只搜索 UI 中的参数名。

The `primvars:` namespace is used for procedural inputs here; the authored properties may be values or relationships and must be checked through normal USD property and composition rules.

对翻译和复刻页面来说，`proceduralSystem`、`primvars:`、`Define()`、`Get()`、`CreateProceduralSystemAttr()` 和 `GetProceduralSystemAttr()` 必须保留英文原形，因为它们分别对应 USD 文件中的 token、命名空间和 C++ / Python API。中文说明负责解释边界，不能把这些标识改写成无法和官方 API 对照的本地化名称。

API and token spellings remain unchanged so the bilingual page can be checked directly against authored USD files and the official API.

因此，本页的完整翻译目标是帮助读者判断“描述是否正确”，而不是承诺任何具体系统一定会执行生成。

The page helps readers judge the authored description, not guarantee execution by a specific system.
