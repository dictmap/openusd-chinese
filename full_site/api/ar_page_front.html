逐段双语理解 / Paragraph-Level Bilingual Coverage

`Ar` 是 OpenUSD 的 Asset Resolution 库，负责 querying、reading 和 writing asset data。它解决的核心问题是：USD 只需要拿到一个可解析的 asset identifier，而不需要知道资产到底存放在本地文件系统、数据库、网络服务、包文件，还是工作室自定义资产管理系统中。换句话说，Ar 把“场景描述中如何引用资产”和“资产后端如何存储与访问”解耦。

The Ar library is responsible for querying, reading, and writing asset data while hiding the physical storage mechanism from USD clients.

官方概览强调 Ar ships with implementations based on a traditional filesystem，但也允许用户提供 custom implementations。默认文件系统 resolver 适合普通文件路径；工作室或生产管线可以通过插件把数据库 id、版本化 asset id、云端 URI、包内路径或专有资源系统接入 USD。这样 `UsdStage`、layer loading 和 asset dependencies 可以保持统一调用模型。

Ar includes filesystem-based implementations, but users can provide custom implementations to integrate their own asset storage systems and behaviors.

`ArResolver` 是本页的核心抽象。它把 authored asset path 或 URI/IRI 输入，结合 resolver context 和 resolver-specific 规则，转换成可读取或可写入的 resolved asset。`ArGetResolver()` 提供当前 resolver 访问入口；自定义 resolver 通常通过 `AR_DEFINE_RESOLVER` 和 `PlugRegistry` 插件机制注册。不要把 resolver 当成简单字符串拼接器，它可以包含版本选择、搜索路径、权限、包展开、缓存和后端查询策略。

ArResolver is the main abstraction that resolves asset identifiers into accessible assets, often through plugin registration and resolver-specific policy.

Primary Resolver 处理默认路径解析语义，通常覆盖普通 asset path、相对路径、搜索路径和项目级规则。URI/IRI Resolvers 处理带 scheme 的资源标识，例如自定义数据库 scheme、网络资源或其他非文件系统形式。阅读本页时要区分二者：Primary Resolver 是默认入口，URI/IRI Resolver 是按 scheme 分派的专门入口；二者都属于 asset resolution，但触发条件和配置方式不同。

Primary Resolver handles default asset path resolution, while URI/IRI resolvers handle identifiers with schemes and specialized backends.

`asset path` 是场景、layer 或 API 中写入的引用字符串；`resolved path` 是 resolver 结合上下文后得到的可访问定位结果。二者不能混用。日志中看到 asset path 只说明 USD 请求了什么；看到 resolved path 才说明 resolver 找到了什么。若资产加载失败，必须同时记录 authored asset path、当前 resolver context、resolved path 结果以及是否使用了 cache。

An asset path is the authored input identifier. A resolved path is the result produced by the resolver under a particular context.

Resolver Contexts 表达解析所依赖的生产上下文，例如项目根目录、搜索路径、asset version、shot context、site policy 或外部资产数据库状态。同一个 asset path 在不同 context 下可以解析到不同资产。`ArResolverContextBinder` 可把 context 绑定到作用域内，`UsdStage` 打开和 layer 解析时会受到当前 context 影响。排查“同一 USD 在两台机器打开结果不同”时，context 是第一类证据。

Resolver contexts carry project, search-path, version, and asset-system state. The same asset path can resolve differently under different contexts.

Resolver Scoped Caches 用于在一个作用域内复用解析结果，降低重复解析、后端查询和 I/O 成本。`ArResolverScopedCache` 与 `ArThreadLocalScopedCache` 表示不同缓存作用域。缓存不是全局真理：如果 context、搜索路径、版本选择或后端资产状态发生变化，旧 cache 可能掩盖问题。调试时要明确是否在 cache scope 内，以及是否需要清空或缩小作用域。

Scoped caches reuse resolution results within a scope. They improve performance but can hide context or backend changes during debugging.

`ArAsset` 表示 resolver 打开的资产访问对象，用于读取或写入 asset data；它和 `ArResolver` 的职责不同。resolver 负责定位和打开，`ArAsset` 负责具体数据访问。把这两个概念混在一起，会导致排查方向错误：找不到文件通常是 resolution 问题；打开后读取失败可能是 asset object、权限、格式或后端 I/O 问题。

ArAsset represents opened asset access, while ArResolver is responsible for locating and opening assets.

Package Relative Paths 说明包文件内资源和外部 asset path 的组合语义。USDZ 或其他 package 相关工作流中，resolver 需要理解包根层、包内相对路径和外部引用的区别。排查包内资源丢失时，应分别检查 package root、package-relative identifier、外部 layer 引用和最终 resolved path，而不是只看一个扁平文件路径。

Package-relative paths require the resolver to distinguish package roots, paths inside packages, external references, and final resolved locations.

Ar 与 `UsdStage`、`SdfLayer` 和 composition 的关系也要分层理解：Ar 负责把资产标识解析成可访问资源；`SdfLayer` 负责读取 layer 内容并提供 layer-level opinions；`UsdStage` 再把多个 layer 和 composition arcs 组合成场景视图。因此，一个 reference 或 payload 加载失败可能发生在多个层次：asset path 无法解析、resolved asset 无法读取、layer 内容格式错误、或 composition 选择未按预期生效。把这些层次拆开记录，能避免把所有问题都误判成 resolver bug。

Ar resolves asset identifiers, SdfLayer reads layer contents, and UsdStage composes layers into a scene view. Asset loading failures should be separated by these layers.

常见调试路径可以固定为：确认当前 resolver 类型和插件是否加载；记录 authored asset path；检查 resolver context 是否符合项目/shot/版本；在禁用或缩小 scoped cache 后重新解析；比较 resolved path 与实际后端资产；最后再检查权限、包内路径、URI scheme 和 `UsdStage` 打开时传入的 context。这样可以把路径拼写、context 错误、插件未注册、缓存污染和后端不可读区分开。

A practical debugging path is resolver plugin, authored asset path, resolver context, scoped cache, resolved path, backend availability, permissions, package paths, URI scheme, and UsdStage context.
