逐段双语理解 / Paragraph-Level Bilingual Coverage

`HdStRenderPassState` 是 Hydra Storm 后端在多个 render pass 之间共享的 rendering-parameter object。官方说明强调这些参数以 GL states、uniforms 或 shaders 表达；因此它不是场景数据、不是 draw item，也不是 `HdSceneDelegate` 的查询接口，而是 pass 执行前后需要绑定到 GPU / Hgi 命令流的一组渲染状态。

`HdStRenderPassState` is the Storm render-pass state object for rendering parameters shared among render passes. It is not scene data or a draw item; it is the state package bound around GPU / Hgi command execution.

与基类 `HdRenderPassState` 相比，`HdStRenderPassState` 是 Storm-specific 的实现层：它连接 `HdRenderIndex`、`HdCamera`、`CameraUtilFraming`、`HgiGraphicsCmdsDesc`、`HgiGraphicsPipelineDesc`、`HgiCapabilities` 和 `Hgi`。阅读本页时应把通用 Hydra render-pass abstraction 与 Storm/HdSt backend state 分开，否则容易把可移植接口、OpenGL 状态和 Hgi pipeline descriptor 混为一谈。

Compared with `HdRenderPassState`, this class is Storm-specific. It connects camera/framing state, render-index inputs, graphics command descriptors, pipeline descriptors, capabilities, and the `Hgi` backend.

`Prepare()` 与 `Bind()` 是理解生命周期的关键。`Prepare()` 通常在 draw 前收集或更新 pass 所需资源，例如 shader、AOV resolve 配置、viewport、framing 和 pipeline hash 相关状态；`Bind()` 则把这些状态实际绑定到当前 graphics commands。把二者当成普通 setter 会误判状态何时进入 GPU 命令序列。

`Prepare()` gathers or updates render-pass resources, while `Bind()` binds the prepared state to active graphics commands. They should be read as render-pass lifecycle operations, not ordinary data setters.

相机和裁剪接口覆盖 `GetWorldToViewMatrix()`、`GetProjectionMatrix()`、`GetCullMatrix()`、`GetClipPlanes()`、`ComputeViewport()` 和 `SetCameraFramingState()`。这些方法描述的是 view/projection/cull 空间、clip planes、viewport rectangle 与 camera framing 如何影响最终 framebuffer 区域；它们不会改变 USD composition，也不会 author 场景层。

Camera and clipping APIs such as `GetWorldToViewMatrix()`, `GetProjectionMatrix()`, `GetCullMatrix()`, `GetClipPlanes()`, `ComputeViewport()`, and `SetCameraFramingState()` describe view, projection, culling, clipping, viewport, and framing state for the render pass.

shader 相关接口包括 `SetLightingShader()`、`GetLightingShader()`、`SetRenderPassShader()`、`GetRenderPassShader()` 和 `GetShaders()`。这些方法管理的是 Storm pass 执行需要绑定的 shader program / shader state；材质网络解析、primvar 数据提供和 scene delegate 查询仍属于上游职责，不能因为这里出现 shader 字样就把它当成材质系统入口。

Shader APIs manage the lighting shader and render-pass shader state consumed by Storm. Material-network resolution and scene-delegate data queries remain upstream responsibilities.

`GetGraphicsPipelineHash()` 和 `HgiGraphicsPipelineDesc` 相关性说明该类也参与 graphics pipeline state caching。任何影响 depth/stencil/blend、shader、attachment、viewport 或 resolve 路径的状态变更，都可能导致不同 pipeline hash 和 pipeline 复用结果；性能排查时应同时检查 hash、Hgi capabilities 和 pass 配置。

`GetGraphicsPipelineHash()` relates the state object to graphics-pipeline caching. Pipeline-affecting state changes can produce a different hash and a different reuse path.

AOV / multi-sample 路径由 `SetResolveAovMultiSample()` 等接口体现。它表达的是 render pass output 在 multi-sample attachment、AOV resolve 和最终读回之间的控制关系；这与 `HdRenderBuffer` 的 allocation、mapping、resolve ownership 是相邻但不同的层次。

AOV and multi-sample behavior is represented by APIs such as `SetResolveAovMultiSample()`. This controls render-pass output resolve behavior rather than render-buffer allocation or ownership.

`AxisAlignedAffineTransform` typedef 出现在本页的 public types 中，提示 viewport / framing 相关计算可能需要在轴对齐的二维区域之间表达 affine transform。它不表示 USD transform op，也不等同于 `GfMatrix4d` 的世界空间变换；它更接近 framebuffer、window policy、data window 或 render viewport 之间的坐标映射辅助。

The `AxisAlignedAffineTransform` typedef should be read in the viewport/framing context, not as a USD transform op or a world-space transform matrix.

本页还有 Additional Inherited Members，因为 `HdStRenderPassState` 继承自 Hydra render-pass state 层。继承成员通常提供通用 render pass 参数或状态访问，而 Storm 子类补上 shader、Hgi pipeline、clip/framing 和 resolve 路径。阅读 member list 时应先判断接口来自基类还是 Storm 子类，再判断它影响通用 Hydra contract 还是只影响 HdSt backend。

Inherited members come from the generic Hydra render-pass state layer. The Storm subclass adds backend-specific shader, Hgi pipeline, clipping/framing, and resolve behavior.

`GetWorldToViewMatrix()` 与 `GetProjectionMatrix()` 的命名方向很重要：前者把 world 空间带到 view 空间，后者描述 view 到 clip/projection 的投影关系；`GetCullMatrix()` 则服务于裁剪判断。调试模型位置正确但画面不可见时，应按 world-to-view、projection、cull、clip planes、viewport 的顺序排查，而不是先怀疑 layer composition。

Matrix names encode space-conversion direction. For visibility bugs, inspect world-to-view, projection, cull, clip-plane, and viewport state before blaming layer composition.

`HgiGraphicsCmdsDesc` 与 `HgiGraphicsPipelineDesc` 的出现说明该类处在 Storm 与抽象 graphics interface 的交界处。OpenGL 术语仍会出现在历史描述中，但现代路径需要按 Hgi backend 能力理解；同一状态在不同 backend 上可能映射到不同 pipeline object 或 command encoding。

The presence of `HgiGraphicsCmdsDesc` and `HgiGraphicsPipelineDesc` places this class at the boundary between Storm state and the abstract graphics interface.

因此，本页更适合作为渲染状态排查入口，而不是场景创作入口。

This page is therefore better read as a render-state debugging entry than as a scene-authoring entry.

调试 Storm 渲染问题时，本页常见阅读顺序是：先确认 camera matrices 与 viewport/framing，再确认 lighting/render-pass shader，随后检查 clip planes、AOV resolve、pipeline hash 和 Hgi backend capabilities。若图像裁剪错误、画面黑屏、AOV 没有 resolve 或 pipeline cache 命中异常，`HdStRenderPassState` 往往是比 USD authoring 层更直接的诊断位置。

For Storm debugging, inspect camera matrices and viewport/framing first, then shader state, clipping, AOV resolve behavior, graphics-pipeline hash, and backend capabilities.

常见误用包括：把 `HdStRenderPassState` 当成 scene delegate 或 USD schema；在没有理解 `Prepare()` / `Bind()` 调度顺序时直接修改状态；把 `ComputeViewport()` 问题误判为 stage composition 问题；把 `GetGraphicsPipelineHash()` 当成稳定的跨后端语义标识。正确边界是：它描述 Storm render pass 执行状态，英文 API 名称和 Hgi/GL 术语必须原样保留以便与源码和 Doxygen 对齐。

Common mistakes include treating this class as scene data, confusing viewport problems with composition problems, or treating `GetGraphicsPipelineHash()` as a stable semantic identifier across backends.
