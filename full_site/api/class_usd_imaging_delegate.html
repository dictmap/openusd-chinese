逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdImagingDelegate` 的核心职责是把 USD scene graph 中的 `UsdPrim`、属性、关系、visibility、purpose、topology、primvars、materials、instancing 和 time-varying data 翻译为 Hydra `HdSceneDelegate` / `HdRenderIndex` 可以同步和查询的数据。它位于 USD composition 结果与 Hydra rendering abstraction 之间：上游看到的是 stage、prim、schema 和 notice；下游看到的是 render index prim、dirty bits、topology、material resource 和 scene delegate query。

`UsdImagingDelegate` translates composed USD scene-graph data into information that Hydra `HdSceneDelegate` and `HdRenderIndex` can synchronize and query. It sits between USD composition results and Hydra rendering abstractions.

官方原文说它是 “The primary translation layer between the Hydra (Hd) core and the Usd scene graph.” 这里的 translation layer 不应理解成简单文件格式转换，而是持续运行的成像桥接层。它要维护 population、resync、dirty propagation、adapter 分发、path mapping 和 time sampling；因此调试显示缺失、拓扑不刷新或材质不同步时，经常要同时检查 `UsdPrim`、adapter、delegate、render index 和最终 render delegate。

The official phrase “primary translation layer” means an active imaging bridge, not a one-time file conversion. It participates in population, resync, dirty propagation, adapter dispatch, path mapping, and time sampling.

`ApplyPendingUpdates()` 是阅读本页时的关键入口。USD 侧 notice、population 或 adapter 变更不会自动等价于 Hydra 已经看到最终数据；pending updates 需要被应用到 render index 同步路径。调试“USD 已改但 viewport 没刷新”时，应检查是否有 pending update、dirty bit 是否正确、对应 prim 是否仍在 render index 中，以及是否发生了 resync 而不是普通 value dirty。

`ApplyPendingUpdates()` is a key synchronization entry point. USD notices or adapter changes are not the same as Hydra already having final data; pending updates must be applied into the render-index synchronization path.

`ConvertCachePathToIndexPath()` 与 `ConvertIndexPathToCachePath()` 体现了 imaging 层的路径边界。cache path 往往更接近 USD prim 或 adapter 内部缓存身份，index path 则是 Hydra render index 中用于定位 rprim/sprim/bprim 的路径。instancing、prototype、adapter-generated prim、picking 和 selection 调试时，路径看起来“不一致”并不一定是错误，关键是确认两套路径是否能被 delegate 正确互转。

`ConvertCachePathToIndexPath()` and `ConvertIndexPathToCachePath()` expose the boundary between imaging cache paths and Hydra render-index paths. Different paths can be valid as long as the delegate maps them correctly.

`Get()` 在这里不是普通从 `UsdStage` 取 `UsdPrim` 的 API，而是 scene delegate 查询面的一部分：Hydra 会通过 delegate 请求某个 id 对应的数据值。类似的 `GetBasisCurvesTopology()`、`GetCameraParamValue()`、`GetCoordSysBindings()`、`GetCategories()`、`GetDisplayStyle()` 等方法，返回的是 Hydra 消费所需的翻译后视图；它们可能经过 adapter、cache、purpose filtering、visibility 计算或 time-sampled value 处理。

`Get()` and the many typed query methods are part of the scene-delegate data access surface. They return Hydra-consumable translated views, not necessarily raw `UsdPrim` fields.

几何查询分组说明 delegate 如何把 USD schema 数据压平到 Hydra 几何输入。`GetMeshTopology()` 与 `GetBasisCurvesTopology()` 负责拓扑视图，`GetIndexedPrimvar()` 和 primvar descriptors 负责按 Hydra interpolation 规则组织 primvar 数据，`SubdivTags` 和 refine level 处理 subdivision 相关信息。这里的重点是：Hydra 要的是渲染同步所需的数据结构，不是完整 USD layer stack 的逐项回放。

Geometry queries such as `GetMeshTopology()`, `GetBasisCurvesTopology()`, indexed primvars, `SubdivTags`, and refine levels convert USD schema data into Hydra geometry inputs rather than replaying the entire USD layer stack.

显示目的和可见性相关查询是另一个常见误读点。`GetDisplayGuides()`、`GetDisplayProxy()`、`GetDisplayRender()`、`GetCullStyle()`、`GetDisplayStyle()` 与 USD purpose、visibility、draw mode 和 viewer 过滤策略相关，但它们不是最终 renderer backend 的 draw state。它们描述 imaging 层告诉 Hydra “哪些数据应该进入哪类显示用途”；最终如何 rasterize、shade 或 cull 仍由 render delegate 和后端决定。

Display-purpose queries describe how the imaging layer exposes USD purpose, visibility, draw mode, and filtering to Hydra. They are not the final renderer backend draw state.

`GetCurrentTimeSamplingInterval()` 和 `UsdTimeCode` 相关查询说明 delegate 必须处理 animated values。Hydra 不一定每帧重新读取所有 USD 属性；delegate 需要根据时间采样、dirty bits、time-varying 标记和缓存策略决定哪些数据更新。调试动画闪烁、延迟更新或 motion blur 采样差异时，应把 time sampling interval、attribute variability 和 render index 同步频率一起核对。

Time-related queries show that the delegate handles animated values and time sampling. Hydra does not necessarily re-read every USD attribute every frame; variability, dirty bits, and cache policy matter.

`RigidXformOverridesMap` 表示 imaging delegate 可以持有 rigid transform overrides；这类覆盖不是 USD authored transform 的替代品，而是 imaging/selection/scene-index 流程中可能需要的变换覆盖数据。类似地，material resource、categories、coord sys bindings、selection/highlight 相关查询都应理解为 Hydra 消费的翻译结果，而不是 schema authoring 的唯一来源。

`RigidXformOverridesMap` and related material, category, coordinate-system, selection, and highlight queries are translated imaging data for Hydra consumption, not replacements for USD-authored schema data.

`UsdImagingDelegate` 与 `UsdImagingAdapterRegistry`、`UsdImagingPrimAdapter` 和 `HdSceneDelegate` 的边界很重要：registry 负责找到和构造 adapter，prim adapter 负责按具体 schema/prim 类型抽取数据，delegate 则汇总 stage、adapter、cache 和 render index 的同步/查询入口。它也不是 `HdRenderDelegate`；后者属于渲染后端实现，例如将 Hydra 抽象转成具体 renderer 的 draw calls 或 GPU resources。

The boundary is: `UsdImagingAdapterRegistry` finds adapters, `UsdImagingPrimAdapter` extracts prim-specific data, `UsdImagingDelegate` aggregates synchronization and query access, and `HdRenderDelegate` belongs to the renderer backend.

常见误读包括：把 delegate 当作最终 renderer；把 cache path 和 render index path 混为一谈；把 `Get*()` 查询当作直接读取 USD 字段；认为 USD notice 已触发就说明 Hydra 数据已刷新；或把 purpose/display filtering 当作 renderer backend 状态。正确理解是：`UsdImagingDelegate` 是一个面向 Hydra 的 USD 数据翻译、缓存和同步边界，问题排查必须沿 `UsdStage -> adapter -> UsdImagingDelegate -> HdRenderIndex -> render delegate` 逐层定位。

Common misreads include treating the delegate as the renderer, conflating cache paths with render-index paths, treating `Get*()` as raw USD reads, assuming notices mean Hydra data is already synchronized, or confusing purpose filtering with backend draw state.
