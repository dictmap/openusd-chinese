逐段双语理解 / Paragraph-Level Bilingual Coverage

`HdInstanceRegistry< VALUE >` 是 Hydra 内部围绕 `HdInstance` 的 dictionary container。官方描述非常短，但关键词是 dictionary container：它用于按 key 管理实例记录或缓存值，而不是直接执行 instancing draw，也不是 USD prim 列表。读取本页时应把它放在 Hydra instance-cache / lookup / lifetime 管理语境中理解。

`HdInstanceRegistry< VALUE >` is a dictionary container of `HdInstance`. It should be read as an internal Hydra registry/cache structure, not as a draw command or a USD prim list.

模板参数 `VALUE` 表示 registry 存储的具体值类型，因此不能翻译成某个固定类型。不同使用场景可以把 `VALUE` 设为实例记录、资源句柄、状态对象或与 `HdInstance` 关联的缓存值。`InstanceType` typedef 则把模板值类型与 registry 中实际暴露的 instance entry 语义连接起来。

The template parameter `VALUE` is supplied by the caller and defines the stored value type. `InstanceType` expresses the instance-entry type exposed by the registry.

`GetInstance()` 与 `FindInstance()` 的边界很重要。`FindInstance()` 通常表示只查找已有条目，找不到时不应隐式创建；`GetInstance()` 则可能服务于“按 key 获取，如果需要则创建或取得可用条目”的缓存模式。阅读调用点时要确认它是否会产生新 instance entry，否则容易把 lookup 误判为纯只读操作。

`FindInstance()` should be read as lookup of existing entries, while `GetInstance()` can participate in get-or-create style cache access depending on the calling pattern.

`Invalidate()` 影响的是 Hydra 内部实例缓存有效性，而不是删除 USD prim、移除 scene delegate 路径或修改 layer。它适合在实例依赖的数据变化、同步结果过期或共享缓存需要重新计算时使用。把 `Invalidate()` 当成 authoring 操作会导致对渲染状态和场景数据边界的误读。

`Invalidate()` invalidates registry/cache state. It is not a USD authoring operation and does not delete USD prims or edit layers.

`GarbageCollect()` 表明 registry 也负责生命周期收敛。未再被引用、已经失效、或与当前 render index / sync 状态不再匹配的 instance entries 可以被清理。该行为通常与 Hydra 同步阶段、resource lifetime 和 instancing cache 管理相关，而不是由 USD composition 直接触发。

`GarbageCollect()` cleans up stale or unused registry entries and should be understood as cache/lifetime maintenance inside Hydra.

`begin()`、`end()`、`size()` 和 `const_iterator` 说明该类也提供标准容器式观察接口。遍历 registry 可用于调试当前缓存中有哪些 instance entries、数量是否符合预期、以及哪些 entries 在 garbage collection 后仍存在；但遍历结果不等同于场景中的 prim 总数或实例化原型数量。

Container-style APIs such as `begin()`, `end()`, `size()`, and `const_iterator` expose registry contents for inspection, not scene population counts.

`HdInstance` 是相邻核心类型。`HdInstanceRegistry< VALUE >` 负责组织、查找和维护 `HdInstance` 相关条目；`HdInstance` 自身才承载具体实例记录语义。二者的关系类似 registry 与 entry：registry 管理 key、lookup 和 lifetime，entry / value 表达某个实例相关状态。

`HdInstance` is the adjacent entry type. The registry manages keys, lookup, and lifetime; instance entries carry the instance-specific state.

dictionary container 的 key 语义通常来自调用方，而不是由 USD 路径天然决定。一个 registry 可能按 prototype、instance source、resource identity 或其他 Hydra 内部 key 组织条目；因此不要把 key 简单等同为 `SdfPath` 或 prim path。排查实例复用错误时，应追踪调用方构造 key 的方式，而不是只看场景层级。

Dictionary keys are defined by the calling code and should not automatically be assumed to be USD prim paths or `SdfPath` values.

生命周期边界也要与线程和同步阶段一起看。Registry 容器提供查找、遍历、失效和回收接口，但不等于自动解决所有并发访问、render-index 同步或资源引用问题。调用方仍需要在正确的 Hydra sync / update 阶段使用它，并确保 `VALUE` 中保存的对象没有越过自己的有效期。

Lifetime must be coordinated with Hydra synchronization and the lifetime of objects stored in `VALUE`; the registry does not by itself solve every concurrency or resource-ownership issue.

调试 Hydra instancing 时，本页有三个常见检查点：首先确认调用方是否用了 `FindInstance()` 还是 `GetInstance()`；其次检查 `Invalidate()` 是否在依赖数据变化后被触发；最后检查 `GarbageCollect()` 后 registry `size()` 是否符合预期。若实例缓存没有更新、实例条目持续增长或失效数据仍被复用，这个 registry 往往比 USD authoring 层更接近问题现场。

For Hydra instancing debugging, inspect lookup behavior, invalidation behavior, and registry size after garbage collection.

常见误用包括：把 `VALUE` 当成固定 OpenUSD 类型；把 registry 遍历结果当成 USD prim traversal；把 `Invalidate()` 当成删除场景对象；或认为 `GarbageCollect()` 会解决所有资源生命周期问题。正确边界是：它是 Hydra 的 C++ template registry，class template、dictionary container、`VALUE`、`InstanceType`、`const_iterator` 和 `HdInstance` 等英文标识必须保持原样。

Common mistakes include treating `VALUE` as a fixed OpenUSD type, reading registry iteration as USD traversal, or confusing invalidation with scene deletion.
