逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdAttributeLimits` 解决的问题不是“读写一个 `UsdAttribute` 的当前数值”，而是读写该属性上的 `limits` dictionary metadata。也就是说，它操作的是 composition 后可见的属性元数据意见，用来描述工具、UI、验证器或导出管线可以参考的值域边界；它不会自动改变 default value，也不会修改 time samples。

`UsdAttributeLimits` works on the `limits` dictionary metadata associated with a `UsdAttribute`; it is not an accessor for the attribute's authored value samples.

官方描述里的 “particular sub-dictionary” 是理解本页的关键。`limits` metadata 可以包含多个语义分组，例如 soft limits 与 hard limits；`UsdAttributeLimits` 实例绑定到其中一个 sub-dictionary，再在该分组内读取或写入 `minimum` 与 `maximum`。因此调用者必须知道自己处理的是哪一类边界，而不是把所有限制都混成一个全局范围。

The phrase “particular sub-dictionary” means the object addresses one named group within the attribute's `limits` metadata rather than every possible limit at once.

在每个 sub-dictionary 内，minimum 与 maximum 分别编码在 `UsdLimitsKeys->Minimum` 和 `UsdLimitsKeys->Maximum` 下。这两个 key 是页面必须保留原样的 OpenUSD token / key 名称；中文说明只解释它们的用途，不翻译或改写字面量。底层值通过 `VtDictionary` 与 `VtValue` 表示，因此类型不匹配、缺少 key 或 authored opinion 被清除时，都应按 metadata 读取失败或 fallback 路径来处理。

Within a sub-dictionary, the minimum and maximum entries are keyed by `UsdLimitsKeys->Minimum` and `UsdLimitsKeys->Maximum`, with values transported through `VtDictionary` and `VtValue`.

soft limits 与 hard limits 的差别通常是使用约定而不是运行时强制执行。soft limits 可用于滑块、编辑器提示或“常用范围”；hard limits 可用于验证或更严格的输入边界。即使页面例子说常用范围是 5 到 10、硬范围是 0 到 15，`UsdAttributeLimits` 本身也不会 clamp 属性值到这些范围内；真正是否拒绝、修正或提示，取决于消费这些 metadata 的工具。

Soft limits are commonly consumed as guidance, while hard limits may be consumed as stricter bounds; neither form automatically clamps authored USD values by itself.

读取 API 分为直接读取和带 fallback 的读取。`GetMinimum()`、`GetMaximum()`、`Get()` 关注 metadata 中是否能取得对应值；`GetMinimumOr()`、`GetMaximumOr()`、`GetOr()` 允许调用方在没有 authored value 或无法解析时传入替代值。调试时要区分“确实写了边界”和“读取函数返回了 fallback”，否则会误以为 layer 中存在某个 limit opinion。

The `Or` variants such as `GetMinimumOr()`, `GetMaximumOr()`, and `GetOr()` are fallback helpers; they do not prove that a limit has been authored.

写入 API 包括 `SetMinimum()`、`SetMaximum()` 以及更通用的写入路径。它们 author 的是 `limits` metadata 中当前 sub-dictionary 下的条目，和 `UsdAttribute` 的类型值、插值样本、连接关系或 schema fallback 不是同一层概念。对导出器或工具插件来说，这些写入应该被当成 metadata opinion，与普通属性值写入分开审计。

The setters author metadata entries inside the selected `limits` sub-dictionary; they are separate from the attribute's value samples and schema fallback behavior.

判断是否存在 authored opinion 时，应优先使用 `HasAuthoredMinimum()`、`HasAuthoredMaximum()` 与 `HasAuthored()`。这些函数回答的是 layer stack 中是否存在该 metadata 意见，而不是某个工具是否计算出了默认范围。这个区别对合成、覆盖和调试很重要：fallback 值可以帮助消费端运行，但它不等于在 USD 文件中写入了限制。

`HasAuthoredMinimum()`, `HasAuthoredMaximum()`, and `HasAuthored()` distinguish authored metadata opinions from consumer-provided fallback values.

清除 API 包括 `ClearMinimum()`、`ClearMaximum()` 与 `Clear()`。这些函数清理的是当前 sub-dictionary 下的 authored limit metadata；它们不会删除 `UsdAttribute`，不会移除该属性的 default value，也不会清空 time samples。多 layer 场景下，清除当前编辑目标中的 opinion 后，较弱或较强 layer 的意见仍可能影响最终 composition 结果。

`ClearMinimum()`, `ClearMaximum()`, and `Clear()` remove authored limit metadata, not the attribute itself or its authored value samples.

`GetSubDictKey()` 用来暴露当前 wrapper 绑定的 sub-dictionary key，`GetAttribute()` 返回其关联的 `UsdAttribute`，`IsValid()` 则帮助确认 wrapper 是否处在可用状态。这些函数的价值在于把“操作哪个属性”和“操作 limits metadata 的哪个分组”分开，让调用方在工具面板、验证器和导出逻辑中能给出更准确的诊断信息。

`GetSubDictKey()`, `GetAttribute()`, and `IsValid()` help callers verify both the target attribute and the selected metadata sub-dictionary.

常见误读有三类：第一，把 limits metadata 当成物理求解器约束；第二，把 soft limits 当成一定会执行的 clamp；第三，把 `GetMinimumOr()` 返回的 fallback 当成已 author 的 USD 数据。正确理解是：`UsdAttributeLimits` 提供的是属性元数据的组织和读写接口，是否检查、显示或强制执行这些边界，由上层应用或验证系统决定。

The common mistake is to treat limits metadata as enforcement. In OpenUSD it is metadata consumed by tools, validators, UI, or pipeline policy.
