逐段双语理解 / Paragraph-Level Bilingual Coverage

UsdUI 是给客户端图形用户界面工具使用的 schema 模块。官方 Overview 明确说，它把信息编码到 USD prims 上，帮助 UI 工具决定如何在界面布局中组织和呈现 prims。这里的主语是“工具界面如何展示 USD 图结构”，不是“3D renderer 如何渲染场景内容”。因此中文读者首先要把 UsdUI 放在 tool-facing metadata 层，而不是几何、材质、物理或渲染结果层。UsdUI encodes information on USD prims for client graphical user interface tools to organize and present prims in UI layouts.

官方特别强调，这些信息不指导 scene data 的 3D rendering；它关注的是如何可视化组成 UsdStage 的 graph nodes。也就是说，UsdUI 可以影响节点图、属性面板、scene graph browser、DCC 工具、review tool 或辅助功能界面的呈现方式，但不应被解释成会改变最终渲染、composition 结果或 prim 的核心语义。UsdUI does not guide 3D rendering; it is about visualizing the nodes in the graph that comprises a UsdStage.

官方列出的 primary classes 分成两组。第一组是 UsdUINodeGraphNodeAPI，用于记录那些通过 relationships 参与 interconnected graph 的 prim 的布局信息，例如 shading network 中的 prim。第二组是 hints 对象：UsdUIObjectHints、UsdUIPrimHints、UsdUIPropertyHints、UsdUIAttributeHints，它们分别面向 UsdObject、UsdPrim、UsdProperty 和 UsdAttribute 的 uiHints metadata dictionary。Primary classes include UsdUINodeGraphNodeAPI and hint APIs for UsdObject, UsdPrim, UsdProperty, and UsdAttribute.

官方还把本模块分成 Accessibility Information 与 UI Hints 两条阅读线。前者描述可暴露给 runtime accessibility frameworks 的辅助信息，例如 voice controls 或 screen readers；后者描述 prims 和 properties 应如何在 UI 中显示，例如名称、隐藏状态、分组、显示条件、值标签、值标签顺序、display group 展开状态和显示条件。UsdUI covers accessibility information for assistive UI and UI hints for presenting prims and properties.

读这页的核心判断是：如果问题是“工具界面为什么这样组织节点或属性”，优先看 UsdUI；如果问题是“stage 为什么这样 compose”，看 Usd；如果问题是“几何为什么这样渲染或变换”，看 UsdGeom；如果问题是“材质网络数据流是什么”，看 UsdShade。这条边界能避免把 UI annotation 误判成场景语义。Use UsdUI for tool presentation behavior, not for composition, geometry, shading, or rendering semantics.

作为 API 模块入口，本页不应只让读者看到类名清单，还要说明这些类如何进入实际工具流程。典型流程是：资产或 schema 作者在 prim、property 或 attribute 上 author 对应 metadata；管线工具在加载 stage 后读取这些 metadata；最终 DCC、review viewer、节点编辑器或属性面板决定是否展示、隐藏、分组、排序或重命名某些 UI 元素。UsdUI 只定义可携带的数据约定，具体 UI 控件、布局样式、辅助功能集成和错误提示仍由应用实现。UsdUI defines portable presentation metadata; applications decide concrete widgets, layout, accessibility integration, and messages.

因此排查问题时要先问“信息写在哪里、由哪个对象类型携带、目标工具是否读取它”。UsdUIObjectHints 更适合 object-level 展示名或隐藏提示；UsdUIPrimHints 更接近 prim 层的展开状态和分组可见性；UsdUIPropertyHints 处理属性面板中的分组与条件显示；UsdUIAttributeHints 则关注 attribute value 的标签、顺序和用户可读表达。把这几层分清楚，才能避免把 attribute 的值标签写到 prim hint，或把 property 的显示条件误放到全局对象提示。Correct placement across object, prim, property, and attribute hints is required for tools to consume the metadata predictably.

Accessibility Information / 辅助功能信息

AccessibilityAPI schema 用于描述 prim 上的 accessibility information，这些信息可以被某个 runtime 的 accessibility frameworks 暴露出来。官方举例包括 voice controls 和 screen readers。中文页要强调：这是为了帮助工具把 USD 场景中的对象用辅助技术可理解的方式呈现，不是为了改变 prim 的几何、材质或物理行为。AccessibilityAPI describes accessibility information on a prim for runtime accessibility frameworks, voice controls, and screen readers.

官方说明 accessibility information 以标准三元组提供：label、description 和 priority。其中 label 更像用户可听到或读到的短名称，description 给出更详细说明，priority 帮助工具决定多个信息之间的呈现顺序或重要程度。翻译时这些字段名必须保留英文，便于和 schema、代码、日志和工具 UI 对照。Accessibility information is provided as a standard triplet of label, description, and priority.

AccessibilityAPI 是 multiple-apply schema，因此同一个 prim 可以有多个带 namespace 的 accessibility triplets。官方说 instance name 可以反映该 triplet 的用途，例如针对 prim 的 size 或 color 提供不同信息。实际管线中，这意味着不要把 accessibility 信息写成一个不可区分的大段字符串；应按用途命名，便于不同工具按场景、属性或用户需求选择合适的信息。AccessibilityAPI is multiple-apply, allowing multiple namespaced accessibility triplets for different purposes.

调试 accessibility 时，先检查 schema 是否应用到正确 prim，再检查 instance name、label、description、priority 是否存在，随后检查目标 runtime 或 DCC 工具是否会把这些 metadata 接入自己的 accessibility framework。若 screen reader 没有读出信息，不应立刻判定 USD 数据错误；也可能是工具未实现消费、runtime 未开启辅助功能，或优先级/命名策略不符合工具约定。Accessibility debugging separates schema authoring, instance naming, metadata values, and runtime framework consumption.

在团队协作里，AccessibilityAPI 还需要和命名规范、资产职责以及 review 工具约定一起看。比如同一 prim 可以同时携带面向尺寸、颜色、用途或交互含义的说明；这些说明既要足够稳定，能在版本迭代后继续被工具识别，也要避免把实时状态或用户界面临时文本硬写进资产。更稳妥的做法是把 schema 中的 label、description、priority 当作可交换的语义描述，而把语言本地化、朗读策略、焦点顺序和平台适配留给消费端。Accessibility metadata should remain portable semantic data, while localization and platform-specific presentation remain consumer responsibilities.

如果某个工具既支持 AccessibilityAPI 又支持自有 UI metadata，排错时不要只看最终界面。先用 USD inspection 工具确认 multiple-apply API 是否存在，再确认 instance name 是否和工具期望一致；再检查 authoring layer 是否被更强 opinion 覆盖；最后检查应用是否过滤了不可见、未加载、变体切换后不存在或 payload 未打开的 prim。这样可以把“USD 数据没写上”“composition 后被覆盖”“工具没有消费”和“runtime 辅助功能没有启用”四类问题拆开。Debug authoring, composition, tool consumption, and runtime accessibility settings as separate failure layers.

UI Hints / 界面提示

官方将 UI hints 概括为与 scene data 在 UI 中呈现有关的 metadata fields。大多数这类字段放在 prims 和 properties 的 uiHints metadata dictionary 中。UsdUI 提供对应 core object type 的 hints 对象来读写这些值：UsdUIObjectHints、UsdUIPrimHints、UsdUIPropertyHints 和 UsdUIAttributeHints。官方也明确说没有只适用于 UsdRelationship 的 unique hints，所以没有 UsdUIRelationshipHints。UI hints are metadata fields for UI presentation, mostly held in uiHints dictionaries on prims and properties.

  • displayName：来自 UsdUIObjectHints，用于 prim 或 property 的用户可见名称。它是界面名，不应替代稳定的 API 名、prim path 或 property name。displayName is a user-facing name for a prim or property.
  • hidden：来自 UsdUIObjectHints，提示某个 object 是否应在 UI 中隐藏。它影响工具展示，不代表数据在 stage 中不存在，也不等于 visibility。hidden hints whether an object should be hidden from UI display.
  • displayGroup：来自 UsdUIPropertyHints，定义 property 所属的显示分组。它帮助属性面板组织字段，不改变 property 的取值或 schema 语义。displayGroup names the display group to which a property belongs.
  • shownIf：来自 UsdUIPropertyHints，是判断某个 property 在 UI 中隐藏或显示的表达式。它是 presentation condition，不是 composition condition。shownIf is an expression for hiding or showing a property in UI.
  • valueLabels 与 valueLabelsOrder：来自 UsdUIAttributeHints，分别把 attribute values 映射到显示标签，并定义这些标签在 combo box 等控件中的用户可见顺序。valueLabels and valueLabelsOrder control labels and ordering for attribute values.
  • displayGroupsExpanded 与 displayGroupsShownIf：来自 UsdUIPrimHints，分别提示 prim 的 display groups 是否默认展开，以及 display group 的显示条件。displayGroupsExpanded and displayGroupsShownIf guide prim display group presentation.

官方还列出一批 UI-relevant metadata，但它们因为各种原因不属于 uiHints dictionary，例如 userDoc、userDocBrief、allowedTokens、propertyOrder、displayGroupOrder、limits 和 arraySizeConstraint。中文读者需要知道：这些字段同样会影响工具如何展示或校验属性，但来源不一定是 UsdUI* hints 对象。Other UI-relevant metadata includes userDoc, allowedTokens, propertyOrder, limits, and arraySizeConstraint.

尤其要注意 displayGroupOrder：官方说明它现在被认为 deprecated，display group order 应由 property order 决定。若工具还依赖旧字段排序，中文页应提醒读者这是兼容性问题，不应把旧字段当成新管线的首选方式。displayGroupOrder is deprecated; display group order should be determined by property order.

uiHints 与这些非 uiHints metadata 的区别很关键。前者是 UsdUI 为 presentation hints 建立的集中 dictionary；后者可能早已存在于 USD 或 schema 生态里，用于文档、可选 token、属性顺序、范围限制或数组尺寸约束。工具作者在实现属性面板时通常会同时读取两类信息：用 propertyOrder 控制基础顺序，用 displayGroup 做分组，用 allowedTokens 生成下拉候选，用 valueLabels 改善显示文本，再用 shownIf 或 hidden 决定是否展示。Tools often combine uiHints with other UI-relevant metadata such as propertyOrder, allowedTokens, limits, and arraySizeConstraint.

这也解释了为什么本页必须保留英文 API 名和字段名：uiHints 的键、allowedTokens 的 token、displayGroup 的组名、shownIf 的表达式和 valueLabels 的映射都需要被程序精确读取。中文解释负责说明使用场景和误读边界，但不能把这些键名翻成中文后写回 USD，否则工具会找不到字段，或者在不同 DCC 之间失去互操作性。Field names, token names, expressions, and mappings remain exact English data keys for interoperability.

写作建议是先定义稳定的对象和属性名称，再用 displayName 增强面向人的标签；先让 schema 和 composition 语义正确，再追加 displayGroup、displayGroupsExpanded 和 displayGroupsShownIf 改善浏览体验；先确认 attribute 的真实取值集合，再添加 valueLabels 与 valueLabelsOrder。如果反过来先写 UI 文案，很容易掩盖底层属性建模错误，或者让工具显示看似友好但语义不稳定的数据。Author schema semantics first, then use UI hints to improve presentation without hiding modeling errors.

边界、误读点与调试路径 / Boundaries, Misreads, and Debugging

  • 不要把 hidden 当作 visibility。前者是 UI 展示提示，后者才和场景可见性/渲染消费相关。hidden is a UI display hint, not scene visibility.
  • 不要把 shownIf 当作 USD composition 规则。它只是 UI 决定是否显示 property 的表达式，不决定 layer stack 或 authored opinions 是否存在。shownIf is a UI presentation expression, not a composition rule.
  • 不要把 displayName 当作稳定标识符。工具可以显示它，但脚本、schema、调试日志和跨工具引用仍应使用真实 prim path、property name 或 API 名。displayName is not a stable identifier.
  • 不要把 UsdUINodeGraphNodeAPI 与 shading dataflow 混为一谈。它记录节点图布局信息；关系网络或 shader 语义仍由对应 schema 表达。UsdUINodeGraphNodeAPI records layout information, not shader dataflow semantics.
  • 不要假设所有 viewer 都消费全部 UI hints。不同 DCC、review tool 或浏览器可能只支持其中一部分字段。Different tools may consume only part of the available UI hints.

调试 UI hints 时，先确认 metadata 是否写在正确对象上：object、prim、property 或 attribute；再确认字段属于 uiHints dictionary 还是其他 UI-relevant metadata；随后检查类型、token、字典键、表达式和标签顺序；最后再检查目标工具是否实现了该字段的消费逻辑。若某个属性在 UI 中没有按预期分组或隐藏，问题可能在 metadata authoring、工具支持、表达式语法、schema registry 或旧字段兼容性，而不一定是 USD 核心数据错误。Debug UI hints through metadata placement, field source, value type, expressions, and tool support.

对 accessibility 信息也要分层：USD 层负责存储 label、description、priority 和 multiple-apply instance；工具层负责把它们映射到 voice control、screen reader 或平台 accessibility framework；用户设置和 runtime 权限也可能影响最终呈现。中文页要让读者知道，AccessibilityAPI 是可读性和辅助功能元数据入口，不是自动保证所有应用都具备完整无障碍体验。AccessibilityAPI stores metadata; tools and runtimes decide how it reaches assistive features.

另一个常见误读是把 UsdUINodeGraphNodeAPI 当作 graph evaluation 机制。官方只把它放在 UI layout 语境里，尤其是通过 relationships 连接成图的 prim，例如 shading network 中的节点；它可以帮助节点编辑器恢复位置或布局提示，但不负责计算 shader、重连关系、决定 shader 输出，也不替代 UsdShade 的 connectable node 语义。调试材质网络时，先看 UsdShade 连接；调试节点图“看起来乱不乱”时，再看 UsdUI 布局数据。UsdUINodeGraphNodeAPI is for node-graph layout presentation, not shader evaluation or relationship semantics.

发布资产时，UsdUI 还要考虑跨工具降级。某些工具可能只支持 displayName 和 displayGroup，不支持 shownIf；某些工具可能展示 userDoc，但忽略 displayGroupsShownIf；某些审阅工具可能完全忽略 UI hints，只显示原始 property name。一个健壮的 USD 资产不应依赖 UI hints 才能表达核心语义；即使 hints 被忽略，真实 prim path、schema 类型、attribute name、token 和 documentation metadata 仍应能让读者和工具理解数据。A robust asset remains understandable even when a consuming tool ignores some or all UI hints.

如果界面表现和预期不一致，可以按四步定位：第一，用本地 source snapshot 或 USD inspection 确认 metadata 是否真的 authored；第二，检查 layer stack、variant、payload 和 reference 后的 composed value；第三，确认字段属于 uiHints 还是其他 UI-relevant metadata，避免查错 API；第四，确认目标应用版本是否支持该字段。这个顺序能把数据缺失、composition 覆盖、API 用错和工具能力不足分开。A practical failure triage separates authored metadata, composed values, field ownership, and application support.

人工审阅这类页面时，还要避免把“工具显示得更友好”误认为“数据已经更正确”。中文主线需要不断提醒：界面提示的价值在于降低查找、浏览、解释和协作成本；数据正确性仍来自清晰的 schema、稳定的命名、可组合的层级意见和可追踪的资产发布流程。只有把这两层分开，读者才能在看到友好名称、分组或隐藏规则时，仍然知道应回到哪个真实属性、哪条 authored opinion 和哪个消费工具去验证问题。这样也能支持后续人工复审、问题回溯和长期维护跟进。Reviewers should separate friendly tool presentation from the underlying correctness of schema data and authored opinions.

相邻阅读路径 / Adjacent Reading Path

建议先读本页建立 UsdUI 的模块边界，再进入 release schema 页面：AccessibilityAPI 处理辅助功能三元组；NodeGraphNodeAPI 处理节点图布局；PrimHints、PropertyHints、AttributeHints 处理不同对象层级上的 UI hints。Read the module page with the release schema pages for AccessibilityAPI, NodeGraphNodeAPI, PrimHints, PropertyHints, and AttributeHints.

如果问题涉及 stage、prim、property、attribute 或 metadata 的核心对象模型，应回到 Usd；如果问题涉及空间层级和 transform，应看 UsdGeom；如果问题涉及 asset preview 或音频媒体在工具界面中的展示，可结合 UsdMedia；如果问题涉及 shading network 的节点图语义，应结合 UsdShade，但把布局 hints 和 shader 语义分开。Use Usd, UsdGeom, UsdMedia, and UsdShade for adjacent concerns.

对工具开发者来说，UsdUI 不是替代 schema 文档的“中文显示层”，而是读取 USD 数据时的一个额外提示源。实现属性浏览器时，可以先从 schema registry、property definitions 和 authored metadata 建立基础列表；再合并 propertyOrder、displayGroup、hidden、shownIf、valueLabels 等展示提示；最后再处理用户偏好、插件覆盖和本地化文案。这样即使某个 hint 无法解析，工具也能退回到稳定的 USD 名称和 schema 信息，而不是让整页 UI 失效。Tool implementations should merge UsdUI hints with schema definitions and user preferences while preserving stable fallbacks.

对资产作者来说，建议把 UsdUI 当作“可读性增强”，而不是“语义补丁”。如果某个属性没有清晰 schema 名或文档，仅靠 displayName 让它看起来友好，后续脚本和跨工具交换仍会困难；如果一个属性必须满足取值范围，应优先在 schema 或 metadata 中表达 limits、allowedTokens 或 arraySizeConstraint，再用 valueLabels 优化展示。中文读者掌握这条顺序，才能把 UI hints 用在提高阅读和操作效率上，而不是掩盖模型设计问题。Asset authors should use UsdUI to improve readability, not to compensate for unclear schema modeling.

本页保留 UsdUI、UsdUINodeGraphNodeAPI、UsdUIObjectHints、UsdUIPrimHints、UsdUIPropertyHints、UsdUIAttributeHints、AccessibilityAPI、uiHints、displayName、displayGroup、shownIf、valueLabels、propertyOrder 等英文标识，便于和官方 Doxygen、schema registry、release user guide、源码和工具日志核对。English identifiers are preserved for source parity and debugging against Doxygen, schema registry, user guides, source, and tool logs.

实践验收清单 / Practical Acceptance Checklist

人工复审时可以用五个问题判断本页是否可读：第一，读者是否能说明 UsdUI 是 UI 工具展示层，而不是渲染层；第二，是否能区分 AccessibilityAPI 与 uiHints；第三，是否能把 displayName、hidden、displayGroup、shownIf、valueLabels 等字段对应到正确的 hints 对象；第四，是否知道哪些 UI-relevant metadata 不在 uiHints dictionary 中；第五，是否能按 metadata placement、字段类型和工具消费能力进行排错。Review checks should cover module scope, AccessibilityAPI, uiHints fields, external UI-relevant metadata, and debugging layers.

本页依据当前本地 source snapshot 与官方 release API 页整理。若未来 UsdUI 增加新的 hints、弃用旧字段或改变 accessibility schema 约定，应重新做 source parity，不应把本轮解释当成永久完整的 UI schema 手册。Future UsdUI changes require a fresh source-parity pass.

验收本页时还应检查阅读路径是否顺畅：读者从总入口进入 API，本页能看到 breadcrumb 和侧栏；从本页能进入 AccessibilityAPI、NodeGraphNodeAPI、PrimHints、PropertyHints、AttributeHints 等 release schema 页；再能回到相邻的 UsdMedia、UsdUtils 或 Usd 模块页。这样本地中文站不是孤立导读，而是能支持连续排查“工具界面为何这样呈现 USD 数据”的阅读网络。Acceptance includes local reading-flow continuity across API and release schema pages.

完成页的目标是让中文读者能在本地站中从 API 模块入口跳到 release schema 页，再返回相邻 API 继续判断问题归属。英文原文和 API 名保留用于核对，但中文主阅读路径已经足够说明页面职责、官方 section、API 分组、误读点和调试方向。The completed page supports local reading, source lookup, issue attribution, and explicit official follow-up.

No Matches UsdUI: USD UI Schemas Overview This module provides schemas for encoding information on USD prims for client graphical user interface tools to use in deciding how to organize and present the prims in UI layouts. This information does not guide the 3D rendering of data contained in a scene, rather it's about how to visualize the nodes in the graph that comprises a UsdStage . The primary classes are: UsdUINodeGraphNodeAPI - Encodes layout information for prims that participate in an interconnected (via relationships) graph, as would prims in a shading network. UsdUIObjectHints - Provides API for reading and writing fields in UsdObject 's uiHints metadata dictionary. UsdUIPrimHints - Provides API for reading and writing fields in UsdPrim 's uiHints metadata dictionary. UsdUIPropertyHints - Provides API for reading and writing fields in UsdProperty 's uiHints metadata dictionary. UsdUIAttributeHints - Provides API for reading and writing fields in UsdAttribute 's uiHints metadata dictionary. UsdUI additionally has schemas that describe accessibility information used in assistive UI, and UI hints that provide hints on how prims and properties should be presented in a UI.
