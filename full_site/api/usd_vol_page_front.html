逐段双语理解 / Paragraph-Level Bilingual Coverage

UsdVol 是 USD 中描述 volumetric geometry 的 schema 模块。官方 Overview 把它分成两条主线：一条是连续体积，比如 smoke、fire，通过 UsdVolVolume 和外部 field asset 表达；另一条是 UsdVolParticleField，用于表达由 3D particles 定义空间存在的 lightfield 技术，例如 3D Gaussian splats，并为 NeRF 及未来粒子场技术保留扩展空间。UsdVol provides schemas for representing volumetric geometry, continuous volumes, and particle fields.

传统 volume 路径中，UsdVolVolume 描述由多个 fields 组成的 volume。官方把 field 类比为 OpenVDB 里的 grid；Volume 本身是 UsdGeomGprim，因此可变换、可渲染，也可以有 material binding。真正的体积数据通常不直接存进 USD layer，而是由 UsdVolFieldAsset、UsdVolOpenVDBAsset 或 UsdVolField3DAsset 指向外部文件。UsdVolVolume is a UsdGeomGprim composed of fields, often backed by OpenVDB or Field3D assets.

UsdVolVolumeFieldBase 用来区分 volume fields 和 particle fields；UsdVolFieldBase 是单个体积 field 的抽象 schema；UsdVolFieldAsset 表示存放在 layer/stage 外部资产中的单个 field。由于 USD 当前不直接表示体积数据结构，volume 通常保存在 OpenVDB、Field3D 等专用格式里，USD 负责描述路径、field 选择和场景关系。FieldBase and FieldAsset separate the USD schema from external volumetric data storage.

OpenVDBAsset 和 Field3DAsset 是具体外部格式 schema。filePath 指向外部体积文件，fieldName 与 fieldIndex 用于从一个可能包含多个 field/grid 的资产中选择目标 field。动画 volume 常通过 filePath.timeSamples 切换不同帧的 VDB 文件，而不是在 USD 中直接采样巨大的体积数组。OpenVDBAsset and Field3DAsset point to external files and select fields via filePath, fieldName, or fieldIndex.

Particle field 路径处理的是快速演进的研究型表示。ParticleField 是所有 particle fields 的 concrete base type；applied schemas 用于描述 position、kernel、radiance、orientation、scale、opacity 等数据如何提供。官方特别提到 3D Gaussian splats 和未来可能的 NeRF/particle field 演进，因此中文页必须把它看成可组合、可验证的 schema 结构，而不是固定成某一种 renderer 私有格式。ParticleField uses a composable schema system for particle positions, kernels, radiance, and related attributes.

Volume schema 分组 / Volume Schema Groups

  • UsdVolVolume：由多个 fields 组成的体积 prim。它继承 UsdGeomGprim 语义，所以与 transform、extent、renderability、material binding、purpose 等几何约定相关。UsdVolVolume is a renderable Gprim composed of named fields.
  • UsdVolFieldBase：单个 volume field 的抽象基类。任何表示体积 field 的 schema 都应继承它，方便消费者识别 field prim 的角色。UsdVolFieldBase is the abstract base for volume field primitives.
  • UsdVolFieldAsset：外部体积资产中的单个 field。它不直接装载体积值，而是描述资产路径、fieldName、fieldIndex 等选择信息。UsdVolFieldAsset describes a field stored outside the USD layer or stage.
  • UsdVolOpenVDBAsset：OpenVDB 资产中的单个 volume field。它让 Hydra、renderer 或 pipeline 工具可以发现 VDB 文件并解释对应 grid。UsdVolOpenVDBAsset points consumers to an OpenVDB-backed field.
  • UsdVolField3DAsset：Field3D 资产中的单个 volume field。它与 OpenVDBAsset 处于同一抽象层，区别在外部文件格式和消费者支持度。UsdVolField3DAsset represents a Field3D-backed field.

这里的关键边界是：USD 描述 volume prim、field prim、relationship 和外部文件路径；真正体积数据结构、稀疏采样、压缩、插值和加载策略属于 OpenVDB/Field3D 文件格式或 renderer/volume loader。不要把 UsdVol 当作体积模拟器，也不要把 OpenVDB 文件误认为会被 USD FileFormat plugin 直接展开成 layer 内容。UsdVol describes the schema and asset references; data loading and sampling belong to volume formats and consumers.

发布体积资产时，建议把 Volume prim 当作“场景中的体积对象”，把 FieldAsset prim 当作“数据入口”，把外部 VDB/Field3D 文件当作“重型数据载体”。这三个层次分别由不同工具负责：USD 校验器检查 schema 和关系，资产解析器检查文件路径与版本，体积渲染器检查 grid、密度、采样和材质。只要明确分层，就能避免把所有问题都归咎于某一个工具。A volume asset should separate the scene object, field entry points, and heavy external data files.

在大型场景中，Volume 还要和 bounds、payload、visibility、purpose、material binding、render settings 和 asset resolver 一起考虑。比如一个 VDB 文件存在并不表示它会显示：可能是 field relationship 没连上，可能是 extent 不正确，可能是 render purpose 被过滤，也可能是材质或渲染器不支持对应 grid。因此中文主阅读路径必须给出排查顺序，而不是只解释类继承。A visible volume depends on schema relationships, bounds, purposes, material binding, render settings, and resolver behavior.

Usage Notes 与关系命名 / Usage Notes and Relationship Naming

官方 Example Usage 中，Volume "wisp" 通过 rel field:density = </wisp/density> 连接到子 prim OpenVDBAsset "density"，后者通过 asset filePath 和 token fieldName 指向外部 VDB 的 grid。这个示例说明 USD 层关心的是“volume 有哪些语义命名的 field”，以及这些 field 在外部资产中如何找到。The example connects a Volume to an OpenVDBAsset field using field:density, filePath, and fieldName.

Namespace Organization and Transformation 说明 field prim 不强制必须是 Volume 的子节点，但通常会作为子节点创建，因为 volume 和 fields 在概念上不可分。这样移动 Volume prim 时，子 field prim 的 local-to-world transformation 会一起变化，再叠加外部资产编码中的变换，消费者就能得到正确世界空间位置。Fields are usually children of the Volume so transforms and organization stay together.

Field Relationships Establish Consumer Field Names 是最容易误读的 section。field:density 关系名和 OpenVDBAsset 上的 fieldName = "density" 看似重复，但目的不同：relationship 名字定义 Volume 或 pipeline 希望消费者看到的 field 名，fieldName 则选择外部 VDB 中的 grid 名。这样可以把 pipeline 中的 velocity 映射到 VDB 里的 vel，也可以复用同一个 Field 作为多个 volume field。The relationship name defines the consumer-facing field name; fieldName selects the grid inside the asset.

Why is OpenVDBAsset not a FileFormat plugin? 给出重要边界：USD 没有直接描述体积数据结构的内建数据模型，这类似 texture 不是 FileFormat plugin。VDB 可以被看作 3D texture；OpenVDBAsset schema 描述可被 Hydra、renderer 或其他消费者发现的 asset path，而不是要求 USD composition 把 VDB 文件变成一堆 layer opinions。OpenVDBAsset is an asset reference for consumers such as Hydra or a renderer, not a USD FileFormat plugin.

这也解释了为什么体积资产的发布检查必须分层：USD 侧检查 prim、relationship、filePath、fieldName、fieldIndex 和 transform；资产侧检查 VDB/Field3D 文件是否存在、grid 名是否正确、帧序列是否完整；渲染侧检查 volume loader、shader、step size、density scale、material binding 和可见性。只要把这些层混在一起，就很容易把路径错误误判成 renderer 不支持，或把 grid 命名错误误判成 USD schema 错误。Validation should separate USD schema, external asset integrity, and renderer-volume consumption.

对动画 volume，filePath.timeSamples 是关键线索。它说明每个时间码要读取哪个外部体积文件，而不是在 USD 中存储体积采样。排查“体积不动”时应先确认 timeSamples 是否覆盖当前帧、资产路径是否能解析、每帧 VDB 是否有同名 grid，再检查渲染器是否按 stage time 读取了对应 filePath。Animated volumes should be debugged through filePath time samples and per-frame asset availability.

官方 section 读者检查表 / Section-by-Section Checklist

  • Overview：先区分 continuous volume 和 particle field 两条路线。前者面向 smoke/fire 等外部 field asset，后者面向 3D particles、Gaussian splats、NeRF 类演进技术。Overview separates continuous volumes from particle fields.
  • Volume Schemas：确认 Volume、FieldBase、FieldAsset、OpenVDBAsset、Field3DAsset 的继承与职责。Volume 是场景中的 Gprim，FieldAsset 是外部体积数据入口，两者不能互相替代。Volume Schemas define the core prim and asset-backed field roles.
  • Example Usage：把示例当作结构模板：Volume 定义 extent 和 field relationship，OpenVDBAsset 定义 filePath 和 fieldName，动画通过 filePath.timeSamples 切换外部文件。Example Usage shows extent, field relationships, filePath, fieldName, and time samples.
  • Usage Notes：重点检查 namespace、transform、field relationship 名称、外部 grid 名称和 OpenVDBAsset 非 FileFormat plugin 的边界。这部分直接决定调试顺序。Usage Notes define namespace, transforms, naming, and asset-plugin boundaries.
  • Particle Field Schemas：确认 ParticleField 是组合基底，applied schemas 提供 position、kernel、radiance 等必要数据。不要把这个 section 当作普通 volume grid 的重复说明。Particle Field Schemas define a composable data model distinct from volume grids.
  • Kernel / Radiance：Kernel 决定 particle 的形状和 falloff，Radiance 决定可见外观。若绑定了材质，官方说明 material description 应优先于本地 radiance，这一点对渲染调试很关键。Kernel and Radiance affect particle field influence and appearance, with material binding precedence.

Particle Field schema 主线 / Particle Field Schema Path

Particle Field Schemas 描述一种动态但结构化的数据模型。所有 particle fields 都基于 concrete base type ParticleField，再用 applied schema 说明必要数据如何提供。这样设计类似 C++ 中抽象基类与派生实现的组合：既允许未来研究继续演进，又能用 base applied schema 做结构验证。Particle fields use a concrete ParticleField base type plus applied schemas for required data categories.

ParticleField Base Schema 要求所有 particle field 提供 position、kernel、radiance 三类基础属性，并可提供 orientation、scale、opacity 等可选属性。position 定义每个 particle 在 local space 中的位置；kernel 定义每个 particle 实例化的形状和 falloff；radiance 定义外观。ParticleField requires position, kernel, and radiance, with optional orientation, scale, and opacity.

Base Applied Schema 中的 ParticleFieldPositionBaseAPI、ParticleFieldKernelBaseAPI、ParticleFieldRadianceBaseAPI 用于验证具体 applied schema 是否补齐了概念结构。Attribute Providing Schema 则列出 ParticleFieldPositionAttributeAPI、ParticleFieldOrientationAttributeAPI、ParticleFieldScaleAttributeAPI、ParticleFieldOpacityAttributeAPI、ParticleFieldSphericalHarmonicsAttributeAPI 等直接用 plain attributes 提供数据的 schema。Base applied schemas validate categories; AttributeAPI schemas provide data directly as attributes.

Kernel Schema 负责定义每个 particle 的 shape 和 falloff function；shape 可由 schema 描述和本地属性完整定义，falloff 在渲染时决定 kernel 影响范围。Radiance Schema 初始提供 ParticleFieldSphericalHarmonicsAttributeAPI，用 spherical harmonic coefficients 表达传统 3D Gaussian splats 的外观。官方也说明这是活跃研究方向，未来可能有外部 material description；如果 ParticleField 位置绑定了材质，应优先使用材质而不是本地 radiance。Kernel schema defines shape and falloff; radiance schema defines particle appearance and may be overridden by material binding.

验证 particle field 时，不应只检查点数。position 决定粒子位置，kernel 决定局部影响区域，scale/orientation/opacity 影响每个 kernel 的变形和权重，radiance 或 material 决定外观。一个 3D Gaussian splat 看起来错误，可能来自 position 坐标系、kernel 尺寸、opacity 权重、SH 系数、材质覆盖或 renderer 支持度，而不是单一 schema 字段。Particle field validation must check position, kernel, scale, orientation, opacity, radiance, material binding, and backend support.

由于官方把 particle fields 称为快速演进的研究领域，中文页必须诚实说明：UsdVol 提供的是可组合 schema 框架和生产可用布局的演进方向，不保证每种新研究表示都已经有固定 schema。管线集成时应记录使用了哪些 applied schema、哪些属性是必需的、哪些属性可以被未来替代表示取代，并为 renderer 或转换工具建立兼容性矩阵。Because particle fields are evolving, pipelines should record applied schemas, required attributes, and consumer compatibility.

对于 3D Gaussian splats，常见问题包括坐标系不一致、scale 单位不一致、opacity 权重解释不同、SH 系数阶数不一致、材质绑定覆盖本地 radiance，以及 renderer 对 kernel 的支持不完整。UsdVol 的目标不是解决每个研究实现的差异，而是让这些数据类别在 USD 中有可组合、可检查、可演进的表达位置。3D Gaussian splat pipelines need checks for coordinates, scale, opacity, SH coefficients, material overrides, and kernel support.

和传统 Volume 不同，ParticleField 的数据不一定来自单个外部 field asset；它可能由多个属性 API 共同提供，也可能未来由 MLP 或其他表示提供。中文读者要先确认当前具体 schema 组合，再判断数据是否完整。若缺少 position、kernel 或 radiance 中任何核心类别，后端即使能加载 prim，也可能无法产生可验证的外观。ParticleField data may be provided by multiple applied schemas rather than one external field asset.

边界、误读点与调试路径 / Boundaries, Misreads, and Debugging

  • 不要把 UsdVol 当成体积渲染器或体积模拟器。它提供 schema 和 asset references；采样、着色、体积 marching、文件加载和缓存由 Hydra、renderer 或专用消费者完成。UsdVol is a schema layer, not a volume renderer or simulator.
  • 不要把 field:density 和 fieldName 混为一谈。前者是 Volume 暴露给消费者的命名关系，后者是外部资产内部的 grid 选择。field:density is a consumer-facing relationship; fieldName selects the asset's internal grid.
  • 不要以为 field prim 必须是 Volume 子节点；这是常用组织方式，不是语法强制。但如果分离组织，必须认真检查 transform、引用路径和关系目标。Field prims are usually children of a Volume but are not required to be.
  • 不要把 particle field 固定理解成某一种 Gaussian splat 文件格式。官方设计目标是可组合、可扩展，并允许未来以不同 schema 表示 position、kernel、radiance 或外观。Particle fields are composable and extensible, not a single fixed file format.
  • 不要把 Volume 的 material binding、ParticleField 的 radiance 和 renderer 的 volume shader 混成一层。USD schema 提供连接点；最终可视外观仍要看 material、radiance、renderer 支持和后端解释。Material binding, local radiance, and renderer volume shading are separate layers.

建议调试顺序是：先确认 Volume 或 ParticleField prim 是否按预期组合出来；再确认 field relationships 是否指向存在的 FieldBase/FieldAsset prim；检查 filePath、fieldName、fieldIndex、timeSamples 和资产解析；随后检查 field prim transform 与外部资产内变换；最后用 UsdRender、Hd、Hdx 或 renderer 日志确认消费者是否识别并渲染了体积数据。Debug from composed prims and relationships through file paths, transforms, and renderer or Hydra consumption.

如果体积看不见，问题可能在 extent、field relationship、VDB grid name、asset path、material/radiance、purpose/visibility、renderer volume support 或 volume step settings；如果动画不对，优先检查 filePath.timeSamples 和帧号映射；如果 particle field 外观不对，检查 position/kernel/radiance applied schemas 是否完整以及是否被 material binding 覆盖。Visibility, animation, and particle appearance failures have different UsdVol checkpoints.

Particle Field Schemas Composable Schema ParticleField Base Schema Base Applied Schema Attribute Providing Schema Kernel Schema Radiance Schema Overview UsdVol provides schemas for representing volumetric geometry. UsdVolVolume can be used to represent continuous volumes (smoke, fire, etc), and can load spatial field definitions directly from OpenVDB and Field3d assets; it can also be extended to support other volume formats. The fields comprising a volume prim are represented as distinct prims whose schema types derive from UsdVolVolumeFieldBase , to distinguish them from particle fields. UsdVolParticleField can be used to represent a family of related lightfield techniques that define their spatial presence through a set of 3d particles. This schema was introduced to support 3d Gaussian splats, but is designed to be extensible and support things like Nerfs as well as future evolutions of this field of technology.
