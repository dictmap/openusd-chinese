中文主阅读路径

Js 模块的官方标题是 Js: JSON I/O，它的职责是为 C++ 提供 JSON 数据的 parsing 和 writing 方法，并在 C++ 侧处理 arbitrary recursive container structures 与 JSON standard 描述的值之间的转换。这里的 Js 不应按名称误读成 JavaScript runtime、JavaScript binding 或 Python binding；官方 Overview 明确说 No Python bindings are provided by this library。Js provides methods for parsing and writing JSON data from C++.

这页的核心不是列出很多类，而是说明一个边界清晰的基础库：它把 JSON parsing/serialization 的实现细节封装起来，让客户端可以依赖稳定、一致的 OpenUSD C++ 接口，而不是直接绑定某个具体 JSON parser 的细节。官方文字说 it abstracts away the details of how the parsing and serialization are performed, in order to allow the implementation to change without affecting clients；中文理解就是，调用方应该面向 js/json.h 入口和该库的值抽象编程，而不是面向内部解析器实现编程。The library abstracts parsing and serialization details so implementation can change without affecting clients.

如果在 Python 中处理 JSON，官方建议使用 Python standard library json module。这个说明非常重要：它把 C++ JSON I/O 和 Python JSON 工作流分开，避免读者寻找不存在的 Js Python bindings。也就是说，本页服务的是 OpenUSD C++ 代码中的 JSON 数据读写、配置文件解析、插件元数据或工具输入输出，而不是 Python 端的 json.loads/json.dumps 文档。Use the Python standard library 'json' module when working with json data in Python.

值抽象和递归容器结构

JSON standard 描述的值通常包括 object、array、string、number、boolean 和 null。官方页没有在 front page 展开具体类型名，但强调 Js 提供的是 values abstraction。这意味着 C++ 调用方可以把 JSON 文档看成递归容器结构：object 映射键值集合，array 映射有序序列，叶子值再承载字符串、数字、布尔或空值。Js 的职责是让这些结构在 C++ 内部和 JSON 文本之间来回转换。Js abstracts values described by the JSON standard and converts recursive container structures.

读者不应把 Js 理解成通用动态语言运行时，也不应期待它承载 USD 场景语义。JSON 只是一种数据交换格式；如果 JSON 内容描述的是 plugin 配置、工具选项、测试输入或外部数据，Js 负责读写文本和值结构，而不是替代 Sdf、Usd、Pcp 或 schema 层去解释场景 composition。场景语义仍由相应 USD 模块负责。Js handles JSON data representation, not USD scene composition semantics.

这种抽象还有一个工程好处：当内部 JSON parser 或 serializer 实现需要调整时，客户端不应该被迫改代码。官方说 implementation can change without affecting clients，中文理解就是模块边界保护调用方。调用方依赖 js/json.h 提供的一致接口，OpenUSD 内部则可以在保持行为契约的前提下优化 parsing、serialization、错误处理或编码细节。The abstraction lets implementation details change while clients continue using the stable interface.

C++ 入口、Python 边界和常见误读

官方唯一明确列出的入口是 js/json.h。因此，C++ 调用方查找 JSON I/O 时应从这个头文件和相关 Doxygen 项开始，而不是从 JavaScript、Node.js、浏览器 API 或 Python 模块找入口。文件名中的 js 是 OpenUSD 的模块名，不能翻译，也不能把它改写成 JSON 或 JavaScript。The top-level entrypoints are in js/json.h.

官方的 Python 边界很清楚：No Python bindings are provided by this library。如果 Python 代码需要处理 JSON，应使用 Python standard library json module。这里不是说 OpenUSD 不能在 Python 生态中出现 JSON，而是说这个 C++ Js library 本身没有提供 Python bindings。调试时若 Python import 或绑定查找失败，不应把它当成当前页缺失翻译或链接错误，而应按官方边界改用 Python 标准库。No Python bindings are provided by this library; use Python's standard json module in Python.

另一个常见误读是把 Js 当成 JSON schema validator。官方页只说 parsing/writing 和 value abstraction，没有承诺验证业务 schema、自动校验字段约束、生成类型安全对象或解释 USD schema。若某个 JSON 文件有业务规则，例如插件配置字段、资产索引字段或工具参数字段，字段合法性仍应由调用方或更高层模块判断，Js 只提供基础 I/O 和值结构。The Js page does not describe a JSON schema validator; domain validation belongs to callers or higher-level modules.

相邻模块关系和调试路径

与 Plug 的关系最直接：插件发现和 plugInfo.json 这类配置数据经常使用 JSON 表达。Js 可以作为 C++ 侧读取或写出 JSON 数据的基础库，而 Plug 负责解释这些数据如何参与插件注册。调试插件元数据时，要把“JSON 是否能被 parse”与“插件 registry 是否接受这些字段”分开看。Plug metadata often uses JSON; Js can parse/write JSON while Plug interprets plugin metadata.

与 Tf 和 Vt 的关系是基础设施层面的：Tf 提供 token、诊断和通用工具，Vt 提供值承载和数组语义，Js 则处理 JSON 值结构和文本 I/O。它们都属于较底层的支持模块，但职责并不重叠。看到 JSON 字符串、token 名或动态值时，需要先判断问题发生在文本解析、值承载还是上层语义解释。Tf, Vt, and Js are adjacent infrastructure modules with separate responsibilities.

与 Sdf 的关系也要区分。Sdf 是 Scene Description Foundations，关注 USD 场景描述、layer、path、spec 等基础数据模型；Js 只是 JSON I/O。即使某个工具把 JSON 用作输入配置，最终写入 USD scene description 的规则也不由 Js 决定。排查数据流时可以先确认 JSON 文本是否有效，再确认解析结果是否符合调用方预期，最后才检查 Sdf/Usd 层写入是否正确。Js is JSON I/O; Sdf is scene description infrastructure.

推荐调试顺序是：先用最小 JSON 输入确认 parser 是否能读；再检查 C++ 容器和值结构是否和预期一致；然后确认 writer 输出是否能被再次读取；最后再进入调用方模块，例如 Plug、工具配置或场景导入逻辑。这样能把 encoding、syntax、container conversion、domain validation 和 USD authoring 错误分开定位。A practical debugging path separates JSON syntax, container conversion, domain validation, and USD authoring.

使用建议和非目标范围

在 C++ 中使用 Js 时，应把它当成 OpenUSD 风格一致的 JSON I/O 门面。它适合处理配置、工具输入输出、测试数据和需要和 JSON 文本互转的递归容器结构。页面没有承诺高层业务对象映射，也没有承诺 Python 或 JavaScript 绑定；如果业务需要更强的类型约束，应在调用方建立清晰的数据结构和校验逻辑。Use Js as a C++ JSON I/O facade with a consistent OpenUSD-style interface.

非目标范围必须说清：本页不解释 JSON standard 的完整语法规范，不替代 Python json 文档，不定义 USD schema，不解释 layer composition，也不说明插件系统如何消费每个 JSON 字段。它只回答“OpenUSD C++ 侧在哪里做 JSON parse/write、为什么要隐藏实现细节、与 Python 和上层模块的边界在哪里”。Out of scope: full JSON specification, Python json docs, USD schema definition, layer composition, and plugin-domain validation.

因此，本地中文页保留 Js、JSON I/O、C++、js/json.h、No Python bindings 和 Python standard library 'json' 等官方短语，中文负责解释它们在工程中的含义。读者可以直接对照官方源页、Doxygen 和本地相邻模块页面，而不会把模块误用到 JavaScript 或 Python 绑定路径上。The page preserves official terms and explains their engineering meaning in Chinese.

工程边界和错误定位

当 JSON 读取失败时，第一层要判断的是文本本身是否符合 JSON syntax，例如括号、逗号、字符串引号和转义是否正确。第二层才是 Js 把文本转换成 C++ 递归容器和值结构是否符合预期。第三层是调用方是否按照自己的业务规则解释这些值。把这三层分开，能避免把业务字段缺失误判成 parser 错误，也能避免把文本语法错误误判成 USD 场景写入错误。Separate JSON syntax, C++ value conversion, and caller domain validation when debugging failures.

写出 JSON 时也一样。Js 可以帮助调用方把内部容器和值结构序列化为 JSON 文本，但它不能替调用方决定字段命名、版本兼容、缺省值策略或业务 schema。若输出文件被另一个工具拒绝，应先确认 writer 生成的 JSON 是否语法有效，再确认字段语义是否符合下游工具契约，而不是直接怀疑 Js 模块改变了 USD 数据模型。When writing JSON, distinguish serialization validity from downstream domain schema acceptance.

官方提到 repository coding standards 和 consistent interface for developers，说明 Js 的价值也包括风格一致性。OpenUSD 内部不同模块如果都各自直接使用不同 JSON 库，错误处理、类型转换和接口风格会分散；统一经过 Js 的 C++ 门面，可以让代码更容易维护，也让未来替换实现时不破坏调用方。The library provides a consistent interface aligned with repository coding standards.

与配置文件相关的场景尤其适合这一路径。例如插件描述、工具选项、测试 fixture 或轻量资产索引可能用 JSON 表达。Js 可以负责把这些 JSON 文本读成可操作的 C++ 值结构；随后 Plug、工具层或业务模块再解释每个字段。这样解析层和语义层分离，问题报告也能更精确。Configuration-like JSON data can be parsed by Js and interpreted by Plug or higher-level tools.

如果同一份 JSON 在 C++ 和 Python 两侧都要处理，推荐明确分工：C++ 代码使用 js/json.h 对应的 Js 接口；Python 脚本使用标准库 json。不要假设两侧共享同一个 binding，也不要因为模块名是 Js 就去寻找 JavaScript runtime 行为。跨语言一致性应通过文件格式和测试用例保证，而不是通过不存在的绑定层保证。For cross-language workflows, use Js in C++ and Python's standard json module in Python.

本页晋级的判断也遵循这个边界：它不是给出 JSON API 的逐函数手册，而是让读者理解模块入口页为什么存在、它保护了哪些实现细节、调用方该从哪里进入、哪些语言绑定不存在，以及排查问题时应按哪几个层次拆分。这样短源页也能形成可独立阅读的中文主路径。This promoted page explains the module boundary rather than pretending to be a full function reference.

最小验证用例应覆盖读取和写回两个方向：准备一个包含 object、array、string、number、boolean 和 null 的小 JSON 文本，确认 C++ 侧能读成预期结构；再把结构写回 JSON，确认下游工具或同一 reader 可以再次读取。这样可以验证基础 I/O，而不把测试扩大成完整业务流程。A minimal validation case should test both reading and writing representative JSON values.

调用方责任也要清楚：如果 JSON 字段代表路径、token、kind、插件名称或其他 OpenUSD 概念，字段含义必须由相应模块或业务代码解释。Js 只保证 JSON 层的读写抽象和一致接口；它不会判断一个 path 是否存在、一个 token 是否注册、或一个插件描述是否满足 pipeline 规则。Callers remain responsible for interpreting paths, tokens, kinds, plugin names, and other OpenUSD concepts.

因此，本页的阅读目标是让开发者先正确定位模块边界，再进入具体函数或头文件细节。只要能区分 JSON I/O、C++ 接口、Python 标准库路径和上层业务校验，就能避免大多数由名称误读导致的排查偏差；后续查看具体函数时，也应以 js/json.h 和对应 Doxygen 条目为准。这样短页仍能成为可靠的本地中文入口，而不是只有标题翻译，也能减少错误跳转和重复排查，利于后续维护，中文路径完整。The reading goal is to locate the module boundary before diving into detailed APIs.
