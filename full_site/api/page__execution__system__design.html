中文主阅读路径 / Chinese Main Reading Path

本页解释 OpenExec execution system 的设计，而不是再演示某个具体 schema computation。它把 OpenExec 描述为一个通用 computation engine：先把 authored scene description 和 registered computation definitions 编译成 execution network，再围绕客户端请求进行 scheduling，最后按 schedule 做 evaluation。这个设计来自 Pixar 在 rigging、animation、layout、crowds、shading、lighting 和部分 simulation 工作流中的生产经验，核心目标是让可交互场景里的大量计算既可复用又可并行。This page explains the execution system design rather than one specific computation.

阅读本页应顺着官方结构展开：先理解 Scene Primitives 中的 attribute、prim、computations；再看 Computations 中 builtin/plugin computation 和 stateless callback 的边界；然后按 Execution Phases 区分 Compilation、Scheduling、Evaluation 三个阶段；最后进入 Execution Engine，理解 Network、Scheduler、Data Managers、Executors 的分工。这样能把 Tutorial 1 的 client request 和 Tutorial 2 的 schema computation authoring 放回完整系统上下文。The recommended order follows scene primitives, computations, phases, and engine components.

本页最重要的边界是“结构成本”和“求值成本”分离。Compilation 负责把高表达力的 USD 场景和 plugin/builtin computations 映射成更轻量的 vectorized data flow network；Scheduling 负责针对一组 requested outputs 构造可重复使用的 execution schedule；Evaluation 负责最频繁地运行节点、命中 cache、处理 invalidation 后重新计算。调试时不要把 authored value 变化、network topology 变化、request 输出集合变化、cache dirty state 混为一个问题。Compilation, scheduling, and evaluation separate structural cost from frequent value computation.

官方 section 顺序 / Official Section Order

  • OpenExec 开场：定义系统定位，强调 execution network 是 vectorized data flow network，许多可唯一识别元素可以沿单条连接流动，因此天然支持并行机会。The introduction frames OpenExec as a vectorized data-flow computation engine.
  • Prelude: Object Model：把 interactive scene 抽象为可持有值的 attribute、拥有属性的高层容器 prims，以及消费 scene data 并赋予对象 executional behaviors 的 computations。Prelude: Object Model defines attributes, prims, and computations.
  • Computations：说明 OpenExec 同时有 Exec_BuiltinComputations::computeValue 这样的 built-in computations，也允许 schema 发布 plugin computations；callback 是 stateless 的，所有输入都必须在 computation definition 中显式供应。Computations distinguish builtin and plugin computations and stateless callbacks.
  • Phases of Execution：把执行拆成三段，按发生频率从低到高、单次成本从高到低分别是 Compilation、Scheduling、Evaluation。Phases of Execution separates compilation, scheduling, and evaluation.
  • Compilation：把 scene objects 转换成 optimized data structures 和 data flow network；拓扑变化会触发增量更新，普通 authored value change 不触发 compilation，只需要 invalidation cached results。Compilation builds and incrementally updates the network topology.
  • Scheduling：基于 compiled network 和客户端 execution request 生成 schedule；例如 viewport 会请求 scene 中所有 visible objects 的输出。schedule 是加速结构，避免每次 evaluation 都重新做 dependency analysis。Scheduling creates a reusable schedule for requested outputs.
  • Evaluation：按 schedule 运行 network nodes，调用相关 callbacks，遇到 cache hit 就截断依赖遍历；默认 parallel executor 可以并行 evaluate nodes，只在 input dependencies 上同步。Evaluation is the frequent pull-based callback execution phase.
  • Execution Engine：说明 engine 由 network、scheduler、data managers、executors 组成，分别负责依赖图、调度、computed data/validity masks 存储和 inner-loop evaluation。Execution Engine decomposes network, scheduler, data managers, and executors.

三阶段执行模型 / Three Execution Phases

Compilation。这是最昂贵但相对低频的阶段。它把复杂的 scene description、plugin computations、builtin computations 和 authoring-friendly 表达映射到一个简化但高效的 data flow network。这个阶段的产物是 nodes 和 connections，描述 computation 结构和依赖。拓扑变化会让 network 局部重编译；但 authored value change 这类非结构变化通常只 invalidates cached results。Compilation maps expressive scene description to a compact data-flow network.

Scheduling。这是中等频率阶段。客户端提交一组 desired outputs，例如 viewport 请求 visible objects 的 posed point positions。scheduler 根据 compiled network 找出需要运行的 nodes，并生成一个 schedule。schedule 不只是节点列表，它还约束内存访问：保证一次 evaluation round 中 data-flow nodes 不会被重复访问，并尽量让数据连续访问，从而把 expensive dependency analysis 摊销掉。Scheduling amortizes dependency analysis for a specific output request set.

Evaluation。这是最高频阶段。它根据 schedule 执行相关 nodes 并生产 requested outputs。evaluation 是 pull-based：从 requested output 的 leaves 开始沿 dependency 向 root 遍历，遇到 cache hit 就截断，然后在回退过程中从 root 和已满足 input dependencies 的 nodes 开始调用 callbacks。默认的 parallel executor 会并行求值，只在 input dependencies 上同步；需要禁用多线程时可设置 os.environ['VDF_ENABLE_PARALLEL_EVALUATION_ENGINE'] = '0'。Evaluation runs scheduled nodes, uses cache hits, and can execute in parallel.

os.environ['VDF_ENABLE_PARALLEL_EVALUATION_ENGINE'] = '0'

这三个阶段对应三类不同 bug：如果新 schema computation 完全不可见，优先查 compilation 和 plugin registration；如果某组客户端输出变化后性能突然下降，优先查 scheduling 是否被频繁失效；如果值陈旧或只在某些输入变化后错误，优先查 evaluation、cache hit、invalidation 和 data manager validity masks。Different failures point to compilation, scheduling, or evaluation boundaries.

Execution Engine 组件 / Execution Engine Components

  • Network。execution network 来自 authored scene objects，用 nodes 和 connections 静态表示 computations 以及它们的依赖。EfLeafNode 是无输出的 terminal node，用来接收 invalidation，也就是追踪和广播 attribute value 变化的机制。Network statically represents computations and dependencies.
  • Scheduler。scheduler 生产 evaluation 使用的 opaque schedule。客户端不能直接检查 schedule，而是在发出 request 后得到一个由 execution system 后续消费的对象。schedule 级优化会分析需要运行的 nodes，并为 executor 提供 task dependencies 等可利用信息。Scheduler produces opaque schedules and dependency information for executors.
  • Data Managers。data managers 存储 network outputs 对应的 computed data。简单理解可以把它看作 node output 到 computed value 的 map，同时还存储 validity masks，用来表示哪些 output 是 dirty、哪些可以命中 cache。Data managers store computed values and validity masks.
  • Executors。executor 是 evaluation 的 inner loop，在 compiled network 和 schedule 已给定时查询、计算并存储结果。executors 可以构成 hierarchy，child executor 可以读取甚至写入 parent 的 computed results，以减少重复计算；不同 executor specialization 也让系统能实验不同求值策略。Executors run the inner loop and may form hierarchies.

这四个组件共同服务一个目标：把丰富但昂贵的 USD authoring 表达转换成可重复、高并发、可缓存的 runtime computation。Network 关心依赖结构，Scheduler 关心“这次请求要跑哪些节点”，Data Managers 关心“哪些结果已缓存或已失效”，Executors 关心“怎样高效执行当前 schedule”。阅读时把它们分清，能避免把所有性能或正确性问题都归咎于 callback。The components separate dependency structure, requested work, cached data, and execution mechanics.

设计验收清单 / Design Checklist

官方开场把 OpenExec 定位为 general-purpose computation engine，这句话不能读成“只服务某个单一模块的工具”。它的目标是让 authored scene description、schema-defined behavior、client request 和 runtime evaluation 可以分层演进。对作者来说，schema 和 plugin 仍然表达业务语义；对执行系统来说，这些语义会被转换为可分析、可调度、可缓存的 execution network。这样做的价值在于让复杂场景里的派生值、可见性、变换或几何结果不必每次从头解释 USD 层，而是按依赖关系精准更新。OpenExec is described as a general-purpose computation engine with explicit opportunities for parallelism.

第一条验收线是交互性能。Compilation 可以昂贵，因为它在结构变化时才发生；Evaluation 必须便宜，因为用户拖动属性、相机或时间码时会反复发生。若一个改动只改变 authored value，却导致大量 topology rebuild，说明 compilation 与 invalidation 边界没有守住。若一个 request 每次都重新做完整 dependency traversal，说明 schedule 没有被正确复用。若 callback 每次都绕过 cache 重新算全部输入，说明 data manager 的 validity masks 或 input dependency 声明需要回查。Fast interaction depends on keeping structural work out of frequent evaluation.

第二条验收线是缓存可靠性。OpenExec 的缓存不是“把所有结果永久存起来”，而是由 execution network、input dependencies、invalidation 和 validity masks 共同维护的临时计算状态。一个 callback 如果偷偷读取未声明输入，就会破坏这个模型：系统不知道哪个输入变化应该让结果 dirty，也无法判断 cache hit 是否安全。因此 Tutorial 2 中的 explicit input declarations 与本页的 Data Managers/Executors 是同一个设计链条的两端。Reliable cached computation depends on explicit dependencies and validity tracking.

第三条验收线是并行边界。官方强调 vectorized data flow network 能给 parallelism 留出机会，但并行不是任意 callback 自动安全。正确的心智模型是：Scheduler 根据 request 组织可执行 work，Executors 根据 dependency order 和 cache state 调度节点，callback 本身应尽量无状态并只依赖显式输入。若关闭 VDF_ENABLE_PARALLEL_EVALUATION_ENGINE 后问题消失，应优先检查 callback 是否有隐藏共享状态、输入声明是否完整、以及 executor hierarchy 是否错误共享了父子 computed results。Parallelism is enabled by the network and schedule, but callbacks still need safe dependency boundaries.

第四条验收线是点击顺序。读者不应从本页随机跳到无关 class reference；更自然的路径是从 OpenExec Overview 建立概念，到 Tutorial 1 学会 requesting values，到 Tutorial 2 学会 defining computations，再到本页理解系统为什么拆成 Compilation、Scheduling、Evaluation 和 Engine components。之后才继续进入 ExecUsd、Vdf、Exec 或 Trace 等实现/诊断入口。这个顺序与本地 reading-flow 导航一致，也避免把“如何使用 API”和“系统内部如何优化执行”混在同一层解释。The local click path follows Overview, Tutorial 1, Tutorial 2, and then System Design.

第五条验收线是调试分层。找不到 computation 时，先看 plugin discovery、schema registration 和 computation token；结果过期时，先看 input dependency 与 invalidation；性能抖动时，先看 request 是否改变 schedule、topology 是否频繁失效、cache hit 是否被过早破坏；多线程差异时，再看 executor、callback state 和同步边界。按这个顺序排查，才能把 OpenExec 的设计收益转化为实际工程判断。Debugging should stay layered: discovery, registration, invalidation, scheduling, cache, and executor behavior.

逐段双语理解 / Paragraph-Level Bilingual Coverage

开场段。OpenExec 的生产背景说明它不是单一 DCC 特性，而是用于 rigging、animation、layout、crowds、shading、lighting、simulation 等交互工作流的通用计算层。这里的 compilation 不是 C++ 编译，而是把 scene description 与 registered computation definitions 解释成 execution network。The opening defines OpenExec as a general-purpose production computation engine.

设计原则。官方用 guiding principles 说明系统设计要支持并行、缓存、增量更新和清晰的 authoring/runtime 分离。虽然原页列表很短，但它是后文三阶段模型的前提：昂贵结构分析要少做，频繁 value evaluation 要快做。The guiding principles motivate separation of expensive structural work from frequent evaluation.

Scene Primitives。attribute 是能持有值的对象，prim 是拥有 attributes 的高层容器；用户通过 prim 和 attribute 表达关系与数据流，computations 消费这些 scene objects 并赋予 executional behaviors。扩展 prims 和 computations 是客户端演进交互环境能力的主要机制。Scene primitives connect authored data with executional behaviors.

Computations。builtin computation 例如 computeValue 用于请求 attribute 的 computed value；plugin computations 由 schema 发布，定义支持的行为集合。stateless callback 的要求很硬：callback 所需输入必须作为 computation definition 的一部分显式提供，这让 execution network 可以优化、缓存和并行调度。Computations are stateless callbacks with explicit inputs.

Compilation。该阶段用较低频率承担高成本，把 authoring-friendly 的表达转换为 optimized network。它不是每次值变化都重做；authored value change 通常只触发 cached results 的 invalidation。这个边界对交互性能很关键，否则用户拖动一个属性也可能引发全网重建。Compilation is expensive and topology-oriented, not triggered by every value change.

Scheduling。schedule 针对 requested outputs 生成。不同 viewport、不同 selection 或不同 downstream request 可能需要不同 schedule；但一旦 schedule 建好，它能避免每次 evaluation 重复 traversal 和 dependency analysis。它也帮助 executor 组织内存访问和 task dependencies。Scheduling amortizes traversal and dependency analysis for a request.

Evaluation。evaluation 是最频繁阶段，适合 cache 和 invalidation 优化。它从 requested outputs 的 leaves 开始向 root 拉取依赖，cache hit 会截断遍历，再在回退过程中调用 callbacks。parallel executor 默认开启，这意味着 callback 设计必须尊重显式输入和同步边界。Evaluation is pull-based, cache-aware, and parallel by default.

Engine components。network、scheduler、data managers、executors 是系统职责拆分，而不是四个同义词。network 解释依赖，scheduler 组织工作，data managers 管理 computed data 和 validity masks，executors 负责高效运行 schedule。调试时应按组件边界定位问题。Engine components split dependency representation, scheduling, data storage, and execution.

边界、误读点与调试路径 / Boundaries and Debugging

  • 不要把 compilation 误解为源码编译；它是把 scene description 和 computation definitions 编译成 runtime data flow network。Compilation means network construction, not C++ compilation.
  • 不要把 authored value change 当成 topology change。前者通常只需要 invalidation cached results，后者才会影响 compiled network 结构。Value changes and topology changes have different costs.
  • 不要把 schedule 当成客户端可检查的公开列表。官方明确说 schedule 是 opaque object，由 execution system 后续消费。Schedules are opaque to clients.
  • 不要在 callback 中隐藏输入依赖。stateless callbacks 和显式 input definitions 是并行、缓存和 invalidation 的基础。Hidden callback dependencies undermine optimization.
  • 不要把 parallel evaluation 失败直接归因于 Work 或线程库；先检查 input dependencies、cache validity 和 callback 线程安全假设。Parallel evaluation issues should be traced through dependencies and cache validity first.

实用排错顺序可以按三阶段走：首先看 compilation，确认 schema/plugin computation 是否进入 network、拓扑是否因结构变化被正确增量更新；其次看 scheduling，确认 request 的 outputs 是否稳定，schedule 是否因为 topology change 被频繁 invalidated；最后看 evaluation，确认 dirty mask、cache hit、callback 输入和值类型是否正确。若设置 VDF_ENABLE_PARALLEL_EVALUATION_ENGINE=0 后问题消失，说明需要进一步检查 callback 的并行安全、输入同步和 executor 层面的数据共享，而不是简单关闭并行作为最终修复。A practical debug path follows compilation, scheduling, and evaluation boundaries.
