逐段双语理解 / Paragraph-Level Bilingual Coverage

官方说明把 `TfToken` 定义为 a handle for a registered string。这个 handle 不是普通 `std::string` 的薄封装，而是指向已注册字符串条目的轻量对象。它的核心价值在于：一旦字符串进入 token registry，比较、赋值和 hashing 可以按 token identity 进行，避免反复逐字符处理已知符号。

The official description defines `TfToken` as a handle for a registered string. This handle is not just a thin wrapper around `std::string`; it is a lightweight object referring to a registered string entry. Its core value is that once a string enters the token registry, comparison, assignment, and hashing can operate on token identity instead of repeatedly processing characters for known symbols.

`TfToken` 适合 bounded number of strings used as fixed symbols。这里的 bounded 很重要：token 用于 schema token、attribute name、metadata key、primvar name、renderer option name 这类固定或可控集合，而不是无限增长的用户输入文本。把任意临时字符串都注册成 token，会让 token table 长期增长并造成内存压力。

`TfToken` is intended for a bounded number of strings used as fixed symbols. The word bounded is important: tokens fit schema tokens, attribute names, metadata keys, primvar names, renderer option names, and similar fixed or controlled sets, not unbounded user text. Registering every temporary arbitrary string as a token can grow the token table for the lifetime of the process and create memory pressure.

constant time comparison 并不表示创建 token 永远是常数时间。首次从字符序列构造 token 时，系统仍需要查找或注册该字符串；优势体现在后续比较和 hash key 使用上。性能敏感路径应该复用已有 `TfToken`，避免在循环中频繁从 `std::string` 或 C 字符串创建新 token。

Constant-time comparison does not mean token creation is always constant time. When a token is first constructed from character data, the system still needs to find or register that string. The benefit appears in later comparisons and hash-key usage. Performance-sensitive code should reuse existing `TfToken` values instead of repeatedly constructing new tokens from `std::string` or C strings inside loops.

`GetString()`、`GetText()` 和 `data()` 都能读取 token 内容，但使用意图不同。`GetString()` 返回字符串值，方便与需要 `std::string` 的 API 对接；`GetText()` / `data()` 面向底层 C 字符串访问。调用方不应为了比较而反复转回字符串，因为那会绕开 `TfToken` 的 identity 比较优势。

`GetString()`, `GetText()`, and `data()` all expose token text, but they serve different call sites. `GetString()` returns a string value for APIs that need `std::string`; `GetText()` / `data()` provide lower-level C-string access. Callers should not convert tokens back to strings merely for comparison, because that bypasses the identity-comparison advantage of `TfToken`.

`Find()` 的阅读重点是“查询已存在 token”而不是“无条件注册新 token”。当代码只想知道一个字符串是否已经在 token registry 中存在时，`Find()` 可以避免无意扩大注册表。这个差别在解析外部数据、过滤可选字段、处理插件扩展名或调试未知 metadata key 时很实用。

The important reading point for `Find()` is lookup of an existing token, not unconditional registration of a new one. When code only needs to know whether a string is already present in the token registry, `Find()` can avoid accidentally expanding the registry. This distinction matters when parsing external data, filtering optional fields, handling plugin extension names, or debugging unknown metadata keys.

`Hash()`、`HashSet` 和 `Set` 体现了 token 作为 key 的用途。`TfToken` 常出现在 unordered containers、token sets 和 name dispatch 表中。与 `std::string` key 相比，它降低了重复 hash 字符串和逐字符比较的成本，但前提仍是 key 集合稳定并且 token 复用充分。

`Hash()`, `HashSet`, and `Set` show how tokens are used as keys. `TfToken` commonly appears in unordered containers, token sets, and name-dispatch tables. Compared with `std::string` keys, it reduces repeated string hashing and character comparison, but this relies on a stable key set and sufficient token reuse.

`IsEmpty()` 和 empty token 要按 API 约定理解。empty token 通常用于表示没有名称、默认标识、缺省路径片段或无效轻量状态，但它不自动等同于所有业务语义上的“未设置”。调用方必须阅读具体 API 文档，区分 empty token、missing attribute、fallback value 和空字符串 token。

`IsEmpty()` and empty-token semantics must be interpreted according to each API contract. An empty token is often used for no name, default identifier, missing path segment, or a lightweight invalid state, but it is not automatically the same as every business-level unset value. Callers must distinguish empty token, missing attribute, fallback value, and an empty-string token in the specific API documentation.

`_ImmortalTag` 和多组构造函数提示 `TfToken` 与字符串生命周期优化有关。一般用户不应把它理解成普通公开业务枚举；它更多服务于底层 token 注册和静态 token 初始化场景。维护代码时，应优先使用官方 token 声明、现成 `TfToken` 常量或明确的构造入口，不要绕过生命周期约定。

`_ImmortalTag` and the constructor overloads indicate that `TfToken` participates in string-lifetime optimization. Most users should not treat it as an ordinary public business enum; it mainly supports lower-level token registration and static-token initialization cases. Maintenance code should prefer official token declarations, existing `TfToken` constants, or explicit construction APIs instead of bypassing lifetime conventions.

在 OpenUSD 中，`TfToken` 是很多 schema 和 composition API 的共同语言。属性名、metadata key、applied API schema 名、primvar 名和 renderer setting 名经常以 token 传递。理解 `TfToken` 可以帮助区分“符号名字相同”与“字符串内容相同”的性能语义，也能解释为什么大量 API 不直接使用 `std::string`。

In OpenUSD, `TfToken` is shared vocabulary across many schema and composition APIs. Attribute names, metadata keys, applied API schema names, primvar names, and renderer setting names are often passed as tokens. Understanding `TfToken` helps separate the performance semantics of symbol identity from raw string content and explains why many APIs do not use `std::string` directly.

调试 token 相关问题时，常见边界有三类：把不受控输入注册为 token 导致表膨胀；在热路径中反复构造 token 导致隐藏开销；把 empty token、空字符串和缺省值混为一谈导致逻辑分支错误。本页的核心阅读目标就是把这些边界与 `TfToken` 的高效比较能力区分开。

When debugging token-related problems, three boundaries appear often: registering uncontrolled input and growing the table, repeatedly constructing tokens on hot paths and creating hidden cost, and conflating empty token, empty string, and fallback value in logic branches. The main reading goal of this page is to separate those boundaries from the efficient-comparison capability of `TfToken`.
