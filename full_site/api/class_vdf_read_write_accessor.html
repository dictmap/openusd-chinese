逐段双语理解 / Paragraph-Level Bilingual Coverage

`VdfReadWriteAccessor< T >` 解决的问题是：在 Vdf computation 或 executor 相关代码中，如何以静态类型 `T` 对 output data 做随机读写访问。官方摘录说它 “allows for random access to output data”；这里的 output data 是 Vdf 执行图中的求值数据，不是 USD authored scene data，也不是拥有长期生命周期的普通 STL container。

`VdfReadWriteAccessor< T >` allows random access to output data. The output data belongs to Vdf evaluation/storage infrastructure, not to authored scene data.

这个类的核心边界是 non-owning accessor。它围绕 `VdfVector` 或 output storage 暴露元素级访问，但不拥有这些数据，也不定义数据的业务语义。拥有输出、决定何时产生输出、以及判断输出是否被请求的是 `VdfContext`、`VdfOutput`、executor 和 node computation 共同参与的执行路径；accessor 只提供读写入口。

The class is an accessor rather than an owning container. Ownership, scheduling, and semantic meaning are handled elsewhere in the Vdf graph.

模板参数 `T` 是访问元素的静态类型约束。它帮助 C++ 代码用 `T` 的形式读写元素，但并不会把图中的 output 改造成另一种业务类型。若 `T` 与实际 output data 的类型不匹配，问题通常来自 output registration、template instantiation、executor wiring 或调用点假设，而不是 accessor 本身改变了数据。

Template parameter `T` is the static element type used by the accessor. It must match the output data representation expected by the graph.

`operator[]()` 是最直接的读写入口，并提供 const 与 non-const 形式。它体现 random access 语义：调用方按索引访问当前 accessor 暴露的元素。使用前应先用 `GetSize()` 或 `IsEmpty()` 检查范围，不能假定 output 一定有某个索引，也不能把 output request 的范围和实际 storage size 混为一谈。

`operator[]()` provides indexed element access. Callers should check size and emptiness before indexing.

`GetSize()` 描述当前 accessor 可访问的元素数量。这个数量要结合 `VdfMask`、`VdfIterator`、`VdfReadWriteIterator` 和本次 evaluation request 理解：它是当前输出数据视图的大小，不一定等于某个 authored array 的长度，也不一定等于下游最终消费的元素数量。

`GetSize()` reports the element count exposed through this accessor. It should be interpreted with masks, iterators, and the active evaluation request.

`IsEmpty()` 是快速判空入口，但空 accessor 的原因可能不同：output 没有数据、本次 request 没覆盖相关元素、上游 computation 没产生值、dirty/invalidation 后尚未重新求值，或 graph wiring 没把数据连到当前输出。排查时应同时看 `VdfContext`、`VdfOutput`、request mask 和 executor 状态。

An empty accessor can result from missing data, request masks, invalidation, or graph wiring. It is not automatically a scene-authoring issue.

读写 accessor 与 `VdfContext` 的关系是分层的：`VdfContext` 是 callback 面对的执行期入口，用于判断 requested output 和写回结果；`VdfReadWriteAccessor< T >` 则是更靠近 output data storage 的 typed random-access view。callback 可以通过 context 间接获得或使用这样的访问机制，但语义上不要把 context 和 accessor 混成同一个对象。

`VdfContext` is the callback-facing execution context, while `VdfReadWriteAccessor< T >` is a typed random-access view over output data.

与 `VdfReadWriteIterator` 的差别在访问模式。iterator 适合顺序遍历或按 mask 遍历；accessor 更适合已知索引的 random access。选择哪一种取决于 computation 的数据访问模式、mask 稀疏性、缓存局部性和是否需要写回元素。

Iterators and accessors serve different access patterns. Iterators are traversal-oriented, while this accessor is random-index oriented.

常见误读包括：把 accessor 保存到 evaluation 之外长期使用，把 `operator[]()` 当成自动边界检查，把 `GetSize()` 当成 authored array size，把空 accessor 当成固定错误，把 `T` 当成可以强制重解释 output 语义的转换器。正确读法是：它是一次 Vdf output data 访问路径上的 typed、non-owning、request-sensitive view。

Common misreads include treating the accessor as persistent storage, assuming unchecked indexing is safe, or using `T` as a semantic conversion mechanism.
