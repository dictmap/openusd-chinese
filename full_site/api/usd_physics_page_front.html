逐段双语理解 / Paragraph-Level Bilingual Coverage

UsdPhysics 是 USD 中描述物理仿真数据的 schema 模块入口。官方开头说它定义了 physics-related prim and property schemas，这句话的重点是“描述”和“交换”：页面讨论的是 USD layer 里如何表达物理场景，而不是某个求解器如何积分、碰撞检测或并行执行。UsdPhysics defines physics-related prim and property schemas that form a physics simulation representation.

Purpose and Scope 说明这个 schema 最初面向 real-time physics 的共同需求，覆盖游戏、Web 3D、机械工程、建筑、人工智能和机器人仿真等场景。它是 rigid body physics 的基线扩展：先提供刚体、碰撞、质量、材质、关节、场景参数等最小公共概念，未来能力可以继续扩展。Purpose and Scope positions UsdPhysics as a baseline rigid-body physics representation for interactive and simulation workflows.

Overall Design Concerns 和 Rigid Body Simulation Primer 提醒读者：刚体仿真器的输入通常是 rigid bodies、constraints 和某个时间点的状态；求解器根据这些输入计算下一时刻状态。USD 页面不实现这个计算，而是把求解器需要的对象、关系和属性编码成可交换的 scene description。Rigid body simulators consume bodies, constraints, and state; USD stores the scene description that feeds such systems.

USD Implementation 是本页核心。这里要把物理 schema 和 USD composition 分开理解：物理场景仍然在 prim hierarchy 中 author，属性仍然受 layer strength、默认值、time samples 和实例化规则影响。UsdPhysics 不绕开 USD 语义，而是把 physics scene、rigid body、collision shape、material、joint、articulation 等概念放进 USD 的 schema/API 体系。USD Implementation explains how physics concepts are authored within ordinary USD composition and prim hierarchy semantics.

Physics Scenes 对应场景级设置，读者应把它理解为物理仿真上下文，而不是几何世界的根节点。一个 USD stage 可以包含用于解析的 physics scene 信息；求解器或导入器会读取这些 schema，决定重力、时间步或 simulation owner 等行为。它和 UsdGeom 的坐标、单位、transform 共同影响最终仿真解释。Physics Scenes provide scene-level physics context rather than replacing the geometric stage root.

Rigid Bodies、Interaction with the USD hierarchy、Kinematic Bodies 和 Animation of Attributes 是最容易误读的区域。rigid body 表示一个会参与刚体仿真的对象；它的 transform、父子层级、是否 kinematic、属性是否随时间变化，都需要和 USD hierarchy 一起解释。不要把 kinematic body 当作普通动画几何，也不要把所有 time-sampled 属性都理解为求解器可任意覆盖的结果。Rigid-body state must be interpreted together with USD hierarchy, kinematic state, and authored animation.

Body Mass Properties、Collision Shapes、Turning Meshes into Shapes、Physics Materials 和 Plane Shapes 组成了刚体可碰撞、可求解的输入。质量属性决定惯性相关信息；碰撞 shape 决定求解器看到的几何近似；mesh 是否能作为 shape 要看官方约束；physics material 负责摩擦、弹性等物理表面参数，而不是渲染材质。Mass properties, collision shapes, and physics materials form the physical input, separate from rendering materials.

Collision Filtering 和 Pairwise Filtering 解释哪些对象应该碰撞、哪些对象应该被过滤。排查“明明穿过了却没有碰撞”时，不应只看几何是否重叠，还要检查 collision group、pairwise filtering、shape 是否有效、层级上的 API 是否被正确应用，以及实例或引用中的 opinion 是否覆盖了预期设置。Collision filtering controls which collision pairs are active and should be debugged separately from geometry overlap.

Joints、Joint Reference Frames、Jointed Bodies、Joint Subtypes、Joint Limits and Drives 与 Articulations 覆盖约束系统。joint 不是普通父子 transform，也不是几何连接线；它描述两个物理 body 之间的约束、参考 frame、限制和驱动。articulation 则面向多关节结构，常见于机器人和机构仿真，应和普通刚体集合区分。Joints describe constraints between bodies, while articulations describe structured multi-joint systems.

官方 Examples 中的 Box on Box、Box on Quad、Spheres with Materials、Group Filtering、Pair Filtering、Joint、Distance Joint 和 Nested Articulation 可作为阅读路径：先看最小刚体和碰撞例子，再看 material 与 filtering，最后看 joint 和 articulation。中文页不逐行翻译示例代码，而是说明每类示例验证哪一组 schema 关系。The examples progress from basic bodies to materials, filtering, joints, and nested articulations.

Physics Parsing Utils Overview 后半部分把页面从 authoring schema 扩展到解析层。Physics Parsing Parameters、Physics Parsing Determinism、Physics Object Descriptor、Physics Scene Descriptor、RigidBody/Shape/Joint/Articulation Descriptor 等 section 用来描述解析器如何把 USD authored scene 变成仿真后端可消费的 descriptor。这里的重点是确定性和 owner 行为，而不是新增一套与 USD 无关的数据模型。Physics parsing utilities describe how authored USD physics schemas are converted into descriptors for simulation owners.

因此，本页的中文主线是：UsdPhysics 用 USD schema 表达刚体物理；UsdGeom 提供空间与层级基础；各类 API 给 body、shape、material、filter、joint 和 articulation 添加物理含义；解析工具再把这些 authored opinions 变成 descriptor。读者只要沿着这个链条排查，就能区分“USD 没 author 对”“解析器没读到”“求解器不支持”三类不同问题。The main path is schema authoring, geometric context, physics APIs, parsing descriptors, then backend simulation behavior.

核心 schema/API 分组 / Core Schema and API Groups

  • Physics Scene：场景级仿真上下文。关注重力、时间步、simulation owner 以及解析入口，不应把它误认为渲染场景或 stage root。Physics Scene carries scene-level simulation context.
  • Rigid Body 与 Mass：描述哪些 prim 参与刚体仿真，以及质量、惯性和运动状态如何 author。它们和 transform、层级、kinematic/animation 状态共同决定物理输入。Rigid body and mass schemas describe simulated bodies and their physical properties.
  • Collision、shape 和 mesh shape：描述碰撞几何，而不是渲染几何本身。mesh 能否成为 shape、plane shape 如何表达、shape 是否有效，都是调试碰撞问题的核心。Collision schemas define solver-facing shapes rather than rendering geometry.
  • Physics Materials：摩擦、弹性等物理表面参数。它和 UsdShade 材质不同；前者给物理求解器，后者给 shading/rendering。Physics materials are distinct from shading materials.
  • Collision Filtering 与 pairwise filtering：控制碰撞对是否启用，常用于机器人、车辆和复杂装配体中的自碰撞排除。Collision filtering controls active collision pairs.
  • Joint、joint subtype、limit、drive 和 articulation：描述约束、自由度、限制和驱动。它们是机构和机器人结构的核心，不应被简化成普通父子层级。Joints and articulations describe constraints, degrees of freedom, limits, and drives.
  • Physics Parsing descriptors：把 USD schema opinions 归约为后端可消费的数据结构。关注 determinism、object descriptor、scene descriptor 和 simulation owner behavior。Parsing descriptors convert authored USD physics into backend-facing data.

边界、误读点与调试路径 / Boundaries, Misreads, and Debugging

  • 不要把 UsdPhysics 当成物理求解器。它定义 scene description；真实积分、碰撞检测、约束求解和稳定性策略属于仿真后端。UsdPhysics is a schema representation, not a solver.
  • 不要把 physics material 和渲染材质混用。一个对象可以有 UsdShade 材质用于外观，同时有 physics material 用于摩擦和弹性。Physics materials and shading materials serve different consumers.
  • 不要只看 visual mesh 判断碰撞。碰撞 shape、mesh-to-shape 规则、过滤组和 pairwise filtering 都可能让视觉上接触的对象不产生碰撞。Visual overlap does not prove that collision shapes or filters are active.
  • 不要把 joint 当成 transform parent。joint 的 reference frames、jointed bodies、limit 和 drive 都是约束语义；父子层级只说明 USD namespace 和 transform composition。Joints are constraints, not ordinary transform parenting.
  • 不要假设所有仿真后端都支持同样的 articulation 或 descriptor 行为。本页提供 exchange schema 和 parsing utilities；后端支持度仍需要用实际 runtime 验证。Backend support for every schema feature must still be validated by the simulation runtime.

建议调试顺序是：先确认目标 prim 上的 physics API 是否 author；再确认 mass、collision shape、material、filtering、joint 或 articulation opinion 是否来自预期 layer；随后检查 scene-level physics context；最后用 parsing descriptor 或后端日志确认 simulation owner 实际读到了哪些对象。这样可以避免把 USD authoring、composition、parsing 和 solver 行为混成一个问题。Debugging should separate authored APIs, layer opinions, scene context, parsing descriptors, and solver behavior.

如果解析 descriptor 与预期不一致，优先检查单位、默认值和 layer strength：官方页面单独列出 Types、Units、Default Values，说明这些基础规则会影响仿真解释。很多“质量不对”“关节方向不对”“碰撞范围不对”的问题，根因并不在后端求解器，而在 USD authored opinion 被更强 layer 覆盖、单位假设不一致，或默认值被误认为显式设置。Descriptor mismatches should be reduced to units, defaults, layer strength, and authored opinions before blaming the solver.

看到 Simulation Owner Behavior 时，要把它当作解析归属和后端消费约定来读；它帮助定位谁负责解释 descriptor，不是新的 authoring schema。Simulation Owner Behavior describes ownership of parsed descriptors rather than a separate authoring schema.
