逐段双语理解 / Paragraph-Level Bilingual Coverage

`HdRenderBuffer` 表示 Hydra 中可以被渲染写入的数据资源句柄。官方定义强调它可以是 draw target 的 2D image，也可以是 auxiliary rendering output；在实际使用里，它常对应 color AOV、depth AOV、id/picking buffer、normal/primvar 输出或 renderer 自己暴露的中间结果。它是 `HdBprim` 类型的 buffer primitive，不是 USD attribute，也不是直接的 GPU texture API。

`HdRenderBuffer` is a Hydra data-resource handle that render passes can render into. It is commonly used for AOVs, draw targets, depth/color/id buffers, or auxiliary renderer outputs.

本页最重要的边界是 indexed prim 与 out-of-band 两种创建方式。作为 indexed prim 时，`HdRenderBuffer` 通过 `HdSceneDelegate` 和 `SdfPath` 参与 `HdRenderIndex` 的同步，buffer properties 可以由场景委托查询得到；out-of-band 创建时，调用方可以直接提供 allocation parameters。这两种路径都会通向具体 backend 资源，但 ownership、dirty tracking 和生命周期入口不同。

A render buffer can be an indexed prim that talks to `HdSceneDelegate`, or it can be created out of band with direct allocation parameters. These paths differ in ownership, dirty tracking, and lifetime control.

`Allocate()` 是资源建立的核心接口。调用者或同步逻辑传入宽度、高度、深度、`HdFormat`、multi-sample 需求等参数，具体实现负责创建底层 CPU/GPU/renderer storage。随后 `GetWidth()`、`GetHeight()`、`GetDepth()`、`GetFormat()` 与 `IsMultiSampled()` 用来核对真实分配结果。调试黑屏或 AOV 尺寸错位时，先检查 allocation 参数是否与 render pass、viewport、AOV descriptor 和 sample count 一致。

`Allocate()` creates the underlying storage. `GetWidth()`, `GetHeight()`, `GetDepth()`, `GetFormat()`, and `IsMultiSampled()` expose the resulting allocation for validation and debugging.

`DirtyBits`、`GetInitialDirtyBitsMask()` 和 `Sync()` 说明 `HdRenderBuffer` 仍然是 Hydra 同步系统的一部分。尺寸、format、多重采样、converged state 或 scene-delegate-provided properties 变化时，需要通过 dirty bits 触发重新同步或重新分配。不要把 render buffer 当成一次性裸资源：如果它作为 indexed bprim 存在，变化通知和 `HdSceneDelegate` 查询仍然决定它何时被更新。

Dirty bits and `Sync()` connect render buffers to Hydra's synchronization model. Allocation changes and delegate-provided properties must be synchronized rather than silently mutated.

`Map()`、`Unmap()` 与 `IsMapped()` 提供 CPU 侧访问 buffer data 的协议。`Map()` 通常返回可读写或可读的内存指针，具体是否阻塞 GPU、是否需要 resolve、是否只允许读取、以及数据生命周期，都由实现类和 renderer backend 决定。调用方必须成对管理 `Map()` / `Unmap()`，不能缓存映射后的临时指针，也不能在未同步或未 resolve 的资源上假设数据已经可用。

`Map()`, `Unmap()`, and `IsMapped()` define the CPU-access protocol for buffer contents. Pointer lifetime, synchronization, and read/write semantics are implementation-dependent.

`Resolve()` 和 multi-sample 相关语义用于处理 MSAA 或 progressive accumulation 的输出。`IsMultiSampled()` 告诉调用方底层资源是否多重采样；`Resolve()` 可能把多采样或累积状态转换成可读取/可展示结果。常见误用是直接 `Map()` 多重采样资源并期待得到最终像素，或者把 resolve 当作永远无成本的拷贝操作。

`Resolve()` is the boundary where multi-sampled or accumulated render-buffer data can be converted into data suitable for reading or presentation.

`IsConverged()` 表示与该 render buffer 相关的渲染结果是否稳定。progressive renderer、路径追踪器、异步 AOV 写入或后台 GPU work 常需要用它向 `HdTask`、render loop 或 UI 汇报当前输出是否仍在变化。它不是图像是否“正确”的语义，也不是所有 renderer 都以同样方式解释的质量指标。

`IsConverged()` reports whether the buffer's output is considered stable for the current renderer/task workflow; it is not a universal image-quality proof.

`GetResource()` 暴露 backend-specific resource handle，方便 render pass、task 或后处理管线接入具体 renderer 资源。这里要特别注意边界：`HdRenderBuffer` 提供 Hydra 层抽象，`GetResource()` 返回的对象含义由实现决定；上层代码不应假设它一定是某种特定 GPU texture、OpenGL object、Vulkan image 或 CPU array。

`GetResource()` exposes an implementation-specific resource handle. Higher-level Hydra code should not assume a fixed GPU or CPU representation.

与相邻类型的关系也要分清：`HdRenderPass` 或 renderer backend 往 buffer 写入；`HdTask` 可能调度 AOV 生成、resolve、readback 或展示；`HdSceneDelegate` 为 indexed buffer 提供 properties；`HdRenderBufferDescriptor` / `HdAovDescriptor` 描述期望的 format、clear value、多重采样和绑定含义；`HdRenderPassState` 管理渲染状态。`HdRenderBuffer` 位于这些对象之间，承担“资源句柄 + 同步 + 映射访问”的职责。

Adjacent objects such as `HdRenderPass`, `HdTask`, `HdSceneDelegate`, `HdRenderBufferDescriptor`, `HdAovDescriptor`, and `HdRenderPassState` define how the buffer is allocated, written, resolved, read, and presented.

常见调试入口包括：AOV 为空时查 `Allocate()` 参数、dirty bits 和 render pass target；尺寸错位时查 viewport、descriptor 与 `GetWidth()`/`GetHeight()`；读取数据全零时查 `Resolve()`、`Map()` 时机和 GPU/CPU 同步；progressive 输出不刷新时查 `IsConverged()`、task scheduling 和 backend 写入完成信号；内存泄漏或崩溃时查 `_Deallocate()`、`Finalize()`、映射指针生命周期和资源所有权。

Debugging usually follows allocation parameters, dirty bits, render-pass targets, resolve timing, mapping lifetime, convergence state, and backend resource ownership.

原文摘录 / English Source Excerpts

A render buffer is a handle to a data resource that can be rendered into, such as a 2d image for a draw target or auxiliary rendering output. The render buffer can be used as an indexed prim, in which case it communicates with the scene delegate to get buffer properties, or it can be created out of band and supplied directly with allocation parameters.
