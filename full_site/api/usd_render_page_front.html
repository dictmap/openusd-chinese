逐段双语理解 / Paragraph-Level Bilingual Coverage

UsdRender 是 USD 中描述渲染任务和输出配置的 schema 模块。官方开头把 rendering 定义为从 scene 生成图像的过程，同时说明渲染也可能产生 depth maps、light probes、point clouds 等相关副产物。因此本页不是只讲“图片文件”，而是讲 render settings 如何把渲染意图、输出产品和 AOV/render var 组织进 USD stage。UsdRender provides schemas and behaviors for describing renders, including images and related byproducts.

Structure and Organization 是本页核心。UsdRenderSettings 描述调用 renderer 所需的顶层设置，并通过 products 指向一个或多个输出；UsdRenderProduct 描述单个产品，可能是图像、点云或其他与图像渲染相关的 artifact；UsdRenderSettingsBase 抽出 settings 与 product 共享的设置；UsdRenderVar 描述一个计算出的输出通道，也就是 AOV。UsdRenderSettings, UsdRenderProduct, UsdRenderSettingsBase, and UsdRenderVar organize render configuration and output channels.

UsdRenderVar 的阅读重点是“通道语义”。一个 product 可以包含多个 related values：camera-space depth、shader 输出、light path expression、renderer 内部度量等。中文读者不要把它只理解成 EXR layer 名称；它还说明 renderer 应如何生成该值，以及该值在后端输出中如何被命名、采样和解释。UsdRenderVar describes computed output channels such as AOVs, shader quantities, LPEs, and renderer diagnostics.

UsdRenderSpec 是对 render settings 求值后的自包含规格，适合后端消费；UsdRenderPass 则把 UsdRender 和更广泛的 renderer 流程连接起来，用来组织多次 renderer 调用、denoising、依赖关系以及随时间变化的一系列输出。这个 distinction 很重要：settings/product/var 描述“要产出什么”，pass 更接近“如何把多个过程组织成最终帧”。UsdRenderSpec is an evaluated backend-facing specification, while UsdRenderPass organizes render invocations and dependencies.

Reading settings 说明一个 UsdStage 可以包含多个 UsdRenderSettings prim，并可通过 layer metadata 中的 renderSettingsPrimPath 指定一个有用默认值。和其他 stage metadata 一样，这个字段有 authoring 位置限制；UsdRenderSettings::GetStageRenderSettings() 和 UsdRenderComputeSpec() 是读取和求值 settings 的关键 API 入口。renderSettingsPrimPath nominates stage render settings; GetStageRenderSettings and UsdRenderComputeSpec read and evaluate them.

组织约定与 pipeline 边界 / Organization Conventions and Pipeline Boundaries

Conventions 解释为什么 render settings 建议放在 /Render 下。这样工具可以只组合渲染相关 prim，降低按整场景扫描的成本；也便于 shot、sequence、production、facility 级别的 render layer 共享和叠加。这个约定不是任意目录风格，而是为大型 pipeline 的快速发现和一致 composition 服务。UsdRender convention places render-related specifications under /Render for fast discovery and consistent composition.

这个约定也带来边界：场景几何和渲染设置可以独立组织。如果把一个 scene 引用进另一个 scene，只引用几何根节点可能不会自动带上 /Render 下的设置；需要单独引用或修正路径。中文页必须把这个 caveat 说清楚，否则读者会误以为 render settings 一定跟随几何资产移动。Referencing scene geometry does not necessarily bring the scene's /Render specifications along.

在实际排查中，先确认 /Render 是否存在、settings prim 是否在正确 layer 中、renderSettingsPrimPath 是否指向预期 prim、products 是否被 settings 关系引用、orderedVars 是否指向存在的 render var。只有这些 USD authoring 与 composition 都成立之后，才进入 renderer 或 Hydra/Hd delegate 的后端问题。Debugging should first verify /Render organization, metadata, products, orderedVars, and composed relationships.

生产管线里常见的错误是把 render settings 和镜头资产、场景资产、lighting layer 混在一起管理，结果引用、变体或 shot override 后找不到预期输出配置。UsdRender 的 /Render 约定提供了一个稳定锚点：几何和材质可以按资产结构组合，渲染输出配置则可以按 shot、facility 或 renderer policy 组合。这样工具能快速定位渲染意图，也能把“场景内容是否正确”和“输出配置是否正确”拆开验证。The /Render convention separates scene content composition from output-configuration composition.

如果一个 show 同时有 beauty、matte、id、deep、denoise input、preview 或 point-cloud 输出，不应把这些都塞成单个模糊设置。更清晰的方式是让 settings 组织 products，让 product 组织 vars，让 pass 描述过程依赖。这样做的好处是每个输出的 camera、resolution、AOV 列表、文件名、sourceName 和 sourceType 都有明确归属，后续自动化检查也能逐层定位。Settings, products, vars, and passes form separate layers for large render-output configurations.

相机、像素与窗口规则 / Camera, Pixels, and Windows

How settings affect rendering 从 camera 和 pixels 开始说明 settings 如何变成最终采样。UsdRenderSettingsBase 通过 relationship 指向 UsdGeomCamera；camera transform 把 world coordinates 变成 camera-relative view coordinates，projection 再映射到 screen coordinates，而 UsdRenderProduct 的 render settings 把 screen coordinates 映射到 pixel coordinates。Render settings connect UsdGeomCamera coordinates, screen coordinates, and product pixel coordinates.

Camera 部分强调 creative camera 属性和 technical render settings 的分工。near/far clipping、additional clipping planes、aperture、focal length、orthographic/perspective projection 影响画面构图；resolution、pixelAspectRatio、data window 等属性则控制渲染输出网格。把这两类设置混淆，会造成“相机看起来没错但输出比例或裁切不对”的问题。Camera attributes define creative composition, while render settings define technical image generation.

Pixels 说明 raster image 是屏幕 aperture 上的矩形像素采样阵列。resolution 与 pixelAspectRatio 一起决定图像总宽高比；非方形 pixel 可以补偿非方形 resolution。这里的关键不是 UI 显示比例，而是像素采样在 aperture 上的密度和位置。Resolution and pixelAspectRatio determine the image aspect ratio and pixel sample placement.

Aspect Ratio Policy 用 aspectRatioConformPolicy 解决 camera aperture 和 image dimensions 的宽高比不匹配。默认策略会扩展 aperture 以填满图像，保证 camera composition 可见，但可能露出额外场景区域。其他策略可能缩小 aperture、计算有效 pixelAspectRatio 或维持其他约束。调试 framing 问题时，这个字段和 camera aperture 必须一起看。aspectRatioConformPolicy resolves mismatch between camera aperture and image dimensions.

Cropping, Tiling, Overscan 由 dataWindowNDC 控制。它用 NDC 表达需要渲染的矩形区域，可用于裁切局部、把大图切成 tiles，或为后处理 kernel 采样 aperture 外的额外像素。Rasterization Rule 则说明像素中心落在 data window 内才被包含，左/下边界包含，右/上边界排除，以避免相邻 interval 重叠或出现缝隙。dataWindowNDC controls cropping, tiling, overscan, and the rasterization rule for pixel inclusion.

这些规则对渲染农场和审片尤其重要。tile 渲染如果 data window 或边界包含规则不一致，拼接时会出现重复像素或缝；overscan 如果没有和后处理 kernel 对齐，合成阶段会缺少边缘采样；aspect policy 如果和镜头 aperture 理解不一致，画面构图会被误判为 camera 错误。UsdRender 把这些技术约定放在 settings/product 层，目的是让不同工具能以同一份 scene description 解释输出窗口。Tile stitching, overscan, and aspect policy must be interpreted consistently across render and post-processing tools.

官方 section 读者检查表 / Section-by-Section Checklist

  • Structure and Organization / Concepts：先建立对象关系。settings 是顶层调用入口，product 是输出对象，var 是输出通道，spec 是求值结果，pass 是多步骤过程。读者应能从这些关系还原一个渲染任务，而不是只背类名。Concepts define the object graph for a render task.
  • Reading settings：确认 stage metadata、root/session layer 限制和 UsdRenderComputeSpec()。这部分告诉工具如何从 stage 找到默认 settings，并把 composition 后的结果归约为后端可用的 specification。Reading settings explains metadata, layer constraints, and evaluated specifications.
  • Conventions：确认 /Render 组织策略是否被遵守。这个 section 是 pipeline 搜索、layer 共享和快速打开渲染设置的依据，不应当当作可忽略的目录建议。Conventions make render specifications discoverable and efficiently composable.
  • Camera：检查 camera relationship 指向的 UsdGeomCamera 是否存在，并把 transform/projection/frustum/aperture 与 render settings 中的技术输出参数区分开。Camera settings combine creative camera state with technical render settings.
  • Pixels / Aspect Ratio Policy：检查 resolution、pixelAspectRatio 和 aspectRatioConformPolicy。很多审片中的“比例不对”并不是素材损坏，而是 aperture 和 image dimensions 的调和策略不同。Pixel and aspect policy explain how image dimensions map to camera aperture.
  • Cropping, Tiling, Overscan / Rasterization Rule：检查 dataWindowNDC、tile 分块和像素中心包含规则。这个 section 直接影响局部渲染、大图分块、overscan 和后处理边界。Data windows and rasterization rules affect tile boundaries and overscan.
  • Extensions：检查 renderer-specific API schema。这里说明可扩展性，但不保证所有 renderer 都支持同一扩展；跨 renderer 发布时必须把扩展支持度作为兼容性检查项。Extensions enable renderer-specific APIs but require backend compatibility checks.
  • Example Usage：把 USDA 示例当作结构模板：metadata 找 settings，settings 指 products，product 指 camera 与 orderedVars，vars 指具体 sourceName/sourceType/dataType，passes 组织过程依赖。The example is a structural template for settings, products, vars, and passes.

核心 API 分组 / Core API Groups

  • UsdRenderSettings：顶层 render settings prim，声明 renderer invocation 需要的全局设置，并通过 relationships 连接 products。UsdRenderSettings stores top-level render settings and product relationships.
  • UsdRenderProduct：单个输出产品，通常是图像，也可以是 point cloud 或其他 artifact。它继承 settings，并可添加 product-specific overrides。UsdRenderProduct describes one output artifact and product-specific overrides.
  • UsdRenderSettingsBase：settings 与 product 共享属性的抽象基础，尤其连接 camera、resolution、pixelAspectRatio 等公共设置。UsdRenderSettingsBase captures shared settings for settings and products.
  • UsdRenderVar：描述 AOV 或 render variable，例如 color、alpha、depth、LPE、shader quantity、renderer diagnostic。重点是 sourceName、sourceType、dataType 等如何让 renderer 生成正确通道。UsdRenderVar describes AOV channels and renderer-produced data.
  • UsdRenderSpec 与 UsdRenderComputeSpec()：把 composed settings 求值成后端可消费的自包含规格，适合在进入 renderer 前做一致性检查。UsdRenderSpec and UsdRenderComputeSpec evaluate composed render settings for backend consumption.
  • UsdRenderPass：组织多 pass 渲染、denoising、依赖关系和随时间变化的输出序列，用于把多个过程组合到最终帧。UsdRenderPass organizes multi-pass renderer invocations and dependencies.
  • UsdRenderSettingsAPI：renderer-specific 扩展的基础 API schema。特定 renderer 可以派生自己的 API schema，让应用通过 API schema 类型层级通用地检查扩展设置。UsdRenderSettingsAPI is the base API schema for renderer-specific extensions.

边界、误读点与调试路径 / Boundaries, Misreads, and Debugging

  • 不要把 UsdRender 当作 renderer 或 Hydra delegate。它描述 settings、products、vars、passes；真正的渲染执行由 renderer、Hydra/Hd 或其他后端完成。UsdRender describes render configuration; renderer execution is backend-specific.
  • 不要把 UsdRenderVar 简化为文件通道名。它还涉及 sourceName、sourceType、dataType、LPE 和 renderer 如何计算该值。UsdRenderVar carries semantic instructions for computed output channels.
  • 不要把 renderSettingsPrimPath 当成普通 subLayer 内任意 metadata。官方说明它可写在 root layer 或 session layer，不在 root 的其他 subLayers 中。renderSettingsPrimPath follows stage metadata authoring constraints.
  • 不要只看 camera 是否正确。输出比例、裁切、tile、overscan、pixelAspectRatio 和 aspectRatioConformPolicy 都可能改变最终图片。Camera correctness does not guarantee pixel, crop, tile, or aspect policy correctness.
  • 不要把 renderer-specific extension 误认为跨 renderer 标准行为。UsdRenderSettingsAPI 允许扩展，但实际支持度仍由具体 renderer 或 pipeline 工具决定。Renderer-specific extensions require backend-specific validation.

建议调试顺序是：确认 /Render 约定和 layer composition；检查 renderSettingsPrimPath；确认 settings 到 products、products 到 orderedVars、products 到 camera 的 relationships；再检查 resolution、pixelAspectRatio、dataWindowNDC、aspectRatioConformPolicy；最后让后端计算或读取 UsdRenderSpec，比较它与实际 renderer 输出是否一致。Debug from /Render organization through relationships, pixel settings, and evaluated UsdRenderSpec before blaming backend output.

与 Hd、Hdx 或 Hydra 调试结合时，要区分 USD 层的 render schema 与 delegate 层的具体执行。UsdRender 可以描述期望输出，但 Hydra/Hd 负责如何把 scene index、render pass、buffer、AOV 或后端资源组织起来。两层都可能出错，不能互相替代。UsdRender schema and Hydra/Hd delegate execution are adjacent but separate debugging layers.

如果最终文件没生成，优先检查 product 的 productName、目标路径权限和 renderer 是否收到该 product；如果 AOV 缺失，检查 orderedVars、sourceName、sourceType 和 renderer 支持的 var 名称；如果输出尺寸错误，检查 resolution、pixelAspectRatio、aspectRatioConformPolicy 和 dataWindowNDC；如果多 pass 输出顺序不对，检查 UsdRenderPass 的依赖关系和时间采样输入。这些排查都应先从 USD authored opinions 开始，再进入 renderer 日志。Missing files, missing AOVs, wrong dimensions, and pass-order issues have different UsdRender checkpoints.

后端边界还包括颜色管理、采样质量、去噪、GPU/CPU 设备选择和 renderer 私有参数。本页可以帮助确认这些设置是否有 USD 侧的表达入口，但不能证明某个 renderer 已按期望执行。最终交付时应同时保存 composed USD、计算出的 render spec、renderer 日志和输出文件元数据，才能把 USD 描述错误、后端配置错误和运行环境错误区分开。Backend validation still requires composed USD, evaluated specs, renderer logs, and output metadata.

对自动化系统来说，最可靠的验收不是“页面里有 RenderSettings 字样”，而是能从中文说明中沿着 settings、product、var、pass、camera、pixel window 和 extension 逐项复核，并能把每个输出问题落到对应的 USD 节点或后端证据上。A useful review path maps each render-output issue to the matching USD node or backend evidence.

这也是本页晋级的标准：中文读者不必依赖英文正文，也能判断渲染配置放在哪里、默认设置从哪里读取、输出产品如何声明、输出通道如何命名、像素窗口如何裁切、多过程渲染如何串联，以及哪些问题应回到 USD composition 层排查。The Chinese path must independently explain placement, defaults, products, channels, windows, passes, and composition debugging.

换句话说，本页把渲染描述、输出声明、后端消费和故障定位拆成可逐项核对的中文路径和验证依据。The page separates render description, output declaration, backend consumption, and debugging.

示例阅读路径 / Example Usage Reading Path

官方 Example Usage 使用 #usda 1.0，在 layer metadata 中设置 renderSettingsPrimPath = "/Render/PrimarySettings"，然后在 /Render 下定义 RenderSettings、RenderProduct、Vars 和 Passes。中文阅读时先看 metadata 如何找到 settings，再看 settings 如何连接 product，product 如何连接 camera 和 orderedVars，最后看 vars 如何定义 color、alpha、directDiffuse、id 等输出通道。The example shows renderSettingsPrimPath, /Render, RenderSettings, RenderProduct, Vars, and Passes.

示例中的 productName、orderedVars、sourceName、sourceType 和 dataType 都应保留英文 token/属性名。中文页只解释它们各自负责的语义，不把它们翻成中文字段名。这样读者可以直接把中文解释映射回 USDA、Doxygen、renderer 日志和 pipeline 配置。Property names and token values are preserved so readers can map the explanation back to USDA and runtime logs.

把示例迁移到实际项目时，建议按四列检查：第一列是 settings 是否能被 renderSettingsPrimPath 找到；第二列是 products 是否列出所有期望 artifact；第三列是 vars 是否覆盖 beauty、alpha、id、depth、LPE 或诊断通道；第四列是 pass 是否说明这些输出在时间和依赖上的执行关系。这样能把“没有输出”“输出通道不全”“文件名错”“pass 没按预期执行”分成不同问题。A practical checklist separates settings discovery, products, vars, and pass dependencies.

如果一个页面或工具只显示最终图像，而没有展示 UsdRenderProduct 和 UsdRenderVar 的关系，读者应回到本页确认输出声明是否完整。OpenUSD 的优势在于这些配置可以被 layer、variant、session layer 和 pipeline override 管理；但这也意味着错误可能来自 composition，而不是 renderer 本身。中文主阅读路径必须提醒读者先核对 USD authored data，再核对后端执行。Render-output errors may originate in USD-authored configuration or composition before backend execution.

No Matches UsdRender : USD Render Schema Table of Contents Structure and Organization Concepts Reading settings Conventions How settings affect rendering Camera Pixels Aspect Ratio Policy Cropping, Tiling, Overscan Rasterization Rule Extensions Example Usage UsdRender provides schemas and behaviors for describing renders. Rendering refers to the process of generating images of a scene. In addition to images, rendering can also produce related byproducts such as depth maps, light probes, or point clouds. UsdRender provides primitives for configuring high-level aspects of the rendering process referred to collectively as render settings . An overview of the process follows. Structure and Organization Concepts UsdRenderSettings describes top-level settings required to invoke a renderer. It specifies one or more products , representing artifacts for the renderer to produce. UsdRenderProduct describes a single product.
