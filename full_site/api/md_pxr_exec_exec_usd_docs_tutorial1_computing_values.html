中文主阅读路径 / Chinese Main Reading Path

本页是 OpenExec 官方教程链中的第一篇，目的不是定义新的 schema computation，而是演示客户端如何使用 OpenExec APIs 请求并读取已经可用的 computed values。官方示例代码位于 USD/extras/exec/examples/computingValues/，阅读时应按页面顺序理解：先准备一个包含 UsdGeomXformable prims 的 UsdStage，再用 ExecUsdSystem 建立执行系统，随后构造 ExecUsdRequest、显式或隐式 prepare、调用 Compute，最后从 ExecUsdCacheView 中按请求顺序取回 VtValue 并转成 GfMatrix4d。This tutorial demonstrates how to request values from OpenExec and read the results.

教程使用的 computation 是 UsdGeomXformable 提供的 computeLocalToWorldTransform。它对某个 Xform 产生一个 4x4 matrix，把该 Xform local-space 的点转换到 world-space。这个结果依赖两个来源：当前 Xform 上 authored 的 transform attribute，以及父级 Xform 上递归计算得到的 computeLocalToWorldTransform 结果。因此调试时不能只看最终矩阵，还要沿着 prim 层级检查 authored transform、父子关系、request 中的 provider 和当前 stage 状态。computeLocalToWorldTransform depends on the authored transform and the parent's computed local-to-world transform.

官方特别提醒这里展示的是 lowest-level API，目标是 imaging system 等性能敏感客户端。也就是说，ExecUsdSystem、ExecUsdRequest、ExecUsdCacheView 这些对象的职责要按性能路径理解：system 维护可复用的 data flow network，request 维护特定值集合的 schedule，cache view 让客户端从已计算结果里提取值。后续 convenience APIs 可能简化调用，但本教程刻意暴露底层顺序，方便读者理解编译、调度和求值成本。The API shown here is the lowest-level API for performance-intensive clients.

官方 section 顺序 / Official Section Order

  • Overview：说明教程目的、示例目录、使用 computeLocalToWorldTransform 的原因，以及 lowest-level API 的性能边界。Overview explains the purpose and performance-oriented API level.
  • Create a UsdStage：从 xformPrims.usda 打开一个 stage，示例中有 /Root、/Root/A1、/Root/A2 三个 Xform，并在 xformOp:transform 上 author 了矩阵。Create a UsdStage opens xformPrims.usda.
  • Create an ExecUsdSystem：用 ExecUsdSystem execSystem(stage); 把 OpenExec 连接到这个 UsdStage，并让 system 持有用于 evaluation 的 compiled data flow network。ExecUsdSystem maintains internal state and a compiled network.
  • Build an ExecUsdRequest：用 ExecUsdValueKey 描述要请求的值；每个 key 由 provider UsdObject 和 computation 名称 TfToken 组成。ExecUsdValueKey identifies a provider and computation name.
  • Prepare the request：调用 ExecUsdSystem::PrepareRequest，确保 network 已为请求编译，并确保 request 的 schedule 是最新的。PrepareRequest compiles the network and updates the schedule.
  • Compute values：调用 ExecUsdSystem::Compute 得到 ExecUsdCacheView，此时 computed results 已经可供读取。Compute returns an ExecUsdCacheView.
  • Extract computed values：按构造 request 时 value keys 的顺序，用 ExecUsdCacheView::Get 取回 VtValue，再用 VtValue::Get<GfMatrix4d>() 得到具体矩阵。Get extracts values by index.
  • Putting it all together：把 includes、stage、system、request、prepare、compute、extract 和 TF_AXIOM 检查放进一个完整 C++ 示例。The final example combines the full workflow.

关键代码路径 / Code Path

下面保留官方示例的关键调用顺序。代码标识符、头文件、token、prim path 和数值断言必须原样保留；中文只解释语义，不替换可编译文本。The identifiers and code tokens are preserved for source parity.

const UsdStageRefPtr stage = UsdStage::Open("xformPrims.usda");
ExecUsdSystem execSystem(stage);

std::vector<ExecUsdValueKey> valueKeys {
    {stage->GetPrimAtPath(SdfPath("/Root/A1")),
     ExecGeomXformableTokens->computeLocalToWorldTransform},
    {stage->GetPrimAtPath(SdfPath("/Root/A2")),
     ExecGeomXformableTokens->computeLocalToWorldTransform},
};

const ExecUsdRequest request = execSystem.BuildRequest(std::move(valueKeys));
execSystem.PrepareRequest(request);
ExecUsdCacheView cache = execSystem.Compute(request);

VtValue value;
value = cache.Get(0);
const GfMatrix4d a1LocalToWorld = value.Get<GfMatrix4d>();
value = cache.Get(1);
const GfMatrix4d a2LocalToWorld = value.Get<GfMatrix4d>();

TF_AXIOM(GfIsClose(a1LocalToWorld.ExtractTranslation(), GfVec3d(1, 2, 0), 1e-6));
TF_AXIOM(GfIsClose(a2LocalToWorld.ExtractTranslation(), GfVec3d(1, 0, 3), 1e-6));

这个顺序可以直接映射到调试步骤：如果 UsdStage::Open 失败，问题在文件路径或层加载；如果 ExecUsdSystem::BuildRequest 得到的 request 不符合预期，检查 SdfPath("/Root/A1")、SdfPath("/Root/A2") 和 ExecGeomXformableTokens->computeLocalToWorldTransform 是否匹配 provider；如果 Compute 后读取失败，检查 cache index 是否和 valueKeys 顺序一致；如果 TF_AXIOM 失败，回到 authored transform 和父级 Xform 的递归结果。The call order also defines a practical debugging order.

注意 ExecUsdSystem::BuildRequest 的输入是移动进来的 std::vector<ExecUsdValueKey>，这说明 request 的语义在构建时就已经固定为一组要一起计算的值。后续 PrepareRequest 和 Compute 复用的是这个 request 的结构，而不是重新扫描整个 stage。对性能敏感客户端来说，这种拆分很重要：可以把“决定要算哪些值”和“实际执行求值”分离，并让 schedule 在多轮求值中被摊销。BuildRequest fixes the requested value set and lets preparation and evaluation reuse request state.

逐段双语理解 / Paragraph-Level Bilingual Coverage

Overview。教程开头强调“请求值”而不是“定义计算”。computeLocalToWorldTransform 是已有 computation，教程只展示怎样把 /Root/A1 和 /Root/A2 作为 providers，请求它们的 local-to-world transform。结果是 GfMatrix4d，语义是 local-space 到 world-space 的点变换矩阵。The Overview frames the tutorial as requesting already provided computations.

Create a UsdStage。示例 USDA 中 /Root 的 transform 平移为 (1,0,0)，A1 增加 (0,2,0)，A2 增加 (0,0,3)。因此最终断言期望 A1 的 world translation 是 (1,2,0)，A2 的 world translation 是 (1,0,3)。这让读者能把矩阵结果和输入 authored values 对上。The authored transforms explain the expected world translations.

Create an ExecUsdSystem。ExecUsdSystem 不是一次性函数调用包装，而是持有内部 state 的系统对象。它会维护 evaluation 所需的 compiled data flow network，因此同一个 stage 上的多组 value requests 可以共享 system 的内部状态，摊薄 network compilation 成本。ExecUsdSystem owns reusable internal state and a compiled data flow network.

Build an ExecUsdRequest。ExecUsdRequest 表示“一批一起计算的值”。ExecUsdValueKey 的 provider 是 stage->GetPrimAtPath(SdfPath("/Root/A1")) 或 stage->GetPrimAtPath(SdfPath("/Root/A2"))，computation 名称是 ExecGeomXformableTokens->computeLocalToWorldTransform。这两部分缺一不可：只有 provider 没有 computation 名称，无法知道要算什么；只有 computation 名称没有 provider，也无法定位 scene object。Each ExecUsdValueKey combines provider object and computation token.

Prepare the request。PrepareRequest 做两件事：确保 system 的 network 已包含计算这些值需要的数据流节点和连接，并确保 request 的 schedule 是最新的。官方说明 prepare 可选，因为 Compute 会在需要时自动 prepare；但性能敏感客户端常显式 prepare，以控制 compilation/scheduling 发生的时机。PrepareRequest updates the compiled network and request schedule.

Compute values。ExecUsdSystem::Compute 执行 request 并返回 ExecUsdCacheView。cache view 的含义是“这次 request 的结果视图”，不是永久的 authored data。读取结果时要记住它对应的是当前 request、当前 stage state 和当前 schedule。Compute returns a cache view for the requested values.

Extract computed values。ExecUsdCacheView::Get(0) 对应 valueKeys 的第一个条目，也就是 /Root/A1；Get(1) 对应 /Root/A2。返回值先放在 VtValue，再转成 GfMatrix4d。常见误读是把 cache index 当作 prim index 或 scene traversal 顺序；正确理解是它严格跟 build request 时 valueKeys 的顺序绑定。Extraction indices correspond to the value key order used to build the request.

Putting it all together。完整示例用 TF_AXIOM 和 GfIsClose 验证提取出的 translation 是否与 authored transforms 的组合结果一致。这不是新的计算定义，而是对 OpenExec value request 工作流的最小闭环：stage 输入、system 状态、request schedule、compute、cache extraction、numeric validation。The full example validates the expected translations after computing values.

示例结果。A1 的本地 transform 是沿 Y 方向移动 2，父级 Root 沿 X 方向移动 1，所以组合后的 world translation 是 GfVec3d(1, 2, 0)；A2 的本地 transform 是沿 Z 方向移动 3，叠加同一个父级 Root 后得到 GfVec3d(1, 0, 3)。教程用 ExtractTranslation 和 GfIsClose 检查这两个值，是为了把 OpenExec 的 computed value 和 USD authored transform 之间的因果关系落到可验证的数值上。The example validates A1 and A2 translations against authored transform composition.

缓存与请求复用。本页虽然没有展开 advanced caching 机制，但已经通过 ExecUsdSystem 和 ExecUsdRequest 暗示了缓存边界。system 复用 compiled network，request 复用 schedule，cache view 暴露当前求值结果。三者不能混为一个对象：system 绑定 stage 级状态，request 绑定一批 value keys，cache view 绑定一次 compute 后的可读结果。读者如果把这三层混在一起，会很难判断错误来自 stage、request 还是 extraction。System, request, and cache view have distinct caching and lifetime roles.

边界、误读点与调试路径 / Boundaries and Debugging

  • 不要把本教程当作 schema computation authoring 教程；它消费 UsdGeomXformable 已提供的 computation。要学习定义新 computation，应继续看 Tutorial 2: Defining Schema Computations。This tutorial consumes existing computations; Tutorial 2 defines computations.
  • 不要把 PrepareRequest 理解为必须每次 compute 前都手动调用。它是可选但有用的性能控制点；显式调用适合希望把 compilation/scheduling 成本提前到某个阶段的客户端。PrepareRequest is optional but useful for controlling when preparation cost occurs.
  • 不要把 ExecUsdCacheView::Get 的 index 当成 prim path 排序。index 来源于 std::vector<ExecUsdValueKey> 的顺序；如果结果错位，优先检查 vector 构造顺序。Cache indices follow valueKeys order.
  • 如果最终 GfMatrix4d 不符合预期，先检查 xformOpOrder、xformOp:transform、父级 /Root 的 transform，再检查 provider prim 是否真实存在。Matrix mismatches should be traced back to xform authored values and provider prims.
  • 与相邻模块的关系：ExecUsd 连接 USD stage 和执行请求，ExecGeom 提供 computeLocalToWorldTransform token，Vt 承载类型擦除结果，Gf 表示矩阵和向量，SdfPath 定位 stage 中的对象。ExecUsd, ExecGeom, Vt, Gf, and SdfPath are all involved in the workflow.

一个实用排查路径是从外到内走：先确认 xformPrims.usda 能被 UsdStage::Open 打开；再确认 /Root/A1 和 /Root/A2 能通过 SdfPath 找到；接着确认这两个 prim 确实是 UsdGeomXformable 语义下可请求 computeLocalToWorldTransform 的 providers；然后检查 ExecUsdValueKey 的 vector 顺序；最后再看 ExecUsdCacheView::Get 的 index 和 VtValue::Get<GfMatrix4d>() 的类型提取是否一致。这样可以避免一上来就把问题归咎于 OpenExec 调度层。A practical debugging path checks stage, prim paths, providers, value key order, cache indices, and VtValue extraction.

本页还要和 OpenExec Overview 区分开：Overview 解释 computation、provider、plugin/builtin computations 和 advanced topic 边界；Tutorial 1 则把这些概念压缩成一个可运行工作流。换句话说，Overview 回答“OpenExec 是什么、有哪些概念”，本页回答“我如何请求并验证一个已有 computed value”。这种分工也是本地点击顺序必须从 Overview 进入 Tutorial 1 的原因。Overview explains concepts; Tutorial 1 turns them into a runnable value request workflow.

实际使用时，本页适合三类读者：第一类是已经有 USD stage 和 schema computations，只想知道怎样请求结果的客户端作者；第二类是需要把 OpenExec 接到 imaging、validation 或工具层的集成者，他们要理解 PrepareRequest 与 Compute 的成本边界；第三类是排查 computed transform 不正确的调试者，他们可以沿着 stage、provider、value key、request、cache view、typed extraction 和数值断言逐步定位。若读者的目标是新增一个 schema computation、注册 callback 或声明 input values，本页只能提供消费端背景，下一步必须跳到 Tutorial 2，而不是在本页寻找注册 API。This page is for requesting values; defining new computations belongs to Tutorial 2.

因此，本地相关链接不是随机目录列表，而是围绕真实任务顺序组织：先用 Overview 建立概念词表，再用本页完成一次值请求闭环，然后用 Tutorial 2 学习定义 computation；需要内部机制时再看 System Design、ExecUsd README 和 Vdf README。这个顺序能避免读者在 class index、source file 和 README 之间来回跳转，却仍不知道哪一页回答“怎么请求值”、哪一页回答“怎么定义值”、哪一页回答“系统为什么这样调度”。The local related links follow the task order: concepts, value requests, computation definitions, and system internals.

本页最容易被忽略的一点是结果读取顺序：cache.Get(0) 和 cache.Get(1) 并不表示场景遍历顺序，而是严格对应 valueKeys 中 /Root/A1 与 /Root/A2 的插入顺序。只要 request 的 key 顺序改变，cache index 的语义也会随之改变。Cache indices follow request key insertion order.

这一点也解释了为什么教程把完整代码放在最后：读者需要看到同一个 valueKeys vector 如何一路影响 request、schedule、cache view 和最终断言，才能确认每个 computed value 的来源。The final code block ties value keys to the returned values.

VtValue 只是结果承载容器，真正的业务类型仍需用 Get<GfMatrix4d>() 明确提取；类型不匹配时应检查请求的 computation，而不是修改矩阵类型。VtValue must be extracted as the expected concrete type.

这样能把类型错误和求值错误分开定位，减少误判。This separates type errors from evaluation errors.
