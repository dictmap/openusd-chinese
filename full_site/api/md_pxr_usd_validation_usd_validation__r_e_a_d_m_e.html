中文主阅读路径 / Chinese Main Reading Path

本页说明 OpenUSD Validation framework 如何验证 assets，覆盖 core rules、schema rules 和 client-provided rules via plugins，目标是让资产在不同 USD workflows 之间更稳健、更可互换。阅读时先把它和普通 schema authoring 区分开：schema 定义可写入的数据模型，Validation 定义运行时或发布前要执行的检查规则；validator 可以报告问题和给出 fixer，但不会自动改写 layer、prim 或 property。The OpenUSD Validation framework provides a system to validate assets, verifying core rules, schema rules, and client-provided rules via plugins.

官方正文的对象层级是：UsdValidationValidator 表示单个 validation test，可产生零个或多个 named validation errors，例如 usdValidation:CompositionErrorTest；UsdValidationValidatorMetadata 描述该测试的 name、pluginPtr、keywords、doc、schemaTypes、isTimeDependent、isSuite；UsdValidationContext 把一组 validators 组织为一次运行；UsdValidationRegistry 是发现、注册、加载 validator 和 validator suite 的中心入口。A single UsdValidationValidator instance represents a single validation test that can result in zero or more named validation errors when run.

本地中文页按用户点击和官方 section 顺序展开：先讲 validators、metadata、context、registry、errors 与 fixers，再讲 Running Validator Tests，然后讲 C++ custom validators 的 plugin/explicit 注册路径、fixers，最后讲 Python validators 的性能、签名、显式注册、插件加载、运行和 suite 分组。这样读者从 API 入口点击到本页后，可以按官方目录逐段理解，而不是只看到摘要表。The page order follows official sections from validator objects through Python custom validators.

核心对象模型 / Validator, Metadata, Context, Registry

  1. UsdValidationValidators 用来运行 validation tests。每个 UsdValidationValidator 对应一个测试，而一个测试可以返回多个 UsdValidationError。因此不要把 validator 和 error 一一对应；一个 composition test 可以同时发现多处问题。UsdValidationValidators are used to run validation tests.
  2. UsdValidationValidatorMetadata 是 validator 的发现和筛选依据。keywords 可用于按主题过滤，schemaTypes 可限制 prim-level validator 适用的 schema，isTimeDependent 标记是否依赖时间范围，isSuite 标记是否是 validator suite。Validator metadata includes name, pluginPtr, keywords, doc, schemaTypes, isTimeDependent, and isSuite.
  3. UsdValidationContext 从一组 validators 构建，用于一次性运行多个检查。它可以通过 UsdValidationRegistry 的 metadata query 按 keywords 或 schemaTypes 选出候选；查询 schema 时可用 includeAllAncestors 让 UsdGeomSphere 同时匹配 UsdGeomGprim 和 UsdGeomImageable。Metadata queries may include ancestor schema types such as UsdGeomSphere, UsdGeomGprim, and UsdGeomImageable.
  4. UsdValidationRegistry 管理 UsdValidationValidator 与 UsdValidationValidatorSuites，也允许注册 custom validators。官方强调 validator 和 registry 在 USD session 内是 immutable、non-copyable、immortal，所以不要在业务逻辑里假设可以临时复制或回收这些对象。The validators and registry are immutable, non-copyable, and immortal within a USD session.

错误和修复器 / Validation Errors and Fixers

UsdValidationErrors 是验证运行返回给调用方的诊断集合。每个 error 包含 Name、Identifier、Error type、Error sites、Message、关联 validator、可选 fixers 和 Error Data。UsdValidationErrorType 里有 None、Error、Warn、Info，它们描述严重度，不等同于是否能自动修复。Validation errors provide name, identifier, error type, error sites, message, validator, fixers, and error data.

Error sites 可指向 SdfLayer、UsdStage、prim 或 property；Error Data 用 VtValue 承载扩展数据。调试时应先看 error site，再看 validator 名称和 message，最后看附加 data；不要只看 message 字符串，因为同一个 validator 可能在多个 site 上产生同名或相近错误。An error site may identify a SdfLayer, UsdStage, prim, or property.

UsdValidationFixer 表示可选的修复动作，包含 name、description、keywords、FixerImplFn、FixerCanApplyFn 和它可修的 error name。验证测试不会自动调用 fixer；调用方必须先用 CanApplyFix() 检查，再在客户端提供的 UsdEditTarget 上调用 ApplyFix()。这条边界很重要：Validation 可以提示修复路径，但不会擅自修改资产。Validation tests will not automatically call any fixers.

Running Validator Tests

运行 validator 有两条基本路径：直接对单个 UsdValidationValidator 调用 Validate()，或把多个 validators 放入 UsdValidationContext 后统一调用 Validate()。Context 可并行运行多个 validators，但调用方仍要保证 stage、layer、prim 等输入对象在验证期间保持有效。Validators can be run by calling Validate() on a UsdValidationValidator or on a UsdValidationContext.

运行时还涉及两个常见筛选边界。其一是 prim-level validator 可结合 Usd_PrimFlagsPredicate 选择要遍历的 prim；其二是 time-dependent tests 默认使用 GfInterval::GetFullInterval()，调用方也可以提供特定 time interval。若验证结果随时间变化，先确认 validator metadata 的 isTimeDependent，再确认实际传入的 time range。Time-dependent tests default to GfInterval::GetFullInterval().

返回的 validation errors 只是诊断结果。若 error 带有关联 UsdValidationFixers，调用方负责决定是否应用 fixer，并提供正确的 edit target。自动化发布管线通常应把验证、报告、人工确认和修复提交分开，以避免 validation run 直接产生不可追踪的 scene edits。If errors provide associated UsdValidationFixers, the caller is responsible for fixing errors.

Creating Custom Validators

自定义 validators 可以通过 OpenUSD plugin infrastructure 懒加载，也可以显式创建并注册。共同点是都要实现一个 validator task function，类型可以是 UsdValidateLayerTaskFn、UsdValidateStageTaskFn 或 UsdValidatePrimTaskFn。选择粒度时要尽量靠近实际检查对象：只需要看 prim 的规则应写成 prim task，不应写成 stage task 后每次全量遍历 stage。A custom validator must implement UsdValidateLayerTaskFn, UsdValidateStageTaskFn, or UsdValidatePrimTaskFn.

若 prim-level validator 只适用于特定 schema，应设置 schemaTypes。这让 framework 跳过非匹配 prim，避免无意义调用。常见误读是把所有检查都写成 stage-level，因为它“看起来最通用”；官方明确提醒过宽的 granularity 会影响性能。Using too broad a granularity can impact performance.

Plugin Validators

插件路径把 metadata 放在 plugInfo.json，例如定义 Validator1 和 ValidatorSuite1。插件代码通过 TF_REGISTRY_FUNCTION(UsdValidationRegistry) 取得 registry，创建 TfToken validatorName("newValidatorPlugin:Validator1")，然后用 RegisterPluginValidator 注册 task function；suite 则通过 GetOrLoadValidatorsByName 找到 validator 后调用 RegisterPluginValidatorSuite。For custom validators created in a plugin, the plugin's plugInfo.json contains custom validator metadata.

TF_REGISTRY_FUNCTION(UsdValidationRegistry)
{
    UsdValidationRegistry &registry = UsdValidationRegistry::GetInstance();
    const TfToken validatorName("newValidatorPlugin:Validator1");
    const UsdValidateStageTaskFn stageTaskFn = [](const UsdStagePtr &usdStage,
        const UsdValidationTimeRange &timeRange) {
        UsdValidationErrorVector errors;
        return errors;
    };
    registry.RegisterPluginValidator(validatorName, stageTaskFn);
    const TfToken suiteName("newValidatorPlugin:ValidatorSuite1");
    const std::vector<const UsdValidationValidator *> containedValidators =
        registry.GetOrLoadValidatorsByName({ validatorName });
    registry.RegisterPluginValidatorSuite(suiteName, containedValidators);
}

Explicit Validators

显式路径适合原型、一次性脚本、测试或运行时生成规则。调用方创建 UsdValidationValidatorMetadata 和 task function，然后调用 UsdValidationRegistry::RegisterValidator()。它没有插件懒加载，也不会在 startup 时让其他工具提前看到 metadata。For custom validators created explicitly, create UsdValidationValidatorMetadata and use UsdValidationRegistry::RegisterValidator().

const UsdValidateStageTaskFn explicitStageTaskFn =
    [](const UsdStagePtr &usdStage, const UsdValidationTimeRange &timeRange) {
        UsdValidationErrorVector errors;
        return errors;
    };
const UsdValidationValidatorMetadata explicitValidatorMetadata = {
    TfToken("ExplicitValidator"),
};
registry.RegisterValidator(explicitValidatorMetadata, explicitStageTaskFn);

Choosing a Registration Path and Adding Fixers

选择 registration path 的核心问题是“其他代码是否需要在 validator 代码加载前发现 metadata”。如果需要按 keywords、schemaTypes 或 suite 信息提前枚举，使用 Plugin registration，因为 metadata source 是 plugInfo.json，startup 就可见且 validator 可 lazy loaded；如果规则只给当前调用方使用，Explicit registration 更简单。If other code needs to query validator metadata before the validator is loaded, use plugin registration.

RegisterPluginValidator() 和 RegisterValidator() 都可传入 std::vector<UsdValidationFixer>。每个 fixer 指明 name、description、FixerCanApplyFn、FixerImplFn、keywords 和 error name。官方示例中的 _ValidatorFixers() 创建 Example Fixer，并关联 ErrorNameAssociatedWithFixer。Each fixer specifies a name, description, FixerCanApplyFn, FixerImplFn, keywords, and the error name it can fix.

const std::vector<UsdValidationFixer> _ValidatorFixers()
{
    std::vector<UsdValidationFixer> fixers;
    FixerCanApplyFn fixerCanApplyFn = [](const UsdValidationError &error,
        const UsdEditTarget &editTarget, const UsdTimeCode &timeCode) -> bool {
        return true;
    };
    FixerImplFn fixerImplFn = [](const UsdValidationError &error,
        const UsdEditTarget &editTarget, const UsdTimeCode &timeCode) -> bool {
        return true;
    };
    fixers.emplace_back(TfToken("Example Fixer"), "An example fixer.",
        fixerImplFn, fixerCanApplyFn, TfTokenVector{},
        TfToken("ErrorNameAssociatedWithFixer"));
    return fixers;
}
registry.RegisterPluginValidator(validatorName, stageTaskFn, _ValidatorFixers());

官方还强调 UsdValidationRegistry does not manage fixers directly；fixers 被各自的 UsdValidationValidator 持有。因此修复逻辑应和产生该错误的 validator 绑定，不应假设有一个全局 fixer registry 可以独立查询和调度。UsdValidationRegistry does not manage fixers directly.

Creating Custom Validators in Python

Python validators 支持两条同构路径。插件注册使用 RegisterPluginLayerValidator、RegisterPluginStageValidator、RegisterPluginPrimValidator，调用方只提供 validator name，metadata 来自 plugInfo.json 并在 registry 初始化时解析。显式注册使用 RegisterLayerValidator、RegisterStageValidator、RegisterPrimValidator，调用方提供完整 ValidatorMetadata，无需插件基础设施。Custom validators can be implemented in Python using plugin registration or explicit registration.

Performance Considerations

官方性能边界非常明确：UsdValidationContext 运行 validators 时，C++ 与 Python tasks 会进入同一个 shared TBB worker thread pool，但 Python task 每次调用都必须获取 Python GIL。这意味着 Python validators 之间不能真正并行，而且大量 Python tasks 可能占满 worker，导致 ready 的 C++ validators 排队等待。因此性能敏感的 pipeline 应优先用 C++，Python 更适合 prototyping、tooling、小场景或低频规则。Python task functions must acquire the Python GIL on each invocation.

Task Function Signatures

  • RegisterLayerValidator / RegisterPluginLayerValidator：callable signature 是 (layer: Sdf.Layer) -> list[ValidationError]。Layer validators receive a Sdf.Layer.
  • RegisterStageValidator / RegisterPluginStageValidator：callable signature 是 (stage: Usd.Stage, timeRange: UsdValidation.TimeRange) -> list[ValidationError]。Stage validators receive a Usd.Stage and timeRange.
  • RegisterPrimValidator / RegisterPluginPrimValidator：callable signature 是 (prim: Usd.Prim, timeRange: UsdValidation.TimeRange) -> list[ValidationError]。Prim validators receive a Usd.Prim and timeRange.

这些 callable 必须返回 UsdValidation.ValidationError 对象的 list 或 iterable；验证通过时返回空 list。不要返回字符串、布尔值或直接抛异常来表达普通验证失败。Return an empty list when the validation passes.

Python Explicit and Plugin Registration Examples

Layer Validator

Layer 示例用 UsdValidation.ValidatorMetadata 注册 myPackage:RequiresDefaultPrim，检查 layer.defaultPrim。错误用 UsdValidation.ValidationErrorType.Warn，site 指向 Sdf.Path.absoluteRootPath。Layer Validator checks whether a layer has a defaultPrim.

from pxr import Sdf, UsdValidation
registry = UsdValidation.ValidationRegistry()
metadata = UsdValidation.ValidatorMetadata(
    name="myPackage:RequiresDefaultPrim",
    doc="Warn when a layer has no default prim set.",
    keywords=["myPackage"],
)
def _CheckDefaultPrim(layer):
    if not layer.defaultPrim:
        return [UsdValidation.ValidationError(
            "MissingDefaultPrim",
            UsdValidation.ValidationErrorType.Warn,
            [UsdValidation.ValidationErrorSite(layer, Sdf.Path.absoluteRootPath)],
            f"Layer '{layer.identifier}' has no defaultPrim.")]
    return []
registry.RegisterLayerValidator(metadata, _CheckDefaultPrim)

Stage Validator

Stage 示例注册 myPackage:RequiresUpAxis，检查 stage.HasAuthoredMetadata(UsdGeom.Tokens.upAxis)。如果缺失，返回 MissingUpAxis 的 Error。它适合 stage metadata 级别检查，不应改写为 prim validator。Stage Validator checks upAxis metadata.

Prim Validator

Prim 示例注册 myPackage:NoPrimsMissingKind，跳过 pseudo-root，用 Usd.ModelAPI(prim).GetKind() 检查每个 prim 是否有 kind。它展示了 prim-level task 如何对每个候选 prim 产生自己的 ValidationErrorSite。Prim Validator checks kind on prims.

Plugin Registration Example

Python plugin 场景中，plugInfo.json 声明 CheckUpAxis 的 doc 和 keywords，实现代码只用 RegisterPluginStageValidator("myPlugin:CheckUpAxis", _CheckUpAxis) 绑定 task function。Plugin validator suites 也类似，可用 RegisterPluginValidatorSuite 加上 GetOrLoadValidatorByName("myPlugin:CheckUpAxis") 注册。When a validator is declared in plugInfo.json, only the name is needed at registration time.

Python Plugin Loading, Running, Suites, and Notes

Python plugin 的 lazy-load flow 是：startup 阶段 ValidationRegistry 解析所有发现插件的 plugInfo.json，metadata 先可见；query 阶段客户端调用 registry.GetOrLoadValidatorByName("myPlugin:CheckUpAxis") 或通过 UsdValidationContext 访问；load 阶段 registry 调用 plugin->Load()，Python-type plugin 会执行 import <module_name>；register 阶段模块的 __init__.py 顶层代码调用 RegisterPluginStageValidator 等函数；最后 registry 返回 fully registered validator。Python plugins rely on module-level registration code in __init__.py instead of TF_REGISTRY_FUNCTION.

插件目录必须让 module directory name 匹配 plugInfo.json 的 Name 字段，目录内有 __init__.py 和 plugInfo.json。该目录要能被 Plug.Registry 发现，可通过 PXR_PLUGINPATH_NAME 或 Plug.Registry().RegisterPlugins()，而 module 的 parent directory 还必须在 sys.path 上，否则 import 会失败。The module directory name must match the Name field in plugInfo.json.

运行 Python validator 时，可以 validator = registry.GetOrLoadValidatorByName("myPackage:RequiresUpAxis") 后直接 validator.Validate(stage)，也可以构造 UsdValidation.ValidationContext([validator]) 后 context.Validate(stage)。suite 分组使用 ValidatorMetadata(isSuite=True)，再调用 RegisterValidatorSuite 把 stage validator 和 prim validator 组合成 baseline checks。Retrieve the registered validator by name and call Validate(), or pass it to a ValidationContext.

Notes 段有几个排错要点：ValidationRegistry 是 singleton，同一进程内注册的 validators 对其他模块可见；validator names 必须唯一；re-registering an existing name will fail silently，可用 HasValidator() 预检；Python exceptions raised inside a task function are converted to Tf errors and the validator returns an empty error list for that invocation；显式注册没有 plugin，不会 lazy loaded，必须每个 session 重新注册。Validator names must be unique across the registry.

官方最后把 usdchecker 和 pxr/usdValidation 下的 schema validators 作为 additional examples。读者如果要落地 pipeline，应先从这些示例确认错误报告格式、validator suite 组织方式和插件 metadata，而不是从零设计一套平行系统。The code for usdchecker has been updated to use validators.

边界、误读点与调试路径 / Boundaries and Debugging

  • 不要把 validator 当成自动修复器。validator 返回 UsdValidationError，fixer 需要调用方明确调用 CanApplyFix() 和 ApplyFix()，并提供 UsdEditTarget。Validation tests will not automatically call any fixers.
  • 不要把所有检查都写成 stage task。若规则只需看 prim，优先实现 UsdValidatePrimTaskFn 并设置 schemaTypes；否则会造成不必要 stage traversal。Using too broad a granularity can impact performance.
  • 不要把 Python validator 误认为能像 C++ 一样并行扩展。Python task 要拿 Python GIL，可能让同一 TBB worker pool 中的 C++ validators 也排队。Python validators do not benefit from parallelism among themselves.
  • 不要把 plugin metadata 和 runtime registration 混淆。plugInfo.json 可让 metadata startup 可见，但 task function 仍要在插件加载时通过 C++ TF_REGISTRY_FUNCTION 或 Python __init__.py 注册。Validator metadata is available immediately before code is loaded.
  • 若 validator 没有被发现，先检查 PXR_PLUGINPATH_NAME、Plug.Registry().RegisterPlugins()、sys.path、plugin Name 与目录名是否一致，再检查 GetOrLoadValidatorByName 的 name 是否带正确 plugin prefix。Plugin directory and module import paths must be discoverable.

发布管线和工具集成建议 / Pipeline and Tool Integration Guidance

在资产发布管线里，本页的最佳使用方式是把 Validation 分成三层：第一层是 validator discovery，用 UsdValidationRegistry 和 metadata query 找到项目需要的 validators 或 suites；第二层是 validation execution，用 UsdValidationContext 在 stage、layer 或 prim 范围内运行；第三层是 report and remediation，把 UsdValidationError 的 severity、site、message、validator 和 fixer 信息写入发布报告。只有第三层经过人工或上层工具确认后，才应调用 fixer 修改目标 layer。A practical validation pipeline separates discovery, execution, and remediation.

对工具作者来说，keywords 和 schemaTypes 是构建 UI 的关键。工具可以先列出所有带有某个 keyword 的 validators，让用户选择与当前任务相关的检查；也可以根据选中 prim 的 schema 自动过滤只适用于 UsdGeomImageable、UsdGeomGprim 或特定派生类型的 rules。这样既能避免全量检查的噪声，也能让错误报告更贴近用户正在编辑的资产区域。Keywords and schemaTypes are useful for validator selection UI.

对插件部署来说，plugInfo.json 的 metadata 是契约，不只是文档。若团队希望 command-line 工具、DCC 插件或 CI 服务在加载 validator 代码之前就能显示规则说明、关键字和 suite 组成，应选择 plugin registration。若规则来自用户脚本、临时测试或交互式工具状态，explicit registration 更合适，因为它不需要插件目录和 discovery 环境，但也不能指望其他进程或下一次 session 自动发现它。plugInfo.json metadata lets tools discover validators before loading implementation code.

对 Python validators 来说，调试重点不只在函数体。若 validator 没有运行，先检查 plugin 是否被发现、module 是否能 import、__init__.py 是否真的执行 registration、validator name 是否与 plugInfo.json 中的 pluginName:validatorName 一致；若 validator 运行但没有错误返回，再检查 callable 是否返回 list[ValidationError]，以及异常是否已被转换成 Tf errors 并导致本次 invocation 返回空列表。Python plugin debugging should check discovery, import, registration, naming, return type, and exceptions.

对性能敏感的批量资产检查，建议把高频、全场景、可并行的规则写成 C++ validators，把低频、项目特定或仍在原型阶段的规则写成 Python validators。运行混合 validator suite 时，要留意 Python GIL 可能占用 TBB worker thread pool，导致 C++ validators 也被延迟。若 CI 中出现耗时异常，应先按 validator name 和 implementation language 分组统计运行时间，而不是只看总耗时。Performance-sensitive suites should separate high-frequency C++ validators from low-frequency Python validators.

对错误归因来说，Error sites 比错误文本更稳定。一个 stage-level validator 可能报告 layer 或 stage 级别问题，一个 prim-level validator 可能报告具体 prim path，一个 property-level 诊断可能落到属性 site。发布报告应保留这些 site，不要只把 message 拼成一行字符串；否则用户很难从报告点击回本地页面、source layer 或编辑目标。Error sites are a durable bridge from validation reports back to authored data.

在 CI 中落地时，推荐把 validator suite 名、validator 名、error type、error site 和 fixer availability 都作为结构化字段保存。这样同一个资产在不同提交之间可以比较新增、消失和持续存在的问题，也能区分 warning policy 的变化与真正的 scene data 退化。若只保存控制台文本，后续很难把 Warn 升级为失败门槛，也很难把同一错误稳定映射回具体 layer 或 prim。Structured validation output makes CI regressions easier to compare.

在人工 review 中，Validation 页面应帮助读者判断“规则在哪里定义、何时加载、检查谁、报告到哪里、是否可修”。这五个问题分别对应 metadata/registration、plugin loading、task function granularity、error site 和 fixer。若某个验证结果难以解释，通常不是单一 API 调用错误，而是这些层次中的某一层没有被记录清楚。A useful review path follows definition, loading, target, reporting, and fixing.

最终验收时还要确认报告能从错误回跳到本地文档和官方页，避免验证结论与读者点击路径脱节。Validation reports should remain connected to local and official documentation.
