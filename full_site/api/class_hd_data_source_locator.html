逐段双语理解 / Paragraph-Level Bilingual Coverage

`HdDataSourceLocator` 表示一个能够识别 Hydra data source 位置的对象。官方定义说 Data Source Locators 是 short lists of tokens；这些 `TfToken` 片段合在一起，表示某个给定 data source 的位置。这里的 locator 更接近 Hydra scene index / data source namespace 中的 tokenized path，不是文件系统路径，也不是 `SdfPath` 或 USD prim path。

`HdDataSourceLocator` represents an object that can identify the location of a data source. Data Source Locators are short lists of tokens that together represent a data source location.

保留 `TfToken` 原名很重要：locator 的每个 path element 都是 token，适合快速比较、hash 和复用。把 locator 降级成普通字符串会丢失它在 Hydra data source 缓存、dirty propagation 和 scene index invalidation 中的性能与语义优势。`GetString()` 适合调试输出，而不是业务逻辑里的主表示。

The token list representation matters for comparison, hashing, and efficient dirty-range checks; `GetString()` is primarily for diagnostics.

多个 `HdDataSourceLocator()` 构造函数提供从不同 token 输入构造 locator 的入口。阅读这些构造函数时，重点不是“字符串格式是什么”，而是调用方要在 Hydra data source 命名空间中构造一个稳定、可比较的路径，例如定位 primvars、xform、material、render settings 或嵌套 container data source 下的字段。

The constructor overloads build stable token-path locators that identify positions inside nested Hydra data sources.

`Append()` 和 `Prepend()` 用于组合 locator 片段。`Append()` 通常把当前 locator 扩展到更深的子数据位置，`Prepend()` 则把局部字段挂到更外层的 namespace 下。它们表达的是 Hydra data source namespace 内的 token path composition，不应误译成文件路径拼接或 USD namespace edit。

`Append()` and `Prepend()` compose locator fragments within the Hydra data-source namespace, not filesystem paths or USD namespace edits.

`GetElementCount()`、`GetElement()`、`GetFirstElement()` 和 `GetLastElement()` 说明 locator 是有序 token 列表。调试 locator 不匹配时，应逐个检查 token element，而不是只比较最终字符串；很多 dirty propagation 问题本质上是某一段 token 命名或嵌套层级不一致。

Element accessors expose the ordered token path so mismatched locator components can be diagnosed precisely.

`HasPrefix()` 和 `GetCommonPrefix()` 是范围判断的核心。Hydra 的变化通知经常需要判断“这个被 dirty 的位置是否覆盖另一个 data source 子树”；prefix 关系可以表达根到子节点的包含关系。例如一个 broader locator 可能表示整个 prim data source 失效，而一个更深 locator 只表示某个 nested field 改变。

`HasPrefix()` and `GetCommonPrefix()` support containment checks between broad invalidation ranges and specific nested data-source fields.

`Intersects()` 用于判断两个 locator 范围是否可能相互影响。它常出现在 scene index 更新、filtering、dirty bit 或 data source invalidation 逻辑中：如果两个 locator 不相交，调用方可以跳过昂贵的更新；如果相交，则需要进一步拉取或重建相关 data source。

`Intersects()` answers whether two locator ranges overlap enough that a data-source update may need to propagate.

`EmptyLocator()` 和 `IsEmpty()` 要结合调用场景理解。空 locator 可能表示没有具体子路径，也可能在变化通知中代表 root-level 或 broad invalidation；它不等于“无事发生”。处理空 locator 时应检查调用方协议，尤其是 scene index observer、data source filtering 或 invalidation fan-out 代码。

`EmptyLocator()` and `IsEmpty()` can represent root-level or broad invalidation depending on the caller protocol.

`operator==()`、`operator!=()`、`operator<()` 和 `Hash()` 让 locator 可以进入 ordered 或 hashed 容器，用于缓存、去重和更新集合。比较语义应基于 token 序列本身；如果先转成字符串再比较，既可能浪费性能，也可能让调试代码和真实运行路径出现差异。

Comparison operators and `Hash()` make locators usable as keys while preserving token-path semantics.

常见误读包括：把 `HdDataSourceLocator` 当成 `SdfPath`；把 `Append()` / `Prepend()` 当成 USD namespace operation；把空 locator 当成可以忽略的空值；只用 `GetString()` 做核心比较；以及忽略 prefix/intersection 语义导致 dirty propagation 过宽或过窄。正确理解是：这是 Hydra data source 和 scene index 更新系统里的轻量 token-path key。

Common misreadings include treating the locator as an `SdfPath`, a filesystem path, or a string-only key. It is a lightweight token-path key for Hydra data-source locations.
