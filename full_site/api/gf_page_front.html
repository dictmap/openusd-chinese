中文主阅读路径

Gf 是 Graphics Foundations 模块：官方说明把它定位为一组 foundation classes and functions，用来处理图形学里最基础的数学方面。读这页时不要把 Gf 当成场景描述层；它不负责 author prim、schema、layer、relationship 或 composition，而是给上层的 UsdGeom、Sdf、Hydra 渲染、物理或工具代码提供可复用的数学值类型和基础运算。Gf contains foundation classes and functions for working with the basic mathematical aspects of graphics.

这页的核心结论是：Gf 解决“怎么表达和计算向量、矩阵、旋转、区间、范围、包围盒、射线、平面、视锥、相机模型”等问题；UsdGeom 再把其中一些数学概念绑定到 USD prim、xform op、extent、camera 或 boundable schema 上；Sdf/Usd 管理 scene description 和 composition；Vt 负责通用值和数组容器。把这些层次分清，调试坐标、包围盒、相机、渲染裁剪和仿真输入时会少很多误判。Chinese guidance: treat Gf as the math/value foundation, not as the scene description or schema authoring layer.

Linear Algebra：向量、矩阵、旋转和复合变换

Linear Algebra 是 Gf 最常被上层模块依赖的一组类型。官方说明提到 vectors of varying numbers and types of components，包括 two, three, and four float components；在实际 API 中你会看到 GfVec*、GfMatrix*、GfQuat*、GfRotation、GfTransform、GfDualQuatf 等类型。它们表达数学对象本身，不自动带 USD prim 路径、time sample 或 xform op 语义。Linear Algebra includes basic types such as vectors, matrices, quaternions, rotations, transforms, and compound linear transformations.

常见误读是把 GfMatrix4d 或 GfRotation 的数值问题直接归因到 UsdGeomXformable。更稳的调试顺序是：先确认 Gf 数学值的维度、精度、单位和组合顺序；再确认 UsdGeom 的 xform op stack 是否按预期 author；最后再看渲染器、物理或工具是否以同一坐标约定消费结果。Debug math values first, then schema authoring, then downstream consumers.

  • GfVec2*/GfVec3*/GfVec4* 关注分量数量与标量类型；不要把向量类型本身等同于位置、法线、颜色或速度的 schema 语义。Vector classes carry components; schema context decides whether those components are positions, normals, colors, or other values.
  • GfMatrix* 与 GfTransform 适合检查变换组合结果；具体的 USD xform op 顺序仍要回到 UsdGeomXformable 和 xform stack。Matrix and transform classes are math carriers; USD xform authoring is handled by UsdGeom.
  • GfQuat*、GfRotation 和 GfDualQuatf 常出现在旋转或骨骼/插值相关路径里；调试时要同时检查归一化、插值方式和坐标系约定。Quaternion-like values require attention to normalization, interpolation, and coordinate conventions.

Basic Mathematical Operations：基础数学运算

官方 Basic Mathematical Operations 覆盖 radians and degrees 的转换、square roots、minimum and maximum values、absolute values 等基础操作。中文阅读时可以把这一组理解成 Gf 为图形学代码提供的轻量工具层：它服务于数值计算和几何判断，但不决定 USD 属性值应该写到哪里，也不替代 schema validation。Basic Mathematical Operations include conversion between radians and degrees, square roots, min/max, and absolute values.

这类函数的风险通常不是“找不到 API”，而是单位、精度和边界条件不一致。例如角度/弧度混用会让相机或旋转看起来像 scene composition 出错；float/double 选择不一致会让大坐标场景下的 bound、ray 或 camera framing 表现不稳定；min/max 或 abs 只修正数值，不会修正上层语义错误。Most failures here are unit, precision, or boundary-condition issues rather than scene graph failures.

Basic Geometry：范围、包围盒、平面、射线、视锥和相机

Basic Geometry 是 Gf 连接上层可见结果的关键分组。官方提到 one-to-three dimensional floating point ranges、three-dimensional bounding boxes、planes、rays、frusta，以及 camera model。对应阅读路径里应重点关注 GfRange*、GfBBox3d、GfPlane、GfRay、GfFrustum、GfCamera 等类型。Basic Geometry includes ranges, 3D bounding boxes, planes, rays, frusta, and a camera model.

边界要讲清楚：GfBBox3d 是数学包围盒表达，UsdGeomBoundable、extent、purpose、visibility 和 Hydra culling 才决定场景里的包围盒如何被 author、组合和消费；GfRay 可用于拾取或几何求交的数学表示，但具体 pick hit、selection outline 或 viewport 行为由工具/渲染层决定；GfCamera 表达相机模型，USD 场景中的相机 prim 和属性 authoring 仍在 UsdGeomCamera。Gf geometry classes are mathematical representations; UsdGeom and rendering or tooling layers define authoring and consumption semantics.

  • 范围类（如 GfRange1d）适合检查数值区间是否为空、是否扩展、是否包含某个值；不要把它们误认为 USD collection 或 layer range。Range classes describe numeric intervals, not USD collections or layer ranges.
  • GfBBox3d 调试时要看空间、变换矩阵和是否已经应用父级变换；仅比较局部数值很容易误判 bound 问题。Bounding-box debugging should account for space, transforms, and parent composition.
  • GfFrustum 和 GfCamera 常用于 framing、裁剪和 view 相关逻辑；当画面裁掉或比例异常时，要同时检查 Gf 视锥、UsdGeomCamera 属性、UsdRender aspect/crop 设置和渲染器实现。Frustum and camera issues often involve both Gf math and higher-level camera or render settings.

Output For Debugging：调试输出

官方把 Output For Debugging 作为独立分组，是因为 Gf 类型经常处在“数学看似正确、上层显示不正确”的排查链条中。快速输出向量、矩阵、范围、包围盒、射线或相机相关值，可以帮助判断问题到底在 Gf 数学值、USD authoring、scene composition、Hydra/renderer 消费，还是仿真系统的输入解释。Output For Debugging includes functions for quickly outputting Gf types for diagnostic purposes.

实用调试路径可以按四步走：第一，打印 Gf 值并确认单位、维度、精度和坐标系；第二，回到 Sdf/Usd 确认属性是否按预期 author 和 composition；第三，用 UsdGeom 检查 xform、extent、camera 或 boundable 语义；第四，再看 Hd/Hdx、UsdRender 或 UsdPhysics 等消费层是否有额外约束。A practical diagnostic chain is math value, authored USD data, schema interpretation, then downstream consumer behavior.

相邻模块关系与边界

Vt 与 Gf 经常一起出现：Gf 给出数学值类型，VtValue 和 VtArray 则用于在 USD 值系统里承载标量或数组。不要把 VtArray<GfVec3f> 当成自动具备 primvar、point、normal 或 velocity 语义；这些语义来自属性名、schema 和读写上下文。Vt can carry Gf values, but schema context supplies meaning.

Sdf/Usd 负责层、路径、属性、composition 和 stage 访问；Gf 不知道 SdfPath、layer stack 或 prim index。把数学对象放入 USD 之前，必须先确定目标 schema、属性类型和 time sample 策略。Sdf and Usd manage scene description and composition; Gf supplies math values.

UsdGeom 是 Gf 最直接的上层消费者之一：transform、extent、bound、camera、point/normal 数据和几何工具都可能使用 Gf 类型。但如果显示结果不对，不应只改 Gf 数值；还要检查 xform op 顺序、purpose/visibility、extent authoring、interpolation 和 renderer 侧处理。UsdGeom consumes Gf-like values but defines scene-level geometry semantics.

Hd/Hdx、UsdRender 和仿真系统会继续消费几何、相机、矩阵和范围信息。Gf 页面能帮助定位底层数学值是否合理，但不承诺渲染器、viewer 或 simulator 一定按同一方式解释每个值。Rendering and simulation consumers may impose additional interpretation or implementation constraints.

API 分组怎么读

从 API 分组角度看，Gf 页面不是让读者逐个背类名，而是先建立一张“数学能力地图”。向量和矩阵类负责线性代数，范围和包围盒类负责空间/区间判断，射线、平面和视锥负责几何查询，相机模型负责把视图参数组织成可计算对象，调试输出函数负责把这些值快速暴露出来。遇到具体类页时，先判断它属于哪一组，再判断它是纯数学值、变换辅助、几何查询对象还是调试工具。Read the Gf API as a map of math capabilities: linear algebra, numeric utilities, geometry helpers, camera/frustum support, and diagnostic output.

例如 GfVec2i、GfMatrix2f、GfMatrix4f、GfDualQuatf、GfRange1d、GfRay 这些已经完成的本地 class 页，可以作为继续阅读的样本。它们各自解决的不是“USD 文件如何组织”，而是“一个数值对象如何被表示、组合、比较或输出”。这也是为什么 Gf 往往出现在低层工具代码里，而不是作为用户直接 author 的 schema 入口。Concrete class pages illustrate value representation and math operations rather than layer or prim authoring.

对中文读者来说，最实用的检查方式是把 Gf 类型与上层语义拆开：GfVec3f 可以承载点、方向、颜色或任意三元数值；只有当它被放进 UsdGeom 的 point/normal、primvar、camera 属性或其他 schema 字段时，才获得具体含义。GfMatrix4d 可以表达变换结果，但 xform op 的 authoring、重置栈、继承和 time sample 仍属于 UsdGeom/Usd 的问题。Meaning is supplied by the higher-level schema and authored attribute, not by the Gf value type alone.

跨层调试路径

调试坐标和几何问题时，建议把问题分成五层。第一层是 Gf 数值层：分量数量是否正确、角度是否用 radians 或 degrees、矩阵组合顺序是否符合预期、四元数是否归一化、范围是否为空。第二层是容器和值系统：VtValue 或 VtArray 是否携带了期望的 Gf 类型，数组长度和元素类型是否与属性声明匹配。第三层是 scene description：Sdf 路径、属性声明、layer opinion、time sample 和 composition 是否把这些值放到了正确位置。A useful debugging path starts with Gf values, then Vt containers, then Sdf/Usd scene description.

第四层是 schema 解释：UsdGeom 如何把数值解释为 xform、extent、bound、camera、points 或 normals；UsdRender 如何从 camera、aspect ratio、crop/tiling/overscan 读取设置；UsdPhysics 或其他仿真系统是否把变换、包围盒或方向值用于碰撞和关节计算。第五层才是消费端实现：Hydra delegate、viewer、renderer 或 simulator 可能有自己的缓存、裁剪、精度和坐标假设。Schema interpretation and downstream consumer behavior are separate from the foundational Gf math layer.

如果一个模型“看不见”，不要只盯着渲染器：先输出 GfBBox3d 或 extent，确认 bound 是否为空或落在错误空间；再检查 UsdGeom 的 visibility、purpose 和 transform；最后看 Hydra culling、camera frustum 和 render settings。如果拾取不准，先检查 GfRay 的 origin/direction 和坐标空间；再看 viewport 到 world 的转换；最后才检查 selection 或 hit decoding。Visibility and picking failures often require checking Gf math values before the renderer or viewer layer.

从 authoring 到 consumer 的边界案例

Gf 页面尤其适合解释“值正确但结果错误”的情况。比如一个 GfVec3f 数组本身可能完全合法，但如果它被写入 points、normals、velocities 或 primvars 时 interpolation、element size、time sample 或属性名不匹配，最终几何或渲染仍然会错。此时 Gf 只能证明底层三元数值存在，不能证明 USD 属性 authoring 正确。Valid Gf values do not guarantee correct authored USD attributes.

再比如 GfCamera 或 GfFrustum 数值可以解释视图和裁剪，但最终画面还要看 UsdGeomCamera、UsdRenderSettings、UsdRenderProduct、render delegate 和 viewer 的约定。一个画面比例或裁剪问题，可能来自 Gf 的视锥参数，也可能来自 aspectRatioConformPolicy、data window、分辨率设置或渲染器实现；所以调试时要沿着本页侧栏进入 UsdRender 和相关 user guide，而不是只停留在 Gf。Camera and frustum issues can involve both Gf math and render settings.

物理和仿真路径也类似。碰撞形状、关节轴、质心、包围盒和变换都可能用到 Gf 数学值，但 UsdPhysics 的 schema、physics scene、collision filtering、mass/material 设置和模拟器后端才决定最终行为。Gf 帮你检查输入向量、矩阵、范围或 bound 是否合理；它不替代 physics schema，也不承诺不同 backend 的求解策略一致。Physics consumers may use Gf values but define their own schema and backend behavior.

因此本页的实用价值不是给出每个类的完整手册，而是建立一条可靠的归因链：数学对象是否正确；值是否通过 Vt 和 USD 属性系统正确承载；schema 是否按预期解释；下游消费端是否有额外约束。只要按这条链排查，就能避免把低层数学、场景描述、几何 schema、渲染和仿真问题混在一起。Use Gf as the first link in a value-to-scene-to-consumer attribution chain.

精度、单位和坐标约定

Gf 里的许多类型都有 float、double、half 或整数变体。选择 GfVec3f 还是 GfVec3d，选择 GfMatrix4f 还是 double 精度矩阵，通常取决于数据规模、误差容忍度和消费端约定。大场景坐标、相机远裁剪面、很小的几何细节、物理求解或多次变换组合，都可能暴露精度问题；这类问题不能简单归因于 USD 文件坏了。Precision choices affect large coordinates, small details, repeated transforms, camera ranges, and simulation input.

单位也是跨层问题。Gf 提供角度/弧度转换和基础数值运算，但它不会替你判断某个 API 期望的是 degrees 还是 radians，也不会替你判断场景单位、线速度、角速度或相机参数是否跟工具一致。调试时要把“数学单位”和“schema 文档约定”分开读：数学函数只保证转换规则，上层 schema 才说明属性语义。Gf math utilities do not replace schema-level unit and semantic documentation.

坐标约定同样如此。向量、矩阵、平面、射线和视锥本身只是数学对象；它们处在 local space、world space、camera space 还是 renderer-specific space，需要由调用方和上层模块决定。只要空间混了，即使 Gf 值本身格式正确，拾取、包围盒、相机 framing、裁剪、碰撞和 debug overlay 仍可能全部错位。Coordinate-space mismatches can make otherwise valid Gf values appear wrong downstream.

定位到具体类页时，可以按问题类型选入口：变换和姿态先看 matrix、rotation、quaternion、dual quaternion；几何查询先看 ray、plane、range、bbox、frustum；相机和视图先看 camera/frustum，再转到 UsdGeomCamera 和 UsdRender；基础数值异常先看角度转换、min/max、abs、sqrt 等工具函数。这样的分类比从 Doxygen 列表从头翻到尾更适合中文读者，也能减少把数学类型误当 schema 的风险。Choose concrete class pages by problem category rather than scanning the Doxygen list linearly.

常见误读与调试清单

  • 误读一：看到 GfVec3f 就认为它一定是位置。正确做法是检查它所在的属性、schema、interpolation 和文档上下文。A vector type alone does not define semantic meaning.
  • 误读二：矩阵结果异常就认定 USD composition 错。先确认矩阵乘法顺序、坐标系、单位、行列约定和精度，再回到 UsdGeom 的 xform stack。Matrix issues should be debugged before assuming composition failures.
  • 误读三：包围盒、视锥或射线不符合预期就直接归咎于渲染器。先检查 GfBBox3d、GfFrustum、GfRay、GfCamera 的输入值，再看上层 schema 和 viewer。Bounding-box, frustum, ray, and camera failures usually cross multiple layers.
  • 误读四：把 Gf 的调试输出当成最终事实。它只说明数学对象在某个时刻的值；最终视觉或仿真结果还取决于 scene graph、composition、render delegate 或 physics backend。Debug output reports values, not final scene behavior.
