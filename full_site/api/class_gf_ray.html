逐段双语理解 / Paragraph-Level Bilingual Coverage

`GfRay` 是 Gf Basic Geometry 中用于 intersection testing 的三维射线值类型。它不是 USD scene graph 中的 prim，也不是可写入 stage 的 attribute，而是一个轻量数学对象，用 origin/start point 和 direction vector 描述从某个起点沿某个方向延伸的 ray。使用它时要先把坐标空间、方向长度和返回 distance 的单位讲清楚，否则 pick、裁剪、包围盒测试和最近点查询都会出现看似随机的偏差。

GfRay is a basic geometry value type used for three-dimensional ray intersection testing. It stores a start point and a direction vector rather than authored USD scene data.

官方描述中特别重要的一句是：默认情况下 `GfRay` 不会把 direction vector 归一化为 unit length。也就是说，`GetPoint()`、`Intersect()` 返回的 distance、以及调用方对“距离”的直觉，都可能随着 direction length 改变。若算法假定 distance 是真实世界单位，调用方应在构造 ray 前自行 normalize direction，或在读取结果时把 distance 与 direction length 一起解释。

The official note says that GfRay does not normalize its direction vector to unit length by default. Distance-like parameters therefore depend on the stored direction length.

ray intersection 还包含 start point：distance 为 0 时也定义为 intersecting。这一点在拾取和碰撞测试中很容易被忽略。如果射线从几何体表面、包围盒边界或裁剪平面上出发，调用方不应把 0 distance 当成“没有命中”，而应根据业务需要决定是否排除自相交、是否使用 epsilon、以及是否只接受正向 distance。

The start point is included in ray intersection computations, so a distance of zero is defined as an intersection.

构造和写入入口分成两类：`GfRay()` 与 `SetPointAndDirection()` 直接给出 start point 和 direction；`SetEnds()` 则从两个端点推导 direction。`SetEnds()` 适合从 segment-like 输入构造 ray，但它并不自动把对象变成有限 `GfLineSeg`。如果后续查询只想在两个端点之间有效，调用方仍要检查返回参数是否落在期望范围内。

SetPointAndDirection() sets a point and direction directly, while SetEnds() derives a ray from two points. A ray remains infinite unless the caller constrains the returned parameter range.

读取方法保持很薄：`GetStartPoint()` 返回起点，`GetDirection()` 返回方向，`GetPoint()` 根据参数返回 ray 上的点。这些 accessor 不会验证坐标系、不做单位换算，也不会自动修正零长度方向。因此在调试时，先打印 start point、direction length、调用空间和期望单位，通常比先怀疑 `Intersect()` overload 更有效。

GetStartPoint(), GetDirection(), and GetPoint() expose stored ray data and parameterized points without changing coordinate spaces or normalizing vectors.

`Intersect()` 有多组 overload，用于不同几何对象，例如 `GfPlane`、`GfRange3d`、`GfBBox3d` 以及球、盒、三角形等常见输入。阅读这些 overload 时要保留参数类型和 overload 编号，因为“与平面相交”“与 axis-aligned range 相交”“与 transformed bbox 相交”的返回约定和失败模式不同。排查命中异常时，先确认输入几何和 ray 是否在同一坐标空间，再看 distance 是否被错误裁掉。

Intersect() overloads operate on different geometry inputs. The input type, coordinate space, and distance convention must be read together.

`FindClosestPoint()` 与 friend functions `GfFindClosestPoints` 处理的是最近点或最短距离问题，不等同于“发生相交”。它们常用于 pick 辅助、测距、线/线段/射线关系判断，返回值需要结合 direction length、参数范围和调用方容差解释。把最近点查询当成相交查询，会让 UI selection 或几何诊断出现“靠近但不该命中”的误判。

FindClosestPoint() and GfFindClosestPoints are closest-point or distance queries, not a universal replacement for intersection tests.

`Transform()` 与 `GfMatrix4d` 配合使用时要区分 point 和 vector：start point 应接受完整 affine transform，包括 translation；direction 是向量，不能把 translation 当作方向偏移应用进去。若矩阵包含 non-uniform scale 或 shear，变换后的 direction length 会改变，之前依赖单位方向的 distance 解释也要重新校准。

Transform() must treat the start point as a point and the direction as a vector. Non-uniform scale or shear can change the direction length and therefore distance interpretation.

比较运算 `operator==()` 和 `operator!=()` 比较的是对象保存的值，而不是“几何意义上等价的射线”。例如两个 ray 可能有相同起点和共线方向，但 direction length 不同；它们在某些命中方向上等价，却不应被值比较视为相同。需要几何等价判断时，调用方应显式 normalize、比较方向夹角并检查起点。

Equality operators compare stored values. Geometric equivalence under scaled directions requires explicit caller-side logic.

相关类型的边界也很关键：`GfVec3d` 承载 point 和 direction 分量；`GfMatrix4d` 负责空间变换；`GfPlane`、`GfRange3d`、`GfBBox3d` 是常见 intersection target；`GfLine` 与 `GfLineSeg` 分别代表无限直线和有限线段。若需求是有限 segment 测试，不要只看 `GfRay` 的 start/end 构造方式，而要选择或补充相应的范围约束。

Related types such as GfVec3d, GfMatrix4d, GfPlane, GfRange3d, GfBBox3d, GfLine, and GfLineSeg define the geometry boundaries around GfRay.

常见问题可以按这条链路排查：先确认 ray 和目标几何是否在同一空间；再确认 direction 是否需要 unit length；然后检查 distance 0 是否被错误排除；接着区分 ray、line 和 line segment 的语义；最后再看 transform 是否把 translation、scale 或 shear 错误应用到 direction。这样能把大多数 pick miss、bbox miss、最近点异常和单位错配问题定位到调用约定，而不是误判为 OpenUSD 内部 bug。

A useful debugging path is coordinate space, direction normalization, zero-distance handling, ray versus line-segment semantics, and transform effects on direction length.
