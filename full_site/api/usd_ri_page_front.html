逐段双语理解 / Paragraph-Level Bilingual Coverage

UsdRi 是 OpenUSD API 中面向 RenderMan 互操作的工具模块。官方页标题是 UsdRi: USD RenderMan Utilities，正文只有一个 Overview，但这个 Overview 给出了非常关键的边界：如果读者要找“对应 RenderMan 概念和插件的 USD schemas”，应当看 usdRiPxr library；本页的 UsdRi 更偏向 authoring 工具和数据类型转换工具，不是完整的 RenderMan schema 手册。UsdRi is the OpenUSD module for RenderMan utility authoring and conversion, while RenderMan schemas and plugins are pointed to usdRiPxr.

官方说明中最重要的一句话是：本模块提供若干 utilities，用于 author 编码了 RenderMan-specific information 的 USD，并在 USD values/datatypes 与 Ri values/datatypes 之间转换。换句话说，UsdRi 的主语不是“渲染一张图”，而是“把 RenderMan 语义可靠地写入 USD，或在 USD 与 Ri 表示之间做清晰转换”。这决定了调试时要先检查 authored USD opinions、tokens、typed values 和 statements，而不是直接去找 Hydra delegate 或 RenderMan runtime 的渲染错误。UsdRi utilities help author RenderMan-specific USD and convert between USD and Ri values and datatypes.

官方还明确说本模块不包含 RenderMan headers。这条边界很实用：它说明 UsdRi 可以编译和使用，即使当前环境没有安装 RenderMan 或没有使用 RenderMan。中文读者不应把它理解成“依赖 RenderMan SDK 的运行时封装”，而应理解成 OpenUSD 侧的轻量互操作层：它让 USD 能记录 RenderMan 相关的 authoring 信息，并给工具链一个可检查、可转换、可发布的结构。UsdRi does not include RenderMan headers, so it remains useful even when RenderMan is not installed or used.

官方列出的 primary classes 中，UsdRiStatements 是核心入口。它提供 API 来编码大多数 RenderMan specific concepts，例如 Ri Attributes 和 scoped coordinate systems。这里的“Statements”不要误读成普通文字注释；它更接近可被 RenderMan 语义消费的结构化 authoring 层，负责把 RenderMan attribute、coordinate system 等概念放入 USD 中，让后续导出器、适配器或 RenderMan 相关工具能够找到这些语义。UsdRiStatements provides APIs for encoding RenderMan concepts such as Ri Attributes and scoped coordinate systems.

官方还指向 usdRi/rmanUtilities.h，并说明它提供 USD/Ri conversion utilities。这个头文件名必须保留英文，因为它是读者从 Doxygen、源码、构建日志或 IDE 中定位工具函数的线索。中文页要表达的是：如果问题发生在类型、token、数组、颜色、矩阵或其他 typed value 的 USD/Ri 表示转换上，应优先检查这里的转换工具和调用方约定，而不是把问题归咎于 UsdRender、UsdShade 或 RenderMan 渲染器本身。usdRi/rmanUtilities.h provides USD/Ri conversion utilities and should be inspected for representation mapping issues.

模块职责与边界 / Module Scope and Boundaries

  • UsdRi：OpenUSD 侧的 RenderMan utility layer。它关注如何 author RenderMan-specific USD information，以及如何在 USD 和 Ri datatypes 之间转换。UsdRi is the USD-side RenderMan utilities layer.
  • usdRiPxr：官方 Overview 指向的 RenderMan concepts/plugins schema library。若要查找具体 RenderMan schema 或 plugin 对应的 USD schema，应从这里继续，而不是把所有 schema 责任放到 UsdRi 模块入口页。usdRiPxr is where the official page points readers for RenderMan concepts and plugin schemas.
  • RenderMan runtime：实际渲染器、插件加载、shader 执行和图像输出属于运行时或 renderer integration。UsdRi 可以帮助描述和转换信息，但不等同于渲染后端。The RenderMan runtime is separate from the UsdRi authoring and conversion utilities.
  • UsdRender：负责通用 render settings、render products、render vars/AOVs 和渲染输出结构。它适合描述跨 renderer 的输出配置；UsdRi 则处理 RenderMan-specific statements 和 USD/Ri 转换。UsdRender describes general render settings and products; UsdRi handles RenderMan-specific statements and conversions.
  • UsdShade 与 UsdMtlx：负责材质网络、shader node、MaterialX 等表达。若 RenderMan specific shader identifier 或 adapter 需要参与转换，读者要分清材质拓扑来自 UsdShade/UsdMtlx，RenderMan 语义或转换帮助来自 UsdRi 或 usdRiPxr。UsdShade and UsdMtlx describe material networks; UsdRi handles RenderMan-specific utility concerns.
  • UsdLux：负责光源、light filters 和 lighting schema。RenderMan renderer 可能有专有光照扩展，但不要把通用光照 schema 和 RenderMan utility layer 混为一谈。UsdLux covers lighting schemas; RenderMan-specific extensions should be separated from generic light schema behavior.

这几层边界是阅读本页的核心。很多 RenderMan 集成问题看起来像“渲染器不认 USD”，实际可能是：USD 中没有正确 author UsdRiStatements；typed value 在 USD/Ri 表示之间没有正确转换；所需 schema 位于 usdRiPxr 而不是 UsdRi；或者通用 UsdRender/UsdShade/UsdLux 结构正确，但 RenderMan-specific adapter 没有消费对应信息。把这些层拆开，才能稳定定位问题。Debugging should separate authoring, conversion, schema location, renderer consumption, and generic render/material/light schemas.

本地中文页因此保留 UsdRi、usdRiPxr、UsdRiStatements、Ri Attributes、usdRi/rmanUtilities.h、RenderMan headers、RenderMan-specific information 等英文标识。它们不是待翻译普通词，而是 Doxygen、源码、构建输出、插件约定和官方文档中的稳定定位锚点。English identifiers are preserved because they are stable anchors in Doxygen, source, build output, and plugin conventions.

Authoring 与转换路径 / Authoring and Conversion Path

使用 UsdRi 时，第一步应确认当前目标是“写入 RenderMan-specific USD 信息”还是“转换 USD/Ri values and datatypes”。前者关注 composition 后的 USD opinions 是否存在、是否位于预期 prim 或 scope、是否符合 UsdRiStatements 的 API 约定；后者关注数值类型、token、数组、颜色、矩阵、坐标系统名或其他 typed value 在 USD 与 Ri 表示之间是否保持语义一致。UsdRi workflows usually split into authoring RenderMan-specific USD information and converting USD/Ri values and datatypes.

UsdRiStatements 的调试重点不是“有没有某个英文段落”，而是检查 API 是否把 RenderMan concepts 写到了合成后的 stage 中，并且下游工具能否在正确的 prim、namespace、scope 或 coordinate system 上找到这些 authored values。若一个 RenderMan attribute 没有生效，应先看 layer stack 中最终 opinion、namespace、scope、强弱顺序和是否被后续 layer 覆盖，再看 renderer 侧日志。UsdRiStatements debugging starts with composed USD opinions, namespace, scope, and layer strength before renderer logs.

usdRi/rmanUtilities.h 更适合处理“值长得对不对”的问题。比如同一个概念在 USD 中可能是 token、TfToken 数组、GfVec、GfMatrix 或 asset path，在 Ri 侧可能有不同的调用和类型习惯。转换工具存在的目的，是把这种跨 API 表示明确化，减少工具作者手写字符串或猜测类型的风险。若某个值转换后丢失精度、顺序、单位或 token 拼写，应优先回到转换函数和调用约定。rmanUtilities.h is the place to inspect representation mapping and conversion behavior.

由于官方强调没有引入 RenderMan headers，本页也要诚实说明：UsdRi 可以在没有 RenderMan SDK 的环境中参与 USD authoring 和检查，但这不代表它能验证所有 RenderMan runtime 行为。运行时插件是否存在、shader 是否可加载、renderer 是否支持特定 attribute 或 coordinate system，仍需要 RenderMan 或相关 adapter 的实测证据。The absence of RenderMan headers makes UsdRi portable, but runtime renderer behavior still needs runtime validation.

从发布角度看，UsdRi 适合放在“RenderMan 兼容信息的 USD 表达和转换”这一层。资产发布者可以在 USD 内保留通用 UsdRender 输出配置、UsdShade 材质结构、UsdLux 光照结构，同时额外 author RenderMan-specific statements。这样非 RenderMan 消费者仍能读取通用结构，RenderMan 相关工具则能读取专有扩展。Pipelines can keep generic render/material/light schemas while adding RenderMan-specific UsdRi statements.

常见误读与调试路径 / Misreads and Debugging

  • 不要把 UsdRi 误读成 RenderMan renderer。它不负责启动渲染、加载 renderer plugin、执行 shader 或生成图像；它负责 USD 侧的 RenderMan utility authoring 与 USD/Ri 转换。UsdRi is not the RenderMan renderer.
  • 不要把 usdRiPxr 与 UsdRi 混为一谈。官方直接指出 RenderMan concepts/plugins 对应 schema 应看 usdRiPxr；本页关注 utility module。RenderMan concept and plugin schemas are pointed to usdRiPxr, not the UsdRi module front page itself.
  • 不要把 RenderMan headers 依赖当成使用前提。官方说明没有包含这些 headers，因此本模块可以在没有 RenderMan 安装的环境里编译和用于 USD authoring 工具。RenderMan headers are not required by this module.
  • 不要把 Ri Attributes 翻译成无法核对的中文属性名。这里的 Ri 和 Attributes 是 RenderMan 语境下的稳定术语，必须保留英文以便和官方文档、代码和日志对应。Ri Attributes should remain a recognizable RenderMan term.
  • 不要把所有 RenderMan-specific 问题都推给 UsdRender。UsdRender 负责通用渲染输出描述，RenderMan statements 和 conversion utilities 属于 UsdRi/usdRiPxr 相关路径。Do not blame generic UsdRender settings for every RenderMan-specific authoring problem.

建议调试顺序是：先检查目标 layer stack 中是否真的 author 了预期 RenderMan-specific USD opinions；再检查 UsdRiStatements 是否写到了正确 prim、scope 或 coordinate system；接着检查 usdRi/rmanUtilities.h 相关转换是否保持了类型、token、顺序和单位；然后确认所需 schema 是否来自 usdRiPxr；最后再进入 RenderMan runtime、plugin availability、shader identifier、renderer-specific token 和 Hydra/Hd adapter 日志。Debug from authored USD opinions, UsdRiStatements scope, conversion utilities, usdRiPxr schemas, and only then runtime or Hydra adapter logs.

如果转换后的值看似正确但 RenderMan 不生效，应把问题拆成四列：USD composition 列，确认 opinion 是否在最终 stage 中；authoring API 列，确认 UsdRiStatements 或相关工具是否按约定写入；conversion 列，确认 USD/Ri datatype mapping 是否正确；consumer 列，确认 RenderMan 或 adapter 是否支持并读取该信息。只有这样才能避免在中文文档中把工具层、schema 层、runtime 层和 renderer behavior 混写成一个模糊概念。A practical diagnosis separates composition, authoring API, datatype conversion, and consumer support.

对于跨 renderer 管线，UsdRi 信息还应被视为可选的 renderer-specific enrichment。资产可以保留可被其他 renderer 读取的通用 USD 结构，同时通过 UsdRi 或 usdRiPxr 提供 RenderMan 的增强语义。这样做的好处是：不支持 RenderMan 扩展的工具仍能读懂核心场景；支持 RenderMan 的工具则能得到额外 statements 和转换语义。UsdRi can enrich a scene for RenderMan while preserving generic USD structures for other consumers.

实践工作流 / Practical Workflow

在实际资产管线中，可以把 UsdRi 的使用拆成三个阶段。第一阶段是 authoring：DCC、导出器或修复脚本把 RenderMan-specific statements 写入 USD，并确保这些 opinions 落在可预期的 layer、prim、scope 和 namespace 上。第二阶段是 conversion：工具通过 usdRi/rmanUtilities.h 或相邻 utility 把 USD typed values 转成 Ri 侧能够理解的表示，或反向把 Ri 侧信息稳定映射回 USD。第三阶段是 consumption：RenderMan 相关插件、delegate、导出器或验证工具读取这些 statements，并决定它们是否会影响最终渲染。A practical pipeline separates authoring, conversion, and consumption.

这三个阶段不能混在一起验收。authoring 阶段的合格标准是：最终合成 stage 中能看到正确的 UsdRiStatements 或等价 RenderMan-specific data，且 layer strength、variant、reference、payload、class/inherit 等 composition 机制没有覆盖或隐藏它们。conversion 阶段的合格标准是：USD/Ri datatype mapping 可重复、可测试，不依赖手写字符串拼接，也不会把 token、数组顺序、坐标系统名或数值单位悄悄改坏。consumption 阶段的合格标准才是 RenderMan 或 adapter 是否按预期读取并使用这些信息。Authoring, conversion, and consumption need separate acceptance checks.

如果一个镜头在非 RenderMan renderer 中仍要可用，建议把通用结构和 RenderMan 扩展分层 author。通用相机、输出、AOV、材质网络和光照结构继续放在 UsdRender、UsdShade、UsdMtlx、UsdLux 等模块负责的 schema 中；RenderMan 专有 attribute、coordinate system 或转换需求再通过 UsdRi 和 usdRiPxr 路径补充。这样做可以减少跨 renderer 迁移时的误伤，也能让中文读者判断哪些信息是 portable USD，哪些信息是 renderer-specific extension。Generic USD structures and RenderMan-specific enrichment should be authored in distinct layers.

发布前的检查表可以按顺序执行：先用 USD 工具打印目标 prim，确认 statements 和关系确实存在；再检查相关 layer 是否有更强 opinion 覆盖；然后对 conversion utilities 做小样本往返测试，确认 USD value 与 Ri value 的语义一致；接着在没有 RenderMan headers 的环境中确认 authoring 侧工具仍能构建或运行；最后在有 RenderMan 的环境中做 runtime smoke test。这样既尊重官方“无 RenderMan headers 依赖”的边界，也不会把运行时验证省略掉。A release checklist should cover composed USD, layer strength, conversion tests, header independence, and runtime smoke tests.

本页之所以把调试路径写得比官方 Overview 更长，是因为官方源页本身非常短，主要提供边界和入口词。中文完成页不能只翻译这几句，而要把读者带到可执行的判断路径：看到 UsdRi 时先想到 utilities；看到 usdRiPxr 时知道那里才是 RenderMan schema/plugin 方向；看到 UsdRiStatements 时检查 authored statements；看到 rmanUtilities.h 时检查 USD/Ri 转换；看到渲染差异时再把问题交给 adapter、Hydra/Hd/Hdx 或 RenderMan runtime 继续定位。The Chinese page expands the short official Overview into an actionable reading and debugging path.
