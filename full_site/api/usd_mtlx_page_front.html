逐段双语理解 / Paragraph-Level Bilingual Coverage

UsdMtlx 是 OpenUSD 中连接 MaterialX 与 USD shading 体系的模块入口。官方 Overview 说它提供 MaterialX file format、shader discovery 和 parsing plugin，并提供读取、交互 MaterialX document 的工具函数。中文阅读时可以把它理解为三件事：让 .mtlx 文档进入 USD 文件格式系统，让 MaterialX node definition 能通过 Sdr 被发现和查询，并把 MaterialX 的材质概念转换到 UsdShade 可表达的对象上。UsdMtlx provides the MaterialX file format, shader discovery, and parsing plugin for USD.

Overview section 的边界是：UsdMtlx 不等同于 MaterialX runtime，也不等同于 renderer 对 MaterialX 的最终执行。它关注的是 file format plugin、shader discovery/parsing plugin 和转换工具。也就是说，它负责把 MaterialX 文档、nodedef、implementation、material 等信息放到 USD/Sdr/UsdShade 能理解的位置，而不是承诺所有 renderer 都按同一种方式执行这些 shader。UsdMtlx is a file-format and discovery bridge, not a renderer runtime.

Concept Mappings 是本页最重要的 section。官方表格把 input 映射到 UsdShadeInput，把 output 映射到 UsdShadeOutput，把 node 与 nodedef 映射到 UsdShadeShader，把 nodegraph 映射到 UsdShadeNodeGraph，把 implementation 映射到 SdrShaderNode。这些英文名必须保留，因为它们既是 MaterialX 概念，也是 USD/Sdr 侧的技术核对点。Concept Mappings connects MaterialX concepts with UsdShade and Sdr types.

同一张映射表也说明几何、材质绑定和 look 的处理方式：geom 与 collection 走 UsdCollectionAPI，material 走 UsdShadeMaterial，materialassign 走 UsdShadeMaterialBindingAPI，variant/variantset/variantassign 对应 USD variants 和 variant selection。这说明 UsdMtlx 会借用 USD composition、collection 和 binding 机制表达 MaterialX 的若干高级概念。Geometry, collections, material binding, and variants map to USD collection, binding, and variant mechanisms.

官方说明还强调，shader discovery 和 parsing plugin 会记录 nodedefs 与 implementations 的信息；同时，file format plugin 也会翻译 nodedef。这样即使某些简单客户端不主动查 registered shader node，也能在 USD 表达里看到 shader 属性，并让 USD composition 使用 nodedef value 作为 fallback。这是理解 UsdMtlx 双通道设计的关键：Sdr registry 负责发现定义，file format conversion 负责让 USD layer 中有可组合的属性表达。Shader discovery records nodedef and implementation information, while the file-format plugin also translates nodedefs for USD composition fallbacks.

MaterialX looks 在 USD 侧会变成单个 USD prim 上的 variants；使用者通常通过 reference 这个 prim，并设置 variant selection 来选择需要的 look。MaterialX 中的 look inherits 与 material inherits 会分别映射到 references 与 specializes，而 nodedef inherits 在转换期间处理，不直接映射成 USD composition。这些规则能帮助读者判断问题属于 MaterialX 文档结构、USD composition，还是材质绑定选择。MaterialX looks become USD variants, while look and material inheritance use references and specializes.

自定义节点是另一个容易出错的点。local custom nodes 需要把关联的 nodedef/nodegraph 放在同一个 material mtlx 文件中，或被该文件 include；shared library custom nodes 通常需要单独 mtlx 文件，并通过 PXR_MTLX_PLUGIN_SEARCH_PATHS 指向包含 custom node nodedefs 的目录。使用纹理的库节点还需要正确暴露 filename typed input，让 renderer 能识别纹理引用。Custom MaterialX nodes require careful nodedef placement, search paths, and texture filename exposure.

Unsupported MaterialX Features section 是生产使用时必须读的边界清单。Geometry Name Expressions 不支持，因为 USD 目前不能表示 glob-like path expressions；light context shader types 不支持，因为 USD 中 light 是对象 schema，通常在 UsdLux 表达，而不是 material 上的 light type；custom type members、geominfo、geompropdef、propertyset、visibility 等元素会被忽略或不能完整转换。Unsupported MaterialX Features lists concepts that are ignored or cannot be represented faithfully in USD.

还要注意若干属性级限制：fileprefix 被忽略；image filename substitutions 目前除 <UDIM> token 外没有通用支持；width、height、bitdepth output attributes 被忽略；disable 会按值转换但在 USD 中没有特殊禁用含义，不会自动禁用 UsdShadeShader prim；enumvalues、uimin、uimax 等 UI 或枚举辅助信息也可能没有直接 USD 对应概念。Several MaterialX attributes are ignored or have no direct USD equivalent.

把本页串起来看，UsdMtlx 的主线是：读取 MaterialX document，把核心概念映射到 USD/UsdShade/Sdr，记录 shader definition 供 Sdr 查询，利用 USD composition 和 binding 表达 look、material、collection 与 variant，同时诚实暴露无法完整映射的 MaterialX 功能。读者不应只记“能读 mtlx”，而要知道哪些语义会被保留、转换、忽略或降级。UsdMtlx reads MaterialX, maps key concepts into USD/UsdShade/Sdr, and documents unsupported or degraded mappings.

实际排查时，可以把问题拆成四层：第一层是 MaterialX 文档是否可读、include 是否完整、自定义节点库是否在搜索路径内；第二层是 UsdMtlx file format 是否把 material、nodegraph、collection、variant 等结构转换成预期的 USD prim 和属性；第三层是 Sdr 是否能发现 nodedef/implementation 并返回正确的 shader node；第四层才是 renderer 是否支持这些定义。这样拆分能避免把所有失败都归咎于 MaterialX 或 USD。Debugging should separate document loading, file-format conversion, Sdr discovery, and renderer support.

在资产交换场景中，UsdMtlx 的价值不是把 MaterialX 每个细节逐字复制到 USD，而是选择 USD 能稳定表达的语义承载方式。例如 collection 路径会被转换并存储到单个 prim，geomprefix 会在转换时烘焙进结果，相同 collection 会共享同一个 UsdCollectionAPI。这些行为让 USD 侧的绑定和组合更可预测，但也意味着回写到原始 MaterialX 时不一定能完全保持同一结构。UsdMtlx maps MaterialX concepts into USD representations that may be stable but not structurally identical to the original document.

阅读 unsupported feature 清单时，重点不是记住所有被忽略的元素，而是建立判断规则：凡是依赖复杂路径表达式、几何绑定自动推断、应用专用 UI 元数据、或 MaterialX 隐含运行时语义的内容，都可能没有直接 USD 对应物。遇到这种情况，中文页应帮助读者确认“这是已知转换边界”，而不是让读者继续在本地链接、编码或翻译质量上误查。Unsupported features should be treated as known conversion boundaries rather than local documentation defects.

完成转换后的核对重点也应明确：USD 文件中是否有期望的 material prim，shader prim 是否带有来自 nodedef 的 fallback 值，collection 是否被统一存放，variant 是否表达了 look 选择，Sdr 是否还能查询到同一个 implementation。只有这些结构都对齐后，才适合继续检查渲染器和视觉结果。Converted USD should be checked for material prims, fallback values, collections, variants, Sdr implementation lookup, and then renderer output.

如果这些检查仍然不能定位问题，再回到官方 unsupported 清单逐项排除：路径表达式、灯光上下文、结构体成员、几何属性、属性集和可见性都可能在转换时被忽略或降级。把这些限制写进中文主阅读路径，是为了让读者知道哪些现象属于设计边界，而不是误判为本地中文站、Sdr 注册表或 UsdShade 网络本身出错。When conversion checks are inconclusive, compare the case against the official unsupported-feature list before blaming the local page, the Sdr registry, or the UsdShade network.

核心概念与 API 分组 / Core Concepts and API Groups

  • MaterialX file format：让 MaterialX 文档作为 USD 可识别的文件格式进入资产读取流程。关注点是文档读取、概念转换和 USD layer 表达。The file-format side reads MaterialX documents into USD-facing data.
  • shader discovery 与 parsing plugin：服务 Sdr registry，让 nodedefs 和 implementations 能变成可查询的 SdrShaderNode 与属性数据。Shader discovery and parsing feed Sdr node definitions.
  • UsdShadeInput、UsdShadeOutput、UsdShadeShader、UsdShadeNodeGraph、UsdShadeMaterial：MaterialX input/output/node/nodegraph/material 在 USD shading schema 中的主要落点。UsdShade types represent the USD-side material network.
  • UsdCollectionAPI 与 UsdShadeMaterialBindingAPI：处理 MaterialX geom、collection、materialassign 等绑定相关概念；读者要把“材质定义”与“材质绑定到哪些几何对象”分开。Collection and binding APIs represent assignment and geometry relationships.
  • PXR_MTLX_PLUGIN_SEARCH_PATHS 与 USDMTLX_PRIMARY_UV_NAME：source 中提到的环境变量。前者帮助定位 shared library custom nodes，后者影响 library custom nodes 使用纹理或纹理坐标时的 primary texture coordinate 名称。These environment variables affect custom-node discovery and primary UV naming.

边界、误读点与调试路径 / Boundaries, Misreads, and Debugging

  • 不要把 UsdMtlx 当成完整 MaterialX 应用运行时。它把 MaterialX 信息带入 USD/Sdr/UsdShade，但具体渲染语义仍取决于 renderer 和 shader implementation。UsdMtlx is not the full MaterialX application runtime.
  • 如果 shader definition 找不到，先检查 MaterialX 文件是否被找到、nodedef 是否被 include、shared library 路径是否在 PXR_MTLX_PLUGIN_SEARCH_PATHS 中、Sdr registry 是否能看到对应 SdrShaderNode。Missing definitions should be reduced to file inclusion, search paths, and Sdr registry results.
  • 如果材质绑定不符合预期，要分别检查 MaterialX look/materialassign、转换后的 UsdCollectionAPI、UsdShadeMaterialBindingAPI 和 variant selection，而不是只看 shader node 是否存在。Binding problems should be debugged separately from shader-node discovery.
  • 如果纹理路径异常，要检查 filename typed input 是否暴露、fileprefix 是否被忽略、是否只依赖了 USD 目前支持的 <UDIM> token，以及 primary UV 名称是否符合 USDMTLX_PRIMARY_UV_NAME 设置。Texture issues involve filename exposure, fileprefix, UDIM support, and primary UV naming.
  • 如果 unsupported feature 被忽略，不应误判为翻译或链接错误。Geometry Name Expressions、light context shader types、custom type members、propertyset、visibility 等限制是官方 source 中的功能边界，应在中文页如实说明。Unsupported features are official conversion boundaries, not local-site defects.

一个可靠的调试顺序是：先打开原始 .mtlx 确认 material 与 nodedef 的组织；再检查转换后的 USD 中是否有期望的 UsdShadeMaterial、UsdShadeShader、UsdShadeNodeGraph 与 UsdCollectionAPI；然后用 Sdr 查询实现节点和属性；最后再交给 renderer 验证视觉结果。每一层失败都应记录不同证据。A reliable debugging order is source document, converted USD, Sdr query, then renderer behavior.

如果用户只关心“材质为什么没有显示出来”，不要直接从 renderer 猜。先看 material binding 是否落到目标几何，variant selection 是否选中了正确 look，shader 属性是否来自 nodedef fallback，纹理 filename 是否仍可解析。这些检查比单纯重新导出 MaterialX 更容易定位真实断点。Material display failures should be reduced to binding, variants, fallback values, and texture resolution before renderer assumptions.
