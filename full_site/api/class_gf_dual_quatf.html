逐段双语理解 / Paragraph-Level Bilingual Coverage

官方定义把 `GfDualQuatf` 称为“Basic type: a real part quaternion and a dual part quaternion”。这里的重点不是它比 `GfQuatf` 多一个字段，而是它把一个 real quaternion 与一个 dual quaternion 组合成 dual quaternion，用来表达 rotation 与 translation 的耦合。它属于 `Gf` 的线性代数值类型，不是 USD scene prim、不是 `UsdAttribute`，也不会自己 author xform op。

The official definition describes `GfDualQuatf` as a basic type with a real part quaternion and a dual part quaternion, used to represent a combination of rotation and translation.

`ScalarType` 对本类固定为 `float`。这意味着它与 `GfDualQuatd`、`GfDualQuath` 是同一数学结构的不同精度变体：`GfDualQuatd` 更适合需要 double 精度的计算，`GfDualQuath` 更偏向紧凑表示，而 `GfDualQuatf` 是常用的 float 精度中间形态。跨精度转换时要注意插值误差、归一化误差和 `GF_MIN_VECTOR_LENGTH` 这类容差常量。

`ScalarType` is the scalar precision for this class. The related `GfDualQuatd` and `GfDualQuath` pages represent double and half precision variants.

本页列出的 7 个 `GfDualQuatf()` 构造函数说明调用者可以从不同输入形式创建 dual quaternion：默认构造、从 real/dual 两个 quaternion 构造、从 rotation 与 translation 相关值构造，或从其他精度变体转换。读构造函数时要先确认输入的数学约定：rotation 是否已经是单位 quaternion，translation 的单位和坐标系是否与项目一致，以及 real/dual 的组合是否代表期望的 rigid transform。

The constructor overloads allow construction from quaternion parts, rotation/translation-style inputs, and related precision variants; callers must preserve the mathematical convention of the input data.

`GetReal()` 与 `GetDual()` 是理解这个类的核心访问器。`GetReal()` 返回 real part quaternion，通常承载旋转；`GetDual()` 返回 dual part quaternion，它编码与平移相关的信息，但不能被当作普通 `GfVec3f` translation 直接使用。把 dual part 误读成平移向量，是调试 dual quaternion 结果错位时最常见的问题。

`GetReal()` returns the real quaternion and `GetDual()` returns the dual quaternion; the dual part is not a raw translation vector.

`GetTranslation()` 是从 dual quaternion 中提取 translation 的便利接口。它通常假定对象表示合理的 rigid transform，并且 real part 已满足旋转 quaternion 的有效性要求。如果先进行了 blend、加权累积或手工修改 real/dual part，应该先检查 `GetLength()`、`Normalize()` 或 `GetNormalized()`，否则提取出的 translation 可能只是非单位 dual quaternion 的数学副产物。

`GetTranslation()` extracts the translation represented by the dual quaternion; correctness depends on the dual quaternion representing a valid normalized rigid transform.

`GetLength()`、`Normalize()` 和 `GetNormalized()` 是本页的稳定性关键。`Normalize()` 修改当前对象，`GetNormalized()` 返回归一化副本；它们不是美化步骤，而是保证 dual quaternion 可解释为有效变换的重要前置条件。插值、累积或数值转换后，未归一化的对象可能会让 rotation 和 translation 同时漂移。

`Normalize()` mutates the object and `GetNormalized()` returns a normalized copy; normalization is essential after blending or accumulation.

`GetInverse()` 与 `GetConjugate()` 都是代数操作，但语义不同。`GetInverse()` 面向变换的逆操作，常用于撤销一个 rigid transform；`GetConjugate()` 是 dual quaternion 的共轭形式，常用于推导、求逆或实现内部计算。不要把 conjugate 简单等同于 inverse，尤其是在对象未归一化或不代表有效刚体变换时。

`GetInverse()` and `GetConjugate()` are related algebraic operations but are not interchangeable in all numerical states.

`GetIdentity()` 与 `GetZero()` 的边界也很重要。identity 表示无旋转、无平移的单位变换；zero 是零值代数构造，通常不应被当作有效 rigid transform。若代码把 zero 当作初始化后的“无变换”状态，后续 `Normalize()`、`GetTranslation()` 或插值结果都可能出错。

`GetIdentity()` represents the identity transform, while `GetZero()` is a zero-valued algebraic object and not a valid rigid transform.

与 matrix transform 相比，dual quaternion 的优势通常出现在刚体变换插值、骨架/蒙皮、避免简单矩阵线性插值带来的剪切或体积问题等场景。它不是 `GfMatrix4f` 的完整替代：matrix 可以表达更一般的 affine transform，例如 non-uniform scale 或 shear；dual quaternion 主要表达 rotation + translation 的刚体语义。

Compared with matrices, dual quaternions are especially useful for rigid transform interpolation and skinning, but they do not represent every affine transform that a matrix can represent.

The linked papers provide the mathematical background. English API names and mathematical terms are preserved so the page remains searchable against source code and Doxygen.

实际调试时可以按以下顺序检查：输入 rotation 是否单位化，translation 是否用正确坐标系写入，构造函数 overload 是否与数据含义匹配，blend 后是否调用 `Normalize()`，`GetLength()` 是否接近预期，`GetReal()` / `GetDual()` 是否被错误解释，最后再比较由 dual quaternion 转出的 translation 或 matrix 是否与 DCC/renderer 中看到的姿态一致。

A practical debugging path is to check normalized rotation input, coordinate-system conventions, constructor overload choice, post-blend normalization, length, real/dual part interpretation, and final extracted transform.

运算符小节也应按值类型来读：`operator==()`、`operator!=()` 比较的是 dual quaternion 两部分的数值状态，不代表两个场景对象、关节或 xform op 语义相同；`operator*` 这类组合运算体现变换组合顺序，顺序错误会导致旋转中心和平移方向都变化。使用它做 skinning 或动画 blend 时，应显式记录乘法顺序和归一化策略。

Operator documentation should be read as value-type algebra: equality compares stored components, while multiplication order affects the represented transform.

`GF_MIN_VECTOR_LENGTH` 链接提醒读者，Gf 数学类型常在接近零长度或退化输入时使用容差思路。对 dual quaternion 来说，接近零长度、real part 非单位、dual part 与 real part 不一致，都会让 `GetInverse()`、`GetTranslation()` 或归一化流程暴露数值问题。工程代码中不要只检查是否为零对象，还应检查是否满足有效 rigid transform 的前置条件。

The `GF_MIN_VECTOR_LENGTH` reference is a reminder that degeneracy and near-zero lengths need tolerance-aware handling in Gf math code.

本类和 USD transform stack 的关系也要分清：`GfDualQuatf` 可以作为计算中的中间表示，但 USD authored transform 通常仍通过 `UsdGeomXformOp`、matrix 或 translate/rotate/scale op 表达。把 dual quaternion 的内部 real/dual part 直接写成 USD 属性，通常不会被标准 USD 消费者理解为 xform stack。

`GfDualQuatf` can be an intermediate math representation, but authored USD transforms are generally represented through standard xform ops or matrices.

所以 Doxygen 中的 “Basic type” 应理解为基础数学值类型，而不是低重要性页面。它经常出现在变换插值、姿态混合、仿真到渲染的数据桥接和动画工具链中；一旦 real/dual 两部分解释错误，最终表现会是姿态漂移、平移翻转或 skinning 体积异常。

Here, “Basic type” means a foundational math value type, not an unimportant API surface.

调试结论：先查归一化。

Diagnostic summary: check normalization first.

因此，读 `GfDualQuatf Class Reference` 的主线是：先把它看作 float 精度 dual quaternion value type，再分清 real part 与 dual part，然后理解构造、归一化、translation extraction、inverse/conjugate、identity/zero 的语义边界，最后再把它放回 `GfQuatf`、`GfVec3f`、`GfMatrix4f`、`GfDualQuatd`、`GfDualQuath` 等相邻类型的坐标系和精度上下文中。
