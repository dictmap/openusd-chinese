This page is promoted to a complete bilingual reference. It keeps official API names, method names, type names, links, and excerpts while adding Chinese-first coverage of Pcp property-index semantics.

官方定义说，`PcpPropertyIndex` 是所有 scene description sites 的索引，这些 sites 会在 composition semantics 下为一个 specific property 贡献 opinions。这里的关键词是 property、sites、opinions 和 composition semantics：它不是最终属性值本身，也不是单个 `SdfPropertySpec` 的包装器，而是 Pcp 在属性级别整理出来的组合来源视图。

`PcpPropertyIndex` is an index of all sites in scene description that contribute opinions to a specific property, under composition semantics.

`PcpPrimIndex` 面向 prim 级别的组合图和 arc；`PcpPropertyIndex` 则面向某个属性，把会影响该属性的 specs、局部贡献和错误组织起来。调试属性为何没有按预期 resolve 时，应先确认 prim index 与 layer stack，再用 property index 查看属性 opinion 的具体来源，而不要把它当作 `UsdAttribute::Get()` 的 resolved value 缓存。

The property index should be read as a property-level composition structure, not as a final value cache.

构造函数 `PcpPropertyIndex()` 建立空索引，另一个构造形式接受 `PcpPropertyNodeRef`，用于从已经计算出的属性节点引用创建索引视图。这个构造边界很重要：`PcpPropertyIndex` 反映的是 composition 计算后的索引状态；它不会写入 layer，也不会创建新的 authored property opinion。

The constructors create an empty property index or initialize one from a `PcpPropertyNodeRef`; they do not author scene description.

`GetPropertyRange()` 是读取属性贡献来源的核心入口。返回的范围通常配合 `PcpPropertyIterator` 使用，用来按 Pcp 组合强弱顺序观察 property specs。这里的 range 不是 C++ 容器快照的语义重点，而是“按 composition 规则可遍历的属性 opinion sites”；调试时应关注每个 iterator item 来自哪个 node、layer stack 和属性路径。

`GetPropertyRange()` provides the iterable range of property specs, typically consumed through `PcpPropertyIterator`.

`GetNumLocalSpecs()` 只统计 local specs 的数量，适合判断当前 local layer stack 对该 property 有多少直接贡献。它不等于整个 composition graph 中所有 opinions 的数量，也不能替代 `GetPropertyRange()`。如果一个属性从 references、inherits 或 variants 等 arc 得到较弱或较强的 opinions，`GetNumLocalSpecs()` 的值仍只说明 local 这部分。

`GetNumLocalSpecs()` reports the number of local specs, not the total number of all composed property opinions.

`GetLocalErrors()` 暴露属性索引计算过程中局部发现的错误。local errors 应与 prim index 错误、layer stack 错误和具体 property path 一起分析；只看最终属性是否存在，容易漏掉 composition path 中被过滤、屏蔽或解析失败的站点。错误列表是诊断信息，不是 authored data。

`GetLocalErrors()` returns local composition errors associated with the property index.

`IsEmpty()` 表示当前属性索引没有可用内容。空索引可能意味着属性确实没有 contributing specs，也可能说明调用方使用了错误的 property path、相关 prim index 没有正确建立，或 composition arc 选择导致该属性没有进入当前索引。不要把 empty 直接等同于“属性值为空字符串”或“属性存在但值为默认值”。

`IsEmpty()` indicates whether the property index contains no usable indexed property content.

`Swap()` 是容器级状态交换，用于高效交换两个 property index 的内部内容。它不会重新计算 composition，不会触发 USD 文件写入，也不会改变任意 `SdfLayer` 中的 authored opinions；如果需要重新得到组合结果，仍应回到 Pcp cache / index 计算流程。

`Swap()` exchanges index contents; it is not a composition recomputation or USD authoring operation.

`PcpPropertyIterator` 被列为 friend，说明该类的主要消费方式之一是 iterator/range 访问。阅读本页时应把 iterator 顺序、composition strength、property node 和 site 的关系放在一起理解：索引告诉你“哪些地方贡献了 opinion”，不是替你解释每个 opinion 在应用层的业务含义。

`PcpPropertyIterator` is a friend and is the natural way to inspect the property sites represented by the index.

常见误用包括：把 `PcpPropertyIndex` 当成 `SdfPropertySpec` 列表直接编辑；把 `GetNumLocalSpecs()` 当作全部 opinions 数；把 `IsEmpty()` 当作 resolved value 为空；忽略 `GetLocalErrors()` 而只看 `UsdAttribute` 最终读取结果；以及把 property index 与 prim index 的职责混在一起。正确做法是用它定位 composition 来源，再结合 `PcpPrimIndex`、`PcpNodeRef`、`SdfPath`、`SdfLayer` 和上层 `UsdProperty` 语义继续排查。

Common misuse includes treating the property index as editable specs, a resolved value, or a complete substitute for prim-index diagnostics.
