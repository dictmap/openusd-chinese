逐段双语理解 / Paragraph-Level Bilingual Coverage

官方把 `UsdStageCache` 描述为 a strongly concurrency safe collection of `UsdStageRefPtr`。它缓存的是已经打开或发布的 `UsdStage` 引用指针，而不是 `SdfLayer`、asset resolver 结果或 composition cache。理解这一点可以避免把 stage 复用问题误判为 layer 加载或资产解析问题。

The official text describes `UsdStageCache` as a strongly concurrency safe collection of `UsdStageRefPtr`. It caches references to already opened or published `UsdStage` objects, not `SdfLayer` objects, asset resolver results, or composition caches. This distinction prevents confusing stage reuse issues with layer loading or asset-resolution problems.

典型用法是把 `UsdStageCacheContext` 绑定到某个 cache，然后调用 `UsdStage::Open()`。在这个上下文中，Open 操作可以先查找匹配的 stage；如果没有合适结果，也可以把新打开的 stage 发布到 cache。也就是说，cache 不替代 `UsdStage::Open()`，而是改变它与共享 stage 集合交互的方式。

The typical pattern is to bind a `UsdStageCacheContext` to a cache and then call `UsdStage::Open()`. In that context, the open operation can first look for a matching stage and, if no suitable result exists, publish a newly opened stage to the cache. The cache does not replace `UsdStage::Open()`; it changes how open operations interact with a shared stage collection.

“strongly concurrency safe” 的含义需要精确阅读：除 construction 和 destruction 外，cache 的操作可以并发执行。但这并不表示应用层业务逻辑自动同步，也不表示共享的 `UsdStage` 内容可以随意并发 mutation。多个线程共享 stage 时，仍要考虑 notice、payload loading、edit target、layer mutation 和调用方读写策略。

The phrase strongly concurrency safe needs a precise reading: operations other than construction and destruction may be performed concurrently on the cache. This does not mean application-level logic is automatically synchronized, nor that shared `UsdStage` contents can be mutated freely from any thread. Shared-stage users still need to reason about notices, payload loading, edit targets, layer mutation, and caller-side read/write policy.

`Find()`、`FindOneMatching()` 和 `FindAllMatching()` 的区别在于匹配范围和选择策略。`Find()` 适合已知 cache id 或明确键值的直接查找；`FindOneMatching()` 适合调用方期待唯一匹配的场景；`FindAllMatching()` 则保留所有候选，便于诊断 root layer、session layer、resolver context 或 load state 造成的多重匹配。

The difference among `Find()`, `FindOneMatching()`, and `FindAllMatching()` is the lookup scope and selection policy. `Find()` is useful for direct lookup by known cache id or explicit key. `FindOneMatching()` fits callers that expect one matching stage. `FindAllMatching()` preserves every candidate, which is useful when root layer, session layer, resolver context, or load state can produce multiple matches.

`Contains()`、`GetAllStages()` 和 `GetId()` 更偏诊断和管理。`Contains()` 可以确认某个 stage 或 id 是否仍在 cache 中；`GetAllStages()` 能枚举当前缓存持有的 stage；`GetId()` 把 stage 与 cache 内部 identity 关联起来。它们反映 cache 中的引用关系，不保证 stage 的 scene 内容没有变化。

`Contains()`, `GetAllStages()`, and `GetId()` are more diagnostic and administrative. `Contains()` verifies whether a stage or id is still present in the cache; `GetAllStages()` enumerates stages currently held by the cache; `GetId()` associates a stage with its cache identity. These calls describe references inside the cache; they do not guarantee that the scene contents of a stage are unchanged.

`Erase()`、`EraseAll()` 和 `Clear()` 只移除 cache 记录，不等于销毁所有 stage 对象。只要其他地方还持有 `UsdStageRefPtr`，对应 `UsdStage` 仍可能继续存活。调试内存或资源释放时，要同时检查 cache 内容和外部强引用，不能只看 cache 是否已经清空。

`Erase()`, `EraseAll()`, and `Clear()` remove cache records; they do not destroy every stage object. As long as another `UsdStageRefPtr` exists elsewhere, the corresponding `UsdStage` can remain alive. When debugging memory or resource release, inspect both cache entries and external strong references, not only whether the cache has been cleared.

stage cache 的匹配语义通常和 root layer、session layer、path resolver context、initial load set 等因素有关。两个 stage 即使指向相同文件名，只要 session layer 或 resolver context 不同，就可能不能当作同一个缓存条目复用。生产工具应明确自己的 cache key 期望，而不是假设 URL 或路径字符串相同就足够。

Stage-cache matching semantics usually depend on root layer, session layer, path resolver context, initial load set, and related factors. Two stages that refer to the same filename may still not be interchangeable if their session layer or resolver context differs. Production tools should define their expected cache key rather than assuming that identical URL or path strings are sufficient.

`UsdStageCache` 常用于长生命周期应用、工具进程、插件宿主或多视图场景中，避免重复打开同一 stage 并让多个客户端共享结果。但共享 stage 也意味着副作用会被更多调用方观察到：payload load/unload、muted layer、variant edit、metadata mutation 等行为都需要按应用边界设计。

`UsdStageCache` is common in long-lived applications, tool processes, plugin hosts, and multi-view workflows, where it avoids repeatedly opening the same stage and lets clients share results. Shared stages also mean side effects can be observed by more callers: payload load/unload, muted layers, variant edits, and metadata mutation need to be designed around application boundaries.

本页和之前晋级的 `UsdPrim`、`SdfLayer`、`SdfPath` 页面直接相关。`SdfLayer` 是底层 layer 数据容器，`SdfPath` 是命名空间定位值，`UsdPrim` 是 composition 后的 prim 句柄，而 `UsdStageCache` 管理的是整个 `UsdStage` 对象的共享与查找。把这些层次分开，是理解 USD 应用缓存策略的关键。

This page connects directly to the previously promoted `UsdPrim`, `SdfLayer`, and `SdfPath` pages. `SdfLayer` is the low-level layer data container, `SdfPath` is the namespace location value, `UsdPrim` is the composed prim handle, and `UsdStageCache` manages sharing and lookup of whole `UsdStage` objects. Keeping these layers separate is key to understanding USD application caching strategy.

常见误读包括：把 cache 的线程安全当成 stage mutation 的线程安全；以为 `Clear()` 会销毁所有外部 stage；把 stage cache 当成 asset resolver cache；或者用 `GetAllStages()` 作为热路径查找接口。正确做法是让 cache 负责共享和查找，让应用层显式管理生命周期、匹配条件和并发读写边界。

Common misreadings include treating cache thread safety as stage-mutation thread safety, assuming `Clear()` destroys every external stage, confusing the stage cache with an asset resolver cache, or using `GetAllStages()` as a hot-path lookup interface. The right model is to let the cache handle sharing and lookup while the application explicitly manages lifetime, matching criteria, and concurrent read/write boundaries.
