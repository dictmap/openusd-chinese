中文主阅读路径

UsdHydra 的官方标题是 UsdHydra : USD Hydra Schemas。这不是完整 Hydra 渲染框架手册，而是 USD 与 Hydra 之间一组 schema/API 连接点的模块入口。当前源页的核心说明是：UsdHydra 提供 UsdHydraGenerativeProceduralAPI schema，用来扩展 UsdProcGenerativeProcedural prim 的定义，使它能够提供 HdGpGenerativeProcedural 插件的 scene description。UsdHydra provides the UsdHydraGenerativeProceduralAPI schema for extending the definition of UsdProcGenerativeProcedural prims to provide scene description of HdGpGenerativeProcedural plug-ins.

因此，阅读本页要先抓住三层关系：UsdProcGenerativeProcedural 是 USD 场景里的 procedural prim；HdGpGenerativeProcedural 是 Hydra 侧的 generative procedural plug-in 语境；UsdHydraGenerativeProceduralAPI 是把两者连接起来的 schema API，负责让 USD prim 带上 Hydra procedural 插件需要的场景描述信息。本页的重点是这个连接边界，而不是讲解 renderer、render pass 或 scene delegate 的完整实现。The useful reading model is USD procedural prim, Hydra generative procedural plug-in, and the API schema connecting their scene description.

源页还有一个很重要的迁移警告：下面描述的 tokens 已经 deprecated，并且很快会被删除。UsdHydra 过去曾经提供基于 UsdShade 创建和连接 shading networks 的 schemas 与 behaviors，但这些 shading-related schemas 已经从该库删除；当前只保留 schema 中曾用到的 token 列表，帮助逐步迁移到通过 shader registry author 的新式 Hydra shaders。这个警告决定了本页不能被当作旧 shading authoring 指南使用。The tokens are deprecated; old UsdShade-based Hydra shading schemas were deleted and only transition tokens remain.

当前职责：USD procedural 到 Hydra procedural 的 schema 桥接

UsdHydraGenerativeProceduralAPI 的角色可以理解为 applied API schema 层的桥接点。它不是直接执行 procedural 的运行时，也不是 Hydra render delegate；它把 UsdProcGenerativeProcedural prim 的定义扩展到足以描述 HdGpGenerativeProcedural plug-in 所需的场景信息。换句话说，它解决的是“USD 场景描述如何携带 Hydra generative procedural 所需信息”的问题。UsdHydraGenerativeProceduralAPI extends UsdProcGenerativeProcedural so the prim can describe HdGpGenerativeProcedural plug-ins.

这条边界有助于定位问题。如果 USD 中 procedural prim 的类型、schema application 或 authored metadata 不对，问题通常在 UsdProcGenerativeProcedural 或 UsdHydraGenerativeProceduralAPI 的场景描述层；如果 Hydra 侧 plug-in 没有被发现、实例化或执行，问题更可能在 HdGpGenerativeProcedural、Hydra 插件注册、render index 或 delegate 消费层。本页只覆盖前者和两者之间的描述接口。Debugging should separate USD-authored procedural scene description from Hydra plug-in discovery and execution.

本页也不是 Hd 或 Hdx 的整体概览。需要理解 Hydra core、render index、scene delegate、render delegate、task、AOV 或渲染输出时，应继续阅读 Hd、Hdx 和 UsdRender。UsdHydra 在这里承担的是 schema 连接和兼容迁移提示，而不是完整渲染管线说明。Use Hd, Hdx, and UsdRender for Hydra framework and rendering-output topics.

deprecated token 与旧 shading schema 边界

官方 Note 必须单独阅读，因为它避免了一个高风险误读：UsdHydra 曾经包含用于创建、连接基于 UsdShade 的 shading networks 的 schemas 与 behaviors，但这些 shading-related schemas 已经被删除。页面中留下的 token 列表不是推荐的新 authoring surface，而是为了帮助从旧 Hydra shader authoring 迁移到新式 shader registry 路径。The Note says the old UsdShade-based Hydra shading schemas were deleted and only transition tokens remain.

因此，看到 UsdShade、shader registry 或 Hydra shaders 时，不应把本页当作材质网络教程。新的 shader authoring 与 definition 查询应优先阅读 Sdr、UsdShade、UsdShaders 或 UsdMtlx 相关页面；UsdHydra 在这个语境中只保留迁移线索和过渡 token，不再拥有旧 shading schema 的主要职责。For new shader authoring, read Sdr, UsdShade, UsdShaders, and UsdMtlx instead of relying on old UsdHydra shading schemas.

deprecated 的实际含义是：这些 token 可能仍在旧数据、旧工具或迁移脚本里出现，但不应作为新资产设计的依据。中文页保留这些英文 token 与 API 名，是为了让读者能核对旧文件和官方 Doxygen，而不是鼓励继续使用已删除的 schema 行为。若迁移过程中发现旧 token，应该把它们视为兼容性信号，再沿 shader registry 的新路径重建 authoring。Deprecated tokens may help migration, but they are not a recommended new authoring interface.

相邻模块阅读路径

第一条路径是 procedural 路径：从本页进入 UsdProc 和 UsdProcGenerativeProcedural，理解 USD 如何表达生成式 procedural prim；再沿 HdGpGenerativeProcedural 进入 Hydra 侧插件模型。这条路径回答“USD 场景描述如何把 procedural 意图交给 Hydra 消费”的问题。The procedural path connects UsdHydra to UsdProc and Hydra generative procedural plug-ins.

第二条路径是 Hydra 消费路径：阅读 Hd、Hdx 和 UsdRender，区分 schema 描述、render index、任务执行、render delegate 与输出设置。UsdHydra 不负责这些系统全部行为，但它的 schema 信息最终可能被 Hydra 侧系统读取或解释。The Hydra path separates schema description from render index, task execution, render delegates, and output settings.

第三条路径是 shader 迁移路径：官方 Note 提到旧 UsdShade shading network schema 已删除，所以应阅读 Sdr 与 UsdShade 来理解 shader registry 和材质网络的新职责。不要把过渡 token 当成当前推荐的 schema authoring 文档。The shader migration path points to Sdr and UsdShade, not to deleted UsdHydra shading schemas.

调试路径与常见误读

调试 generative procedural 时，先确认 USD prim 是否是预期的 UsdProcGenerativeProcedural，再确认是否应用或携带了 UsdHydraGenerativeProceduralAPI 所需的描述信息。若这一步不成立，Hydra 侧很可能根本拿不到正确的 procedural scene description。只有当 USD 描述正确后，才继续检查 HdGpGenerativeProcedural plug-in 是否存在、是否注册、是否被 Hydra 消费。First validate USD procedural scene description, then check Hydra plug-in registration and consumption.

调试旧 shader 迁移时，先识别 token 是否来自已废弃的 UsdHydra shading-related schemas。若是，应把它当作迁移提示，而不是继续追问为什么旧 schema 行为在新库里不可用。新式 Hydra shader authoring 应沿 shader registry 方向排查：definition 是否能被发现、Sdr 是否能查询、UsdShade 网络是否正确引用、renderer 是否支持相应 implementation。For old shader data, treat UsdHydra tokens as migration hints and use the shader registry path for new authoring.

常见误读包括：把 UsdHydra 当成完整 Hydra renderer 文档；把 deprecated tokens 当成新 schema；把已删除 shading schemas 当成仍可 author 的接口；把 HdGpGenerativeProcedural 插件执行失败归因到 USD schema 页面；或者把 UsdHydraGenerativeProceduralAPI 当成 procedural 运行时本身。本页的中文主路径把这些边界拆开，帮助读者把错误归因放在正确层级。Misreads include treating UsdHydra as the renderer manual, deprecated tokens as new schema, or the API schema as runtime execution.

工程使用提示

在资产或工具链中使用这类 schema 时，建议把证据分成三份保存：USD 侧 authored scene description、Hydra 侧 plug-in discovery/execution 证据、shader registry 或 material network 迁移证据。这样能避免“procedural 没出现”或“shader 不工作”这类笼统结论，直接压到单一模块上。Keep separate evidence for USD scene description, Hydra plug-in behavior, and shader-registry migration.

如果团队正在清理旧资产，UsdHydra 页面中的 deprecated token 信息适合用作审计线索：它能帮助发现旧数据中仍在引用过时 Hydra shading schema 的位置。但修复方案不应是继续扩大旧 schema 使用，而应迁移到 shader registry、Sdr、UsdShade 或新的 renderer-supported shader definition 路径。Deprecated token information is useful for auditing old assets, not for expanding old schema usage.

最小验收清单

判断一个 UsdHydra 相关问题是否落在本页职责内，可以使用一个最小清单。第一，问题是否涉及 UsdProcGenerativeProcedural prim 的 USD 场景描述；第二，问题是否涉及 UsdHydraGenerativeProceduralAPI 这一 schema API 是否被正确应用或解释；第三，问题是否涉及把这些描述传递给 HdGpGenerativeProcedural plug-in。只有同时靠近这些点时，本页才是主要入口。Use the page when the issue is about USD procedural scene description and the schema bridge into HdGpGenerativeProcedural.

如果问题是 “Hydra 为什么没有画出来”，清单应继续向下分流：是否有 render delegate、是否有 render index 同步、是否有 task 执行、是否有 render product 或 buffer、是否有 renderer 后端能力。这些属于 Hd、Hdx、UsdRender 或具体 delegate 页面，而不是 UsdHydra module front page。这个分流能防止把渲染层错误误报成 schema 错误。Rendering failures should be routed to Hd, Hdx, UsdRender, or a concrete render delegate after schema description is verified.

如果问题是 “旧 shader 数据为什么不能继续 author”，清单要转向迁移语义：是否使用了已 deprecated 的 token，是否依赖已删除的 shading-related schemas，是否仍假定 UsdHydra 提供基于 UsdShade 的旧 shading network 行为。若答案为是，修复方向应是迁移到 shader registry 和当前 shader definition 路径，而不是在本页寻找旧接口的替代调用。Old shader data should be treated as migration work toward shader registry rather than continued use of deleted UsdHydra shading schemas.

如果问题是 “procedural 插件为什么没有 scene description”，清单才回到本页：先检查 prim 类型和 authored API，再检查对应属性或 token 是否符合当前 schema 预期，最后检查 Hydra 侧 plug-in 是否能读取到这些信息。本地中文页保留这些步骤，是为了让读者在短源页之外仍能建立可操作的诊断顺序，并知道下一步该跳向 UsdProc、Hd、Sdr 还是 UsdShade。For missing procedural scene description, verify prim type, authored schema API, authored data, and then Hydra plug-in consumption.

同时也要保持克制：官方源页没有列出函数签名、属性表或完整 token 表时，本地中文页不应伪造细节。这里补充的是阅读顺序、职责边界和迁移判断，而不是替代 class reference。需要逐项 API 成员时，应进入对应 class 页、source 页或官方 Doxygen；module front page 只负责把官方短说明翻译成可靠的工程上下文。The local page adds reading context and boundaries; it does not invent member-level reference details absent from the source page.

这也是本轮晋级的质量边界：补足中文理解，但不越过源页证据；读者能判断职责、迁移风险和下一跳位置即可，并能更安全地避免误用旧接口。The quality boundary is stronger Chinese comprehension without exceeding source evidence.
