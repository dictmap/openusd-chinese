逐段双语理解 / Paragraph-Level Bilingual Coverage

`SdfUsdzFileFormat` 是 Sdf 层用于 package `.usdz` files 的 file format 实现。官方摘录的关键词是 “File format for package .usdz files”，所以它首先属于 `SdfFileFormat` / `SdfLayer` 的底层 I/O 机制，而不是 `UsdStage` 的场景遍历、composition 或渲染 API。它让 USDZ package 能以 layer/file format 的形式被识别、读取和写出。

The official page describes this class as the file format for package `.usdz` files.

`CanRead()` 回答的是“给定路径或文件是否能由该 file format 识别和读取”的问题。它不等同于完整打开 stage，也不保证包内所有 referenced assets 都可解析。调试时应把 format recognition、resolved path、package contents 和后续 `SdfLayer` 打开流程分开检查。

`CanRead()` is about file-format readability, not a guarantee that every packaged asset or composed stage will load successfully.

`Read()` 与 protected 的 `_ReadDetached()` 面向 layer data 载入。`Read()` 是常规 file-format 读取入口；`_ReadDetached()` 暗示读取逻辑可以与当前 layer/外部文件系统状态解耦，适合理解 package 内数据被拷贝、缓存或独立解析的场景。它们处理的是 Sdf 层数据，不负责解释 `UsdStage` 的 payload loading、variant selection 或 composition arcs。

`Read()` and `_ReadDetached()` load Sdf layer data; stage-level composition behavior remains outside this class.

`GetPackageRootLayerPath()` 是 USDZ package 的关键 API：它定位 package 内部的 root layer 路径。普通 `.usd` / `.usda` / `.usdc` 文件通常直接就是 layer 文件；`.usdz` 则是 package，需要先确定包内哪个 layer 是入口。排查 USDZ 打不开时，root layer path、package-relative path 和 resolver identifier 往往比单纯文件扩展名更关键。

`GetPackageRootLayerPath()` locates the root layer inside a package, which is central to package-aware layer opening.

`IsPackage()` 明确该 file format 代表 package 语义。工具可以用它判断是否需要 package-aware asset resolution、包内 root layer 展开、以及对 package-relative paths 的特殊处理。它不表示该 layer 一定处于只读或可写状态；实际写出能力还要结合 layer identifier、文件系统权限、包结构约束和 writer 的支持范围。

`IsPackage()` identifies package semantics; writeability and asset resolution still depend on environment and package structure.

写出路径包括 `WriteToFile()`、`WriteToStream()` 和 `WriteToString()`。这些方法体现它仍遵循通用 `SdfFileFormat` 写出接口，但 `.usdz` 的归档结构、资源打包规则、外部依赖限制和目标路径权限会影响结果。`WriteToString()` 适合检查 layer 文本表达，不应被误认为已经生成完整可发布的 USDZ package。

Write APIs share the general file-format contract, but package layout and external-resource rules affect practical USDZ output.

`ReadFromString()` 可用于从内存字符串读取 layer 内容，常见于测试、转换或中间数据流。对于 USDZ 来说，它不能替代真实 package 的资源存在性检查；材质、贴图、子层和其它包内 asset 是否存在，仍需要通过 package 内容和 resolver 结果验证。

`ReadFromString()` is useful for in-memory layer content, not proof that a real package contains all required resources.

`FileFormatArguments`、`InitData()` 和 `SDF_FILE_FORMAT_FACTORY_ACCESS` 说明本类仍是 Sdf 插件工厂体系的一部分。用户通常不会手动 new `SdfUsdzFileFormat`；实际使用中是 `SdfLayer`、resolver 和 file-format registry 根据文件扩展名、identifier 或插件注册信息选择它。

Factory and initialization hooks show that this class participates in the Sdf file-format plugin system.

常见误读包括：把 `SdfUsdzFileFormat` 当成 stage API；把 `CanRead()` 当成“所有包内资源都没问题”；把 `WriteToString()` 当成完整 package 写出；只检查 `.usdz` 扩展名而忽略 root layer；或把 package-relative paths 当成普通文件系统路径。正确排查路径是先看 layer identifier，再看 resolved path、root layer path、package contents、Sdf data、write target 和 resolver 日志。

Common debugging separates layer identifier, resolved path, package root layer, package contents, Sdf data, write target, and resolver diagnostics.
