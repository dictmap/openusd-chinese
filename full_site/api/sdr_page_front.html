逐段双语理解 / Paragraph-Level Bilingual Coverage

Sdr 是 OpenUSD 的 Shader Definition Registry，用于发现、延迟解析并查询 shader definition。它不直接运行 shader，也不在 stage 上 author 材质网络；它提供一个可扩展的 registry 层，让管线在真正需要执行或展示 shader 之前，先以统一方式找到 shader 的实现、输入输出属性和元数据。Sdr provides an extendable framework to discover, lazily parse, and ask for information about shaders.

Background section 的重点是 shader 系统差异。shader 是接收 input 并产生 output 的程序，但它们可以来自不同语言、运行时和 renderer-specific 表达。复杂 production pipeline 不希望每一步都直接耦合某个 shading system 的内部结构，所以需要一个中间 registry：保留足够信息供后续使用，同时把具体实现差异推迟到需要解析时再处理。A shader can be implemented in many languages and runtimes, so pipelines avoid coupling directly to every shading-system representation.

官方页说明，使用 Sdr 时，管线通常只需要抓住两项关键数据：identifier 与 shadingSystem。前者标识要找的 shader definition，后者说明它属于哪个 shading system。通过这两个键，Sdr 可以找到原始 shader implementation、shader 的 input/output properties，以及 Sdr 标准化的 common metadata 与 source-specific custom metadata。With Sdr, pipelines need only hold on to identifier and shadingSystem to find implementation, properties, and metadata.

Usage section 的核心读法是按对象分层：SdrRegistry 是发现和查询入口，负责把 discovery/parser 机制暴露为 registry 查询；SdrShaderNode 表示一个可查询的 shader definition；SdrShaderProperty 表示 shader 的输入、输出或其他 property 描述。读者应把三者看成 registry、node definition、property definition 的关系。Usage points readers to SdrRegistry, SdrShaderNode, and SdrShaderProperty.

Sdr 的 “lazy parse” 很重要：registry 可以先发现 shader definition 的存在和基本 identity，等客户端真正查询 node 或 property 细节时再解析更重的内容。这样大型 shader 库不会因为启动时一次性解析所有文件而拖慢工具，也能让 UI、材质浏览器、renderer bridge 和验证工具只为当前需要的 shader 支付解析成本。Lazy parsing keeps large shader libraries discoverable without parsing everything upfront.

换句话说，shader discovery 与 lazy parsing 是 Sdr 的两个核心动作：discovery 先告诉系统有哪些 shader definition 可以被找到，lazy parsing 再把昂贵的解析延迟到查询发生时。这个设计让插件可以暴露很多定义，同时仍保持启动和浏览阶段轻量。shader discovery identifies available definitions, while lazy parsing delays heavier parsing until query time.

在 OpenUSD 栈里，Sdr 与 Ndr、UsdShade、UsdMtlx 的边界要分清。Ndr 是更底层的 node definition registry 基础设施；Sdr 面向 shader definition 加上 USD shading 语义需要的查询包装；UsdShade 负责在 USD scene 中表达 material、shader prim 和连接网络；UsdMtlx 则把 MaterialX 相关定义接到 USD shading 工作流中。Sdr should be read next to Ndr, UsdShade, and UsdMtlx, but each layer has a different responsibility.

因此，Sdr 页面的主阅读路径不是“如何写一个材质网络”，而是“一个 shader definition 如何被发现、注册、延迟解析、查询并交给上层工具使用”。如果你正在调试 UsdShadeShader 上的 authored connection，先看 UsdShade；如果你已经有 shader identifier 但找不到 node definition、property 列表或 metadata，则应该回到 Sdr。Sdr answers discovery and definition queries, while UsdShade answers authored material-network questions.

对于工具开发者，Sdr 的价值在于统一查询面：材质 UI 可以通过 SdrShaderProperty 枚举输入输出，资产验证可以检查 identifier 是否能解析到 node definition，renderer integration 可以把 shadingSystem 作为桥接条件，离线转换器可以从 metadata 中读取 family、source type、implementation URI 或自定义字段。Tools use Sdr to query properties, metadata, implementation identity, and shading-system-specific definitions.

对于错误定位，常见路径是：确认 identifier 字符串是否正确，确认 shadingSystem 是否匹配，确认相关 discovery/parser plugin 是否被加载，确认 shader source 或 asset path 是否可访问，再检查 SdrRegistry 返回的 SdrShaderNode 是否包含预期的 SdrShaderProperty。不要把 renderer 不支持某个 shader 的 runtime 行为，误判成 Sdr registry 一定失败。Debugging starts with identifier, shadingSystem, plugin loading, source accessibility, and returned node/property data.

如果一个 pipeline 同时支持 GLSLFX、MaterialX 和 renderer-specific shader 描述，Sdr 让这些来源能以相同的 node/property 查询模型出现。这样工具层可以统一读取 name、identifier、property、metadata 和实现来源，而把“这个定义来自哪个文件格式、哪个 parser、哪个 renderer 插件”的差异留在 registry 和插件实现里。Sdr presents different shader sources through a common node/property query model.

本页是模块入口，不是完整 API reference。继续深入时应打开具体类页核对构造、查询和返回类型；但作为阅读起点，它必须先解释 Sdr 的职责边界：发现 definition、统一 metadata、延迟解析、返回 node/property 描述、服务材质工具和渲染桥接，而不是替代 shader authoring、compiler 或 renderer runtime。This module front page positions Sdr before readers move to detailed class references.

生产环境里还要注意版本和来源一致性：同一个 identifier 可能因为插件版本、搜索路径、renderer 包或 MaterialX 库不同而解析到不同定义。排查这类问题时，先记录 registry 看到的 definition 来源和 property 列表，再比较 USD scene 中 authored shader prim 的 identifier，这比直接修改材质网络更稳。Production debugging should compare registry results with authored shader identifiers before editing material networks.

因此，本页完成后应能让读者独立判断：问题属于 registry discovery、definition parsing、UsdShade authoring，还是 renderer runtime 支持范围，并能选择下一步排查入口。The page separates registry, parsing, authoring, and runtime concerns.

核心 API 分组 / Core API Groups

  • SdrRegistry：shader definition 的发现、缓存和查询入口。它把 identifier、shadingSystem、source type、metadata 等查询组织成 registry API，并负责与 discovery/parser 扩展机制协同。SdrRegistry is the primary discovery and query entry point.
  • SdrShaderNode：表示一个 shader definition。它通常提供名称、identifier、family、source type、implementation 信息、metadata，以及一组可查询的 shader properties。SdrShaderNode represents a discovered shader definition.
  • SdrShaderProperty：描述 input/output/property 的类型、默认值、连接能力、UI 相关提示和自定义 metadata。材质编辑器和验证工具经常从这里构建属性面板或规则检查。SdrShaderProperty describes shader inputs, outputs, and property metadata.
  • identifier 与 shadingSystem：官方页明确提出的两项最小定位信息。前者回答“找哪个 shader”，后者回答“在哪个 shading system 的语义里找”。identifier and shadingSystem are the minimal lookup facts called out by the official page.
  • Ndr/Sdr 关系：实际实现中 Sdr 建立在 node discovery/registry 基础之上，但面向 shader definition 提供更贴近 shading pipeline 的类型和查询语义。中文页保留两者英文缩写，避免把它们翻成不可核对的抽象术语。Sdr layers shader-specific meaning over node-definition registry concepts.

边界、误读点与调试路径 / Boundaries, Misreads, and Debugging

  • 不要把 Sdr 当成 shader compiler。Sdr 发现和描述 shader definition，但不承诺编译、优化或在 renderer 中执行 shader code。Sdr is not a shader compiler.
  • 不要把 Sdr 当成 UsdShade 的替代品。UsdShade 管 scene 中的 material network authoring；Sdr 管 registry 中的 shader definition 查询。Sdr does not replace authored UsdShade material networks.
  • 如果材质 UI 少了某个 input，先检查 Sdr 是否能返回对应 SdrShaderProperty，再检查 UsdShade prim 上是否正确 author 了 input 和 connection。Missing UI inputs should be reduced to Sdr property discovery and UsdShade authoring separately.
  • 如果一个 identifier 在某台机器可用、另一台不可用，优先检查插件路径、discovery result、parser 支持、asset search path 和 shadingSystem 字符串，而不是直接修改 USD 文件里的材质网络。Machine-specific discovery failures usually point to plugins, search paths, or shadingSystem mismatches.
  • 如果 MaterialX 相关 shader 解析异常，应把问题拆成三层：MaterialX 文件或定义是否可访问，UsdMtlx/Sdr 桥接是否能产出 node definition，UsdShade 网络是否引用了正确 identifier。MaterialX issues should be separated into source availability, registry bridge, and authored network references.
  • 如果 renderer runtime 不支持某个 shader，不一定是 Sdr 错。Sdr 只说明 definition 能否被发现和查询；renderer 是否实现该 shader、如何解释 metadata、是否支持某个 source type，是下游 renderer integration 的责任。Renderer runtime support is downstream from Sdr discovery.
