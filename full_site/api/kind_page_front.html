中文主阅读路径

Kind 库提供的是 runtime-extensible taxonomy，也就是运行时可扩展的分类体系。官方页说 kinds are just TfToken symbols，但关键价值不在 token 本身，而在 KindRegistry 可以把这些 token 组织成互相关联、可细化的 taxonomy，并通过 KindRegistry::GetBaseKind() 与 KindRegistry::IsA() 查询层级关系。中文阅读时应把它理解成“场景图对象的分类语义”，不要把它误读成 C++ 类继承、USD schema 类型、prim typeName 或 composition arc。The Kind library provides a runtime-extensible taxonomy known as kinds.

Kind 最常见的用途是给 scenegraph objects 分类，尤其是标记模型根节点代表哪一种 model。官方页明确说 builtin kind hierarchy 的动机来自 scenegraph taxonomy 和 model hierarchy；也就是说，model、component、group、assembly、subcomponent 不是几何类型，也不是资产文件格式，而是帮助工具、浏览器、资产系统和 pipeline 识别“这里是一个模型边界、容器、发布资产或组件内部停止点”的分类标签。Kinds are useful for classifying scenegraph objects such as roots of models.

本页的核心是两件事：第一，读懂内置 model hierarchy；第二，知道如何通过 PlugRegistry 和 plugInfo.json 扩展 KindRegistry。中文页保留所有 token、API 名和 JSON 字段名，例如 model、component、group、assembly、subcomponent、Kinds、baseKind、description、chargroup、charprop 与 newRootKind，因为这些名字需要和官方 Doxygen、插件配置和资产元数据逐字对应。This page focuses on the builtin model hierarchy and extending the KindRegistry through PlugRegistry and plugInfo.json.

The Core Kind Hierarchy / 核心 Kind 层级

model 是整个 model hierarchy 的根概念。官方定义是 Everything in the model hierarchy is a kind of model。理解它时不要把 model 只等同于某个三维对象；在 USD 资产组织里，它更像一个承认“这个 prim 位于模型组织体系中”的分类根。后续的 component、group、assembly 和 subcomponent 都需要放在这个 hierarchy 中理解。model is the root kind for everything in the model hierarchy.

component 是 terminal model，也就是模型层级里的终端模型，官方强调 it can have no child models。它适合表示一个可作为完整单元交付或引用的模型边界。常见误读是把 component 当成“任何组件子树”；更准确地说，component 可以包含内部结构，但不应再包含 child models。工具在展开模型层级、构建资产浏览器或判断引用边界时会依赖这个语义。component is a terminal model in the model hierarchy and can have no child models.

group 是 simply a container for other models。它的重点是容器语义，而不是发布资产的重要性。一个 group 可以帮助把多个 model 组织在一起，但它本身未必是 pipeline 中需要被当作独立发布资产处理的重要实体。读者可以把它看成“模型集合的普通容器”。group is a model that is simply a container for other models.

assembly 是 important group model，官方示例是 often because it is itself a published asset。它仍然是 group 的一种，但比普通 group 更适合作为重要资产边界。比如一个角色组合、场景块或发布单位可能是 assembly；它不仅包含其他 model，还表达“这个容器本身有资产意义”。这一区分对资产发现、打包、引用和工具 UI 非常重要。assembly is an important group model, often because it is itself a published asset.

subcomponent 位于 component model 内部，用来标识重要的、通常可 articulable 的子树。官方说 subcomponents are stopping points when dynamically unrolling or expanding a component。也就是说，subcomponent 不应被误写成 child model；它是 component 内部的停止点或可辨识部分，适合工具在展开复杂组件时知道哪里应当停下来、哪里值得给用户显示。subcomponent identifies important sub-trees within a component and acts as a stopping point when expanding a component.

Extending the KindRegistry / 扩展 KindRegistry

官方扩展方式依赖 PlugRegistry：在模块的 plugInfo.json 中添加 Kinds sub-dictionary。每个条目可以声明 baseKind，也可以添加 description；如果没有 baseKind，就可以形成新的 root kind taxonomy。页面示例里的 chargroup 以 assembly 为 base kind，charprop 以 component 为 base kind，newRootKind 则没有 base kind。The kind registry can be extended by adding a Kinds sub-dictionary to plugInfo.json through PlugRegistry.

这段内容的实现边界很重要：扩展 registry 不等于修改内置 hierarchy。官方 Note 明确说明，不能通过 extension file 改变 core kinds、core kinds 在 hierarchy 中的位置，或相关数据；尝试覆盖会导致 registration error。中文页把这个限制放在主阅读路径里，是为了避免团队为了适配本地命名而重定义 model、component 或 assembly，从而破坏跨工具的一致理解。Core kinds cannot be altered or overridden through the extension file; attempting to do so is a registration error.

如果需要自定义 taxonomy，推荐做法是保留官方 core hierarchy，并在业务域里新增清晰的 child kind 或独立 root kind。例如角色 pipeline 可以把 chargroup 放在 assembly 下，把 charprop 放在 component 下；但不要改变 assembly 本身的含义。这样 KindRegistry::IsA() 查询仍能判断自定义 kind 是否属于既有层级，工具也能在不了解所有业务扩展的情况下做合理降级。Custom kinds should extend the registry while preserving the builtin model hierarchy.

官方示例中的 chargroup、charprop 和 newRootKind 还展示了三种不同扩展意图：细化重要组、细化终端模型、创建全新 taxonomy 根。评审 plugInfo.json 时不要只检查 JSON 是否能解析，还要检查每个条目的语义是否和 baseKind 相称。把一个仍包含 child models 的复杂资产标成 component，或者把普通容器标成 assembly，都可能让资产发现工具产生错误假设。The sample entries demonstrate refinement under assembly, refinement under component, and a new root taxonomy.

扩展描述字段 description 不是运行时推理所需的唯一依据，但它对人类评审很有帮助。建议在自定义 kind 中写清“它为什么属于这个 baseKind、工具应如何解释它、是否代表发布资产边界”。这样即使后续维护者没有读过原始 pipeline 约定，也能从 registry 元数据理解 taxonomy 的设计意图。Descriptions help human reviewers understand why a custom kind belongs under a chosen baseKind.

因此，本地中文页采取“token 原样、语义中文解释”的策略：配置文件、API 查询和资产 metadata 中出现的英文 token 不能翻译，否则会破坏实际数据契约；但每个 token 的使用场景、边界和审核方法必须用中文讲清楚。这样读者既能直接对照官方 Doxygen 和源文件，又能在不依赖英文正文的情况下判断应该 author 哪一种 kind。The local page preserves tokens while explaining their semantics and authoring checks in Chinese.

如果排查结果显示 taxonomy 查询正确，但工具界面仍然不按预期分组，就应继续检查工具层是否使用了相同的 composed stage、是否读取了正确 prim、是否把未加载 payload 或实例代理误当成模型根。Kind 本身只提供分类语义；它不会替代 stage 加载策略、资产解析策略或 UI 层的过滤规则。这个边界能帮助读者把问题分派给 registry、composition、asset resolver 或客户端工具，而不是在一个层里反复试错。If taxonomy queries are correct but tools group assets incorrectly, inspect stage loading, prim selection, payload state, and UI filtering.

边界、误读点和相邻模块关系

Kind 不是 Sdf 层的路径、layer 或 composition 语法；Sdf 负责场景描述基础结构，Pcp 负责 composition 和 PrimIndex，Kind 则给场景图对象附加分类语义。实际阅读时可以先在 Sdf 理解 layer/path/spec，再在 Pcp 理解 composition，然后回到 Kind 判断某个 prim 在资产组织里应被视为 model、component、group 还是 assembly。Kind is taxonomy metadata, not Sdf path syntax or Pcp composition logic.

Kind 也不是 UsdGeom 的几何 schema。一个 prim 的 typeName 可以是几何、灯光、材质或其他 schema，Kind 关注的是这个 prim 在模型组织体系中的分类角色。几何页面帮助解释形状、xform、bound 和图元表示；Kind 页面帮助解释模型根、终端模型、容器模型、重要组和组件内部停止点。两者可以同时存在，但职责不同。Kind metadata is separate from UsdGeom schema type information.

Kind 与插件系统的关系来自 PlugRegistry，不是来自运行时随意注册字符串。扩展 KindRegistry 时要检查对应插件是否被发现、plugInfo.json 是否在可见搜索路径内、Kinds 字典是否拼写正确、baseKind 是否指向已存在且允许作为父 kind 的 token。调试时如果 KindRegistry::IsA() 返回不符合预期，应先检查 registry 是否真的加载了自定义 kind，而不是怀疑 USD composition 本身。KindRegistry extension depends on plugin discovery and plugInfo.json data.

Kind 与 TfToken 的关系也需要分清。官方说 kinds are just TfToken symbols，意思是 kind 的标识符是 token；但分类语义来自 KindRegistry 中的层级和 base kind，而不是 token 字符串天然拥有类型关系。一个字符串只有被 registry 认识并放进 taxonomy，GetBaseKind() 和 IsA() 才能用于可靠推理。Kind names are TfToken symbols, while hierarchy reasoning comes from KindRegistry.

调试和阅读路径

排查 Kind 问题时，第一步是确认目标 prim 是否位于预期的模型边界上。不要只看 prim 名称或文件路径；需要查看作者是否给模型根写入了正确 kind metadata，以及工具查询时是否拿到了同一个 composed prim。若模型浏览器没有把资产显示成预期层级，先检查 kind 是否写在根 prim 上、是否误把 component 内部子树写成 child model、是否把 group 和 assembly 的资产意义混在一起。Debugging starts by verifying where kind metadata is authored and whether it is on the intended model root.

第二步是检查 registry。自定义 kind 不生效时，先看插件是否被 PlugRegistry 发现，再看 plugInfo.json 中 Kinds 字典的结构。baseKind 拼错、父 kind 未注册、试图覆盖 core kind、或者插件不在搜索路径内，都会导致查询层级时失败或降级。官方 Note 中的 registration error 是硬边界，不能通过本地中文页把它解释成允许覆盖。For custom kind failures, inspect PlugRegistry discovery, plugInfo.json structure, baseKind spelling, and core-kind override errors.

第三步是把 Kind 查询和 composition 查询分开。KindRegistry::GetBaseKind() 和 KindRegistry::IsA() 回答的是 taxonomy 层级；composition 是否引入某个 prim、payload 是否加载、reference 是否成功、layer stack 是否符合预期，则属于 Pcp/Sdf/Usd 的问题。把这些层级混在一起会让调试方向变得很散。Separate taxonomy queries from composition, payload, reference, and layer stack debugging.

建议阅读顺序是：先读本页 overview 建立 Kind 的 taxonomy 心智模型；再读核心层级，尤其区分 component、group 和 assembly；随后读扩展 registry，理解 PlugRegistry 和 plugInfo.json；最后跳到 Sdf、Pcp、UsdGeom、Plug 和 Tf 页面，把分类语义放回资产路径、composition、几何表示、插件发现和 token 基础设施中。Recommended reading connects Kind to Sdf, Pcp, UsdGeom, Plug, and Tf pages.

在资产组织和工具链中的使用方式

在真实 pipeline 中，Kind 的价值通常体现在“工具如何连续阅读一个 stage”。资产浏览器需要知道哪些 prim 是可展示的模型根，打包工具需要知道哪些边界是发布资产，展开工具需要知道 component 内部哪些子树是 subcomponent 停止点。若没有 Kind，工具只能从路径、命名约定或 schema 猜测资产组织；若 Kind 写得稳定，工具可以用 taxonomy 查询做更可靠的降级和筛选。In pipeline tools, Kind helps identify model roots, published asset boundaries, and stopping points for expansion.

作者在写 Kind 时应先决定 prim 在模型层级里的角色，而不是先挑一个听起来接近的 token。一个可以独立交付、没有 child models 的资产根更接近 component；一个只是装多个模型的普通容器更接近 group；一个本身也是重要发布单元的容器更接近 assembly；component 内部需要被工具识别但不应成为 child model 的局部结构更接近 subcomponent。这种判断比“它是不是一团几何”更重要。Authoring should start from the model-hierarchy role rather than from a superficial token name.

自定义 Kind 的设计也应保持可推理性。把 chargroup 放在 assembly 下，意味着不了解 chargroup 的工具仍可把它当作一种 assembly 处理；把 charprop 放在 component 下，意味着通用工具仍可把它当作终端模型处理。如果业务需要一个完全不同的分类体系，可以使用没有 baseKind 的新 root kind，但要清楚这条 taxonomy 与官方 model hierarchy 的关系会变弱。Custom kinds should preserve reasoning through baseKind when they are refinements of builtin kinds.

评审资产时可以按四个问题快速检查：kind 是否写在正确的模型根上；component 是否真的没有 child models；assembly 是否确实是重要 group 或发布资产；自定义 kind 是否通过 plugInfo.json 被注册，并且 KindRegistry::IsA() 能落回预期的 builtin kind。任何一个问题不成立，都可能让浏览、打包、展开或资产发现工具表现出看似随机的层级错误。Reviewing kind metadata means checking authoring location, child-model rules, assembly importance, and registry fallback behavior.

本地中文页没有翻译或改写官方 token，是因为这些 token 本身就是数据契约的一部分。中文说明负责解释它们的语义和风险；model、component、group、assembly、subcomponent、baseKind 等字符串必须原样保留，才能让读者把本文、source snapshot、官方 Doxygen 和实际资产文件对应起来。The official tokens are preserved because they are part of the data contract.

非目标范围也要明确：本页不定义新 schema，不说明如何 author payload 或 reference，也不替代资产命名规范；它只解释 Kind taxonomy 如何被注册、查询和用于模型层级判断。Out of scope: schema definition, payload/reference authoring, and asset naming policy.
