逐段双语理解 / Paragraph-Level Bilingual Coverage

`HdStDispatchBuffer` 是 Storm / HdSt 后端用于 indirect dispatch 的 GPU-side command/data buffer。官方描述说它是 “a VBO of a simple array of unsigned integers”，这句话的重点不是“普通数组”，而是：该数组被组织成 GPU 能直接消费的 unsigned integer command layout，用于 `MultiDrawIndirect` 或 `DispatchComputeIndirect` 这类间接绘制 / 间接计算调度。

`HdStDispatchBuffer` is an unsigned-integer VBO used by Storm for indirect dispatch data consumed by `MultiDrawIndirect` or `DispatchComputeIndirect`.

这个类不表示 USD primvar、scene delegate 数据，也不是 `HdRenderBuffer` 那种 AOV / framebuffer 存储。它的职责更靠近 Storm backend 的 draw/compute scheduling：把命令数量、命令 stride、interleaved data 和 resource views 放到 GPU 资源中，让 shader 或 graphics API 可以按预期读取。

It is not a USD primvar buffer or an AOV render buffer. Its role is backend scheduling data for draw/compute dispatch.

`AddBufferResourceView()` 是理解本页的核心接口之一。官方说明提到同一 uint array 的 interleaved subsets 会以不同方式绑定；每个绑定都在 uint array 上定义一个 `BufferResourceView`。也就是说，底层存储可以是一段连续 VBO，但 shader 看到的是多个有名称、类型、offset 和 binding method 的资源视图。

`AddBufferResourceView()` defines resource views on top of the same unsigned-integer array, so different interleaved subsets can be bound in different ways.

`HdBufferArray` 与 `HdResourceBinder` 描述资源如何被聚合和绑定：`HdBufferArray` 聚合这些 views，`HdResourceBinder` 按指定 binding method 与 interleaved offset 把它们绑定给 shader。相关类型 `TfToken`、`HdTupleType`、`HdStBufferResource` 和 `HdStResourceRegistry` 分别承担命名、数据布局、具体 GPU resource 和资源注册 / 复用职责。

`HdBufferArray` aggregates views and `HdResourceBinder` binds them using binding methods and interleaved offsets. Names, tuple layout, concrete resources, and registry ownership are represented by adjacent types.

这也意味着资源名、tuple type 与 binding method 必须同时一致；只检查 buffer 是否存在，不能证明 shader 实际按正确语义读取。

Resource names, tuple types, and binding methods must agree; buffer existence alone does not prove correct shader-side interpretation.

这个边界在后端排错时尤其关键。

This boundary is especially important during backend debugging.

`GetCommandNumUints()` 与 `GetCount()` 共同解释 indirect command layout。前者说明每条 dispatch/draw command 占用多少个 `uint`，后者说明当前有多少条命令。若 command uint stride 或 count 错误，GPU 可能从错误 offset 读取参数，表现为 draw 数量异常、compute group 数错误、shader 读到错位数据或间歇性后端错误。

`GetCommandNumUints()` describes the unsigned-integer stride per command, while `GetCount()` reports the command count.

因为 indirect command 通常由 GPU 后端按固定 binary layout 解释，`HdStDispatchBuffer` 的中文阅读不能只停在“数组”层面。要同时核对每条命令的字段顺序、每字段的 `uint` 数量、各 view 的 offset、shader 端期望的 tuple type，以及 buffer array range 是否覆盖了完整命令区域。任何一个环节不一致，表面症状都可能只是渲染少了对象或 compute 没有发出。

Indirect command data is interpreted through a fixed backend layout, so the command fields, uint stride, view offsets, shader-side tuple type, and buffer-array range must agree.

`Reallocate()`、`CopyData()` 和 `GarbageCollect()` 体现 GPU resource lifecycle。`Reallocate()` 处理容量变化，`CopyData()` 把 CPU-side command data 上传或复制到 buffer，`GarbageCollect()` 清理过期资源。它们影响的是 Storm GPU 资源状态，不是 USD authored data；调试时应区分“场景数据是否正确”和“后端 command buffer 是否已经按新数据重建”。

`Reallocate()`, `CopyData()`, and `GarbageCollect()` manage GPU-resource lifetime: allocation, data upload/copy, and stale-resource cleanup.

`GetEntireResource()`、`GetResource()`、`GetResources()` 和 `GetBufferArrayRange()` 暴露不同粒度的访问路径。`GetEntireResource()` 面向整段 buffer，`GetResource()` / `GetResources()` 面向某个或多个 resource view，`GetBufferArrayRange()` 则让调用方定位 buffer-array range。它们适合用于排查 shader 读取范围、resource view 是否创建、以及 binder 是否拿到了预期资源。

The resource accessors expose the whole buffer, individual resource views, resource lists, and the buffer-array range used by Storm binding code.

受保护的 `_AddResource()` 表明 view/resource 注册有内部维护步骤，外部代码通常不应绕过 public API 直接拼接资源。构造函数与析构函数也应放在 `HdStResourceRegistry` 的所有权语境中理解：dispatch buffer 可以引用或创建 backend resources，但长期管理、共享、回收和失效检测仍要与 registry 及 buffer array range 配合。

The protected `_AddResource()` hints that resource registration is internally maintained. Ownership and reuse should be read together with `HdStResourceRegistry` and buffer-array ranges.

`DebugDump()` 是本页很有价值的诊断入口。Storm draw/compute 出现 dispatch 数量不对、indirect command 参数错位、shader 读取 view 为空、或 GPU 后端看起来没有执行命令时，应优先核对 `GetCommandNumUints()`、`GetCount()`、resource view 列表、interleaved offset 与 `DebugDump()` 输出，而不是直接怀疑 USD layer composition。

`DebugDump()` is useful when indirect command layout, resource views, interleaved offsets, or command counts need to be inspected.

常见误用包括：把 `HdStDispatchBuffer` 当作普通 vertex attribute / primvar buffer；把 `BufferResourceView` 当成会复制数据的新 buffer；忽略 `uint` command stride 与 count 的一致性；在未重新分配或未复制数据时期待 GPU 看到新命令；把 backend indirect-dispatch 问题误判为 USD authoring 问题。正确边界是：它是 Storm 内部 GPU dispatch resource，VBO、indirect dispatch、`MultiDrawIndirect`、`DispatchComputeIndirect`、`BufferResourceView` 等英文术语必须原样保留。

Common mistakes include treating this as an ordinary primvar buffer, assuming resource views copy data, ignoring command stride/count consistency, or confusing backend dispatch issues with USD authoring problems.

原文摘录 / English Source Excerpts

A VBO of a simple array of unsigned integers. This buffer is used to prepare data on the GPU for indirect dispatch i.e. to be consumed by MultiDrawIndirect or DispatchComputeIndirect. At the same time, interleaved subsets of the array are bound in several different ways to provide additional data interface to shaders. For each binding, we define 'BufferResourceView' on top of the uint array. HdBufferArray aggregates those views and HdResourceBinder binds them with specified binding method and interleaved offset....
