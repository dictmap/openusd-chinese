逐段双语理解 / Paragraph-Level Bilingual Coverage

`GlfDrawTarget` 是 `Glf` OpenGL helper 层中的离屏绘制目标对象。官方描述说它是 “a GL render target with multiple image attachments”，意思不是 USD prim 的集合，而是一个封装了 framebuffer、attachment textures、depth component 和相关 GL 状态的渲染资源容器，用来承接某个 custom render pass 的输出。

`GlfDrawTarget` represents an OpenGL render target with multiple image attachments. It is a rendering resource container, not a USD scene-graph object.

“custom render pass” 和 “arbitrary variables” 是本页容易误读的词。这里的变量是 shader 可以写入的渲染输出，例如颜色、法线、ID 或其他中间图像数据；这些输出之后可以作为 `GLSL shaders` 的 `texture samplers` 再被读取。它解决的是图形管线中的中间缓冲和采样问题，而不是 USD composition、schema authoring 或 Hydra task 调度问题。

The official wording about arbitrary variables refers to shader outputs that can later be sampled as textures, not authored USD data.

附件模型是理解该类的核心。`AttachmentsMap` / `AttachmentsContainer` 管理 named attachments，`AttachmentRefPtr` 表示引用计数持有的 attachment 对象；每个 attachment 通常对应一个 `GL_TEXTURE_2D` image。`AddAttachment()` 创建或登记新的输出附件，`DeleteAttachment()` 删除指定附件，`GetAttachment()` / `GetAttachments()` 用于读取当前集合；这些方法改变的是 OpenGL render target 的 texture resources，不是 stage 上的 property 或 material binding。

Named attachments map application-level names to `GL_TEXTURE_2D` images. Attachment operations manage GL resources rather than USD properties.

默认 depth component 也很关键：DrawTarget 会创建可在 draw pass 中作为 depth buffer 使用的深度资源，并且后续也能像普通 `GL_TEXTURE_2D` 数据一样访问。阅读 `GetDepthAttachment()`、depth buffer 相关说明或英文摘录中的 `depth component` 时，应把它理解为 GPU framebuffer 的深度附件，而不是 `UsdGeom` 中的几何属性或场景层级深度。

The depth component is a framebuffer depth attachment that can also be accessed as texture data after rendering.

`Bind()` / `Unbind()` / `IsBound()` 描述 framebuffer binding 状态。调用 `Bind()` 后后续 GL draw calls 会写入这个 draw target；`Unbind()` 解除绑定；`IsBound()` 用于检查当前上下文状态。它们依赖有效的 OpenGL context，因此不能脱离 GL 线程、上下文 lifetime 或资源创建顺序来理解，也不能把绑定状态当成 Hydra render pass 是否被调度的同义词。

Binding methods control the OpenGL framebuffer binding state for subsequent draw calls and are tied to the active GL context.

`GetFramebufferId()` 与 `GetFramebufferMSId()` 暴露 framebuffer object 标识。带 `MS` 的接口通常对应 multi-sample framebuffer，`HasMSAA()` 表示当前目标是否使用 MSAA；`Resolve()` 则把多重采样结果解析到普通可采样附件中。调试锯齿、空纹理或采样结果过期时，应检查 MSAA framebuffer、resolve 时机、attachment size 和 shader sampling 路径是否一致。

The MS framebuffer is resolved into sampleable attachments through `Resolve()`. MSAA state, attachment sizes, and sampling timing must match.

尺寸和矩阵状态也影响使用方式。`GetSize()` 反映 draw target 的 pixel dimensions；相关链接中的 `GfVec2i` 指向整数尺寸表示，`GfMatrix4d` 常用于保存绘制或投影相关矩阵状态。尺寸改变后需要重新考虑 attachment allocation、viewport、framebuffer completeness 和 shader 采样坐标，否则可能出现只渲染局部、采样错位或 framebuffer invalid 的问题。

Size, viewport, attachment allocation, and matrix state must stay coherent for valid rendering and sampling.

`CloneAttachments()` 和 `ClearAttachments()` 体现附件集合的生命周期边界。复制附件通常用于构造具有相同输出布局的新目标，清理附件则释放或断开当前集合；这些操作会影响后续 render pass 能否写入指定 attachment，也会影响 shader sampler 是否还能读取旧资源。它们不是“清空 USD layer”，也不是“删除 prim”。

Cloning and clearing attachments affect render-target resources and subsequent sampling, not USD layers or prims.

`IsValid()` 应作为 GL resource readiness 的检查来读：framebuffer、attachments、depth resource、MSAA target 和 size 必须共同满足 OpenGL framebuffer completeness。若 `IsValid()` 失败，排查顺序通常是 attachment format/size 是否一致、depth component 是否创建、MSAA 与非 MSAA attachments 是否匹配、是否在正确 GL context 中创建资源。

`IsValid()` is about GL framebuffer/resource validity. Mismatched formats, sizes, depth resources, MSAA state, or context lifetime can invalidate the target.

与相邻系统的边界要明确：`GlfDrawTarget` 属于 `Glf` 图形辅助层，提供低层 OpenGL draw-target 管理；Hydra / HdSt 可能在更高层使用类似 render targets，但该类本身不是 `HdTask`、不是 `HdRenderBuffer`，也不负责把 USD prim 转成 draw items。它可以服务于调试视图、离屏渲染、中间 pass、shader preview 或 picking/selection 管线中的图像附件，但不定义场景语义。

`GlfDrawTarget` is a low-level OpenGL helper. It is distinct from Hydra task scheduling, `HdRenderBuffer`, and USD scene semantics.

常见误用包括：把 attachment name 当成 USD attribute name；在没有 active GL context 时创建或绑定 draw target；忘记在 MSAA 模式下调用 `Resolve()` 就直接采样；修改 size 后没有重新创建 attachments；把 `GetFramebufferId()` 返回的整数长期保存到跨 context 对象中；或把英文摘录中的官方拼写 `mutliple` / `mages` 当成新术语。中文层按原意解释为 multiple image attachments / images，但保留英文原文便于核对。

Common mistakes include confusing attachment names with USD attributes, sampling before resolve, using stale framebuffer ids across contexts, or ignoring attachment reallocation after size changes.
