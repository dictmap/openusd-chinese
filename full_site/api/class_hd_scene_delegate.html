逐段双语理解 / Paragraph-Level Bilingual Coverage

`HdSceneDelegate` 是传统 Hydra 架构里的抽象场景数据查询接口。官方摘要说它是 “Adapter class providing data exchange with the client scene graph”，重点在 “data exchange”：它不负责创作 USD、不负责最终渲染，也不是 renderer backend；它把宿主场景图中的数据按 Hydra 约定交给 `HdRenderIndex`、`HdRprim`、`HdSprim`、`HdBprim`、`HdTask` 和相关同步逻辑读取。

`HdSceneDelegate` is the classic Hydra scene-data query adapter. It provides data exchange with the client scene graph; it is not a USD authoring layer, not the final renderer, and not the renderer backend.

阅读本页时要先分清边界：`UsdImagingDelegate` 是常见的 USD-to-Hydra 适配实现，负责把 composed USD scene 翻译成 Hydra 可查询数据；`HdRenderIndex` 是 Hydra 保存 render primitives 和调度同步的索引；render delegate 才是面向具体 renderer/GPU 后端的实现；`HdSceneIndex` 和 data source pipeline 是 Hydra 2.0 更偏数据流的新架构。`HdSceneDelegate` 位于这些概念之间，提供查询入口，但不拥有所有渲染状态。

Keep the boundary clear: `UsdImagingDelegate` adapts composed USD data into Hydra queries, `HdRenderIndex` owns the indexed render primitives, render delegates implement renderer-facing behavior, and the scene-index/data-source pipeline is the newer Hydra 2.0 data-flow model.

本页大量接口以 `SdfPath` 作为 id 参数。这里的 path 是 Hydra render primitive、instancer、camera、material、computation 或其他对象的 identity key，并不总是与 authored USD prim path 一对一相同。调试 delegate 时，如果一个 `SdfPath` 找不到、实例路径错位或 material binding 不生效，应先确认该 path 在 `HdRenderIndex` 中登记的对象类型，而不是直接假设它就是某个 USD stage 上的 prim。

Many methods use `SdfPath` as an identity key for Hydra objects. That key often relates to USD namespace paths, but it is not always a one-to-one authored USD prim path.

几何查询族包括 `GetMeshTopology()`、`GetBasisCurvesTopology()`、`GetExtent()`、`GetTransform()`、`GetVisible()`、`GetPurpose()`、`GetCullStyle()`、`GetDoubleSided()` 和类似方法。它们共同回答“这个 render primitive 是什么形状、在何处、是否可见、按什么 purpose 分类、边界盒如何、双面或剔除策略如何”等问题。Hydra sync 阶段会在 dirty bits 表示数据变化时重新调用这些查询，因此 delegate 返回值要稳定、可复现，并且与变更通知保持一致。

Geometry-oriented queries such as `GetMeshTopology()`, `GetBasisCurvesTopology()`, `GetExtent()`, `GetTransform()`, `GetVisible()`, and `GetPurpose()` answer the shape, placement, visibility, purpose, bounds, and display state that Hydra needs during synchronization.

primvar 和通用值查询是另一组核心接口。`GetPrimvarDescriptors()` 声明某个 primitive 在指定 interpolation 下有哪些 primvar；`GetPrimvar()` 或 `GetIndexedPrimvar()` 返回实际数据；`Get()` 返回以 `VtValue` 包装的通用值。这里要保留 `TfToken`、interpolation token、role 和 element size 的官方语义，因为 renderer 或 imaging adapter 会依赖这些名字解释数组长度、插值方式和 shader 输入。

Primvar queries describe and return per-primitive data. `GetPrimvarDescriptors()`, `GetPrimvar()`, `GetIndexedPrimvar()`, and generic `Get()` values must preserve the token and interpolation semantics expected by Hydra and renderer code.

instancing 相关接口包括 `GetInstancerId()`、`GetInstancerPrototypes()`、`GetInstanceIndices()` 和 instance category 查询。delegate 需要同时解释 prototype、instance id、indices 和类别过滤，否则画面上常见的问题是实例数量正确但 transform 或 material 错、某些实例被 pick/selection 逻辑漏掉，或者 nested instancing 路径无法追踪。

Instancing queries connect prototypes, instance ids, indices, and categories. Correct answers are required for drawing, selection, picking, and nested instancing behavior.

material、light、camera、task 和 resource 类查询体现了 `HdSceneDelegate` 的横向覆盖面。`GetMaterialId()` 把几何对象接到 material primitive；`GetCameraParamValue()` 提供 camera 参数；`GetRenderBufferDescriptor()` 等 buffer/resource 查询描述渲染缓冲；task aspects 则让 task 从 delegate 读取执行所需数据。它们说明 delegate 是场景数据的供给层，而不是单纯的 mesh topology helper。

Material, light, camera, render-buffer, resource, and task-related queries show that a scene delegate supplies broad scene data, not just mesh topology.

dirty/sync 边界是调试 `HdSceneDelegate` 时最容易误读的部分。delegate 回答数据查询；`HdChangeTracker` 和 dirty bits 告诉 Hydra 哪些数据需要重新拉取；`HdRprim::Sync()`、`HdSprim::Sync()` 或具体 primitive 的 sync 逻辑才会把这些值消化进 render index / renderer-facing state。不要在 delegate 的 getter 中偷偷修改 render index，也不要认为 getter 被调用就代表 renderer 最终状态已经更新。

Dirty bits and sync are separate from the getter itself. The delegate answers queries, while change tracking and primitive sync decide when those answers are consumed.

时间采样接口和 value lifetime 需要特别谨慎。`HdTimeSampleArray`、sample count、sample times 与返回的 `VtValue`/数组数据应有清晰生命周期，不应依赖临时对象地址或调用后马上失效的缓存。运动模糊、animated transform、deforming primvars 或 render buffer 更新异常时，常要同时检查 sample time、dirty bits、缓存 invalidation 和 delegate 返回数据的所有权。

Time-sampled values and returned arrays need stable lifetime and clear invalidation behavior. Motion blur and animated data bugs often come from inconsistent sample times, dirty tracking, or transient storage.

常见误用包括：把 `HdSceneDelegate` 当成 USD authoring API；绕过 dirty bits 直接期待 renderer 自动重读数据；把所有 delegate `SdfPath` 都当成 stage 上的 authored prim path；在 getter 里返回不稳定的临时数据；混淆 `UsdImagingDelegate`、scene-index adapter、unit-test delegate 和 renderer backend；以及在新代码里忽略 scene-index/data-source 架构与传统 delegate API 的差异。

Common mistakes include treating `HdSceneDelegate` as a USD authoring API, bypassing dirty bits, assuming every delegate path is an authored USD prim path, returning unstable transient data, and confusing the delegate with imaging adapters, scene-index adapters, or renderer backends.
