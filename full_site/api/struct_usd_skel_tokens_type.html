骨骼与层级 / Skeleton and Topology

`Skeleton`、`SkelRoot`、`joints`、`jointNames`、`bindTransforms`、`restTransforms`、`skelSkeleton` 和 `skelJoints` 描述骨骼 prim、绑定关系和 joint 拓扑。`joints`/`jointNames` 关注 joint 的顺序与命名，`bindTransforms`/`restTransforms` 关注绑定时和 rest pose 的变换数组；这些数组的顺序不匹配时，常见症状是蒙皮点位、法线或动画姿态整体错乱。Skeleton topology tokens identify skeleton prims, joint order, bind transforms, rest transforms, and relationships from skinned prims to skeleton data.

成员数据逐项语义 / Member Data Semantics

这一层按官方 `Member Data Documentation` 的点击顺序补中文用途说明。目的不是把 Doxygen 表格另起一套中文名称，而是说明每个 token 在 authoring、query、validation 或调试中的位置。This section follows the official member data sequence and explains how each token is used without renaming it.

  • `allTokens`：面向工具和诊断的全集入口，可用于枚举 `UsdSkelTokens`，但不表示某个 prim 必须 authored 所有字段。allTokens enumerates the generated token set for tooling and diagnostics.
  • `bindTransforms`：绑定时 joint 到 skeleton/geom 语境的矩阵数组，常与 `joints` 顺序一起排查；顺序错比数值错更隐蔽。bindTransforms should be debugged with joint ordering.
  • `restTransforms`：rest pose 的 joint 变换数组，用于与动画采样比较；它不是当前帧姿态，也不是几何点位置。restTransforms represents rest pose transforms, not current frame point positions.
  • `joints` 与 `jointNames`：前者常用于 skeleton 拓扑列表，后者用于名字表达；工具链若混用两者，应先确认消费端期待的是 path-like joint order 还是 display-friendly name。joints and jointNames should be interpreted according to the consuming schema or tool expectation.
  • `rotations`、`scales`、`translations`：这些是 `SkelAnimation` 上的动画数组 token。它们的 time sample、数组长度和 joint 顺序必须一起验证，不能只看 token 是否存在。rotations, scales, and translations are animation array tokens and require length and time-sample checks.
  • `primvarsSkelJointIndices`：描述每个点或 element 受哪些 joint 影响；它和 `primvarsSkelJointWeights` 成对出现时才有完整意义。primvarsSkelJointIndices identifies the joint influence indices.
  • `primvarsSkelJointWeights`：描述对应 influence 的权重。调试时要看 normalization、elementSize 和 interpolation，不能把它与 `blendShapeWeights` 混为同一类权重。primvarsSkelJointWeights stores influence weights and is separate from blend shape weights.
  • `primvarsSkelGeomBindTransform`：记录几何绑定空间，常用于把局部几何点转到 skeleton/skin 计算需要的空间；它和 `GfMatrix`、`UsdGeomXformable` 调试关系很近。primvarsSkelGeomBindTransform records geometry bind space information.
  • `primvarsSkelSkinningMethod`：选择 skinning method 的 token-valued primvar；`classicLinear` 与 `dualQuaternion` 是它的 allowedTokens 值。primvarsSkelSkinningMethod carries the allowed skinning method token values.
  • `skelAnimationSource`：把可蒙皮对象或 skeleton 连接到 `SkelAnimation` 的关键入口；relationship 没解析时，动画数组存在也不会自动驱动目标。skelAnimationSource connects targets to SkelAnimation data.
  • `skelSkeleton`：指向 skeleton 资源的关系 token，常用于确认绑定目标是否被 layer composition、payload 或 reference 改写。skelSkeleton names the relationship to the skeleton resource.
  • `skelBlendShapes` 与 `skelBlendShapeTargets`：前者描述可用 shape 名称集合，后者连接具体 target；二者错位会让权重曲线找不到实际形变。skelBlendShapes and skelBlendShapeTargets must agree with the available target set.
  • `offsets`、`normalOffsets`、`pointIndices`：用于 BlendShape target 的点级数据；排查形变穿帮时，应同时核对索引范围和点数。offsets, normalOffsets, and pointIndices describe blend shape point-level data.
  • `weight`：用于单个 BlendShape 或 inbetween 的权重语义；它不是 joint influence weight，也不是材质或物理权重。weight is a blend shape weighting concept, not a joint influence or physics weight.

Authoring 边界与常见误读 / Boundaries and Misreads

这些 token 是稳定命名工具，不是运行时算法。`UsdSkelTokens->joints` 帮助你访问属性名，真正的数据仍在 USD layer 的 attribute、relationship、time sample 或 primvar 中。不要把 token 表当成 `UsdSkelSkeleton`、`UsdSkelAnimation` 或 `UsdSkelBindingAPI` 的完整 API 文档；具体创建、校验和计算仍需要跳转到相邻 schema/class 页。Tokens are stable names, not the data itself and not the full API for evaluating a skeleton or applying skinning.

区分 property names、schema identifiers 与 `allowedTokens` 很重要。`Skeleton`、`SkelRoot`、`SkelAnimation`、`SkelBindingAPI` 更接近 schema 或 type/API 标识；`joints`、`restTransforms`、`rotations` 等是属性名；`classicLinear`、`dualQuaternion` 是 token-valued attribute 的允许值。调试时先确认自己要查的是 prim type、属性、关系还是 allowed token，再到本页查精确拼写。The page helps distinguish schema identifiers, property names, relationships, primvar names, and allowed token values.

在 Python 或 C++ 中，token 拼写错误通常不会表现为编译错误，而是表现为 attribute 查不到、relationship 未解析、array 长度不匹配或 skinning method 回退。本页的价值在于把字符串拼写集中到 `UsdSkelTokens`，再由 schema API 或 `UsdPrim::GetAttribute()` 使用。Using generated tokens reduces string spelling errors when querying with schema APIs or UsdPrim::GetAttribute().

错误归因边界 / Failure Attribution

遇到 UsdSkel 资产显示异常时，不要把所有问题都归因到 token 表。`UsdSkelTokensType` 只能证明“应该使用哪个名字”；它不能证明 layer 中已经 authored 了正确值，也不能证明 composition 后的值没有被覆盖。正确的归因顺序是：先用本页核对 token 拼写，再用 `Sdf`/`Usd` 查询 composed attribute 或 relationship，再用 `Vt`/`Gf` 检查数组与矩阵数值，最后用 `UsdGeom`、Hydra 或下游工具确认消费端行为。UsdSkelTokensType identifies names; authored values, composed values, numeric validity, and consumer behavior must be debugged in adjacent layers.

例如 `primvarsSkelSkinningMethod` 拼写正确但资产仍没有 dual quaternion 效果，可能是目标 renderer 不支持 `dualQuaternion`，也可能是 primvar authored 在错误层、interpolation 不符合预期，或几何没有有效 joint influences。再例如 `skelSkeleton` relationship 存在但解析失败，问题通常在 target path、reference/payload、namespace edit 或 layer stack，而不是 token 本身。A correctly spelled token can still point to unsupported consumer behavior, wrong layer authoring, or unresolved relationships.

当需要了解某个 schema 的构造函数、`Get`/`Create` 方法、fallback 值或 authored value 行为时，应继续点击对应 class/API 页；当只需要在 layer、日志、调试器或代码审查中确认字段名是否准确时，才回到本 token 页。这个边界能避免把 `UsdSkelTokensType` 误当成教程、算法说明或完整 schema reference。Use schema and class pages for behavior, fallback values, and authoring APIs; use this token page for exact names.

调试路径 / Debugging Path

  1. 先检查 prim 是否位于 `SkelRoot` 范围内，并确认 `Skeleton` prim、`skelSkeleton` relationship 与 `skelJoints` token 对应的 joint 列表都存在。Start by checking SkelRoot scope, Skeleton prims, skeleton relationships, and joint lists.
  2. 再检查 `joints`、`jointNames`、`bindTransforms`、`restTransforms` 的数组顺序和长度。任何顺序不一致都会让后续 `rotations`、`scales`、`translations` 映射到错误 joint。Validate array order and length across joints, jointNames, bindTransforms, and restTransforms.
  3. 对几何变形问题，检查 `primvarsSkelJointIndices`、`primvarsSkelJointWeights` 和 `primvarsSkelGeomBindTransform` 是否存在，interpolation 是否符合消费端预期，以及 joint influence 数量是否被 renderer 或工具链支持。For deformation issues, inspect joint indices, joint weights, geom bind transforms, interpolation, and consumer support.
  4. 对动画不动或姿态错误，检查 `skelAnimationSource` 指向的 `SkelAnimation`，并核对 `rotations`、`scales`、`translations` 的 time samples 与 joint 顺序。For animation issues, verify the SkelAnimation source relationship and sampled animation arrays.
  5. 对 BlendShape 问题，检查 `skelBlendShapes`、`skelBlendShapeTargets`、`blendShapes`、`blendShapeWeights`、`offsets`、`normalOffsets` 和 `pointIndices` 是否指向同一组 shape target 语义。For blend shape issues, validate target relationships, weights, point offsets, normal offsets, and point indices together.
