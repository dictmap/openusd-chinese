逐段双语理解 / Paragraph-Level Bilingual Coverage

`UsdImagingAdapterRegistry` 是 USD Imaging 层的 adapter factory registry。官方说明 “Registry of PrimAdapter plug-ins” 容易被误读成它保存已经可直接使用的 adapter objects；实际重点是它保存 factories 和注册信息，用于在需要时构造 `UsdImagingPrimAdapter` 或 `UsdImagingAPISchemaAdapter`。它是 USD schema/type 到 imaging adapter 的映射入口，不是渲染器、不是 Hydra render task，也不是 USD stage 的组成部分。

Registry of PrimAdapter plug-ins.

官方备注 “this is a registry of adapter factories, and not adapter instances” 是本页最关键的边界。factory 是根据 prim type 或 API schema key 创建 adapter 的机制；adapter instance 通常通过 `ConstructAdapter()`、`ConstructAPISchemaAdapter()` 或 `ConstructKeylessAPISchemaAdapters()` 创建，并由 per-stage data、`UsdImagingDelegate` 或相关 imaging 数据结构管理生命周期。把 registry 当成 adapter cache，会导致错误的生命周期假设。

This is a registry of adapter factories, and not adapter instances.

`GetInstance()` 表示该 registry 以 singleton 方式使用。`TfSingleton< UsdImagingAdapterRegistry >` 作为 friend 说明对象实例由 Tf singleton 基础设施管理，而不是由调用者 new/delete。调用方通常先取得全局 registry，然后查询 adapter key 或构造 adapter；这类全局注册表描述的是插件和类型能力，不描述某个 stage 当前已经实例化了哪些 adapter。

GetInstance() returns the singleton registry used by USD Imaging.

`ConstructAdapter()` 面向 prim adapter。它根据 USD prim 的 schema/type 选择对应 `UsdImagingPrimAdapter` 工厂，用来把 prim 的属性、拓扑、材质绑定、transform、visibility 等信息转成 Hydra 可消费的数据。它不会直接绘制 prim，也不会直接修改 `HdRenderIndex`；真正的同步和数据提交仍由 `UsdImagingDelegate`、adapter instance 和 Hydra 流程协作完成。

ConstructAdapter() creates prim adapters from registered factories.

`ConstructAPISchemaAdapter()` 与 `ConstructKeylessAPISchemaAdapters()` 面向 API schema adapter。前者处理有 key 或可按 schema key 定位的 API schema adapter，后者处理不依赖特定 key 的 API schema adapters。它们的作用是把 applied API schema 对 imaging 行为的影响接入适配层，例如某些 schema 需要额外 dirty tracking、visibility/purpose/material 语义或数据源扩展。

API schema adapter construction handles registered API-schema-specific imaging behavior.

`ApiSchemaAdapters` typedef 表示 API schema adapter 集合。阅读 member list 时要区分 prim adapter 与 API schema adapter：prim adapter 主要围绕 prim type；API schema adapter 主要围绕 applied API schema 的附加行为。一个 prim 可以同时受 typed schema 和多个 API schema 影响，因此 registry 查询和构造路径需要同时覆盖这两类 adapter。

ApiSchemaAdapters groups API schema adapters associated with registered schemas.

`GetAdapterKeys()` 与 `GetAPISchemaAdapterKeys()` 是调试 registry 覆盖范围的入口。前者列出 prim adapter keys，后者列出 API schema adapter keys。它们不是 stage traversal，也不保证某个具体 prim 一定会实例化 adapter；它们反映的是当前进程中注册表通过内置注册和插件发现得到的 adapter 能力集合。

GetAdapterKeys() and GetAPISchemaAdapterKeys() expose the registered adapter key sets.

`HasAdapter()` 与 `HasAPISchemaAdapter()` 是轻量能力查询。它们适合在 fallback 路径中判断某个 key 是否有注册项，再决定构造专用 adapter、使用默认 adapter，或忽略某个 schema 的 imaging 扩展。它们不是对插件加载失败的完整诊断；如果返回 false，还应检查 plugin metadata、搜索路径、外部插件开关和 build 配置。

HasAdapter() and HasAPISchemaAdapter() query whether matching registered factories exist.

`AreExternalPluginsEnabled()` 说明 registry 与 plugin discovery 有关。外部插件是否参与发现会影响自定义 `UsdImagingPrimAdapter` 或 `UsdImagingAPISchemaAdapter` 是否出现在 registry 中。排查自定义 schema 没有可见效果时，不能只看 prim 是否存在；还要确认插件是否加载、adapter key 是否匹配 schema/type、registry 是否启用了外部插件，以及对应 adapter 是否实际被 delegate 构造。

AreExternalPluginsEnabled() reports whether external adapter plug-ins participate in discovery.

与相邻类型的关系需要分层理解：`UsdImagingAdapterRegistry` 负责 factory registration 和 lookup；`UsdImagingPrimAdapter` / `UsdImagingAPISchemaAdapter` 负责具体适配逻辑；`UsdImagingDelegate` 负责把 stage/view 的变化同步给 Hydra；`TfToken` 和 `TfTokenVector` 表示 adapter keys；`TfSingleton` 管理 registry 生命周期；Hydra 的 `HdRenderIndex` 和 scene-index/data-source 层消费适配后的数据。registry 本身不保存 USD composition opinions，也不替代 adapter 的 per-stage 状态。

Related types separate registry lookup, adapter behavior, delegate synchronization, token keys, singleton lifetime, and Hydra consumption.

常见误读包括：把 registry 当成 adapter instance cache；把 `ConstructAdapter()` 当作 render submission；把 API schema adapter 当成 USD schema authoring；把 `GetAdapterKeys()` 当成 stage 中所有 prim 类型列表；把 `AreExternalPluginsEnabled()` 当成某个具体插件已成功加载的证明。正确排查顺序通常是：确认 schema/type key，确认 registry 是否有对应 key，确认外部插件发现是否启用，确认 adapter factory 能构造实例，确认 delegate 是否在正确 stage/context 中使用该 adapter。

Common debugging separates registered factory availability from stage contents, adapter instances, delegate use, and Hydra rendering.
