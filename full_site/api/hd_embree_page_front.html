中文主阅读路径

HdEmbree 是一个 Embree-based Hydra renderer plugin，基于 Intel's Embree raytracing kernels。它的定位和 HdStorm 不同：HdStorm 面向实时 Hydra renderer plugin 和多图形后端，HdEmbree 更像一个最小但完整的 renderer plugin 示例，用来展示 Hydra renderer plugin API 应该怎样接入、同步场景、提交资源、执行任务并生成图像。HdEmbree implements a Hydra renderer plugin based on Intel's Embree raytracing kernels.

官方 Overview 明确说 HdEmbree is provided as a minimal example of how to write a renderer plugin for hydra，并且 aims to be living documentation of that API。因此读这页时不能只把它当成“另一个渲染器”，而要把它当成 renderer plugin 编写路径的示例文档：它展示类如何分组、render delegate 如何创建 prim 表示、render pass 如何输出图像，以及插件类如何把 delegate 暴露给 Hydra runtime。HdEmbree is a minimal example and living documentation for writing a Hydra renderer plugin.

Embree API 的前缀是 rtc，来自 raytracing core。HdEmbree 的中文阅读重点不是翻译 Embree 外部文档，而是解释 OpenUSD/Hydra 侧如何把 Hydra 场景数据转换为 Embree scene，并在 render pass 中通过 raycasting 生成输出。所有 HdEmbree* 类名、HdEngine::Execute、HdRenderIndex::SyncAll()、CommitResources() 和 HdxRenderTask 等名称都保留原样，便于和官方 Doxygen 对照。Embree APIs are prefixed by rtc; this page focuses on the Hydra side of the Embree renderer plugin.

类职责路线图

可以把 HdEmbree 的类分成四组来读。第一组是插件和 delegate：HdEmbreeRendererPlugin 只负责让 Hydra runtime 能发现并创建 HdEmbreeRenderDelegate，而 HdEmbreeRenderDelegate 才是 renderer plugin 在运行时的中心对象。第二组是场景对象表示：HdEmbreeMesh 和 HdEmbreeInstancer 把 Hydra prim 数据转成 Embree scene 可用于 raycast 的对象或实例。第三组是执行输出：HdEmbreeRenderPass 负责把 task 映射到一次绘制，HdEmbreeRenderer 执行实际 raycasting 并写入 HdRenderBuffer。第四组是配置和传参：HdEmbreeConfig 暴露 Embree 配置，HdEmbreeRenderParam 在 Sync 期间把 scene 句柄等状态传给 prim。The HdEmbree classes can be read as plugin/delegate, scene representation, execution/output, and configuration/parameter groups.

这种分组能避免常见误读：看到 RendererPlugin 不代表它负责渲染像素；看到 RenderPass 不代表它拥有所有 Embree scene 数据；看到 RenderDelegate 不代表它直接执行每一次 raycast。Hydra 的设计把发现、数据同步、资源提交、任务执行和输出写入拆成多个对象，HdEmbree 的价值正是把这些对象以最小示例串起来。This grouping separates discovery, synchronization, resource commit, task execution, and output writing.

如果调试时某个 mesh 没有出现在结果里，优先沿着 HdSceneDelegate 到 Sync()，再到 HdEmbreeMesh/HdEmbreeInstancer 和 Embree scene 的路径检查；如果 render pass 被执行但图像为空，再检查 HdEmbreeRenderer、raycasting、HdRenderBuffer 和后续 Hdx blit。类职责路线图比逐个打开类列表更能帮助读者定位下一页该看哪里。The class map provides a practical debugging path from scene delegate data to Embree raycasting output.

Hydra 图像生成三阶段

官方页把 Hydra image generation algorithm 分成三个主要阶段：sync、commit resources 和 execution of tasks。这个结构是 HdEmbree 页的核心。Hydra 通过这些阶段把场景状态、资源提交和实际绘制解耦；renderer plugin 在每个阶段承担不同责任。读者应先理解这三阶段，再看 HdEmbree 的具体类如何参与。The Hydra image generation algorithm has three main phases: sync, commit resources, and execution of tasks.

在 sync 阶段，Hydra tracks scene state in the HdRenderIndex datastructure，并在生成图像前调用 HdRenderIndex::SyncAll() 更新场景状态。每个 prim 绑定到 HdSceneDelegate，它作为 datasource；SyncAll 会对每个 prim 调用 Sync()，从 scene delegate 拉取新数据，并更新 renderer 可用的 renderable representation。Sync uses HdRenderIndex::SyncAll() to pull data through HdSceneDelegate and update renderable representations.

HdRenderDelegate 在这里负责创建具体 renderer 的 prim classes。HdEmbree 提供 HdEmbreeRenderDelegate，它可以创建 HdEmbreeInstancer 和 HdEmbreeMesh。这些类把 mesh objects 和 mesh instances 插入 Embree scene，让后续 drawing code 能对它们进行 raycast。为了支持标准 Hydra render task，scene delegate 还支持基本 camera 类型 HdCamera。HdEmbreeRenderDelegate creates HdEmbreeInstancer and HdEmbreeMesh, which populate the Embree scene for raycasting.

把 sync 理解成“读取并转换场景状态”会更准确。它不是最终绘制，也不是一次性初始化；Hydra 会根据脏标记和 scene delegate 数据让 renderer plugin 更新自己的 renderable representation。HdEmbree 在这个阶段把 Hydra 的 mesh 和 instance 信息准备成 Embree 能查询的结构，所以如果几何体缺失或变换不更新，sync 路径往往比 render pass 更值得先检查。Sync is the phase that reads scene data and updates the renderer-specific representation.

Commit Resources 与 Executing tasks

在更新 render index 之后、运行 tasks 之前，HdEngine::Execute() 会调用 render delegate 的 CommitResources()。官方说明这个阶段给 renderer 一个提交 sync 结果的机会，例如复制 buffers 到 GPU，或最终重建 intersection datastructures。HdEmbree 不使用这个阶段，而是把 BVH rebuilds 推迟到下一次 Render() 调用，这说明不同 renderer plugin 可以选择不同资源提交策略。HdEngine::Execute() calls CommitResources(); HdEmbree defers BVH rebuilds to the next Render() call.

Executing tasks 是 HdEngine::Execute() 的最后阶段。Hydra tasks 是 schedulable callbacks，可用于渲染、维护场景状态，或与其他 renderers/libraries 交错执行。Hydra 提供 HdxRenderTask 作为简单渲染场景的方式；它接收 collection 和 render parameters，并把它们映射到由 render delegate 创建的 render pass。The last phase is running Hydra tasks such as HdxRenderTask.

Collections 指定场景中哪些部分要绘制，包括 include/exclude paths，以及如何通过 repr token 绘制。HdEmbree 提供 HdEmbreeRenderPass，它通过 raycasting 进入由 HdEmbree prim objects 填充的 Embree scene 来绘制。真正的 raycasting 在 helper HdEmbreeRenderer 中完成，输出写入 HdRenderBuffer，这些 buffer 表示图像数据；通常 Hdx 的辅助任务会把它们 blit 到 GL framebuffer。HdEmbreeRenderPass raycasts into the Embree scene and writes outputs into HdRenderBuffer objects.

任务阶段的关键是“Hydra 调度的是任务，renderer plugin 提供 task 需要的 render pass 和实现”。这意味着 HdEmbree 可以作为教学页解释 Hydra task system：HdxRenderTask 本身不直接知道 Embree 内部如何组织 BVH，它把 collection 和 render parameters 映射到 render pass；render pass 再进入 HdEmbree renderer。这个拆分让不同 renderer delegate 能接入同一个 Hydra 执行模型。HdxRenderTask maps task inputs to a render pass, while the renderer plugin implements the renderer-specific drawing.

限制与插件边界

官方 Note 很重要：HdEmbreeRenderPass currently doesn't support collection include/exclude paths，所以它在每个 render pass 中绘制 everything。多个 Embree render passes 也 unsupported，因为它们会互相覆盖，而不是 compositing。这个限制说明 HdEmbree 更偏示例和 living documentation，不应被误读成完整生产级 renderer。HdEmbreeRenderPass currently does not support collection include/exclude paths, and multiple Embree render passes are unsupported.

Renderer Plugin section 说明 HdEmbree 作为 plugin 进入 Hydra runtime，可发现类是 HdEmbreeRendererPlugin。这个 plugin class 的唯一责任是创建 HdEmbreeRenderDelegate 实例。它不负责直接执行 raycasting，不拥有所有场景同步细节，也不替代 render delegate；这和 HdStorm 的“插件入口 vs 实现库”边界类似，但 HdEmbree 文档更详细地展示了 delegate、render pass 和 renderer helper 如何协作。HdEmbreeRendererPlugin's only responsibility is to create HdEmbreeRenderDelegate instances.

因此，评估 HdEmbree 时要诚实看待它的范围：它非常适合学习 renderer plugin 编写、验证 Hydra 的 sync/task 调用顺序、理解 Embree scene 如何被填充；但它不是用来说明所有 Hydra renderer 都应采用同样的资源提交策略，也不是用来证明复杂 collection、compositing、多 render pass 或生产级图像管线已经完整支持。中文页把这些限制写出来，是为了避免把示例文档当作产品承诺。The limitations are part of the learning value: HdEmbree demonstrates the API shape without claiming full production-renderer coverage.

Embree Scene Ownership 与配置

Embree Scene Ownership section 说明 render passes raycast against 的 Embree scene 位于 HdEmbreeRenderDelegate 中。因此，每个 render index 应该有 distinct render delegate。这个所有权边界很关键：Embree scene 不是全局共享对象，也不是 render pass 自己随意拥有的对象；它通过 render delegate 管理，并通过 HdEmbreeRenderParam 在 Sync() 期间传给 prims。The Embree scene lives in HdEmbreeRenderDelegate, so each render index should have a distinct render delegate.

同一 section 还说明 scene 会在 renderer object 创建时传入，而 renderpass 在创建时获得 renderer handle。这个链条可以按“render delegate owns scene -> render param passes scene to prims -> renderer raycasts scene -> render pass invokes renderer”来记。理解这条所有权链，有助于排查为什么 mesh/instance 没进入 Embree scene、为什么 render pass 看不到新数据，或为什么多个 render index 不应共享同一个 delegate。The ownership chain connects render delegate, render param, prim sync, renderer, and render pass.

Configuration section 说明 Embree 通过 HdEmbreeConfig 暴露一些配置参数，并且可由 environment variable 控制。中文页保留 HdEmbreeConfig 和 environment variable 的英文表达，因为这些是实际调试和搜索时需要匹配的标识。HdEmbreeConfig exposes Embree configuration parameters controllable by environment variable.

Unit Test 与相邻模块阅读路径

Unit Test section 提到 testHdEmbree，它是写 minimal Hydra application 的示例。它使用 HdSt_UnitTestGLDrawing 创建窗口并提供 GL framebuffer，用 Hd_UnitTestDelegate 提供简单 scene graph/data source，然后通过 HdEngine::Execute() 运行 HdxRenderTask 渲染场景。这个测试是学习 Hydra renderer plugin 最有价值的入口之一。testHdEmbree shows a minimal Hydra application using HdSt_UnitTestGLDrawing, Hd_UnitTestDelegate, HdxRenderTask, and HdEngine::Execute().

相邻阅读路径建议是：先读 Hd 理解 render index、scene delegate、render delegate 和 task 模型；再读 HdEmbree 看一个 Embree-based plugin 如何实现这些接口；如果目标是实时 viewport 和多 backend 图形接口，对比 HdStorm 和 HdSt；如果目标是渲染输出 schema，转到 UsdRender；如果目标是耗时或并发，结合 Trace 和 Work。Related reading: Hd, HdStorm, HdSt, UsdRender, Trace, and Work answer different questions around Hydra rendering.

如果要自己写一个最小 Hydra renderer plugin，建议按官方页顺序复现：先建立可发现的 renderer plugin 类，再实现 render delegate，让它能创建 mesh、instancer、render pass 和 render param；然后走通 sync，把 scene delegate 数据放入 renderer-specific 表示；最后通过 HdxRenderTask 执行 render pass，并确认输出进入 HdRenderBuffer。testHdEmbree 提供了从窗口、GL framebuffer、简单 scene graph 到 HdEngine::Execute() 的端到端线索。A minimal renderer plugin can follow the official order from plugin discovery to render delegate, sync, render pass execution, and testHdEmbree.

调试和排查路径

如果 HdEmbree 无法被选择，先检查插件发现和 HdEmbreeRendererPlugin，不要直接进入 Embree raycasting。此时问题可能是插件注册、动态库加载或运行环境，而不是 mesh 同步或 render pass。若 plugin 已被发现但 delegate 创建失败，再看 HdEmbreeRenderDelegate 的初始化和 Embree 配置。Plugin discovery and delegate creation should be debugged before raycasting internals.

如果 delegate 创建成功但几何体不显示，沿着 HdRenderIndex::SyncAll()、HdSceneDelegate、Sync()、HdEmbreeMesh/HdEmbreeInstancer 到 Embree scene 的路径检查。这个阶段要确认 scene delegate 是否提供了正确拓扑、点数据、实例和变换，也要确认 HdEmbreeRenderParam 是否把 scene 句柄传到了需要的对象。Missing geometry should be debugged along SyncAll, scene delegate, prim sync, and Embree scene population.

如果几何体进入 scene 但输出异常，再看 task execution：HdxRenderTask 的 collection 和 render parameters 是否正确，HdEmbreeRenderPass 是否被执行，HdEmbreeRenderer 是否写入 HdRenderBuffer，Hdx 辅助任务是否把结果 blit 到 GL framebuffer。性能问题则结合 Trace 记录 sync、BVH rebuild、render pass 和 blit 阶段的耗时。Output and performance issues belong in task execution, render buffer, blit, and Trace analysis.

运行时数据流理解

HdEmbree 的运行时数据流可以按“Hydra 场景状态到 Embree 查询结构，再到图像输出”理解。Hydra 先用 render index 管理 prim 状态，scene delegate 提供每个 prim 的数据，render delegate 创建对应的 HdEmbree prim 类；这些 prim 在 Sync 阶段把 mesh 和 instance 信息放入 Embree scene。之后 render pass 不再直接向 scene delegate 查询完整场景，而是对已经准备好的 Embree scene 进行 raycasting。The runtime data flow moves from Hydra scene state into the Embree query structure and then into image output.

这种数据流解释了为什么官方要单独讲 Embree Scene Ownership。Embree scene 放在 HdEmbreeRenderDelegate 中，意味着 delegate 生命周期和 render index 之间有明确对应关系；如果共享或复用不当，就可能出现旧数据、错误实例或 render pass 之间互相覆盖的问题。调试时要把“谁拥有 scene”和“谁临时使用 scene”分开看。Embree scene ownership explains the lifecycle relationship between render delegate, render index, and render passes.

输出路径也需要分层理解。HdEmbreeRenderer 负责 raycasting 并把像素写到 HdRenderBuffer，但用户最终看到的窗口图像还依赖 Hdx 辅助任务把 buffer blit 到 GL framebuffer。也就是说，图像为空可能是 raycast 没命中、render buffer 没写入、blit 没执行，或 framebuffer/window 流程出错；这些不是同一个层级。Empty output can come from raycasting, render buffer writes, blit, or framebuffer/window handling, and these layers should be separated.

常见误读和调试提示

  • 不要把 HdEmbree 当成完整生产 renderer。官方定位是 minimal example 和 living documentation。HdEmbree is a minimal example and living documentation, not necessarily a full production renderer.
  • 不要忽略 CommitResources() 策略差异。HdEmbree 将 BVH rebuilds 延迟到下一次 Render()，这和其他 renderer 的资源提交策略可能不同。HdEmbree defers BVH rebuilds to Render(), so resource commit behavior differs by renderer.
  • 不要假设 collection include/exclude paths 在 HdEmbree 中完整支持。官方 Note 已说明当前 render pass 会绘制 everything。HdEmbreeRenderPass currently does not support collection include/exclude paths.
  • 不要让多个 render index 共享同一个 Embree scene 所有权模型；官方建议每个 render index 使用 distinct render delegate。Each render index should have a distinct render delegate because the Embree scene lives in the delegate.
  • 不要把 HdEmbreeRendererPlugin 与 HdEmbreeRenderer 混淆。前者创建 render delegate，后者是 raycasting helper。HdEmbreeRendererPlugin creates delegates; HdEmbreeRenderer performs helper raycasting work.
