逐段双语理解 / Paragraph-Level Bilingual Coverage

`SdfPrimSpec` 表示某个 `SdfLayer` 内的 prim description。它是 layer-level authored data：记录一个 prim 在单个 layer 中直接写下的 specifier、typeName、metadata、properties、child prims、references、payloads、inherits、specializes 和 variant selections。它不是 `UsdPrim`，也不是 composition 后的最终场景视图。

`SdfPrimSpec` represents a prim description in an `SdfLayer`. It stores authored layer-level data and is not the composed `UsdPrim` view seen on a `UsdStage`.

官方描述强调每个 `SdfPrimSpec` 都定义在一个 layer 里，并由该 layer namespace hierarchy 中的 `SdfPath` 标识。这个路径是 spec 在 layer 内的位置，不等于“该 prim 一定会在最终 stage 上出现”。最终 `UsdPrim` 是否存在、类型是什么、哪些 metadata 生效，还要经过 composition、load rules、variant selection 和其他 layer opinions。

Each `SdfPrimSpec` is defined in a layer and identified by an `SdfPath` in that layer's namespace hierarchy. Composition determines what appears as a final `UsdPrim`.

`New()` 可以在 containing `SdfLayer` 下创建 root-level prim spec，也可以在 parent `SdfPrimSpec` 下创建 child prim spec。`SdfCreatePrimInLayer()` 是常见 helper，可快速补齐路径上的 primSpecs hierarchy；但 helper 只是写入 spec，不会替调用者决定 schema type、specifier 语义、metadata 合法性或 composition 结果。

`New()` creates root-level or nested prim specs, and `SdfCreatePrimInLayer()` can create a hierarchy quickly. These APIs author specs; they do not solve composition or validate intended schema semantics for the caller.

`specifier` 和 `typeName` 是阅读本页的核心字段。`specifier` 描述这个 spec 是 def、over 还是 class 等 authored intent；`typeName` 记录 schema type 名称。两者都只是 layer opinion：它们会参与 composition，但不等于最终 `UsdPrim` 的完整类型判断，也不能替代 `UsdSchemaRegistry` 或 schema wrapper 的运行时解释。

`specifier` and `typeName` are authored opinions on a prim spec. They participate in composition but are not the same as final schema interpretation on a composed prim.

`NameChildrenView`、`PropertySpecView`、`AttributeSpecView` 和 `RelationshipSpecView` 暴露的是 authored child specs。它们适合遍历和编辑 layer 中直接存在的 children/properties，但不要把这些 view 当成 stage 上的 composed children。一个 composed child 可能来自别的 layer、reference、payload 或 variant，也可能被更强 opinion 覆盖或隐藏。

The child and property views expose authored specs in the layer. They are not equivalent to composed children or properties on a stage.

`SdfPrimSpec` 也是多种 list op 和 metadata opinion 的容器。references、payloads、inherits、specializes、variantSets、variant selections、relocates、permissions、active、instanceable、kind 和 assetInfo 等字段都在这里存放为 authored data。`ClearReferenceList()`、`ClearPayloadList()`、`ClearInheritPathList()`、`BlockVariantSelection()` 之类 API 操作的是具体 opinion 或 list-editing 状态，不是求解后的结果。

`SdfPrimSpec` stores list-editing and metadata opinions such as references, payloads, inherits, specializes, variants, active, instanceable, kind, and assetInfo. Clear and block APIs modify authored opinions, not resolved composition results.

ordering 相关 API 如 `ApplyNameChildrenOrder()` 和 `ApplyPropertyOrder()` 处理 authored ordering metadata。它们影响 name children 或 property children 的显示/遍历顺序，但不改变 child prim 或 property 的语义内容。调试顺序问题时，应同时查看 order metadata 和实际 children view，不要只看 UI 展示。

Ordering APIs apply authored order metadata. They affect traversal or display order, not the semantic content of the child prims or properties themselves.

与 `SdfLayer`、`SdfPath`、`SdfAttributeSpec`、`SdfRelationshipSpec`、`SdfChildrenView`、`TfToken`、`VtDictionary`、`VtValue` 和 `UsdPrim` 的关系需要分层理解：`SdfLayer` 是持有 spec 的文件/层，`SdfPath` 是 layer namespace 中的地址，attribute/relationship specs 是 property-level children，`UsdPrim` 是 stage composition 后的访问视图。

Related types sit at different layers: `SdfLayer` owns specs, `SdfPath` identifies them, attribute and relationship specs are property children, and `UsdPrim` is the composed stage-level view.

排查“为什么最终 prim 不对”时，建议同时打印 `SdfPrimSpec::GetPath()`、所在 `SdfLayer` identifier、specifier、typeName、reference/payload/list op、variant selection，以及最终 `UsdPrim` path 和 composed metadata。这样可以判断问题发生在 authoring、layer strength、composition arc、variant selection、payload loading 还是 stage view 层。

When debugging prim mismatches, compare the prim spec path, layer identifier, authored fields, list operations, variant selections, and the final composed `UsdPrim` view.

常见误用包括：把 `SdfPrimSpec` 当作可直接代表最终 scene object；在多个 layer 参与时只看一个 layer 的 spec；清掉 list op 后误以为 composition 结果必然消失；把 `NameChildrenView` 当成 composed traversal；以及忘记 `SdfPrimSpec` 是底层 authoring API，通常需要比 `UsdPrim` / `UsdStage` API 更谨慎地处理 layer mutation。

Common mistakes include treating a prim spec as the final scene object, inspecting only one layer, confusing authored child views with composed traversal, and assuming list-op edits directly describe final stage results.
