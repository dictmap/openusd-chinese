逐段双语理解 / Paragraph-Level Bilingual Coverage

`GfMatrix4f` 是 Gf Linear Algebra 模块中的 4x4 `float` matrix 值类型。它用于表达基础矩阵运算和 3D transformation，但它本身不是 USD 场景中的 authored transform op，也不是 `UsdGeomXformOp` 或 prim schema；它只是可被上层 API、渲染代码、相机代码或数学工具传递的数值对象。

`GfMatrix4f` is a 4x4 `float` matrix value type in Gf Linear Algebra. It can represent 3D transformations, but it is not a USD-authored transform operation such as `UsdGeomXformOp`.

官方摘录强调矩阵按 row-major order 定义，`matrix[i][j]` 表示第 `i` 行第 `j` 列。这个约定影响构造函数参数顺序、`operator[]` 的访问方式、调试打印结果和矩阵乘法顺序；把它和 column-major / column vector 数学库混用时，最容易出现转置、乘法顺序反了或平移列/行放错的问题。

The official reference states that matrices are row-major and `matrix[i][j]` indexes row `i`, column `j`. That convention affects construction, `operator[]`, debug output, and multiplication order.

`ScalarType` 对本类来说是 `float`，这意味着它比 `GfMatrix4d` 更轻、更贴近 GPU/渲染常见数据格式，但精度和数值稳定性也更低。涉及远离原点的大场景、长链路 transform composition、相机投影、逆矩阵或重复分解时，应主动判断是否需要 `GfMatrix4d`，不要默认把 `GfMatrix4f` 当成 double precision 容器。

For this class `ScalarType` is `float`. It is compact and useful for rendering-style data, but it has lower precision than `GfMatrix4d` and should be chosen deliberately.

构造与设置函数覆盖多个入口：默认矩阵、从标量或数组构造、从向量行构造，以及 `SetIdentity()`、`SetZero()`、`SetRotate()`、`SetScale()`、`SetTranslate()`、`SetLookAt()` 等就地设置方法。阅读这些 API 时要区分“把当前矩阵设置成某种矩阵”和“在已有矩阵上追加组合”；如果需要组合，通常应显式使用矩阵乘法并核对顺序。

Construction and setup APIs create or assign a matrix state. Calls such as `SetRotate()`, `SetScale()`, `SetTranslate()`, and `SetLookAt()` should not be read as implicit append operations.

3D transform 方法是本页的核心。`Transform()` 通常用于点或齐次坐标，平移分量会参与结果；`TransformDir()` 用于方向向量，通常不应让平移改变方向；`TransformAffine()` 则面向 affine transform 语境。调试光照方向、法线方向、射线方向、相机视线和 point position 时，必须先判断输入是 point 还是 direction，否则矩阵看似正确但结果会偏移。

The transform APIs distinguish points, homogeneous coordinates, and directions. `Transform()` and `TransformDir()` have different translation semantics and are not interchangeable.

`GetDeterminant()`、`GetInverse()` 和相关 inverse / determinant 逻辑用于判断矩阵是否可逆以及构造逆变换。对 singular matrix 或 near-singular matrix，`float` 精度下的容差会直接影响结果；在包含非均匀 scale、shear、投影成分或退化轴的矩阵上，不应无条件相信逆矩阵，而应检查 determinant、返回值和调用方的误差容忍度。

`GetDeterminant()` and `GetInverse()` are central for invertibility. Singular or near-singular matrices require explicit tolerance and error handling, especially with `float` precision.

`Factor()`、`ExtractTranslation()`、`ExtractRotation()`、`ExtractRotationMatrix()` 和 `DecomposeRotation()` 体现了从 4x4 矩阵中提取 transform 成分的需求。它们适合把已有矩阵解释为平移、旋转、缩放或其他分量，但含 shear、非均匀 scale、负 scale、投影项或数值噪声时，分解结果可能不符合直觉；这类 API 应和调用场景的矩阵来源一起阅读。

Factorization and extraction APIs interpret matrix contents as transform components. Shear, non-uniform scale, negative scale, projection terms, and numerical noise can make the decomposition subtle.

矩阵乘法和比较操作应按 OpenUSD/Gf 的 row vector 约定理解。矩阵乘法不是可交换运算，`A * B` 与 `B * A` 的变换顺序不同；当矩阵来自外部引擎、DCC、GPU shader 或其他数学库时，先确认 row/column convention、坐标系 handedness、单位和是否需要 transpose，比事后猜测结果更可靠。

Matrix multiplication order matters. When exchanging matrices with other engines, DCC tools, or shaders, confirm vector convention, handedness, units, and transpose requirements.

相关类型说明了本类的边界：`GfMatrix4d` 是 double precision 对应类型，`GfMatrix3f` 常用于 3x3 rotation / linear 部分，`GfVec3f`、`GfVec4f` 承载点、方向或齐次向量，`GfRotation` 与 `GfQuatf` 承载旋转表达。把这些类型互相转换时，应明确是否丢失平移、是否丢失投影项、是否从 double 降到 float、以及旋转表示是否归一化。

Related types include `GfMatrix4d`, `GfMatrix3f`, `GfVec3f`, `GfVec4f`, `GfRotation`, and `GfQuatf`. Conversions can drop precision, translation, projection terms, or rotation normalization assumptions.

常见误读包括：把 `GfMatrix4f` 当作 USD 文件里的可编辑 xform op；忘记 `float` 精度边界；用 `TransformDir()` 处理 point；对含 scale/shear 的矩阵直接提取 rotation；未检查 singular matrix 就调用 inverse；以及把其他库的 column-major 矩阵直接塞进 `GfMatrix4f`。本页的中文层正是为了把这些 API 边界提前讲清楚。

Common mistakes include treating the type as an authored USD transform op, ignoring `float` precision, mixing point and direction transforms, extracting rotation from scaled or sheared matrices, and using inverse operations without singularity checks.
