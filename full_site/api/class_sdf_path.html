逐段双语理解 / Paragraph-Level Bilingual Coverage

官方开头把 `SdfPath` 定义为 used to locate objects in layers or scenegraphs 的 path value。这里的 path value 表示它是一个可复制、可比较、可作为 key 使用的值对象；它本身不拥有 prim、property 或 layer 数据，也不保证目标对象一定存在。`SdfPath` 的职责是用统一语法表达 USD 命名空间中的对象位置。

The official introduction defines `SdfPath` as a path value used to locate objects in layers or scenegraphs. Path value means that it is copyable, comparable, and usable as a key; it does not own prim, property, or layer data, and it does not guarantee that the target object exists. Its role is to express object locations in the USD namespace using a uniform syntax.

第一种官方用途是作为 `SdfLayer` 中寻址和访问值的 storage key。底层 layer 保存的是 specs 和 fields，很多读写操作需要先用 `SdfPath` 指定“哪个 prim、property 或 target”。因此同一个 path 在不同 `SdfLayer` 中可能命中不同 authored opinions；路径值相同不代表底层数据来源相同。

The first official use is as a storage key for addressing and accessing values held in an `SdfLayer`. A low-level layer stores specs and fields, and many read/write operations need an `SdfPath` to identify which prim, property, or target is being addressed. The same path can resolve to different authored opinions in different layers, so equal path strings do not imply the same data source.

第二种用途是作为 scenegraph objects 的 namespace identity。对 `UsdPrim`、`UsdProperty`、`UsdAttribute` 和 `UsdRelationship` 这类高层对象来说，path 常常是用户最稳定的标识形式。它描述对象在 stage namespace 中的位置，但对象是否 active、loaded、defined、abstract 或 composed 成功，还需要通过 `UsdPrim`、`UsdProperty` 或 composition 查询判断。

The second use is as a namespace identity for scenegraph objects. For high-level objects such as `UsdPrim`, `UsdProperty`, `UsdAttribute`, and `UsdRelationship`, a path is often the most stable user-visible identifier. It describes the object's location in the stage namespace, but whether the object is active, loaded, defined, abstract, or successfully composed must be checked through `UsdPrim`, `UsdProperty`, or composition queries.

第三种用途是通过 relative paths 引用其他 scenegraph objects。relative path 的含义取决于包含它的 prim 或 property；例如 relationship target 如果是 relative path，通常相对于持有该 relationship 的 prim object 解释。调试 target path 时，必须先问“这是 authored relative path，还是已经解析后的 absolute path”。

The third use is to refer to other scenegraph objects through relative paths. A relative path is interpreted relative to the prim or property that contains it. For example, a relationship target authored as a relative path is usually interpreted relative to the prim object holding that relationship. When debugging target paths, first ask whether the value is an authored relative path or an already resolved absolute path.

absolute path 和 relative path 是本页最基础的区分。absolute prim path 从 `AbsoluteRootPath()` 开始，例如 `/World/Model`；relative path 不从根开始，常用于 relationship target、relocation 或局部引用。不要把 relative path 直接当成 stage 绝对路径使用，也不要仅凭字符串是否包含 `/` 判断语义。

The distinction between absolute and relative paths is fundamental. An absolute prim path begins at `AbsoluteRootPath()`, such as `/World/Model`; a relative path does not begin at the root and is often used for relationship targets, relocations, or local references. Do not treat a relative path as an absolute stage path, and do not infer semantics solely from whether the string contains `/`.

prim path、property path、target path 和 relational attribute path 表达不同层级。`/World/Cube` 是 prim path，`/World/Cube.size` 是 property path，relationship target 或 mapper/relational attribute 会在 property path 之后继续附加目标语法。`ContainsPropertyElements()`、`ContainsTargetPath()`、`IsPrimPath()` 等方法用于在工具代码中明确区分这些情况。

Prim paths, property paths, target paths, and relational attribute paths describe different levels. `/World/Cube` is a prim path, `/World/Cube.size` is a property path, and relationship target or mapper/relational-attribute syntax can extend beyond a property path. Methods such as `ContainsPropertyElements()`, `ContainsTargetPath()`, and `IsPrimPath()` help tools distinguish these cases explicitly.

variant selection path 是 USD 特有的路径语法之一，表示某个 prim 在某个 variant set 中选择了特定 variant。`AppendVariantSelection()` 和 `ContainsPrimVariantSelection()` 等方法应优先用于构造和检查这类路径。手写字符串很容易破坏 path grammar，尤其是在嵌套 variant、property path 和 target path 混合时。

Variant selection paths are a USD-specific part of path syntax and represent the selection of a variant inside a variant set on a prim. Prefer methods such as `AppendVariantSelection()` and `ContainsPrimVariantSelection()` for constructing and checking these paths. Hand-written strings can easily violate path grammar, especially when nested variants, property paths, and target paths are mixed.

路径构造 API 的选择会影响代码稳健性。普通代码应优先使用 `AppendChild()`、`AppendProperty()`、`AppendTarget()`、`AppendVariantSelection()`、`MakeRelativePath()`、`MakeAbsolutePath()` 这类语义明确的 helper；`AppendElementString()` 和 `AppendElementToken()` 更适合底层工具，调用者必须已经理解 path element syntax。

The choice of path-construction API affects robustness. General code should prefer semantic helpers such as `AppendChild()`, `AppendProperty()`, `AppendTarget()`, `AppendVariantSelection()`, `MakeRelativePath()`, and `MakeAbsolutePath()`. Lower-level helpers such as `AppendElementString()` and `AppendElementToken()` are appropriate when the caller already understands path element syntax.

thread-safety 提示应和 `SdfPath` 的值对象语义一起理解。路径对象通常适合跨线程共享和比较，因为它们表示不可变路径值；但这不等于所有围绕路径的外部缓存、全局 path table、layer 查询或 stage 查询都自动线程安全。多线程工具应把 `SdfPath` 的共享和被路径索引的对象生命周期分开处理。

The thread-safety note should be read together with `SdfPath` value semantics. Path objects are generally suitable for sharing and comparison across threads because they represent immutable path values. This does not mean that every external cache, global path table, layer query, or stage query around those paths is automatically thread-safe. Multi-threaded tools should separate sharing `SdfPath` values from managing the lifetime of objects indexed by those paths.

在 composition 调试中，`SdfPath` 通常是从高层症状回到底层数据的入口。`UsdPrim::GetPath()` 给出 composed prim 的 namespace identity，`SdfLayer` 使用相同或相关路径查找 authored specs，`Pcp` 相关结构则解释这些路径如何跨 layer stack 和 composition arcs 组合。路径是线索，不是完整解释；真正的强弱顺序和来源仍要结合 layer、prim index 和 resolver 结果分析。

In composition debugging, `SdfPath` is often the entry point from a high-level symptom back to low-level data. `UsdPrim::GetPath()` provides the namespace identity of a composed prim, `SdfLayer` uses the same or related paths to find authored specs, and `Pcp` structures explain how those paths compose across layer stacks and composition arcs. A path is a clue, not the whole explanation; strength ordering and source attribution still require layer, prim-index, and resolver analysis.
