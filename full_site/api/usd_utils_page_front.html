逐段双语理解 / Paragraph-Level Bilingual Coverage

官方 Overview 的第一句说明 `UsdUtils` contains utility classes and functions for managing, inspecting, editing and creating USD assets。这里的关键词是 utility、assets 和 workflow。`UsdUtils` 不是单一 scene schema，也不是 composition 引擎；它是围绕 USD 资产处理、工具链、命令行批处理和管线胶水代码的实用函数集合，常把 `Sdf`、`Usd`、`Ar`、`Tf`、file format 和 packaging 能力组合起来。

UsdUtils provides utility classes and functions for managing, inspecting, editing, and creating USD assets.

Authoring 分组提供 copying metadata between layers 的能力。它适合处理 layer 之间的 metadata 迁移、工具生成 layer 或批量修正 layer metadata 的场景。它不是任意强度的 composition override，也不会替用户决定最终 composed value；它只是在 authoring 工具层辅助拷贝或整理元数据。

Authoring provides functionality for copying metadata between layers.

Diagnostic delegate 相关工具有两类：Coalescing diagnostic delegate 聚合来自 `Tf` diagnostic management system 的 warnings 和 statuses；Conditionally abort diagnostic delegate 可以根据 inclusion / exclusion rules，在 non fatal USD error 或 warning 发生时中止操作。批处理资产、转换 USDZ、发布管线或持续集成校验时，这些 delegate 可作为质量门禁，而不只是把消息打印到终端。

Diagnostic delegates aggregate warnings/statuses or abort operations based on non-fatal USD errors or warnings.

Stitching 分组提供 merging layers together 的能力，也可以从一组 layers 生成 `value clips`，相关入口包括 `stitchClips`。它常用于把多段时间采样、缓存片段或分批输出合并成可消费的 USD 表达。阅读时应把 stitching 与 composition 区分：stitching 是工具生成或合并 layer 内容，composition 是运行时按照 arcs 解析 opinions。

Stitching merges layers and can generate value clips from sets of layers.

Dependency analysis and asset packaging 分组用于提取给定 layer 的 external dependencies，例如 `references`、`payloads` 和 `sublayers`，并支持 `UsdUtilsExtractExternalReferences`、`UsdUtilsCreateNewUsdzPackage` 等流程。这里的 asset dependency 不只是文件系统路径字符串，还要结合 `Ar` resolver、`SdfLayer`、package-relative path、asset path authored value 和 layer stack 语义理解。

Dependency analysis and asset packaging extract external dependencies such as references, payloads, and sublayers.

Introspection 分组用于理解 `UsdStage` 的各种特征，包括 memory usage、prim counts、prototype counts 等。它适合工具报告、性能诊断、资产规模评估和回归监控。它读取并总结 stage 特征，但不负责修改 asset，也不替代 `UsdStage`、`Pcp` 或 schema 页面对 composition 和 scene semantics 的解释。

Introspection helps understand characteristics of a UsdStage such as memory usage, prim counts, and prototype counts.

Pipeline 分组提供 establishing pipeline conventions 的 utilities。它是把项目约定、资产命名、路径策略或发布流程固化进工具的入口，而不是 USD 核心规范的一部分。因此同一个 `UsdUtils` API 在不同 studio 或 pipeline 中可能被不同层包装；阅读时要区分 OpenUSD 提供的工具能力和本地管线约定。

Pipeline utilities help establish pipeline conventions around USD workflows.

StageCache 分组提供 `UsdUtilsStageCache`，用于管理一个可被多个 clients 共享的 singleton `UsdStageCache` instance。它和 `UsdStageCache` 的核心页面相邻，但 `UsdUtilsStageCache` 更偏应用或工具层便利封装。若问题是 stage 是否被复用、缓存生命周期、client 间共享，先看 cache 使用方式；若问题是 stage composition 本身，再回到 `UsdStage` / `Pcp` / `Sdf`。

StageCache provides UsdUtilsStageCache for managing a shared singleton UsdStageCache among clients.

`complianceChecker.py` 提供 Python classes and methods，用于检查给定 USD asset 或 USDZ package 的 compliance。它适合发布前验证、自动化质量门禁和包结构检查。compliance failure 不一定表示文件无法被 `UsdStage` 打开，也可能是发布规范、包内容、依赖组织或工具约定不满足。

complianceChecker.py checks compliance of USD assets or USDZ packages.

User Processing Functions 是本页最容易误读的部分。Localization routines 可以接受自定义 processing function；该函数会在 dependency traversal 发现 asset paths 时被调用。它允许修改 asset path、添加额外 dependencies，或从输出中移除 asset path / dependencies。它的目的不是重写任意 layer 内容，而是在本地化或打包依赖遍历中定制输出。

User processing functions customize localization routines as asset paths are discovered during dependency traversal.

定义 processing function 时，官方强调函数针对每个 recursive traversal 发现的 asset path 调用，参数包括 containing `SdfLayerHandle` 和 `UsdUtilsDependencyInfo`。返回值传回需要对 asset path 做的修改：返回新的 `UsdUtilsDependencyInfo` 表示替换或添加依赖，返回空 asset path 可以移除引用。C++ 返回 `UsdUtilsDependencyInfo` struct，Python 返回 `UsdUtils.DependencyInfo` object。

A processing function receives the layer and UsdUtilsDependencyInfo for each discovered asset path and returns dependency changes.

Modifying AssetPaths and Dependencies 的示例说明了常见用法：把 `.jpg` 转成 `.png`，或为 `.obj` transient dependency 自动加入对应 `.mtl`。这类函数适合资产格式转换、发布包整理和临时依赖补全，但必须保持 dependency traversal 语义清晰，否则可能生成无法加载或路径不一致的包。

Processing functions can transform asset paths or add related transient dependencies such as sidecar files.

官方特别警告：processing function 的意图是修改当前 asset path；在 processing 期间修改包含该 path 的 layer 可能导致 assets behave unexpectedly or fail to load。也就是说，函数应返回修改后的依赖信息，而不是直接在回调里改 layer 内容。对已有 package 中的 asset path 进行修改也可能触发 coding error，需要用 `SdfFileFormat::IsPackage` 和 `ArIsPackageRelativePath` 判断当前 layer/reference 是否处于 package 场景。

Processing functions should modify the current asset path through returned dependency information, not mutate the containing layer during traversal.

Removing Asset paths 小节说明，如果希望从 localization 输出中排除某些 assets，可以让返回的 `UsdUtilsDependencyInfo` object 的 `assetPath` 为空字符串。例如 `.txt` extension 可以被排除。这一机制适合去掉临时文件、无关 sidecar 或不应打包的资源，但也要确认下游 asset 是否仍能打开，否则会把依赖裁剪成运行时缺失。

Returning an empty assetPath removes that asset path from the localization output.

调试 `UsdUtils` 时应把 utility failure 拆成底层模块：路径解析看 `Ar`，layer I/O 和 package 判断看 `SdfLayer` / `SdfFileFormat`，composition 和 stage 行为看 `UsdStage` / `Pcp`，诊断通道看 `Tf`，打包输出看 USDZ/package 工具。`UsdUtils` 常把这些能力组合起来，因此错误表面在 utility 层，根因可能在 resolver、layer、composition、file format plugin 或用户 processing function。

UsdUtils failures should be decomposed into asset resolver, layer I/O, composition, diagnostics, packaging, and user-processing-function layers.

English API names, class names, function names, and dependency terms are retained so readers can map the Chinese explanation to code and Doxygen references.
