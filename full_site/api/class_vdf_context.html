逐段双语理解 / Paragraph-Level Bilingual Coverage

`VdfContext` 解决的问题是：当 Vdf evaluation 调用一个 computation callback 时，如何把这次调用所需的输入访问、输出写回、请求信息和诊断入口集中交给 callback。官方摘录把它称为传给 computations callbacks 的 parameter bundle，并强调函数访问 inputs 的唯一 API 就是 context；因此它是执行期上下文，不是持久化 graph state，也不是 USD stage metadata。

`VdfContext` is the parameter bundle passed to computation callbacks. It is the only API through which a computation function accesses its inputs.

`VdfContext` 与 `VdfNode` 的边界是本页最重要的读法。`VdfNode` 描述 data-flow graph 中节点的拓扑、端口和依赖关系；`VdfContext` 则存在于一次 callback 执行期间，告诉 callback 哪些输入值可读、哪些输出被请求、结果应该写到哪里，以及出错时如何携带 node debug name 报告。一个定义图结构，一个承载执行调用现场。

`VdfNode` defines graph structure, while `VdfContext` is the runtime callback context used for value access, output production, and diagnostics.

输入读取相关 API 要分清“端口存在”和“值可用”。`GetInputValue()` 用于以类型化方式读取输入值，`GetInputValuePtr()` 提供指针形式访问，`HasInputValue()` 用来判断当前 computation 中输入值是否可用。缺少输入值时，可能是上游没有产生输出、依赖 mask 没有请求该输入、连接未建立，或者本次 evaluation scope 并不需要它；不要简单理解为 authored USD 数据缺失。

Input access APIs distinguish available runtime values from merely declared ports or authored data. Missing values should be debugged through scheduling, masks, and connections.

输出写回相关 API 体现 Vdf 的按需求值。`IsOutputRequested()` 让 callback 判断某个 output 是否真的被本次 evaluation 请求；多输出或昂贵计算节点应先检查它，避免计算不需要的结果。`SetOutput()` 的多个 overload 写回普通结果，`SetEmptyOutput()` 明确写回空输出，`SetOutputToReferenceInput()` 表示输出引用某个输入以表达 pass-through 或避免复制。

Output APIs support demand-driven evaluation. A computation can ask whether an output is requested, set concrete values, set empty outputs, or reference an input as output.

`SetEmptyOutput()` 和“不调用 `SetOutput()`”不是同一件事。前者显式表达本次 computation 产生了一个 empty output，后者可能表示 callback 没有写结果、没有被请求或发生了逻辑错误。调试空结果时，应同时查看 `IsOutputRequested()`、输入值可用性、callback 分支和 warning/coding error。

An empty output is an explicit result, not the same as failing to set an output. Debugging should check requested outputs, available inputs, and diagnostics.

`SetOutputToReferenceInput()` 的语义是让输出引用输入，而不是复制成一个全新的 owned value。它适合 pass-through 节点或转发场景，但必须确保引用的输入类型、生命周期和 evaluation ownership 与下游预期一致。若误用，可能把本应独立的输出变成依赖上游存储的引用。

`SetOutputToReferenceInput()` expresses reference-style output forwarding. It should be used only when type, lifetime, and ownership expectations are correct.

诊断接口把错误和 warning 绑定到当前 callback 上下文。`Warn()` 用于可恢复但需要提示的问题，`CodingError()` 用于调用或实现层面的错误，`GetNodeDebugName()` 给出当前节点可读名称。复杂 Vdf graph 排错时，保留 node debug name、输入端口名、请求输出名和 callback 分支，通常比只看最终 value 更有效。

Diagnostics are tied to the active computation context. Node debug names and port names are important evidence when debugging graph evaluation.

与 `VdfEvaluationState`、`VdfReadWriteAccessor<T>` 的关系也要分层理解。evaluation state 更像调度和缓存状态的底层容器，read/write accessor 提供具体 typed data 读写机制；`VdfContext` 则是 callback 作者面对的稳定入口，把这些底层机制包装成输入读取、输出设置和诊断 API。

`VdfEvaluationState` and accessors are lower-level execution/storage mechanisms. `VdfContext` is the callback-facing API that packages them into input, output, and diagnostic operations.

常见误读包括：把 context 当作跨 evaluation 可保存的对象，把 `IsOutputRequested()` 当成节点永久状态，把 `HasInputValue()` 当成端口存在性检查，把 `SetEmptyOutput()` 当成没有设置输出。正确读法是：`VdfContext` 只描述当前 callback 调用现场；它的值访问和输出写回都受当前 request、dependency mask、evaluation state 和 graph 连接关系约束。

Common misreads come from treating context as persistent state. `VdfContext` is scoped to the active callback invocation and constrained by the current request and graph state.
