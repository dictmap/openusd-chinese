中文主阅读路径 / Chinese Main Reading Path

CLI::App 是 OpenUSD API 文档中随 CLI11.h 暴露的命令行应用类参考。它不是 USD scene description 的数据模型类，也不是插件注册或 JSON 解析模块；它的职责是描述一个命令行程序如何组织顶层 app、子命令、选项、flag、帮助文本、配置文件、回调和解析结果。阅读这页时应把它看成工具入口的参数解析基础设施：上层工具把命令行语义放进 App，再由 parse()、parse_from_stream()、remaining()、count()、get_subcommands() 等接口取回解析后的状态。CLI::App represents a command line program and its parsing surface, not an OpenUSD scene model.

解析生命周期

1. 建模命令行

构造 App(std::string app_description, std::string app_name) 后，用 name()、alias()、description()、footer()、group() 组织用户可见的命令说明。这个阶段只定义命令模型，不读取命令行输入。Create and describe the command model.

2. 添加选项与 flag

add_option() 用于带值参数，add_flag() 用于布尔或计数式 flag，add_option_function()、add_flag_callback() 和 add_option_group() 负责更复杂的转换、回调或分组。不要把这些接口理解成 USD attribute 创建；它们只服务命令行 parser。Options and flags describe parser inputs, not USD attributes.

3. 组织 subcommands

add_subcommand()、require_subcommand()、get_subcommands()、got_subcommand() 和 protected 的 _find_subcommand()、_parse_subcommand() 决定多层命令结构。OpenUSD 工具常见的 “主命令 + 子操作” 模式可以用这类接口表达。Subcommand APIs organize nested command behavior.

4. 执行 parsing

parse()、parse_from_stream()、_parse()、_parse_arg()、_parse_positional()、_parse_config()、_parse_stream() 等接口处理 argv、stream 和配置来源。解析结果再由 count()、remaining()、remaining_for_passthrough() 等读取。Parsing consumes command line, stream, and config inputs.

核心 API 分组

Public Member Functions

  • 回调组：callback()、final_callback()、parse_complete_callback()、preparse_callback()、immediate_callback() 分别对应解析前、解析完成、最终处理或即时回调。调试命令行工具时，应确认回调是在解析前、解析后还是处理阶段触发。Callbacks mark pre-parse, post-parse, final, and immediate execution points.
  • 解析容错组：allow_extras()、allow_config_extras()、prefix_command()、fallthrough()、validate_positionals() 控制未知参数、配置文件多余项、前缀命令和 positionals 的处理。它们决定命令行工具是严格失败还是把未识别参数留给后续系统。Parsing policy APIs decide how strict the parser is.
  • 帮助与格式组：formatter()、formatter_fn()、config_formatter()、help_ptr、get_help_ptr()、set_help_all_flag() 影响 help 文本和配置输出。这里的 formatter 是命令行帮助格式，不是 Hydra/renderer formatter。Formatter APIs control CLI help and config formatting.
  • 查询组：count()、count_all()、get_option()、get_option_no_throw()、get_subcommands()、got_subcommand()、remaining() 让调用者读取解析结果。查询结果来自命令行 parser，不应被误认为 stage、layer 或 plugin registry 状态。Query APIs inspect parser state.

Protected Member Functions 与 Protected Attributes

protected 区域包含 _configure()、_validate()、_process()、_process_callbacks()、_process_config_file()、_process_env()、_process_extras()、_process_help_flags()、_process_requirements()、_recognize()、_move_option() 等内部阶段。普通读者不需要逐行进入实现，但要知道这些函数解释了错误从哪里来：配置文件、环境变量、help flag、requirements、extras 或 subcommand 验证失败，都可能在这些阶段被报告。Protected functions document the parser's internal stages and failure points.

Doxygen 成员表阅读策略

这类页面的英文正文很大一部分是自动生成的重载列表。中文读者不需要把每个重载逐字翻译一遍，而应该先识别“同名重载属于同一能力”。例如 add_option() [1/4] 到 add_option() [4/4] 都属于添加带值选项；差别通常是目标变量、回调、默认值、类型推导或转换函数的入口不同。add_flag() [1/4] 到 add_flag() [4/4] 同理，都是 flag 建模接口，不应被拆成四个互不相关的概念。Overload groups should be read as one capability with several entry forms.

另一个关键是区分“配置 parser 的函数”和“读取 parser 结果的函数”。allow_extras()、fallthrough()、prefix_command()、validate_positionals()、require_subcommand() 属于规则配置；count()、get_option()、get_subcommands()、got_subcommand()、remaining() 属于读取解析结果。混淆这两组接口，会让调试时误以为某个查询函数能改变解析行为。Configuration APIs and query APIs have different responsibilities.

callbacks 也要按阶段阅读：preparse_callback() 是解析前的钩子，适合做输入规模或上下文准备；parse_complete_callback() 在解析完成后执行，适合做解析状态确认；callback() 与 final_callback() 代表处理完成后的执行点；immediate_callback() 则改变触发时机，适合需要在解析链中尽早响应的子命令。若工具运行顺序异常，先画出这些回调的阶段，再看每个 subcommand 是否覆盖了父级行为。Callbacks should be debugged by lifecycle phase.

配置文件相关接口需要和命令行参数分开看。config_formatter()、config_to_str()、allow_config_extras()、_parse_config()、_process_config_file() 说明 CLI::App 可以从配置文本读取或输出参数形态；但它不负责解析 USD layer、Alembic、Draco、OpenVDB 或 JSON scene data。若用户报告“配置项没有生效”，先确认是 CLI config 没进 parser，还是 parser 输出已经正确但后续工具没有把结果应用到 OpenUSD 操作。Config-file parsing is separate from USD data loading.

典型使用场景

在 OpenUSD 工具链里，这类命令行 parser 常见于离线转换、检查、调试、批处理和测试程序。典型流程是：创建 App，添加全局 flag，添加输入/输出路径 option，按功能拆分 subcommand，配置 help 和 formatter，调用 parse()，再根据 count()、remaining()、get_subcommands() 或具体 option 的值进入业务逻辑。也就是说，本页读懂的是“工具入口怎么把用户输入变成结构化状态”，不是“这些状态如何修改 stage”。The page explains how command line input becomes structured parser state.

如果一个 OpenUSD 命令支持 “严格模式”和 “透传模式”，通常会涉及 allow_extras()、prefix_command() 或 fallthrough()。严格模式适合发现拼写错误和不支持的参数；透传模式适合 wrapper 命令把剩余参数交给下游程序。中文页特别保留这些函数名，是为了让读者能在源码和调用栈中直接搜索，不把调试线索藏在意译词里。Strict and passthrough modes are controlled by parser policy APIs.

如果一个工具需要多层子命令，例如 tool inspect、tool convert、tool validate，应重点看 add_subcommand()、require_subcommand()、get_subcommands() 和 protected subcommand 解析路径。父级 app、子 app、option group 和 callback 之间的关系，决定了参数到底由哪一层消费。很多“参数明明写了但没有生效”的问题，本质是参数挂在了错误的 app 或 subcommand 上。Nested subcommands determine which app consumes each argument.

边界与常见误读

  • CLI::App 不是 USD composition、Sdf layer、Prim、Attribute 或 PluginRegistry。它只处理命令行输入到程序配置的映射。CLI::App is not a USD scene description API.
  • add_option() 和 add_flag() 的 “option/flag” 不等于 USD property。它们是 CLI 参数；值可能后续影响 USD 操作，但本类不负责写 stage。CLI options are not USD properties.
  • allow_extras()、prefix_command() 和 fallthrough() 很容易造成“未识别参数没有报错”的现象。调试时先查这些策略，再看子命令是否吞掉了参数。Extras, prefix commands, and fallthrough affect unknown argument handling.
  • config_formatter() 与 formatter() 分别关注配置输出和 help 输出，不是 JSON formatter、shader formatter 或 renderer formatter。Formatters here are CLI help/config formatters.
  • missing_t 和 startup_mode 是源页保留的 typedef/enum 文档入口。中文页解释语义，但不重命名枚举或类型。Typedef and enum names are preserved.

调试路径

  1. 如果命令没有进入预期子命令，先查 add_subcommand() 的创建位置，再看 require_subcommand()、_valid_subcommand()、get_subcommands() 和 got_subcommand()。For missing subcommands, inspect subcommand creation and validation.
  2. 如果参数被认为是多余项，检查 allow_extras()、allow_config_extras()、prefix_command()、fallthrough()、remaining() 和 remaining_for_passthrough()。For extras, inspect strictness and passthrough policies.
  3. 如果 help 或配置输出不符合预期，检查 formatter()、formatter_fn()、config_formatter()、help_ptr 和 set_help_all_flag()。For help/config output, inspect formatter APIs.
  4. 如果 callback 顺序错乱，按 preparse_callback()、parse_complete_callback()、callback()、final_callback()、immediate_callback() 的阶段顺序排查。For callback order, separate pre-parse, parse-complete, final, and immediate callbacks.

失败归因时不要直接把所有错误归到 parse()。如果报错发生在参数识别前，通常与 app 名称、alias、subcommand 注册或 token 拆分有关；如果报错发生在识别之后，通常是 option 类型、required、excludes、needs、positionals 或 config extras 的约束失败；如果报错发生在业务逻辑开始前，可能是 _process_callbacks()、_process_requirements() 或 help flag 提前结束。把错误放回这些阶段，才能判断应该修改命令行声明、解析策略、callback，还是修改后续 OpenUSD 工具代码。Classify failures by parser phase before changing tool logic.

本页还保留 clear()、clear_aliases()、disabled()、enabled_by_default()、excludes()、failure_message()、ignore_case、ignore_underscore 等名称，是为了提醒读者：一个命令行 app 的行为不仅由 option 列表决定，也由启用状态、互斥/依赖关系、名称匹配规则和错误消息策略共同决定。中文说明覆盖这些语义边界，但不改写 Doxygen 成员名。Parser behavior also depends on enablement, exclusions, name matching, and failure-message policies.
