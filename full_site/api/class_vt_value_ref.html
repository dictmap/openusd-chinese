This page has been promoted to a complete bilingual reference. The Chinese coverage explains `VtValueRef` as a non-owning, type-erased value view interoperating with `VtValue`, while preserving English API names and excerpts for direct comparison.

逐段双语理解 / Paragraph-Level Bilingual Coverage

`VtValueRef` 解决的问题是：当 API 需要接受“任意类型的值”时，不必总是构造拥有数据的 `VtValue`，也不必为每个类型写一套重载。它提供一个 type-erased view，把普通 typed value 或已有 `VtValue` 暂时包装成统一的引用入口。这个设计的收益是轻量、少复制、少分配；代价是调用方必须严格尊重被引用对象的生命周期。

`VtValueRef` lets APIs accept values of arbitrary type without always constructing an owning `VtValue` or adding many overloads. It is a lightweight type-erased reference view, with lifetime responsibility left to the caller.

官方原文最重要的一句是 non-owning：`VtValueRef` 不拥有被引用的值，不延长对象生命周期，也不保证底层存储仍然有效。它通常应作为函数参数或 automatic variable 使用；把它保存到长期容器、异步任务、缓存对象或跨线程延迟执行回调中，通常都是危险用法。需要跨作用域保存时，应使用 owning `VtValue` 或明确复制底层数据。

The most important official constraint is non-ownership. `VtValueRef` must not persist beyond the lifetime of the viewed value; use owning `VtValue` or an explicit copy when data must outlive the call.

`VtValueRef` 与 `VtValue` 的边界是：`VtValue` 是拥有型 type-erased value container，可保存、复制和跨作用域传递；`VtValueRef` 是非拥有型 view，适合只读或短期访问。普通 typed values 和 `VtValue` 都可隐式转换为 `VtValueRef`，这让通用 API 可以统一入口，但也容易掩盖临时对象生命周期问题。

`VtValue` is an owning type-erased value container, while `VtValueRef` is a non-owning view. Both typed values and `VtValue` can convert implicitly to `VtValueRef`, which is convenient but makes temporary lifetimes important.

`Get<T>()` 是 typed access 的主要入口。调用它之前应使用 `GetTypeid()`、`GetTypeName()`、`GetType()` 或 `_TypeIs()` 确认实际类型，不能只依赖调用者的静态预期。类型不匹配时的行为应按底层 API 约定理解；对外部工具而言，更稳妥的模式是先诊断类型，再选择相应读取路径。

`Get()` is the main typed access path. Callers should verify runtime type information with `GetTypeid()`, `GetTypeName()`, `GetType()`, or `_TypeIs()` before assuming a static type.

`GetType()`、`GetTypeid()`、`GetTypeName()` 和 `GetKnownValueTypeIndex()` 分别服务不同层次的类型诊断：`typeid` 面向 C++ RTTI/类型比较，type name 面向日志和人类可读信息，`TfType` 或 known value type index 面向 OpenUSD 自身类型系统和高性能分发表。调试 `VtValue` / `VtArray` / Python 包装值混用时，这些方法通常比直接猜测模板类型更可靠。

The type-query methods serve different diagnostic layers: C++ `typeid`, human-readable type names, `TfType`, and known value type indices used by OpenUSD dispatch paths.

数组相关方法体现了 `VtValueRef` 对 scalar 与 array 的统一处理。`GetArraySize()` 返回被引用值的数组规模，`GetElementTypeid()` 返回数组元素类型；但这些信息描述的是底层值，不表示 `VtValueRef` 自己拥有数组存储。对非数组值调用时，应遵循 API 约定并避免把返回值误读成一个空数组。

`GetArraySize()` and `GetElementTypeid()` describe the underlying viewed value, not storage owned by `VtValueRef`. They help handle scalar and array values through a uniform API.

`CanHash()` 与 `GetHash()` 表示底层值是否支持通用 hash 操作。并非每种 type-erased value 都可 hash；调用 `GetHash()` 前应先检查 `CanHash()`，尤其在把值用作 cache key、change detection 或 deduplication 时。hash 能力取决于底层类型及其在 Vt 系统中的注册/实现，而不是取决于 `VtValueRef` 本身。

`CanHash()` and `GetHash()` describe whether the underlying type supports generic hashing. Hashability belongs to the viewed value’s type, not to `VtValueRef` itself.

`CanComposeOver()` 和 `CanTransform()` 表示该值是否支持通用 composition 或 transform 操作。它们常用于 USD 值合成、array edit、变换应用或 generic value processing 的前置检查。这里的 “can” 是能力查询，不是执行操作本身；调用方仍需选择正确的 composition/transform API，并处理类型不支持时的退路。

`CanComposeOver()` and `CanTransform()` are capability checks for generic composition and transformation workflows. They do not perform the operation themselves.

`_Get()`、`_GetMutable()`、`_TypeIs()`、`_ArrayHelper` 和 `_TypeInfoFor` 属于更底层的实现支撑接口。普通调用者应优先使用公共 typed API 和类型查询方法；阅读这些成员时，重点是理解 `VtValueRef` 如何在 type-erased 外壳下保存指针、类型信息和数组 helper，而不是把它当作拥有型 variant。

Protected helpers such as `_Get()`, `_GetMutable()`, `_TypeIs()`, `_ArrayHelper`, and `_TypeInfoFor` support the implementation of the type-erased view. Normal callers should prefer the public typed API.

常见误读包括：把 `VtValueRef` 当成可长期保存的 `VtValue`；把临时对象隐式转换成 `VtValueRef` 后延迟使用；未检查类型就调用 `Get<T>()`；把 `GetArraySize()` 结果当成 ref 自己的数组所有权；或在未检查 `CanHash()` / `CanComposeOver()` / `CanTransform()` 时直接执行 generic operation。正确理解是：它是短生命周期、非拥有、轻量的任意类型值视图。

Common mistakes include storing `VtValueRef` long-term, delaying use of references to temporaries, calling `Get()` without type checks, assuming array ownership, or running generic operations without capability checks.
