逐段双语理解 / Paragraph-Level Bilingual Coverage

`SdrShaderProperty` 解决的问题是：在 shader discovery registry 中用结构化对象描述一个 shader node 的接口，而不是执行 shading computation。它告诉工具和渲染集成“这个节点有哪些输入/输出、每个端口叫什么、类型是什么、默认值是什么、如何显示、能否与另一个端口连接”。因此它处在 shader definition、USD attribute authoring、节点图 UI 和 renderer plugin 之间的接口层。

`SdrShaderProperty` describes a shader-node interface in the shader discovery registry. It does not execute shading; it exposes names, types, defaults, display metadata, and connection compatibility for node-graph and renderer integration.

官方原文强调 property 可以是 input 或 output，并且必须有 name 和 type。`GetName()` 给出稳定的 property 标识，通常用于 node graph、connection target 和 UI 查找；`GetLabel()` 则更偏向人类可读显示名称。不要把 label 当成稳定 API id，也不要把 implementation name、info string 或 UI label 混同为同一个概念。

The official definition says a property is an input or output and must have a name and type. `GetName()` is the stable interface identifier, while `GetLabel()` is display-oriented and should not be treated as the stable API key.

`GetType()` 与 `GetTypeAsSdfType()` 体现 shader registry 类型和 USD/Sdf 值类型之间的边界。registry 侧类型可能来自 shader definition 或 renderer-specific 类型系统，而 `GetTypeAsSdfType()` 把它转换为更适合 USD attribute authoring 的 `SdfValueTypeName` 视角；转换时还要关注 `SdrSdfTypeIndicator`，因为 array、matrix、asset、token 等类型在 UI、USD 和 renderer 之间可能有不同约定。

`GetType()` reports the shader-registry type, while `GetTypeAsSdfType()` converts that information into an `SdfValueTypeName`-oriented view for USD authoring. `SdrSdfTypeIndicator` matters when registry and Sdf type semantics differ.

`GetDefaultValue()` 与 `GetDefaultValueAsSdfType()` 是默认值读取的核心方法。前者保留 registry/discovery 侧的默认值表示，后者更适合需要写入 USD 或与 authored value 对齐的工具链。常见误读是把 default value 当作用户已经 authored 的值；实际上默认值描述的是 shader property 的接口默认行为，是否写入 layer、是否覆盖、是否参与 composition 还要看具体 authoring 工具。

`GetDefaultValue()` preserves the registry-side default representation, while `GetDefaultValueAsSdfType()` is more useful for USD-authored values. A default value is interface metadata, not proof that the value has been authored into a layer.

`GetArraySize()` 和 `GetTupleSize()` 用来补充类型形状信息。它们帮助工具区分标量、固定长度 tuple、数组属性和 UI 控件布局，但不等同于运行时 shader 参数缓冲区的实际内存布局。构建材质编辑器时，应把 name/type/default/array size/tuple size 组合起来生成控件，而不是只看 type 字符串。

`GetArraySize()` and `GetTupleSize()` add shape information used by UI and authoring tools. They should be interpreted together with name, type, and default values rather than as renderer memory layout.

metadata 相关方法需要特别区分新旧路径。`GetMetadata()` 返回旧式 `SdrTokenMap`，官方说明它已 deprecated；新代码应优先使用 `GetMetadataObject()` 与 `SdrShaderPropertyMetadata`。metadata、hints、options、page、shown-if 规则主要服务 shader discovery、UI 分组、条件显示和 renderer/plugin 集成，而不是直接改变 shading 结果。

Metadata access has an old and new path. `GetMetadata()` returns deprecated `SdrTokenMap` metadata, while `GetMetadataObject()` and `SdrShaderPropertyMetadata` are the preferred direction for newer code.

`GetHelp()`、`GetHints()`、`GetOptions()`、`GetPage()`、`GetShownIf()` 和 `GetLabel()` 面向 authoring/UI 体验。它们可以告诉节点编辑器如何分组参数、如何给用户展示帮助文本、哪些选项可选、在什么条件下显示某个控件。它们不是 shader evaluation，也不是 USD composition 规则；如果 UI 与 renderer 行为不一致，应分别检查 discovery metadata、USD authored values 和 renderer implementation。

`GetHelp()`, `GetHints()`, `GetOptions()`, `GetPage()`, `GetShownIf()`, and `GetLabel()` support authoring and UI behavior. They do not evaluate shaders and do not define USD composition semantics.

`CanConnectTo()` 是本页连接语义的核心：它判断另一个 `SdrShaderProperty` 是否可以连接到当前 property。这个判断通常需要考虑 input/output 方向、type compatibility、array/tuple 形状、renderer 或 source type 约定，以及 node graph 所处的 render context。它只能回答 property 层面的兼容性，不代表完整材质网络一定能在目标 renderer 中成功编译。

`CanConnectTo()` checks whether another `SdrShaderProperty` can connect to this property. The decision involves direction, type compatibility, shape, source type, render context, and renderer conventions, but does not guarantee full material-network compilation.

`GetImplementationName()` 与 `GetInfoString()` 适合诊断和实现层映射。implementation name 可与 renderer/plugin 内部参数名关联，info string 可用于调试或显示补充信息；两者都不应替代 `GetName()` 作为稳定接口标识。维护跨渲染器 shader definition 时，必须区分 stable property name、UI label、implementation-side name 和 metadata field。

`GetImplementationName()` and `GetInfoString()` are useful for implementation mapping and diagnostics. They should not replace `GetName()` as the stable interface identifier.

`SdrShaderProperty` 与相邻类型的边界是：`SdrShaderNode` 表示整个 shader node，`SdrShaderProperty` 表示 node 上的单个端口或参数，`SdrShaderPropertyMetadata` 表示该 property 的结构化元数据，`SdfValueTypeName` 是 USD/Sdf authoring 侧的值类型视角。把这些层次分清，才能避免把 shader registry 的发现结果、USD layer 中的 authored value 和 renderer 运行时参数混为一谈。

The boundary is: `SdrShaderNode` represents the whole node, `SdrShaderProperty` represents one port or parameter, `SdrShaderPropertyMetadata` stores structured metadata, and `SdfValueTypeName` is the USD/Sdf authoring type view.

常见误读包括：把 `GetDefaultValue()` 当成已经写入 USD 的 authored value；继续依赖 deprecated `SdrTokenMap` metadata；用 label 或 help 文本当稳定标识；只凭 type 字符串判断连接合法；或把 property metadata 当作 shader 执行结果。正确理解是：`SdrShaderProperty` 是 shader 节点接口描述对象，服务 discovery、authoring、UI、连接检查和 renderer integration。

Common misreads include treating defaults as authored USD values, relying on deprecated `SdrTokenMap` metadata, using labels as stable identifiers, checking only type strings for connections, or confusing property metadata with shader evaluation results.

Represents a property (input or output) that is part of a SdrShaderNode instance. A property must have a name and type, but may also specify a host of additional metadata. Instances can also be queried to determine if another SdrShaderProperty instance can be connected to it. Note SdrTokenMap metadata is deprecated in favor of SdrShaderPropertyMetadata .
