逐段双语理解 / Paragraph-Level Bilingual Coverage

`EsfPropertyInterface` 是 Exec Scene Framework 中面向 scene adapter implementation 的 property abstraction。官方说它 “closely resembles the read-only interface of `UsdProperty`”，这句话的边界很重要：它借用了 `UsdProperty` 的只读查询形态，但并不是 authoring API，也不负责写入 USD layer。它的职责是在 Esf adapter 与 exec network compiler 之间提供一个稳定、可 journal 的 property 读取接口。

`EsfPropertyInterface` is a property abstraction for scene adapter implementations. It resembles the read-only `UsdProperty` interface, but it is not a USD authoring wrapper.

官方还说明该类的 public methods 会被 exec network compiler 调用。因此它不是普通应用代码随意使用的 property helper，而是编译 OpenExec / ExecUsd 网络时读取场景属性信息的抽象边界。compiler 通过这个接口读取 property name、namespace、所属对象和类型相关信息，同时不需要知道底层来自真实 `UsdProperty`、adapter cache，还是其他 scene representation。

The public methods are called by the exec network compiler, making this class an adapter boundary rather than a general-purpose property utility.

每个 public method 都带有 `EsfJournal*` 参数，这是本页最容易被忽略的设计点。`EsfJournal` 捕获 “conditions for recompilation”：也就是某次属性查询依赖了哪些 scene 状态、命名、类型或关系。后续 stage、prim 或 property 发生变化时，系统可以依据 journal 判断 exec network 是否需要重新编译。这里的 journal 是依赖记录机制，不是用户日志，也不是调试文本输出。

The `EsfJournal*` argument records recompilation conditions for dependency tracking; it is not merely a diagnostic log.

`GetBaseName()` 与 `GetNamespace()` 体现了 property 命名语义。`GetBaseName()` 面向 property 的末端名称，`GetNamespace()` 面向 `foo:bar:baz` 这类 namespaced property 的前缀部分。这里的 namespace 是 USD property namespace，不是 C++ namespace，也不是 prim path namespace。翻译中保留这两个方法名，可以避免把 property name 拆分规则误解成文件或 C++ 作用域规则。

`GetBaseName()` and `GetNamespace()` refer to USD property naming semantics, not C++ namespaces or prim path namespaces.

因为 `EsfPropertyInterface` 是 abstract class，页面的重点不在对象内部存储，而在派生类必须实现的查询契约。scene adapter implementation 可以把 `EsfAttribute`、`EsfRelationship` 或其他 property-like 对象包装成这个接口；exec network compiler 只依赖抽象方法和 journaling 行为。这样可以把编译器与具体 USD scene representation 解耦。

As an abstract interface, it defines the query contract that adapters implement so the compiler does not depend on a concrete USD object layout.

本页的相邻类型说明了接口所在层次：`EsfObjectInterface` 是更通用的对象接口，`EsfPrim`、`EsfStage`、`EsfAttribute`、`EsfRelationship` 描述 adapter 侧的 scene abstraction；`SdfPath` 和 `TfToken` 承载路径和 tokenized name；`VtValue` 与 `TfType` 参与值和类型描述；`UsdProperty` 是它所模仿的官方只读接口。理解这些关系比逐个查方法签名更重要。

Related types place this interface at the adapter/compiler boundary across objects, paths, tokens, values, and type metadata.

常见误读包括：把 `EsfPropertyInterface` 当成可写 `UsdProperty` wrapper；认为 `EsfJournal*` 只是可选调试参数；把 `GetNamespace()` 理解为 C++ namespace；或者在 adapter 外长期保存接口对象并假定 scene 变化不会影响其语义。正确读法是：每次 compiler 通过它查询 property 时，都应同时记录会影响重新编译的依赖条件。

Common mistakes come from treating this interface as a writable USD property wrapper or ignoring journaling as part of the compiler dependency model.
