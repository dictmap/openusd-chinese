Houdini USD Example Workflow

本页把官方 Houdini USD Example Workflow 教程整理成中文主阅读路径，并保留必要英文原文、命令、路径和产品名。它不是现代 Houdini 集成的最新推荐流程，而是一个历史教程，用来说明 USD 中心管线怎样让 DCC 工具只写入局部覆盖层，再把覆盖层合入 shot 的组合结果。

This page provides a Chinese-first reading path for the official Houdini USD Example Workflow while retaining the original English terms, command, paths, and product names needed for comparison.

逐段双语理解 / Paragraph-Level Bilingual Coverage

页面首先要读成一个历史示例，而不是当前最佳实践。官方说明指出，这个教程没有定期测试；更关键的是，Houdini USD plugin 已经在 USD 20.05 中从 USD 发行版移除，因为 Houdini 现在在 Solaris 中提供原生 USD 支持。因此，本地复刻页保留它的原因不是建议读者安装旧插件，而是保留一个清楚的 USD pipeline 示例：场景数据留在 USD layer stack 中，DCC 工具负责编辑局部意见，最终仍由 USD 组合规则决定结果。

The first reading boundary is historical. The official page states that this tutorial is not regularly tested, and that the Houdini USD plugin was removed from USD in version 20.05 in favor of Houdini Solaris native USD support.

教程的入口是示例 shot，而不是一个孤立的 Houdini 文件。命令 usdview USD/extras/usd/tutorials/Houdini/shot.usda 用来打开原始 USD shot。读者应该先在 usdview 中观察场景结构，再进入 Houdini。这样做的目的，是让编辑者先确认 prim path、layer composition 和可见的资产关系，避免在 DCC 中凭视觉猜测去编辑错误对象。

The workflow starts from a USD shot, not from a standalone Houdini scene. The command usdview USD/extras/usd/tutorials/Houdini/shot.usda opens the original USD file so the scene structure can be inspected before editing.

官方步骤要求在 usdview 中找到 /World/sets/house。这个 prim path 是后续编辑的关键锚点：如果路径写错，Houdini 导出的覆盖层即使语法正确，也不会影响目标房屋。场景还包含地形等上下文对象，导入这些上下文对象不是为了把整场景重新烘焙，而是为了在 Houdini 中判断 house 移动后的相对位置和视觉关系。

The important prim path is /World/sets/house. The terrain and other context objects are imported for spatial context, while the house prim is the main target of the authored transform opinion.

进入 Houdini 后，核心动作是用 USD import workflow 把 house 和地形带入 Houdini 场景。中文读者需要把这里理解为“读取 USD 中已经存在的场景描述”，而不是“把 USD 转成 Houdini 私有资产并丢失组合语义”。如果只需要移动房屋，理想输出是一个稀疏的 override layer，只包含相对于原始 shot 的差异意见。

Inside Houdini, the house and terrain are imported for editing. The intended output is a sparse USD override layer rather than a full copy of the shot.

移动 house 是教程最容易误读的地方。实际要学习的不是某个 Houdini 按钮，而是 USD 的意见强度模型：新导出的 layer 应该在更强的位置表达 transform opinion，并且只覆盖需要改变的 xform 数据。这样，原始资产、材质、引用、payload、变体和其他未变内容仍来自原有 layer stack，Houdini 只贡献局部修改。

The main lesson is the USD opinion model. The Houdini-authored layer should provide a stronger transform opinion for the house while leaving unrelated asset data in the original layer stack.

导出后的 USD 文件通常作为 overlay 或 sublayer 被插入到 shot 中。这里的检查重点是 layer order：如果覆盖层太弱，原始 shot 的意见会继续获胜；如果覆盖层太强或内容过多，可能无意覆盖原有资产结构。正确的交付物应该能用 usdview 对比原始 shot 和新 shot，并清楚看到 house 位置变化，同时其他场景内容保持稳定。

The exported USD is inserted back into the shot as an overlay or stronger layer. The result should be inspected in usdview to compare the original and updated composition.

现代 Houdini 用户需要特别注意 Solaris 边界。由于旧 Houdini USD plugin 已从 USD 发行版移除，本页不应被当作安装或运行旧插件的操作手册。它更适合作为迁移阅读材料：旧教程中的 USD import、局部 transform override、导出 USD layer、再由 USD 组合到 shot 的思想，仍然对应 Solaris 和现代 USD 管线中的常见工作方式。

Modern Houdini users should treat the page as a pipeline concept example. Solaris is the contemporary Houdini-side USD environment, while the old plugin-specific details are historical.

如果结果没有变化，排查顺序应从 USD 语义开始，而不是先怀疑渲染。先检查目标 prim path 是否仍是 /World/sets/house，再检查导出的 layer 是否真的包含 transform opinion，然后检查该 layer 是否放在足够强的位置，最后用 usdview 的 layer browser 或 composition inspection 确认最终 stage 中哪一层提供获胜意见。

If the result does not change, debug the prim path, the exported transform opinion, the layer strength, and the composed stage before looking at rendering-specific causes.

从管线角度看，教程的价值在于“可追溯修改”。原始 shot.usda 继续描述基础场景，Houdini 产物只记录一次艺术调整。这样做可以让下游复查时清楚回答三个问题：谁移动了 house，移动意见写在哪一层，这个意见为什么在最终 stage 中获胜。若把整场景重新导出成一个大文件，这些问题会变得模糊，USD 的分层协作优势也会被削弱。

From a pipeline perspective, the tutorial demonstrates traceable edits. The base shot remains the source, and the Houdini-authored layer records the local artistic change.

读者还应区分“导入用于观察”和“拥有用于改写”。导入 house、terrain 和上下文 geometry，并不表示 Houdini 应该成为这些资产的新权威来源。USD 中常见的生产约定是让资产部门、布景部门、灯光部门分别拥有不同层或不同 namespace 的意见；本教程中的 Houdini 步骤更像一个 layout/shot 调整层，而不是资产发布层。

Importing context does not mean taking ownership of all imported assets. The Houdini step is best read as a layout or shot-adjustment layer, not as a replacement asset publish.

在实际复刻或迁移到 Solaris 时，最重要的是保留“输入 USD stage、局部编辑、输出 USD layer、重新组合验证”这四段流程。旧教程中的具体插件菜单可以变化，但四段边界不能混在一起。输入阶段负责读取现有 stage，编辑阶段负责生成用户意图，输出阶段负责写入可组合 USD，验证阶段负责确认 layer strength、路径和视觉结果。

When reproducing the idea in Solaris, preserve the four-stage boundary: input USD stage, local edit, output USD layer, and recomposition verification.

这也是为什么本页保留英文命令和路径。usdview、USD/extras/usd/tutorials/Houdini/shot.usda、/World/sets/house、Houdini、Solaris、layer stack、sublayer、overlay 和 xformOp 都是可核对的技术坐标。中文解释负责说明它们之间的关系，英文标识负责让读者在工具、文件和官方文档中准确定位同一个对象。

The retained English identifiers act as technical coordinates. The Chinese text explains the relationships, while the original names let readers locate the same objects in tools and official documentation.

如果要把这个教程放进真实生产流程，应把输出层命名、保存位置和接入方式都做成可审查约定。比如可以让 Houdini 输出位于 shot 层附近的调整层，再由 shot.usda 或更上层的 session/shot edit layer 引用它。提交评审时，评审者不应该只看最终画面，还应该能打开 layer stack，看到新层只包含 house 的位移或旋转意见，并确认没有额外写入材质、拓扑、payload 或无关 prim。这个习惯能降低 DCC 之间互相覆盖的风险。

In production, the authored layer should be named, stored, and connected in a reviewable way. Reviewers should be able to inspect the layer stack and see that the new layer contains only the intended house edit.

本页的中文阅读顺序因此是：先确认它是历史 Houdini 插件教程，再用 usdview 建立原始 shot 的语义坐标；然后把 /World/sets/house 当作局部修改目标，把 terrain 当作空间参照；接着理解 Houdini 只生成稀疏 transform override；最后回到 usdview 和 USD 组合规则检查 layer strength 是否让新意见获胜。只要抓住这条线，即使具体 Houdini UI 已迁移到 Solaris，读者仍能复用教程背后的 USD 思维。

The recommended reading order is: identify the tutorial as historical, inspect the original shot in usdview, treat /World/sets/house as the local edit target, author a sparse transform override, and verify the composed result.

成功标准也应具体化：不是“能从 Houdini 导出一个 USD 文件”就算完成，而是最终 composed stage 中 house 的位置按预期变化，未编辑对象保持来源不变，新增 layer 可以被单独禁用或移除以恢复原始 shot，并且所有参与者都能通过 prim path 和 layer stack 解释变化来源。这个标准比单次视觉检查更可靠。

Success means the composed stage changes the house as intended, leaves unrelated objects sourced from the original layers, and makes the new edit removable and explainable through the layer stack.

因此，本地中文版把这页归为“历史教程 + 管线概念”而不是“当前安装步骤”。后续读者如果在新 Houdini 版本中实践，应优先寻找 Solaris 中等价的 USD 导入、编辑、导出和 layer 组合节点，并把本页当作验证组合语义的路线图，尤其关注路径、层序和覆盖范围。

The localized page therefore treats the tutorial as historical pipeline guidance, and directs modern users to equivalent Solaris USD workflows.

工作流检查表 / Workflow Checklist

  1. 用 usdview 打开 USD/extras/usd/tutorials/Houdini/shot.usda，先确认原始 shot 的结构、视图和 layer 关系。
  2. 在场景树中定位 /World/sets/house，并把这个 prim path 作为后续覆盖层写入的目标。
  3. 把 house 和必要的 terrain context 导入 Houdini，确保 DCC 中看到的是可判断空间关系的上下文，而不是无边界的整场景复制。
  4. 在 Houdini 中移动 house，只表达需要改变的 transform，而不要把未修改的资产内容重新导出成强覆盖。
  5. 把 Houdini 生成的 USD layer 作为 overlay 或更强 sublayer 接入 shot，并用 usdview 比较组合结果。
  6. 把旧 plugin 相关步骤当作历史背景；在现代 Houdini 管线中，应映射到 Solaris 的 USD 原生节点和导出流程。

概念边界 / Concept Boundaries

不是完整 Houdini 手册

本页只解释官方教程中的 USD 组合思想，不替代 Houdini 或 Solaris 的产品文档。具体 UI 名称可能随 Houdini 版本变化。

不是整场景导出

目标是稀疏覆盖层。移动 house 时，应只写目标 prim 的 transform 差异，避免把原始资产、材质和引用结构全部复制进新层。

不是绕过 USD 组合

Houdini 输出仍由 USD layer stack、subLayer 顺序、reference 和 opinion strength 决定最终结果。调试时必须回到 composed stage。

不是当前插件安装指南

旧 Houdini USD plugin 已在 USD 20.05 后从 USD 发行版移除；现代工作应优先理解 Solaris 的原生 USD 支持。

命令与层片段 / Command And Layer Fragments

官方示例的基础检查命令保留原样。命令、路径和 prim path 不翻译，因为它们是可以直接核对的技术对象。

usdview USD/extras/usd/tutorials/Houdini/shot.usda

概念上，Houdini 生成的覆盖层应只包含类似“对 /World/sets/house 写入更强 transform opinion”的内容。实际语法会随导出设置变化，但阅读时应检查三个问题：目标路径是否正确、transform 是否写入、layer 是否在足够强的位置参与组合。

常见误读与调试路径 / Common Misreadings And Debugging

  • 如果 usdview 中 house 没有移动，先确认 overlay layer 是否真的被加入 shot，而不是只在磁盘上生成了一个独立文件。
  • 如果导入到 Houdini 后对象层级变化，先检查导入节点是否保持了 USD prim path；不要把 Houdini 内部节点名误认为最终 USD path。
  • 如果原始资产的材质或引用被意外改变，检查导出层是否过于完整。稀疏 override 应只写修改点。
  • 如果同一个 transform 在不同工具中结果不同，检查 xformOp 顺序、单位、轴向和 parent transform，而不是只看单个 translate 值。
  • 如果教程中的插件步骤无法复现，优先确认 USD 版本和 Houdini 版本。旧 plugin 已移除，现代流程应转到 Solaris 概念映射。

术语与保留英文 / Terms Retained

usdview

USD 的交互式检查工具，用于打开 shot、查看 scenegraph、检查 layer 和组合结果。

/World/sets/house

教程中的目标 prim path，后续 transform override 应该落在这个语义目标上。

overlay layer

覆盖层或叠加层，用来写入局部更强意见，而不是替代原始资产层。

Solaris

Houdini 中现代 USD 原生工作环境，是旧 Houdini USD plugin 之后应优先参考的方向。
