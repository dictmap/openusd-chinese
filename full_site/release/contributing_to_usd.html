本页是向 OpenUSD 项目贡献代码和文档的流程说明。中文主阅读路径应覆盖 CLA、coding conventions、pull request guidelines、git workflow、GitHub issues 和 major changes 的五步流程。它不是一般 Git 教程，而是告诉贡献者如何在 OpenUSD 项目规范内提交可评审、可测试、可落地的改动，并理解 Pixar/项目维护者在测试和 landing 阶段的职责。This page has been promoted to bilingual_complete after checking the official/local source structure and adding a Chinese main reading path.

主要 section 中文说明

  • CLA 是贡献前置条件。中文说明应强调这是法律和项目治理要求，贡献者应按官方链接完成，不要把它当作可跳过的普通表单。Official section anchor: Contributor License Agreement
  • Coding Conventions 约束代码风格、命名、注释、测试和维护一致性。保留 C++、Python、header、namespace、API 名称等英文术语，中文解释为什么这些规则影响 review 和长期维护。Official section anchor: Coding Conventions
  • PR guideline 的重点是小而清晰的改动、可复现测试、问题说明、兼容性风险和 review 便利性。不要把多个不相关修复塞进一个 PR。Official section anchor: Pull Request Guidelines
  • Git workflow 提供 branch、commit、rebase/merge、同步 upstream 等路径。命令和分支名保留英文，中文说明每一步的目的和失败时的排查方向。Official section anchor: Git Workflow
  • Issue 不是随意聊天区，而是缺陷、需求、设计讨论和复现信息的入口。好的 issue 应包含版本、平台、复现步骤、预期/实际行为和相关日志。Official section anchor: GitHub Issues
  • 重大改动必须先达成共识，再改代码、测试、提交 review，最终由 Pixar/维护者测试并合入。中文应强调设计共识先于大规模实现，避免完成大量代码后才发现方向不被接受。Official section anchor: Making Major Changes

操作和排查路径

  1. 小修复先开 issue 或关联已有 issue，确认范围后提交最小 PR，并附测试结果。Workflow note for local Chinese reading path.
  2. 大改动先按 major changes 流程写清动机、设计、兼容性和替代方案，再进入实现。Workflow note for local Chinese reading path.
  3. 提交后根据 review 反馈迭代，保持 commit 和说明清晰，避免把无关格式化混入功能改动。Workflow note for local Chinese reading path.
