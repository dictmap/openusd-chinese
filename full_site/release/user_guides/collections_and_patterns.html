逐段双语理解 / Paragraph-Level Bilingual Coverage

`Collections and Patterns` 解决的是“如何在 USD 场景里表达一组对象”的问题。这里的一组对象可以是 prim，也可以是 property，还可以通过包含其他 collection 来组合更大的逻辑集合。它不是单纯的目录分组功能，而是一种面向下游工作流的集合表达机制，常见用途包括 light-linking、material binding、渲染选择、批量检查和工具侧的逻辑过滤。

Collections describe heterogeneous groups of scene objects, including prims, properties, and other collections.

官方页首先强调 collection 的灵活性：一个模型的“逻辑部件”不一定沿 scene hierarchy 连续分布，某个灯光影响的几何体也可能来自多个 layer 或多个命名空间分支。如果只靠父子层级，很难表达这些跨层级、跨类型、跨资产的集合。因此 `CollectionAPI` 提供了一个独立于物理层级的 membership 描述层，让工具可以用同一套规则询问“哪些对象属于这个集合”。

Collections support workflows that need flexible sets of objects beyond simple scene hierarchy.

成员关系有两条路线。第一条是 relationship-mode，它用 target-based rules 表示包含和排除，核心属性是 `rel collection:instanceName:includes`、`rel collection:instanceName:excludes`、`bool collection:instanceName:includeRoot` 和 `uniform token collection:instanceName:mode`。第二条是 expression-mode，也叫 pattern-based collections，它用 `pathExpression collection:instanceName:membershipExpression` 来匹配对象，并把 `mode` 设置为 `expression`。

Membership can be authored with relationship-mode include/exclude rules or expression-mode path expressions.

同一个 collection instance 不能同时按两种模式工作。即使文件里同时写了 relationship-mode 相关属性和 `membershipExpression`，实际消费时也只能根据 `mode` 或 automatic fallback 选择一种解释方式。这个规则很关键：如果你以为 `includes` 和 `membershipExpression` 会叠加，就会误判最终 membership。审查 collection 时，第一步应先确认 `collection:instanceName:mode` 或 fallback 推导出的模式。

A collection works in one membership mode at a time; authored properties for the inactive mode are ignored.

`CollectionAPI` 是 multiple-apply schema，因此同一个 prim 可以应用多次，例如 `CollectionAPI:relCollection` 和 `CollectionAPI:expCollection`。`instanceName` 就是冒号后的实例名，它会进入属性命名空间，形成 `collection:relCollection:includes` 或 `collection:expCollection:membershipExpression` 这样的属性。阅读时不要把 `relCollection` 或 `expCollection` 翻译成普通说明词，它们是 schema instance name，会直接影响属性名。

`CollectionAPI` is a multiple-apply schema; the instance name becomes part of the collection property namespace.

Basic Usage 的创建流程可以拆成两步。第一步是在承载 collection 的 prim 上应用 `CollectionAPI`，可以在 USD 数据里通过 `prepend apiSchemas = ["CollectionAPI:myCollection"]` 写入，也可以在 Python 里用 `Usd.CollectionAPI.Apply(collectionPrim, "myCollection")`。第二步是根据选择的模式配置 membership：relationship-mode 写 `includes` / `excludes`，expression-mode 写 `membershipExpression`，同时用 `mode` 明确当前 collection 的解释方式。

Basic usage applies `CollectionAPI` to a prim, then configures membership according to the chosen mode.

查询流程也分层。先调用 `ComputeMembershipQuery()` 得到 `CollectionMembershipQuery`，它捕获当前 collection 配置解析后的查询规则；再用 `Usd.CollectionAPI.ComputeIncludedObjects(query, stage)` 或 `ComputeIncludedPaths(query, stage)` 在具体 stage 上计算对象或路径。若 collection 配置发生变化，应重新计算 query。若已经在遍历场景，也可以用 `query.IsPathIncluded(path)` 判断某个 path 是否属于集合。

Querying first computes a `CollectionMembershipQuery`, then uses it to compute included objects, paths, or path membership.

默认查询只匹配 `Usd.PrimDefaultPredicate` 下的对象。如果需要匹配 abstract class prims，必须传入包含 abstract prims 的 traversal predicate，例如 `Usd.PrimAllPrimsPredicate`。这个细节常见于资产库、class-based 复用或模板层检查：collection 规则看起来正确，但查询结果缺少 class prim，原因可能不是 membership 写错，而是 traversal predicate 过滤掉了目标。

`ComputeIncludedObjects()` and `ComputeIncludedPaths()` accept a traversal predicate; matching abstract prims requires an appropriate predicate.

relationship-mode 的 mental model 是“先包含，再按规则排除，并按 `expansionRule` 展开”。`includes` 可以直接 target prim 或 property，也可以通过 target 另一个 collection property 来合并其他 collection 的规则。合并并不保证对方集合的所有内容都保留，因为本集合的 `excludes` 仍可能排除其中一部分。若多个 include target 对同一路径产生冲突，后加入的 target 通常更强。

Relationship-mode uses include and exclude targets, and included collection rules may be merged rather than copied wholesale.

`excludes` 用来排除已包含路径下的对象，可以 target prim 或 property，但不能 target 另一个 collection。官方这样限制是为了让 membership 推理保持简单、高效、可解释。孤立的 excluded path 不会让额外对象进入集合，但可能拖慢 membership 查询；如果想表达“包含全部但排除少数路径”，relationship-mode 允许通过非 descendant 的 exclude 触发 pseudo-root `/` 的包含语义，并可用 `includeRoot` 表示 root 是否作为 included target 参与计算。

`excludes` removes paths under included targets; special handling supports collections that mean everything except some paths.

`expansionRule` 决定 target 如何展开。`expandPrims` 表示包含 include target 下的 descendant prims，并保留被直接 include 的 property path；`expandPrimsAndProperties` 还会把 included prims 上的所有 properties 纳入集合；`explicitOnly` 只保留明确 include 且未被 exclude 的路径。调试 membership 时，必须同时看 `includes`、`excludes`、`includeRoot`、`mode` 和 `expansionRule`，只看其中一个属性很容易误判。

`expansionRule` controls whether prim descendants, properties, or only explicit targets belong to the collection.

pattern-based collections 适合层级复杂、规则随命名和 schema 特征变化的场景。它不用列出 include/exclude target，而是 author 一个 `membershipExpression`，让 `CollectionAPI` 用 path expression 在 stage 中匹配对象。light-linking 是典型例子：灯光和几何体可能分布在许多 layer 与 hierarchy 中，手写 include/exclude 容易漏掉未来新增对象；用路径通配、predicate 和逻辑操作可让规则更具适应性。

Pattern-based collections use `membershipExpression` path expressions and are useful when hierarchy and composition are complex.

path expression 由 patterns、operators 和 expression references 组成。pattern 遵循 `Sdf.Path` 语法并可带 glob-style wildcards：`*` 匹配单个 path segment 内的零个或多个字符，`?` 匹配一个字符，`//` 匹配任意长度的层级序列。它们是 USD path 匹配语义，不是文件系统 glob；例如 `/primA//leafB` 可以跨多级匹配 descendant path。

Path expressions combine `Sdf.Path`-like patterns, glob-style wildcards, operators, and expression references.

predicate functions 给 pattern 加上额外条件。`{hasAPI:MaterialBindingAPI}` 可要求匹配对象应用了指定 API schema；`{abstract}` 可匹配 abstract prim；`{specifier:class,over}` 可限制 specifier；`{variant(myVariantSet=blue)}` 可按 variant selection 过滤。predicate 还可以用 `not`、`and`、`or` 和 parentheses 组合，例如用 `hasAPI` 同时排除 `isa:Sphere` 或 `isa:Cube` 的对象。

Predicate functions such as `hasAPI`, `abstract`, `specifier`, `isa`, and `variant` refine path matches.

expression operators 提供集合代数：`+` 是 union，空白也可表示 implied union；`&` 是 intersection；`-` 是 difference；`~` 是 complement；parentheses 用来控制组合顺序。把这些符号看成集合运算比看成字符串拼接更准确。表达式过于复杂时，建议先用简单 pattern 验证，再逐步加 predicate 和 operator，避免一次性写出难以诊断的 membershipExpression。

Operators provide set algebra: union, intersection, difference, complement, and grouping.

expression references 用 `%/primPath:name` 或 `%:name` 引用其他 expression，适合复用基础集合。例如先定义 allFruit，再在 allFood 中引用它并加上 cheese、meat 等 pattern。注意，expression-mode 不能直接引用 relationship-mode collection；如果需要互通，应先把 relationship-mode membership query 转成等价 path expression，再把该表达式写入 `membershipExpression`。

Expression references reuse named expressions, but expression-mode cannot directly reference relationship-mode collections.

`%_` 是 weaker expression reference，用来从较弱 composition arc 中取同名 expression opinion 并合成当前规则。它适合 layer 覆盖场景：强层写入新 membershipExpression，同时希望保留 referenced layer 中的较弱表达式结果。若 OpenUSD 无法解析该 weaker expression，`%_` 会被替换为空表达式，因此调试时要检查 composition 后的完整 stage，而不是只看当前 layer 文本。

The `%_` syntax references weaker expression opinions from composition and resolves to empty if unavailable.

OpenUSD 会在可能时简化表达式，例如把某些 difference 形式归约为 complement 形式；如果需要查看简化后的文本，可以使用 `Sdf.PathExpression.GetText()`。同时，`Sdf.PathExpression.Nothing()` 表示匹配 nothing，`Sdf.PathExpression.Everything()` 等价于 `//`。这些 API 对调试 expression-mode collection 很有用，因为它们能把最终用于匹配的表达式显式化。

`Sdf.PathExpression` can expose simplified expressions and provides convenience values for nothing and everything.

配置 pattern-based collection 时，关键属性是 `uniform pathExpression collection:instanceName:membershipExpression` 与 `uniform token collection:instanceName:mode = "expression"`。`expansionRule` 在这里也存在，但语义略有不同：`expandPrims` 表示 `ComputeIncludedObjects()` 和 `ComputeIncludedPaths()` 只把 matching prims 视作成员；`expandPrimsAndProperties` 会连同所有 properties 纳入；`explicitOnly` 对 expression-mode 等价于 `expandPrims`。

Pattern-based configuration uses `membershipExpression`, `mode = "expression"`, and an expansion rule for prim/property matching.

如果已经有 relationship-mode collection，但想在 expression-mode 里引用等价规则，可以先调用 `ComputeMembershipQuery()`，再用 `Usd.ComputePathExpressionFromCollectionMembershipQueryRuleMap(collectionQuery.GetAsPathExpansionRuleMap())` 计算等价 expression。这样可以把 include/exclude 规则迁移到 `membershipExpression`，但这是一种显式转换，不是自动混用两种模式。

Relationship-mode collections can be converted to an equivalent expression with `ComputePathExpressionFromCollectionMembershipQueryRuleMap()`.

本页最容易误读的地方有三个。第一，把 collection 当成 scene hierarchy 的替代品；实际上它是逻辑 membership 层。第二，把 `mode` 留空后忘记 automatic fallback 会根据 includes/excludes 或 includeRoot 推断 relationship-mode。第三，认为 pattern expression 可以引用任何 collection；实际上 expression references 只能直接引用其他 expression，relationship-mode 需要先转换成等价 expression。

Common mistakes include treating collections as hierarchy, ignoring automatic mode fallback, and mixing relationship-mode with expression references.

完整读懂本页后，读者应能判断：什么时候用 explicit include/exclude，什么时候用 pattern；如何应用 `CollectionAPI` 并命名 instance；如何重新计算 `CollectionMembershipQuery`；如何用 `ComputeIncludedObjects`、`ComputeIncludedPaths` 或 `IsPathIncluded` 消费结果；以及当匹配结果异常时，应检查 `mode`、`expansionRule`、predicate、traversal predicate 和 composition 后的 resolved expression。

A complete reading covers mode choice, API application, query computation, result consumption, and debugging of membership results.

操作路径 / Workflow Checklist

  1. 选择承载 prim：决定哪个 prim 持有 `CollectionAPI:instanceName`，并保持 instanceName 可读、稳定。Choose the prim that carries `CollectionAPI:instanceName`.
  2. 选择 membership mode：稳定、可审计的明确列表用 `relationship`；层级复杂或规则化匹配用 `expression`。Choose `relationship` for explicit rules or `expression` for pattern matching.
  3. 配置属性：relationship-mode 写 `includes`、`excludes`、`includeRoot`、`expansionRule`；expression-mode 写 `membershipExpression`、`mode`、`expansionRule`。Author the properties required by the selected mode.
  4. 计算 query：调用 `ComputeMembershipQuery()`，配置变化后重新计算。Call `ComputeMembershipQuery()` and recompute after configuration changes.
  5. 消费结果：用 `ComputeIncludedObjects()`、`ComputeIncludedPaths()` 或 `IsPathIncluded()` 查询具体 stage 上的 membership。Use the query to compute objects, paths, or path membership.
  6. 调试异常：同时检查 `mode` fallback、`expansionRule`、predicate、abstract prim traversal 和 expression references。Debug mode fallback, expansion, predicates, traversal predicates, and references together.

代码与配置摘录 / Code and Configuration Excerpts

def "CollectionPrim" (
    prepend apiSchemas = ["CollectionAPI:relCollection", "CollectionAPI:expCollection"]
)
{
    rel collection:relCollection:includes = [
        </World/Clothing/Shirts>,
        </World/Clothing/Pants>,
    ]
    uniform token collection:relCollection:mode = "relationship"

    pathExpression collection:expCollection:membershipExpression = "/World/Clothing/Shirts/Red*//"
    uniform token collection:expCollection:mode = "expression"
}
collectionPrim = stage.DefinePrim("/CollectionPrim")
myCollection = Usd.CollectionAPI.Apply(collectionPrim, "myCollection")
query = myCollection.ComputeMembershipQuery()
matchedObjects = Usd.CollectionAPI.ComputeIncludedObjects(query, stage)
matchedPaths = Usd.CollectionAPI.ComputeIncludedPaths(query, stage)
pathExpression collection:myCollection:membershipExpression = "/World/Light*{hasAPI:LightAPI}"
uniform token collection:myCollection:mode = "expression"
