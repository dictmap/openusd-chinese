本页是 OpenUSD release 用户指南中 usdLux `LightAPI` 的完整双语参考页。中文优先解释 schema 作用、connectable 约束、light-linking/shadow-linking、核心光照输入、`normalize` 语义、filter/material/shader 属性和调试边界，同时保留所有 API 名、属性名、token、数学公式、类型名和链接原样。

This local bilingual page explains the usdLux `LightAPI` guide in Chinese first while retaining official identifiers, property names, tokens, formulas, and links for comparison.

逐段双语理解 / Paragraph-Level Bilingual Coverage

`LightAPI` 是给 prim 增加“灯光能力”的 API schema。应用后，该 prim 不只是多了几个普通属性，而是被视为能够参与 usdLux 光照语义的对象：它可以有 `inputs:color`、`inputs:intensity`、`inputs:exposure`、`inputs:diffuse`、`inputs:specular` 等输入，也可以通过 `collection:lightLink`、`collection:shadowLink`、`light:filters` 等关系与场景几何、阴影和 filter 网络发生联系。

When `LightAPI` is applied to a prim, that prim gains light capabilities such as color, intensity, specular and diffuse multipliers, light-linking, and light filters.

官方页的第一层意思是：`LightAPI` 抽取了多种灯光 schema 的共同能力。无论具体灯光是 `DiskLight`、`RectLight`、`SphereLight`、`DistantLight`、`DomeLight`，还是 geometry light 相关的 API，它们都需要某些通用输入来描述颜色、亮度、曝光和可见影响范围。理解 `LightAPI` 后，再读具体灯光页面时就不会把每个属性当成孤立字段，而能看出它们在统一光照模型中的角色。

The API provides common light attributes shared by concrete light schemas.

第二层意思是 connectable 约束。带有 `LightAPI` 的 prim 被视为 UsdShade networks 中的 connectable prim，因此应遵循 UsdShade node encapsulation 规则。它不能随意放在 `UsdShade NodeGraph`、`UsdShade Material`、`UsdLux LightFilter` 或其他普通 connectable container 下面。例外是父 prim 自身也应用了 `LightAPI`，因为某些灯光概念确实需要灯光嵌套，例如 `PortalLight` 位于 `DomeLight` 下。

A prim with `LightAPI` is considered connectable for UsdShade networks and must respect node encapsulation rules.

`collection:lightLink:includeRoot` 和 `collection:shadowLink:includeRoot` 是两个容易影响渲染结果的 collection 开关。它们的 USD type 都是 `bool`，fallback value 都是 `True`。`lightLink` 决定哪些 geometry 与这盏灯关联，也就是哪些 prim 会被照亮；`shadowLink` 决定哪些 geometry 与该灯的阴影行为关联，也就是哪些 prim 会为这盏灯投射阴影。因为 fallback 是 `True`，默认含义是照亮所有对象、让所有对象投影。

`collection:lightLink:includeRoot` and `collection:shadowLink:includeRoot` control light-linking and shadow-linking, and their fallback value is `True`.

调试 light-linking 时要先看 collection 语义，而不是只看灯光本体。`includeRoot=True` 会把 pseudo-root `/` 当作 included target path，因此没有额外收窄规则时会覆盖全场景。如果某盏灯“照到了不该照的对象”，先检查 `collection:lightLink:*`；如果“阴影来自不该投影的对象”，先检查 `collection:shadowLink:*`。这与上一轮已完成的 `Collections and Patterns` 页面直接相连。

Debugging linking issues requires inspecting the relevant collection, especially when the root path is included by fallback.

颜色输入分两层。`inputs:color` 的 USD type 是 `color3f`，fallback 是 `(1, 1, 1)`，它在渲染色彩空间中指定灯光颜色，并与标量 emission 相乘：`L_Color = L_Scalar * color`。`inputs:colorTemperature` 的 USD type 是 `float`，fallback 是 `6500.0`，表示 Kelvin 色温；它只有在 `inputs:enableColorTemperature` 为 true 时生效。`6500` 对应 D65 常用标准白点。

`inputs:color` multiplies the scalar emission, while `inputs:colorTemperature` is enabled by `inputs:enableColorTemperature` and uses a D65-normalized fallback of `6500.0`.

亮度控制也分线性与指数两种。`inputs:intensity` 的 USD type 是 `float`，fallback 是 `1.0`，按物理 falloff 线性缩放亮度，并被解释为未乘其他控制前的 base luminance。`inputs:exposure` 的 USD type 是 `float`，fallback 是 `0.0`，以 `2^exposure` 方式指数缩放，类似摄影中的 F-stop。二者同时存在时，`intensity` 给出基础量级，`exposure` 用来快速做倍数调整。

`inputs:intensity` scales brightness linearly, while `inputs:exposure` scales brightness by a power of two.

官方对亮度单位给了规范性说明：当 `intensity=1`、`exposure=0`，直接可见的灯光以法线方向照到 sensor plane 时，在 RGB renderer 中通常产生 `[1, 1, 1]` 像素值，并对应 1 nit 的 luminance。`intensity=1`、`exposure=2` 则对应 4 nits；`intensity=2`、`exposure=0` 对应 2 nits。保留这些英文和数学字面量，是为了让读者能和 renderer 行为核对。

The page gives normative examples connecting `intensity`, `exposure`, pixel value, and luminance in nits.

`inputs:diffuse` 和 `inputs:specular` 都是 `float`，fallback 都是 `1.0`，分别控制 diffuse contribution 和 specular contribution。官方明确说这是 non-physical control。也就是说，它们更像艺术调节旋钮，不应被误读成严格物理单位。生产中如果出现“镜面太强”或“漫反射被过度照亮”，这些属性是快速定位目标之一，但要注意它们可能破坏物理一致性。

`inputs:diffuse` and `inputs:specular` are non-physical multipliers for diffuse and specular contributions.

`inputs:normalize` 的 USD type 是 `bool`，fallback 是 `False`。启用后，灯光会按形状或角度的 sizeFactor 调整 luminance，使改变灯光尺寸时总功率或接收照度保持更稳定。对于 area lights，sizeFactor 来自世界空间表面积；对于 `DistantLight`，官方定义了基于 `distantLightAngle`、`sin(theta_max)^2` 和 `pi` 的公式。`DomeLight` 与 `PortalLight` 忽略该属性，sizeFactor 始终视为 1。

`inputs:normalize` adjusts luminance by a size factor; area lights use world-space surface area, distant lights use an angular formula, and DomeLight/PortalLight ignore the attribute.

`normalize` 的调试含义很重要。关闭时，增大 area light 通常会改变总照明；开启时，目标是让尺寸变化不直接改变总能量或接收照度。第三方或自定义 light schemas 必须自己说明 `normalize` 的预期行为。若渲染器之间亮度不一致，要核对具体灯光家族、sizeFactor 规则、transform scale、`intensity`、`exposure` 和 renderer 对该 schema 的实现。

When renderer results differ, inspect the light family, size factor, transform scale, intensity, exposure, and schema-specific normalize behavior.

`light:filters` 是 relationship，用于把 `LightFilter` prim 关联到灯光，进而调制该灯的效果。它不是颜色或强度值，而是连接到 filter 网络的关系。`light:shaderId` 是 token，用于在没有 render-context-specific shader ID 时定义灯光 shader ID；对于 intrinsic lights，默认通常是灯光类型名，并对应一个 Sdr shader node。读者应把它们看作渲染器解释层的入口，而不是普通艺术参数。

`light:filters` connects light filters, and `light:shaderId` identifies the shader for the light when no render-context-specific shader ID is available.

`light:materialSyncMode` 用于 LightAPI 应用到带有 bound Material 且材质全部或部分 emissive 的 geometry 时，描述材质响应与灯光响应的关系。有效 token 包括 `materialGlowTintsLight`、`independent` 和 `noMaterialResponse`。fallback 是 `noMaterialResponse`，表示几何体表现得像没有绑定 Material，光线 base color 完全由 `inputs:color` 控制。

`light:materialSyncMode` controls how emissive material response relates to lighting response when `LightAPI` is applied to geometry.

这三个 `materialSyncMode` token 的差别会直接影响 mesh light 或 emissive geometry 的外观。`materialGlowTintsLight` 允许灯光颜色 tint 几何体 glow，同时保留 `intensity` 等 LightAPI 控制；`independent` 让 primary/secondary rays 看到材质的 emissive/glow，但 light rays 的 base color 只由 `inputs:color` 决定；`noMaterialResponse` 则让材质响应不参与 diffuse、specular 或 transmissive 行为。部分 emissive geometry 如果没有把合适 pattern 连接到 `inputs:color`，可能会均匀发光。

The three material-sync modes determine how emissive material response and light-ray base color interact.

把 `LightAPI` 应用到 geometry 时，还要区分“几何可见外观”和“灯光发射行为”。材质可能决定摄像机看到的 glow 或表面响应，`LightAPI` 则决定 light rays 如何从该对象发射、如何被 `inputs:color`、`inputs:intensity`、`inputs:exposure` 和 filters 调制。`light:materialSyncMode` 正是这两个通道之间的同步策略，不应被简单理解成材质绑定开关。

When applied to geometry, `LightAPI` separates camera-visible material response from light-emission behavior.

connectable 规则也要和材质节点图分开理解。灯光可以连接到 shading networks，但带 `LightAPI` 的 prim 本身不能被随意塞进普通 `UsdShade Material` 或 `NodeGraph` 里当作内部节点。若场景层级中出现这种嵌套，消费端可能无法按 UsdShade node encapsulation 正确解释网络边界。例外的灯光嵌套应来自明确的灯光概念，而不是任意组织层级。

Connectability does not mean a light prim can be arbitrarily nested inside a material or node graph.

从作者ing 角度看，`LightAPI` 页不是某一种灯光形状的说明书，而是一张公共属性地图。具体灯光页面会继续定义几何形状、尺寸或角度，例如 `DiskLight` 的半径、`RectLight` 的宽高、`DistantLight` 的角度；本页说明这些具体灯光如何共享颜色、亮度、曝光、归一化、linking、filter 和 shader 标识。维护中文页时应保留这种分层，不要把具体灯光形状的知识提前混入 `LightAPI`。

`LightAPI` is the common property map; concrete light schemas add shape-specific details.

因此验收本页时，应确认中文解释已经覆盖“如何成为灯”“如何连接网络”“如何限制受光对象”“如何调亮或调色”“如何与材质和 filter 协作”这五个问题。

A useful review checklist covers light identity, network connectivity, linked geometry, brightness/color controls, and material/filter interaction.

本页的属性清单很长，但可以归纳成五组阅读：linking collection、颜色与色温、亮度与曝光、尺寸归一化、filter/material/shader。实际排查问题时也应按这五组走：先确认是不是 linking 导致对象受光范围错误，再看 color/intensity/exposure，之后看 normalize 是否因为尺寸或角度造成亮度变化，最后检查 filters、materialSyncMode 和 shaderId 是否被渲染器解释为预期结果。

The properties can be read in groups: linking, color, intensity/exposure, normalization, and filter/material/shader integration.

完整读懂 `LightAPI` 后，读者应能回答：这个 API schema 如何让 prim 成为灯；为什么带 `LightAPI` 的 prim 是 connectable prim；light-linking 与 shadow-linking 的默认 root 行为是什么；`intensity` 与 `exposure` 如何共同控制亮度；`normalize` 为什么依赖灯光家族；以及材质、filter、shader 属性如何把灯光接入更大的 UsdShade 与渲染器生态。

A complete reading explains light capabilities, connectability, linking defaults, brightness controls, normalization, and material/filter/shader integration.

属性阅读路径 / Property Reading Path

  1. 先看 `collection:lightLink:includeRoot` 与 `collection:shadowLink:includeRoot`，判断全场景默认照明和默认投影是否符合预期。Start with light-link and shadow-link collection roots.
  2. 再看 `inputs:color`、`inputs:colorTemperature`、`inputs:enableColorTemperature`，确认颜色来源和色温开关。Inspect color and color-temperature controls.
  3. 然后看 `inputs:intensity` 与 `inputs:exposure`，区分线性基础亮度和指数曝光调整。Separate linear intensity from exponential exposure.
  4. 再检查 `inputs:normalize`，确认尺寸、角度或自定义 schema 是否改变能量解释。Check normalize and size-factor behavior.
  5. 最后检查 `light:filters`、`light:materialSyncMode`、`light:shaderId`，定位 filter、emissive material 和 shader ID 解释问题。Finish with filters, material sync, and shader identity.

代码与配置摘录 / Code and Configuration Excerpts

def Xform "LightPrim" (
    prepend apiSchemas = ["LightAPI"]
)
{
    color3f inputs:color = (1, 0.95, 0.8)
    float inputs:intensity = 2
    float inputs:exposure = 1
    bool inputs:normalize = true
    bool collection:lightLink:includeRoot = true
    bool collection:shadowLink:includeRoot = true
}
# Brightness intuition retained from the official page:
L_Color = L_Scalar * color
L_Scalar = intensity
L_Scalar = L_Scalar * 2^exposure
L_Scalar = L_Scalar / sizeFactor
