逐段双语理解 / Paragraph-Level Bilingual Coverage

`CylinderLight` 是 usdLux 中的 intrinsic light schema，用来描述从圆柱侧面向外发光的光源。官方定义里最关键的空间约束是：圆柱以原点为中心，major axis 位于局部 X axis，并且不从 flat end-caps 发光。中文阅读时要把它理解成“只有圆柱侧壁发光的管状面积光”，而不是一个封闭圆柱体整体发光，也不是两端圆面也贡献照明的体积光。这个边界会直接影响建模、阴影、采样和调试：如果你期望两端也亮，`CylinderLight` 本身并不表达这个语义，需要改用其他几何、材质或 renderer 特定方案。

`CylinderLight` is an intrinsic light that emits outward from the side of a cylinder whose major axis is the local X axis.

典型用途是模拟 tube-shaped fluorescent lights、类似管状光源、linear lights、light panels，以及建筑室内常见的 commercial lighting。和 `SphereLight` 相比，它更适合表达有明显长度方向的灯管或长条光；和 `RectLight` 相比，它不是平面矩形，而是圆柱侧面；和 `DiskLight` 相比，它不是单个圆盘发光面；和 `DistantLight`、`DomeLight` 相比，它是场景中有位置、有长度、有半径的局部光源。换句话说，选择 `CylinderLight` 时，读者应先确认“管状/线状面积光”是设计意图，再去调整强度、颜色和 shaping 参数。

Use it for tube-shaped, line-like, panel-like, or commercial interior light sources that have cylinder-side emission.

官方示例把 `CylinderLight "Light1"` 放在一个 `Sphere` 和一个 `Cube` 附近，设置 `inputs:length = 3`、`inputs:radius = 0.25`，并通过 `inputs:shaping:cone:angle = 45`、`xformOp:rotateXYZ = (0, 0, 45)`、`xformOp:translate = (0, 0, 1)` 控制光带方向和位置。这个示例说明了三层信息：第一，`inputs:length` 和 `inputs:radius` 定义发光圆柱的形状；第二，`ShapingAPI` 风格的 `inputs:shaping:cone:angle` 可以限制光扩散，让结果成为较窄光带；第三，transform 决定圆柱在世界中的摆放和朝向。不要把这些职责混在一起：形状由 length/radius 表达，位置方向由 transform 表达，发射约束由 shaping 属性表达。

The official example combines length, radius, shaping angle, rotation, and translation to produce a thin band of light.

`inputs:length` 是本页最重要的几何属性之一，USD type 为 `float`，fallback value 为 `1.0`，表示圆柱在 local X axis 上的长度。它不是世界空间长度的唯一来源，因为 prim 的 transform 仍然可能缩放、旋转或移动该圆柱；它也不是亮度参数，增加长度通常改变的是发光形状和面积直觉，而不是简单等价于提高 `inputs:intensity`。调试时应同时查看 authored length、`xformOpOrder`、父层级 transform 和 renderer 如何把 light units/area light 解释为最终照明。

`inputs:length` is the cylinder length in the local X axis, with fallback value `1.0`.

`inputs:radius` 是圆柱半径，USD type 为 `float`，fallback value 为 `0.5`。它定义管状发光面的粗细，和 `inputs:length` 一起决定 `CylinderLight` 的几何外观和采样意义。半径很小时，视觉上可能接近线光；半径较大时，它更像有明显厚度的管状面积光。不要把 `inputs:radius` 当作照明距离或亮度范围；若画面过暗，应先检查 `inputs:intensity`、`inputs:color`、曝光、单位约定、距离和遮挡，再决定是否调整半径。

`inputs:radius` is the radius of the cylinder, with fallback value `0.5`.

`light:shaderId` 是 renderer-facing 的 shader 标识，USD type 为 `token`，fallback value 为 `CylinderLight`。官方说明还指出，USD 会注册一个 identifier 为 `CylinderLight`、source type 为 `USD` 的 `Sdr` shader node，用来对应该灯光的 inputs。这个 token 必须原样保留，不能翻译成“圆柱灯”，也不要和 prim name `Light1` 混淆：prim 可以叫任何合法名称，但 shader ID 仍然是用于 schema 与渲染实现对接的稳定标识。

`light:shaderId` keeps the `CylinderLight` shader identity and corresponding USD `Sdr` node mapping.

`treatAsLine` 是一个 renderer hint，USD type 为 `bool`，fallback value 为 `False`。当它为 true 且 renderer 支持 non-area lighting 时，renderer 可以把这个灯视为 line light，也就是 effectively a zero-radius cylinder；如果 renderer 只支持 area lights，则会忽略该属性。因此它不是跨 renderer 强制行为，也不是把 schema 改成另一种 prim type。跨 DCC 或跨渲染器交付时，应把 `treatAsLine` 写成“可用则采用”的优化/兼容提示，并在验证中记录目标 renderer 是否实际支持。

`treatAsLine` is a hint that the light may be treated as a zero-radius line light when the renderer supports non-area lighting.

继承属性同样属于中文主阅读路径。`extent` 来自 `Boundable`，用于边界和包围盒相关计算；`xformOpOrder` 来自 `Xformable`，决定 `xformOp:translate`、`xformOp:rotateXYZ` 等变换操作的应用顺序；`proxyPrim`、`purpose` 和 `visibility` 来自 `Imageable`，分别影响代理 prim 关系、渲染用途分类和可见性继承。很多“灯不亮”问题并不是 `CylinderLight` 本体属性错了，而是父层级 visibility、payload、variant、layer strength、purpose 过滤或 transform 覆盖导致结果不符合预期。

Inherited `Boundable`, `Xformable`, and `Imageable` properties affect bounds, placement, purpose, and visibility.

实际排错可以按固定顺序进行：先确认 prim type 确实是 `CylinderLight`，再确认 `inputs:length` 和 `inputs:radius` 是否表达了目标灯管形状；接着检查 `xformOpOrder`、rotation 和 translation 是否把局部 X 轴圆柱放在正确位置；然后检查 `inputs:intensity`、`inputs:color`、light units、camera exposure 和场景尺度；之后检查 `visibility`、`purpose`、payload、variant、collection/linking 和 stronger layer opinion；最后再检查 renderer 对 `treatAsLine`、area light sampling、shaping cone 和 shadow 控制的支持。这个顺序能避免把形状、朝向、能量、可见性和渲染器能力混为同一个问题。

A practical debugging path is schema type, shape inputs, transform, energy/color, visibility/composition, then renderer support.

和相邻 usdLux 页面一起阅读时，`CylinderLight` 的职责边界会更清楚。`SphereLight` 适合球形或点状局部光；`RectLight` 适合矩形平面面光；`DiskLight` 适合圆盘面光；`PortalLight` 用于把环境光采样引导到开口区域；`ShapingAPI` 控制 cone、focus、IES 等发射塑形；`ShadowAPI` 控制阴影艺术属性；`LightAPI` 管理通用 light 行为和 linking。`CylinderLight` 只负责“管状/线状具体光源”的 schema 语义，不应承载所有 shaping、shadow、linking 或 renderer policy。

Adjacent pages separate cylindrical emission geometry from shaping, shadows, linking, portals, and other light shapes.

从制作流程看，`CylinderLight` 常见于三类任务。第一类是室内灯管或霓虹管建模，重点是长度、半径和摆放方向；这时 `inputs:length` 与 `xformOp:rotateXYZ` 的组合比单纯调亮度更重要。第二类是舞台、展台或产品照明，重点是让管状高光沿某个方向拉伸；这时要用 transform 控制圆柱主轴，并用 `inputs:shaping:cone:angle` 或相关 shaping 属性限制扩散。第三类是跨 renderer 交付，重点是明确 `treatAsLine` 只是 hint：支持 line light 的 renderer 可以把它近似为零半径线光，不支持的 renderer 仍可能按面积圆柱处理或忽略该提示。把这三类任务分开写，可以避免读者把形状、采样策略和艺术控制混为一谈。

Production use cases separate tube shape, placement direction, shaping behavior, and renderer-specific line-light support.

`CylinderLight` 的“端盖不发光”还会影响资产建模决策。如果艺术目标是完整玻璃灯管在镜头中可见，通常需要额外的几何体和材质来表现灯管外观；`CylinderLight` prim 负责照明语义，不自动生成可见灯管模型，也不自动让端盖发光。如果目标是一个两端也发亮的实体灯具，可以把可见几何、emissive material、`GeometryLight`/插件灯或 renderer 特定方案分开评估，而不是强行把这种语义塞进 `CylinderLight`。中文页面必须说清这一点，因为英文属性表本身只列出字段，读者很容易误以为圆柱所有表面都参与发光。

Use separate visible geometry or renderer-specific approaches when the lamp body or end caps themselves must be visible or emissive.

当 `CylinderLight` 和 `LightAPI`、collection-based light linking 一起使用时，还要区分“灯的形状”与“灯影响哪些对象”。`inputs:length`、`inputs:radius` 和 transform 决定管状光源本体；light linking、shadow linking 或 collection membership 决定哪些 prim 接收照明或阴影影响。如果某个物体没有被照亮，不一定是圆柱长度或半径错误，也可能是 collection、relationship、purpose 过滤或 layer composition 使它不在受影响集合内。可靠的中文主阅读路径应把这些排查入口串起来，而不是只翻译字段名称。

Light shape is separate from light linking and collection membership; both affect debugging.

官方示例理解 / Official USDA Example

官方示例的核心是：在 `Scope "Lights"` 下定义 `CylinderLight "Light1"`，设置 `inputs:length = 3`、`inputs:radius = 0.25`、`inputs:shaping:cone:angle = 45`、白色 `inputs:color`、`inputs:intensity = 20.0`，再用 `xformOp:rotateXYZ = (0, 0, 45)` 和 `xformOp:translate = (0, 0, 1)` 放置灯。随后 `Xform "TestGeom"` 下的 `Sphere` 和 `Cube` 用于观察管状光和窄光带效果。这个示例保留了属性名、prim 名和 transform token，中文只解释职责和调试顺序。

The official USDA example positions a `CylinderLight` near a sphere and cube, with length 3, radius 0.25, and shaping/transform controls.

#usda 1.0
(
    upAxis = "Y"
)

def Scope "Lights"
{
    def CylinderLight "Light1"
    {
        float inputs:length = 3
        float inputs:radius = 0.25
        float inputs:shaping:cone:angle = 45
        color3f inputs:color = (1, 1, 1)
        float inputs:intensity = 20.0
        float3 xformOp:rotateXYZ = (0, 0, 45)
        double3 xformOp:translate = (0, 0, 1)
        uniform token[] xformOpOrder = ["xformOp:translate", "xformOp:rotateXYZ"]
    }
}

如果渲染结果不是窄光带，先检查 `inputs:shaping:cone:angle` 是否被目标 renderer 支持，再检查 `xformOp:rotateXYZ` 和 `xformOpOrder` 是否让局部 X 轴圆柱处于预期方向。若形状不对，检查 `inputs:length` 和 `inputs:radius`；若亮度不对，检查 `inputs:intensity`、`inputs:color`、曝光和单位；若完全不显示，检查 `visibility`、`purpose`、payload、variant 和 layer composition。

Use the example to separate shape inputs, shaping controls, transform placement, energy/color, and composition visibility.

边界与相邻页面 / Boundaries and Adjacent Pages

  • SphereLight：用于球形或点状局部光；如果目标是灯泡、头灯或点光近似，通常比 `CylinderLight` 更合适。Use `SphereLight` for spherical or point-like emitters.
  • RectLight：用于矩形面光、soft box、灯板或贴图矩形光；它不是圆柱侧面发光。Use `RectLight` for rectangular area emitters.
  • DiskLight：用于圆盘发光面；不要把圆盘端盖发光误读成 `CylinderLight` 的 end-cap emission。Use `DiskLight` for circular disk emission.
  • ShapingAPI：控制 cone、focus、IES 等发射塑形；`CylinderLight` 示例中的 `inputs:shaping:cone:angle` 属于这类读法。Use `ShapingAPI` for emission shaping controls.
  • ShadowAPI：控制阴影距离、falloff、颜色等艺术属性；不要把 shadow 行为写入 `CylinderLight` 本体属性。Use `ShadowAPI` for shadow controls.
  • LightAPI：管理通用 light 行为、linking 和 collection 语义；`CylinderLight` 只定义具体管状发光几何。Use `LightAPI` for common light behavior and linking semantics.
