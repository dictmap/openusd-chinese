逐段双语理解 / Paragraph-Level Bilingual Coverage

`DiskLight` 是 usdLux 中的 intrinsic light schema，用来表达一个位于局部 `XY plane` 的圆形发光盘。它不是球形光，也不是矩形面光，而是从圆盘的一侧沿局部 `-Z axis` 发光的面积光。中文阅读时最先要抓住两点：第一，圆盘几何由 `inputs:radius` 决定；第二，默认发光方向和 prim 的 transform 共同决定最终照向哪里。如果画面里光照方向不符合预期，通常应先检查 `xformOpOrder`、父层级 transform 和局部轴向，而不是把 `DiskLight` 当成无方向的点光源。

`DiskLight` is an intrinsic light that emits from one side of a circular disk in the local XY plane.

官方使用场景把 `DiskLight` 放在摄影 soft boxes、圆形 light panels、fluorescent lights 和类似线性或面状照明的语境里。这里的重点不是把所有真实灯具都简化成圆，而是让创作者在 USD 中获得一个稳定的“圆盘面积光”表达：它有位置、有半径、有一面发光的朝向，并能和 `LightAPI` 的通用光照属性、`ShapingAPI` 的光束塑形属性、`ShadowAPI` 的阴影控制属性一起使用。和 `RectLight` 相比，它没有矩形的 width/height；和 `SphereLight` 相比，它不是全方向发光；和 `CylinderLight` 相比，它没有沿 X 轴延伸的管状侧面。

Typical uses include soft-box-like, panel-like, and circular area-light setups.

示例层中有一个 `def DiskLight "Light1"`，设置 `float inputs:radius = 0.8`、`color3f inputs:color = (1, 1, 1)`、`float inputs:intensity = 20.0`，再通过 `double3 xformOp:translate = (4, 0, 1)` 把灯放到 `Sphere` 和 `Cube` 附近。这段示例说明了职责边界：`inputs:radius` 负责圆盘大小，`inputs:intensity` 和 `inputs:color` 负责能量和颜色，`xformOp:translate` 负责位置，`xformOpOrder` 负责变换顺序。中文页解释这些字段时必须保留 API/token/属性名原样，不能把可复制的 USD 语法翻译成中文标识符。

The official example places a `DiskLight` near a sphere and cube with authored radius, color, intensity, and transform opinions.

`inputs:radius` 是本页最关键的专有属性，USD type 为 `float`，fallback value 为 `0.5`。它表示圆盘半径，半径越大，视觉上越接近宽面积光，阴影边缘通常也会更柔和；但它不是单独的亮度旋钮，也不是照明距离。官方说明提到更大的半径会增加光的 overall reach，这在中文里应理解为圆盘发光面更大、影响范围和采样表现可能更宽，而不是保证所有 renderer 都以同一种物理单位或同一种曝光模型输出相同亮度。排查“太暗”或“太亮”时，应同时看 `inputs:intensity`、`inputs:color`、场景尺度、曝光、单位约定、遮挡和 renderer 的面积光采样。

`inputs:radius` is the disk radius, with fallback value `0.5`.

`light:shaderId` 是 renderer-facing 的 shader 标识，USD type 为 `token`，fallback value 为 `DiskLight`。官方页说明 USD 会注册一个 identifier 为 `DiskLight`、source type 为 `USD` 的 `Sdr` shader node，用来对应该灯光的 inputs。这个字段不应翻译，也不应和 prim 名 `Light1` 混淆：prim 名可以由资产作者决定，`light:shaderId` 则是 schema 和渲染实现之间的稳定连接点。跨 renderer 或跨 DCC 交付时，如果灯光没有按预期识别，除了检查 prim type，还要确认 `light:shaderId` 没有被错误覆盖。

`light:shaderId` keeps the `DiskLight` shader identity and the corresponding USD `Sdr` node mapping.

继承属性同样属于主阅读路径。`extent` 来自 `Boundable`，用于边界和包围盒相关计算；如果可见性、选择、裁剪或某些 Hydra delegate 的包围行为异常，需要检查 authored extent、自动计算和父级 transform。`xformOpOrder` 来自 `Xformable`，决定 `xformOp:translate`、旋转、缩放等操作的应用顺序；对于单面发光的 `DiskLight`，顺序错误会直接改变圆盘在世界中的位置和朝向。`proxyPrim`、`purpose`、`visibility` 来自 `Imageable`，分别影响代理 prim、渲染用途分类和可见性继承；很多“灯不生效”问题其实来自 visibility、purpose 过滤、payload/variant 或更强 layer opinion，而不是 `inputs:radius` 写错。

Inherited `Boundable`, `Xformable`, and `Imageable` properties affect bounds, placement, purpose, proxying, and visibility.

调试顺序建议固定下来：先确认 prim type 是 `DiskLight`，再确认 `inputs:radius` 是否表达目标圆盘大小；接着检查局部 `XY plane` 和局部 `-Z axis` 是否经 transform 后朝向目标；然后检查 `inputs:intensity`、`inputs:color`、曝光、场景尺度和遮挡；之后检查 `visibility`、`purpose`、collection/linking、payload、variant 和 layer strength；最后再看目标 renderer 对面积光、阴影软化、采样噪声和 light units 的支持。这个顺序能避免把几何、能量、朝向、可见性和 renderer 能力混为同一个问题。

A useful debugging path is schema type, radius, transform direction, energy/color, visibility/composition, then renderer support.

和相邻 usdLux 页面一起读时，`DiskLight` 的边界会更清楚。`CylinderLight` 表示局部 X 轴方向的管状侧面发光，`RectLight` 表示矩形面光，`SphereLight` 更适合球形或近似点状的局部光源，`DistantLight` 表示远处方向光，`DomeLight` 表示环境或贴图驱动的穹顶光，`PortalLight` 用于引导环境光采样穿过开口区域。`DiskLight` 只负责“圆盘、单面、面积光”的具体语义，不应承担所有 shaping、shadow、linking 或 renderer policy；这些行为应转到 `ShapingAPI`、`ShadowAPI`、`LightAPI` 或 renderer 文档中判断。

Adjacent usdLux pages separate circular disk emission from cylindrical, rectangular, spherical, distant, dome, portal, shaping, shadow, and linking concepts.

制作流程中常见三类使用方式。第一类是摄影棚软箱或圆形灯盘，需要一个有明确半径和方向的柔和面积光；这时应优先调 `inputs:radius`、transform 和 `inputs:intensity` 的组合。第二类是产品、角色或室内场景中的圆形补光，需要在镜头中稳定控制高光形状；这时要注意圆盘一侧发光的方向，而不要期待它像 `SphereLight` 一样全方向贡献。第三类是跨工具交换，需要让 DCC、Hydra delegate 或离线 renderer 都能识别同一个 schema；这时 `light:shaderId`、`purpose`、`visibility` 和本地/官方链接核对比单纯翻译说明更重要。

Production use cases distinguish circular soft-box lighting, controllable round highlights, and cross-renderer schema exchange.

本轮已对比官方页和本地 source snapshot：标题 `DiskLight`、定义段、用途段、USDA 示例、`Properties`、`Inherited Properties (Boundable)`、`Inherited Properties (Xformable)`、`Inherited Properties (Imageable)`、previous/next 链接均纳入中文解释。页面保留 `DiskLight`、`inputs:radius`、`inputs:color`、`inputs:intensity`、`light:shaderId`、`extent`、`xformOpOrder`、`proxyPrim`、`purpose`、`visibility`、`CylinderLight`、`DistantLight` 等 API/schema/token/属性名原样；中文负责建立独立可读的主路径，英文只作为核对。

The official page and local source snapshot were checked for the title, prose, example, properties, inherited properties, and adjacent links.

还需要明确一个容易误读的点：`DiskLight` 的圆盘几何本身并不等同于可见灯具模型。它描述的是光照 schema，而不是自动生成一块在镜头中可见的圆形灯板。如果资产需要让灯具外壳、灯罩、支架或发光面在画面中可见，通常还要另建 `Mesh`、`GeomSubset`、材质或 renderer 特定的 emissive 表达；`DiskLight` prim 负责照明贡献，几何资产负责可见外观。把这两层分开，能避免美术资产看起来没有灯具、或者灯具可见但实际不照明这两类混乱。

Visible lamp geometry is separate from the `DiskLight` schema that contributes illumination.

在组合、引用和变体场景中，`DiskLight` 的 authored opinion 也要按 USD composition 规则读取。弱层里写了 `inputs:radius` 不代表最终 stage 一定采用该值；更强 layer、variant selection、payload、reference 或 session layer 都可能覆盖它。若调试时本地 HTML 与 usdview/renderer 结果不一致，先用属性面板或 `usdcat`/`usdtree` 查看最终 composed value，再判断是不是翻译、文档或 renderer 的问题。中文主阅读路径必须提醒读者：schema 字段说明回答“这个属性是什么意思”，composition 检查回答“当前 stage 实际用了哪个值”。

Authored values must still be interpreted through USD composition to find the composed value used by the stage.

因此，中文读者现在可以独立判断：什么情况下选择 `DiskLight`，如何解释圆盘半径和单面发光方向，如何把 radius、intensity、color 与 transform 分开调试，继承属性为什么会影响可见性和阅读路径，以及何时应该跳转到 `RectLight`、`SphereLight`、`CylinderLight`、`ShapingAPI` 或 `ShadowAPI` 查看相邻概念。这个判断路径也是后续检查渲染结果和资产交付问题的基础。

USDA 示例阅读 / USDA Example Reading

下面保留官方示例中的关键结构，便于和本地 source snapshot 对照。代码标识符保持英文原样；中文说明只解释职责，不替换语法。

#usda 1.0
(
    upAxis = "Y"
)

def Scope "Lights"
{
    def DiskLight "Light1"
    {
        float inputs:radius = 0.8
        color3f inputs:color = (1, 1, 1)
        float inputs:intensity = 20.0
        double3 xformOp:translate = (4, 0, 1)
        uniform token[] xformOpOrder = ["xformOp:translate"]
    }
}
  • `def DiskLight "Light1"` 表示 prim type 是 `DiskLight`，`Light1` 只是 prim name。The prim type is `DiskLight`; `Light1` is the prim name.
  • `inputs:radius = 0.8` 扩大圆盘发光面；它和 `inputs:intensity = 20.0` 是两个不同维度。Radius and intensity are separate authored opinions.
  • `xformOp:translate = (4, 0, 1)` 只给出位置；如果要改变朝向，需要额外 transform op 并维护 `xformOpOrder`。Translation places the light; orientation requires transform opinions.
