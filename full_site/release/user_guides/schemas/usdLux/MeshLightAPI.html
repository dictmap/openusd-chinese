逐段双语理解 / Paragraph-Level Bilingual Coverage

MeshLightAPI 是本轮 usdLux 短页小批量冲刺中的一个完成页。中文主阅读路径覆盖页面职责、官方 section、属性/API 分组、边界、误读点、调试路径和相邻 usdLux 类型关系；英文术语用于和官方页核对，不作为主要阅读路径。

Use this schema to apply light behavior to a Mesh prim, preferring MeshLightAPI over applying LightAPI directly to a Mesh.

官方源页核心说明：Use this schema to apply light behavior to a Mesh prim.

MeshLightAPI 的职责是把灯光行为应用到一个 Mesh prim 上。阅读时不要把它理解成新的几何形状灯类型；它是针对 mesh 发光场景的 API schema，用来把 mesh 自身、绑定材质和灯光 shader 身份放到同一条可解释路径里。

官方明确说，MeshLightAPI 优先于直接把 LightAPI 应用到 Mesh。原因是它会设置常用内建行为：默认把 light:materialSyncMode 覆盖为 materialGlowTintsLight，让 Mesh 的 bound material 能参与灯光颜色或发光外观的同步。

另一个核心默认值是 light:shaderId 为 MeshLight。这不是中文名，也不是用户随手写的标签；它是渲染器和 Sdr shader node 体系识别 mesh light 的关键标识，保留英文 token 才能和官方 USD 文件、插件和 render delegate 对齐。

本页的主要 section 很短，但实际阅读路径要覆盖三层：第一层是将 mesh 作为发光源；第二层是与材质绑定同步，避免 mesh 外观和照明贡献脱节；第三层是插件可以在 MeshLight shader identity 上挂接额外 mesh-light 属性。

常见误读是把 MeshLightAPI 当成 GeometryLight 的同义词。官方当前更推荐在 Mesh 上使用 MeshLightAPI；而 GeometryLight 已经 deprecated。需要迁移旧资产时，应优先确认是否可以把几何发光关系表达为 mesh 上的 MeshLightAPI。

调试路径应从 prim 类型和 applied API schema 开始：确认目标 prim 确实是 Mesh，确认 MeshLightAPI 已应用，再检查 bound material、light:materialSyncMode、light:shaderId、渲染器是否支持 mesh light，以及材质 glow 或 emission 是否符合预期。

与相邻 usdLux 类型的关系也要分清：LightAPI 是通用灯光 API；GeometryLight 是旧式几何灯；PluginLight 通过外部 Sdr node 表达插件灯；VolumeLightAPI 处理体积 prim。MeshLightAPI 的位置是“Mesh prim 上的推荐发光 API”。

本轮已核对官方页与本地 source snapshot：标题、用途段落、light:materialSyncMode、materialGlowTintsLight、light:shaderId、MeshLight 和指向 Mesh Lights 的语义都已纳入中文主阅读路径。

从 section coverage 看，MeshLightAPI 的中文主阅读区不只翻译开头用途，还把官方 Properties、继承属性、相邻页面链接和 source parity 放在同一条阅读路径里。这样读者即使不先读英文正文，也能知道本页在 usdLux schema 域中的位置、应该看哪些属性、哪些内容只是继承上下文，以及哪些英文 token 必须保持原样。

从 API/schema 分组看，本页需要重点核对的属性或结构是：light:materialSyncMode、light:shaderId。这些名称都是 USD 文件、Doxygen 表格、Python/C++ API 和 renderer 解释共同使用的稳定标识，中文说明只能解释用途、边界和排查方式，不能把属性名、token 值、schema 名或 shader identity 翻译成中文别名。

从继承结构看，本页需要同时注意：无额外继承属性表需要展开。继承属性通常不是本页的新增业务功能，但它们会影响 transform、可见性、purpose、代理显示或 bounds 语义；如果只看直接属性，容易把 renderer 不显示、灯方向不对、缓存不生效等问题误判成 schema 本身缺失。

从边界和误读点看，MeshLightAPI 不应被当成所有 usdLux 灯光问题的通用答案。它与 LightAPI、GeometryLight、PluginLight、VolumeLightAPI、overview 等相邻页面存在职责分工：有的负责具体 intrinsic light 形状，有的负责 applied API，有的负责插件 shader node，有的负责 light list 缓存，有的只是 deprecated 旧入口。中文页必须把这些边界写清楚，避免读者在旧 schema、新 schema 和插件扩展之间来回跳转。
