范围说明 / Scope Note

本页是 OpenUSD release 用户指南中 usdLux `PluginLightFilter` 的完整双语参考页。中文优先解释它如何把 `LightFilter` 的通用过滤语义连接到外部 `SdrShadingNode` definition，并通过 `UsdShadeNodeDefAPI` 让 render delegates 识别插件实现；同时保留官方标题、section、API/schema/token/属性名、代码和链接原样。

This local bilingual page explains the usdLux `PluginLightFilter` guide in Chinese first while retaining official identifiers, section names, property names, tokens, and links for comparison.

逐段双语理解 / Paragraph-Level Bilingual Coverage

`PluginLightFilter` 的核心职责不是重新定义一种固定的灯光过滤算法，而是给插件型 light filter 提供一个可发现、可声明、可交给渲染端解释的 schema 入口。官方第一段说它提供一些属性，使 prim 能够通过 `UsdShadeNodeDefAPI` 标识一个外部 `SdrShadingNode` definition，并把这个 definition 提供给 render delegates。换句话说，OpenUSD 在这里表达的是“这个 filter 的实现由外部 Sdr 节点定义”，而不是把每一种 renderer-specific filter 都固化成独立 schema。

`PluginLightFilter` identifies an external `SdrShadingNode` definition through `UsdShadeNodeDefAPI` so render delegates can consume a light-filter implementation without a dedicated schema for every filter type.

阅读本页时应先把 `PluginLightFilter` 和上一页 `LightFilter` 分层理解：`LightFilter` 负责通用过滤器语义，包括 `filterLink` collection、`lightFilter:shaderId`、与灯光的 `light:filters` 关系以及从 `Xformable` / `Imageable` 继承的通用 scene graph 属性；`PluginLightFilter` 则进一步说明“具体实现如何通过外部节点定义暴露给 renderer”。因此它是扩展层，不是基础过滤语义的替代品。

Read `PluginLightFilter` as an extension layer on top of `LightFilter`, not as a replacement for the base filtering semantics.

`SdrShadingNode` 是这里的关键技术边界。Sdr 节点定义通常描述 renderer 或 shading system 可识别的节点类型、输入输出和实现标识。`PluginLightFilter` 通过引用或携带这类 definition，使 render delegate 能在运行时找到对应的 filter 实现。这样做的价值是让 USD 文件可以声明插件 filter 的身份和连接方式，同时把具体参数、算法和视觉行为留给 Sdr 定义与 renderer/plugin 实现。

The external `SdrShadingNode` definition carries implementation-facing information that the render delegate can interpret.

`UsdShadeNodeDefAPI` 应被理解为“把 prim 与 node definition 关联起来”的桥，而不是普通 UI metadata。它让 `PluginLightFilter` 可以参与 UsdShade 式的节点定义发现流程。排查这类页面时，不要只看 prim type 是否是 `PluginLightFilter`；还要检查该 prim 是否具备正确的 `UsdShadeNodeDefAPI` 相关 authored data，以及当前 renderer delegate 是否支持被引用的 `SdrShadingNode`。

`UsdShadeNodeDefAPI` provides the bridge from the prim to a node definition; renderer support is still required.

官方页的 `Properties` section 对 `PluginLightFilter` 自身没有列出新增 direct property。这一点很重要：它提示读者本页的独特性主要来自 `UsdShadeNodeDefAPI` 和外部 node definition，而不是一组新的内建 USD 属性。具体 renderer 的 gobo、barn door、projection、decay 或其他 light filter 参数，通常应来自对应 Sdr 节点的 inputs，而不是被误认为本页必须列出的通用 USD 属性。

The direct `Properties` section does not list additional built-in properties for this schema.

继承自 `LightFilter` 的 `collection:filterLink:includeRoot` 仍然是作用范围控制的核心属性。它的 USD type 是 `bool`，fallback value 是 `True`。当 `includeRoot=True` 时，pseudo-root path `/` 会被视为 included target path，因此该 light filter 默认会影响与它关联的灯所照亮的所有对象，除非 `filterLink` collection 通过 include/exclude 规则缩小范围。插件 filter 的存在不会绕过这套 collection 语义。

Inherited `collection:filterLink:includeRoot` remains a `bool` with fallback `True`, controlling the filterLink collection's default root inclusion.

`filterLink` collection 和 `PluginLightFilter` 的关系容易被误读：`PluginLightFilter` 说明 filter 的实现来源，`filterLink` 说明 filter 影响哪些 geometry prims。前者解决“这个 filter 是什么实现”，后者解决“这个 filter 作用到哪里”。如果渲染结果看起来 filter 影响范围过大，优先检查 `collection:filterLink:includeRoot` 和 collection membership；如果 filter 完全没有效果，再检查 `lightFilter:shaderId`、`UsdShadeNodeDefAPI`、Sdr definition 与 renderer delegate 支持。

Implementation identity and affected-geometry membership are separate concerns.

继承的 `lightFilter:shaderId` 是 `token`，fallback value 为空，也就是 no shader ID。对 `PluginLightFilter` 而言，它通常是连接到 renderer/plugin 可识别实现的入口之一，但它本身不保证实现存在。一个场景可以正确 author `lightFilter:shaderId`，却仍因为插件没有加载、Sdr registry 没有找到节点、或 render delegate 不支持该节点而没有预期效果。

Inherited `lightFilter:shaderId` is a `token` whose fallback is no shader ID; the renderer/plugin must still resolve it.

从 authoring 角度，`PluginLightFilter` 应先被灯光通过 `LightAPI` 的 `light:filters` relationship 引用，之后再通过 inherited `LightFilter` 属性限定作用范围并指定 shader 身份，最后通过 `UsdShadeNodeDefAPI` / `SdrShadingNode` 把具体实现暴露给 renderer。这个顺序有助于定位问题：关系未连上属于 scene graph authoring 问题，collection 不对属于作用范围问题，shader ID 或 Sdr definition 不对属于实现解析问题。

Practical authoring separates light association, collection membership, shader identity, and Sdr implementation lookup.

`xformOpOrder` 继承自 `Xformable`，USD type 是 `token[]`。对插件 light filter 来说，transform 是否有意义取决于具体 filter 实现：某些投影、遮挡或空间化过滤器可能使用 prim transform；某些全局调制 filter 可能不关心位置和方向。本页保留 `xformOpOrder` 是因为它是继承属性，而不是因为所有 `PluginLightFilter` 都必然依赖 transform。

Inherited `xformOpOrder` controls transform op ordering; whether it affects a plugin filter depends on the implementation.

`proxyPrim` 继承自 `Imageable`，USD type 是 relationship。它通常用于显示或代理表现，不替代 `light:filters`、`filterLink` 或 Sdr definition。若工具中看不到 filter 的可视化 representation，可以检查 `proxyPrim`；若渲染贡献不对，应优先检查灯光关系、collection、shader ID 和 renderer 支持。

Inherited `proxyPrim` is a relationship for representation concerns and does not replace filtering relationships.

`purpose` 继承自 `Imageable`，USD type 是 `token`，fallback value 是 `default`。它属于通用 imageable 用途分类，不是决定插件 filter 是否生效的主开关。不要把 `purpose` 与 `filterLink` collection 混为一谈：`purpose` 影响的是场景对象用途分类和消费策略，`filterLink` 影响的是这个 filter 的目标 geometry 范围。

Inherited `purpose` is a general imageable-purpose token with fallback `default`.

`visibility` 同样继承自 `Imageable`，USD type 是 `token`，fallback value 是 `inherited`。它控制 scene graph 层面的可见性继承，但官方页并没有把它定义为 light filter 贡献的核心控制项。排查时应把“这个 prim 是否在工具中可见”和“这个 filter 是否被 renderer 用于调制灯光”分开看。

Inherited `visibility` has fallback `inherited`; do not confuse view or scene-graph visibility with filter application.

完整读懂本页后，读者应能回答四个问题：为什么 `PluginLightFilter` 不需要为每种 filter type 提供单独 schema；`SdrShadingNode` 和 `UsdShadeNodeDefAPI` 分别承担什么角色；`collection:filterLink:includeRoot` 与 `lightFilter:shaderId` 作为继承属性如何继续参与插件 filter；以及 `xformOpOrder`、`proxyPrim`、`purpose`、`visibility` 哪些只是通用继承属性，哪些可能因 renderer 实现而影响实际表现。

A complete reading should distinguish schema extensibility, Sdr discovery, inherited LightFilter properties, and generic inherited scene-graph properties.

生产环境中的常见误判是把所有问题都归咎于插件本身。更稳的调试顺序是：先确认 light 的 `light:filters` target 是否指向该 `PluginLightFilter`；再确认 `filterLink` collection 是否覆盖目标 geometry；接着确认 `lightFilter:shaderId`、`UsdShadeNodeDefAPI` 与 `SdrShadingNode` 是否匹配；最后检查 renderer delegate/plugin 是否加载并支持对应实现。这样可以避免把 collection 范围错误误判成 shader 错误，也避免把插件加载失败误判成 USD schema 书写错误。

Debugging should move from USD relationships and collections to shader identity, Sdr definition, and renderer/plugin support.

和 `PluginLight` 对照时，`PluginLightFilter` 的模式也更容易理解：两者都让插件或 renderer-specific 实现进入标准 USD 场景描述，但作用对象不同。`PluginLight` 面向发光体本身，`PluginLightFilter` 面向调制光贡献的过滤器。作者不应把插件 light 的参数迁移到 filter 上，也不应把 filter 的 linking 规则当作 light 的发光属性；两者通过 `light:filters` relationship 组合，而不是互相替代。

Compared with `PluginLight`, `PluginLightFilter` extends filtering behavior rather than the light-emitting prim itself.

和 `PortalLight` 等具体 schema 对照时，本页的“空 Properties”也有明确含义。具体 schema 会把稳定、跨实现的属性列在页面中；插件 schema 则更像开放扩展点，把 renderer/plugin 的差异留在外部 node definition 里。这样可以减少 USD core schema 的膨胀，同时允许不同 renderer 暴露自己的 filter 类型。中文阅读时要保留这种边界，不把外部插件参数翻译成 OpenUSD 标准属性。

Unlike concrete schemas, the plugin schema leaves renderer-specific parameters in the external node definition.

对资产交换来说，`PluginLightFilter` 的可移植性也需要谨慎理解。USD 层能够保留 prim、relationship、collection 和 shader/node definition 引用，但另一个软件是否能复现同样视觉效果，取决于它是否理解相同的 Sdr 节点和 renderer delegate 语义。也就是说，本页保证的是结构化表达路径，不保证所有消费端都有相同渲染结果。

USD can carry the structured reference, but visual portability depends on consumer support for the same plugin definition.

因此，一个可读的 `PluginLightFilter` 页面必须同时覆盖三层信息：第一层是继承自 `LightFilter` 的基础过滤连接和作用范围；第二层是 `UsdShadeNodeDefAPI` 与 `SdrShadingNode` 提供的插件发现路径；第三层是 renderer delegate 对外部定义的消费边界。只翻译标题和属性名无法让读者独立理解这些层次，所以本页把它们作为中文主阅读路径，而英文保留为核对依据。

The Chinese reading path therefore covers base filtering, plugin discovery, and renderer-consumption boundaries.

属性阅读路径 / Property Reading Path

  1. 先看 `Properties`：本页没有新增 direct property，因此不要期待这里列出 renderer-specific filter 参数。The direct `Properties` section does not introduce renderer-specific parameters.
  2. 再看继承自 `LightFilter` 的 `collection:filterLink:includeRoot`：它决定 `filterLink` collection 是否默认包含 pseudo-root `/`。Read inherited `collection:filterLink:includeRoot` for collection root-inclusion semantics.
  3. 继续看继承自 `LightFilter` 的 `lightFilter:shaderId`：它提供 filter shader 身份，但实际解析依赖 renderer/plugin。Read inherited `lightFilter:shaderId` for shader identity.
  4. 然后看 `UsdShadeNodeDefAPI` 和外部 `SdrShadingNode` definition：它们说明插件 filter 的具体实现如何被发现。Use `UsdShadeNodeDefAPI` and `SdrShadingNode` to understand implementation discovery.
  5. 最后看 `xformOpOrder`、`proxyPrim`、`purpose`、`visibility`：这些是通用继承属性，不能替代 filter-linking 或 Sdr definition。Inherited transform, proxy, purpose, and visibility properties do not replace filter-linking or Sdr semantics.

代码与配置摘录 / Code and Configuration Excerpts

def PluginLightFilter "RendererSpecificFilter"
{
    token lightFilter:shaderId = "SomeRendererPluginLightFilter"
    bool collection:filterLink:includeRoot = true
}
def SphereLight "KeyLight" (
    prepend apiSchemas = ["LightAPI"]
)
{
    rel light:filters = </Looks/RendererSpecificFilter>
}

上面的片段只表达结构：light 通过 `light:filters` 指向 `PluginLightFilter`，继承的 `filterLink` 属性控制作用范围，`lightFilter:shaderId` 和 `UsdShadeNodeDefAPI` / `SdrShadingNode` 相关 authored data 帮助 renderer 找到具体实现。实际 token 值、Sdr inputs 和渲染行为由对应 renderer/plugin 定义。

The snippets illustrate structure only; concrete shader identifiers, Sdr inputs, and visual behavior are renderer- or plugin-defined.

常见误读与调试边界 / Common Misreadings and Debugging Boundaries

  • 不要把 `PluginLightFilter` 当成固定视觉算法。它是插件实现的 schema 入口，具体算法来自外部 `SdrShadingNode` 和 renderer/plugin。`PluginLightFilter` is not a fixed visual algorithm.
  • 不要把 `Properties` 为空理解成页面无意义。这里的重点是外部 node definition 和继承属性，而不是新增 direct properties。An empty direct `Properties` section does not mean the schema has no role.
  • 不要把 `lightFilter:shaderId` 当成保证生效的开关。它只是身份线索，仍需要 Sdr registry、plugin 和 render delegate 支持。`lightFilter:shaderId` is not a guarantee of renderer support.
  • 不要把 `filterLink` 与材质绑定、visibility 或 purpose 混淆。`filterLink` 控制被 filter 影响的 geometry 范围。`filterLink` controls affected geometry membership, not material binding or visibility.
  • 如果 filter 没有效果，按 `light:filters` relationship、`filterLink` collection、`lightFilter:shaderId`、`UsdShadeNodeDefAPI`、`SdrShadingNode`、renderer/plugin 加载顺序排查。Debug relationships, collections, shader identity, node definition, and renderer/plugin loading in order.
