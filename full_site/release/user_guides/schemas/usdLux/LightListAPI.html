逐段双语理解 / Paragraph-Level Bilingual Coverage

LightListAPI 是本轮 usdLux 短页小批量冲刺中的一个完成页。中文主阅读路径覆盖页面职责、官方 section、属性/API 分组、边界、误读点、调试路径和相邻 usdLux 类型关系；英文术语用于和官方页核对，不作为主要阅读路径。

Use this schema to discover and cache lights in a scene during traversal.

官方源页核心说明：Use this schema to discover lights in a scene during traversal.

LightListAPI 的职责不是定义某一种灯，而是在 scene traversal 过程中发现、缓存和复用灯光列表。它适合大模型、payload 或层级较深的场景，因为每次递归查找所有灯都可能很昂贵。

官方给出的第一个能力是 gather a list of lights in the scene。调用 ComputeLightList()，并传入 ComputeModeIgnoreCache，可以递归遍历并收集灯，例如收集带有 LightAPI schema 的 prim。中文阅读时要保留这些函数名和模式名，因为它们对应真实 API。

官方 Python 示例从 &lt;/World&gt; 开始计算 descendant lights。读者可以把它理解为“从某个模型层级或世界根节点开始，得到该范围内所有灯的目标列表”。这一步是查询，不会自动创建灯，也不会改变灯的强度、颜色或 linking。

lightList 是缓存或手工 author 的灯光列表，USD type 是 rel relationship。它必须 authored on a model hierarchy prim。空的 lightList 不是“没有写完”，而是一个有意义的缓存：表示该范围内没有灯。

lightList:cacheBehavior 控制使用缓存时的解释方式。consumeAndHalt 表示相信该列表是本 prim 及以下所有灯的最终权威声明，并停止递归；consumeAndContinue 表示读取缓存后继续向下递归，以允许引用或 layering 添加更多灯；ignore 表示忽略缓存，相当于让缓存失效。

常见误读是把 LightListAPI 当成 light-linking。它不是控制哪些对象受哪些灯影响；那类关系属于 LightAPI 的 linking/shadow-linking 语义。LightListAPI 只是帮助遍历系统更快、更明确地找到灯。

调试路径是先确定 LightListAPI applied 在模型层级 prim 上，再看 lightList 是否为空、是否指向正确灯 prim、lightList:cacheBehavior 是否导致递归提前停止，以及是否需要用 ComputeModeIgnoreCache 验证缓存和真实遍历结果是否一致。

与相邻类型的关系：ListAPI 是 deprecated 的旧名；LightAPI 是灯光通用行为；MeshLightAPI 和 VolumeLightAPI 是给具体 prim 类型应用灯行为。若新写资产，应使用 LightListAPI 而不是 ListAPI。
