逐段双语理解 / Paragraph-Level Bilingual Coverage

RectLight 是 usdLux 中的矩形面光源 schema，用来描述从矩形一侧发光的 intrinsic light。官方定义里最重要的空间约定有三个：矩形中心位于局部 XY plane，默认尺寸在 X 和 Y 方向各为 1 unit，发光方向沿局部 -Z axis。也就是说，RectLight 的可见几何尺寸由局部 X/Y 方向上的宽高决定，而照射朝向不是由某个额外的方向属性决定，而是由 prim 的 transform 把局部 -Z 旋到场景中的目标方向。阅读或调试这个页面时，先确认局部坐标，再看 `xformOp:rotateXYZ`、`xformOp:translate` 和 `xformOpOrder`，通常比先调 `intensity` 更有效。

RectLight is an intrinsic light that emits from one side of a rectangle. The rectangle is centered in the XY plane, emits along the -Z axis, and is 1 unit long in X and Y by default.

RectLight 的常见用途是模拟摄影中的 soft boxes、linear lights、fluorescent lights 和 light panels。它和点光源或球形灯不同：RectLight 的亮部来自有面积的矩形，所以阴影边缘、反射高光和照明覆盖范围会体现出面光源尺寸。它也不同于 `PortalLight`：`PortalLight` 通常服务于 dome light 采样入口，重点是帮助渲染器在门窗等矩形开口处采样环境光；RectLight 本身就是发光体，用 `inputs:width`、`inputs:height`、`inputs:texture:file`、`inputs:color` 和 `inputs:intensity` 直接描述矩形发光面的尺寸、颜色贴图和强度。

Use RectLights to illuminate objects, simulating soft boxes used in photography, linear lights, fluorescent lights, and light panels.

官方示例把一个 `RectLight` 放在 `Sphere` 和 `Cube` 附近，设置 `inputs:width = 5`、`inputs:height = 5`，并用 `checkerboard.png` 作为 `inputs:texture:file`。这个示例同时展示了三层信息：第一层是发光面积，5x5 的矩形比默认 1x1 更像大面积灯板；第二层是表面颜色，texture color map 会让发光颜色在矩形面上变化；第三层是空间变换，`xformOp:rotateXYZ = (0, 90, 0)` 和 `xformOp:translate = (3.5, 0, -1)` 把局部 -Z 发光方向和场景摆位联系起来。若渲染结果看起来像没有照到目标，优先检查 transform 和 `xformOpOrder`，再检查 renderer 是否支持相应的 texture/light shaping 组合。

The official example positions a RectLight near a Sphere and Cube, gives it a width/height of 5x5, and uses a colored checkerboard texture as a color map.

`inputs:width` 是矩形灯在局部 X axis 上的宽度，类型为 `float`，fallback value 为 `1.0`。`inputs:height` 是局部 Y axis 上的高度，类型同样是 `float`，fallback value 也是 `1.0`。这两个属性共同决定发光面积和可见面光源尺度，但不替代 transform：如果想让矩形变长或变高，改宽高；如果想改变照射方向，改 transform；如果想改变整体能量或曝光感，再调 `inputs:intensity` 或渲染器相关的 exposure/单位设置。把宽高当作世界空间尺寸、把旋转当作灯光形状参数，都是常见误读。

`inputs:texture:file` 是 `asset` 类型的灯光颜色贴图。官方说明指出，这张 texture 会作为矩形表面的 color map 使用，并且 texture 坐标有固定方向：默认 texture 的 min 坐标在 `(+X, +Y)`，max 坐标在 `(-X, -Y)`。这意味着贴图看似翻转时，不一定是图片文件错了，也可能是你对 RectLight 局部 X/Y 方向、旋转或 viewer 显示方向的预期不一致。中文实现应保留 `inputs:texture:file` 和 `checkerboard.png` 原样，因为它们是 schema 属性名和示例资产路径，不应翻译。

`light:shaderId` 是 RectLight 的 shader 节点标识，类型为 `token`，fallback value 为 `RectLight`。这不是普通用户随便改名的中文标签，而是 USD 通过 Sdr 注册和查找 light shader node 的关键标识。官方页面说明 USD 会注册 identifier 为 `RectLight`、source type 为 `USD` 的 shader node。对使用者来说，通常只需要理解：当 renderer 或 delegate 需要把 USD 灯光 schema 转成可渲染的灯光实现时，`light:shaderId` 提供了这类桥接信息。不要把 `RectLight` token 翻译成“矩形灯”，否则会破坏可解析性。

继承属性也要按职责理解。`extent` 来自 `Boundable`，表示用于包围盒和可见范围计算的 `float3[]` 数据；`xformOpOrder` 来自 `Xformable`，决定多个 transform op 的组合顺序；`proxyPrim`、`purpose` 和 `visibility` 来自 `Imageable`，分别影响代理 prim 关系、渲染用途分类和可见性继承。它们不是 RectLight 独有属性，但在 RectLight 上同样可能影响调试：`visibility` 隐藏了 prim 不等于改掉灯光尺寸，`purpose` 的过滤规则可能影响某些流程看到的对象，`extent` 或 transform 错误会让调试视图中的灯板位置和范围不符合预期。

把 RectLight 与相邻 schema 放在一起看，选择会更清楚：需要圆盘状发光面时看 `DiskLight`，需要球形或柱形发光体时看 `SphereLight` 或 `CylinderLight`，需要把环境光通过矩形入口引导采样时看 `PortalLight`，需要控制遮挡、阴影距离或 light/shadow 行为时再看 `ShadowAPI`。本页的中文主线不是替代官方英文，而是让读者能先用中文建立正确的 mental model，再用英文属性名、代码片段和官方链接核对实现细节。

从制作流程看，RectLight 通常出现在三类任务里。第一类是 lookdev 或产品展示，需要大面积柔和高光，此时宽高比例和灯面朝向比单纯提高强度更关键。第二类是室内或舞台照明，需要把矩形灯板放在窗户、广告牌、灯箱或天花板附近，此时要明确它是主动发光的矩形面，而不是只帮助环境光采样的 portal。第三类是带贴图的灯光设计，需要用 `inputs:texture:file` 给灯面提供空间变化的颜色。三类任务都要求保留 USD 属性名原样，因为 pipeline、renderer delegate、validator 和 DCC 插件都依赖这些 token。

如果页面被用于排查渲染差异，可以按“形状、方向、颜色、能量、继承状态”五步检查。形状检查 `inputs:width` 和 `inputs:height`，方向检查局部 -Z 经过 `xformOpOrder` 后是否指向目标，颜色检查 `inputs:color` 与 `inputs:texture:file` 是否叠加或被 renderer 忽略，能量检查 `inputs:intensity` 与场景单位是否匹配，继承状态检查 `visibility`、`purpose`、collection 过滤或 layer opinion 是否覆盖了期望值。这样读者不需要先读完整英文正文，也能定位 RectLight 最常见的误配来源。

官方页面中的 “Properties”、“Inherited Properties (Boundable)”、“Inherited Properties (Xformable)” 和 “Inherited Properties (Imageable)” 是 Doxygen/Sphinx 结构标题，本复刻页保留它们的英文含义并用中文解释读法。属性表不是把每个字段翻成中文变量名，而是解释字段在 USD 场景中的职责边界：哪些决定矩形灯的物理外观，哪些决定 transform 组合，哪些来自通用可见对象能力。对于 OpenUSD 文档读者来说，这种双语结构比逐词替换更可靠，因为它既能让中文成为主阅读路径，也能保留与官方 schema、代码和生成工具的可核对性。

本页晋级时只处理 `RectLight.html` 一页，源页对比重点覆盖官方定义段落、使用场景段落、示例说明、四个本地属性和三组继承属性。`RectLight`、`inputs:height`、`inputs:width`、`inputs:texture:file`、`light:shaderId`、`checkerboard.png`、`xformOp:rotateXYZ`、`xformOpOrder`、`extent`、`proxyPrim`、`purpose`、`visibility`、`PortalLight` 和 `ShadowAPI` 都保持英文原样。这样既避免把 API/schema/token 翻坏，也避免只在开头写一段泛泛导读后让读者继续依赖英文正文。完成态的判断以中文是否能独立解释用途、属性、示例、继承边界和调试路径为准，而不是简单比较英文和中文字符比例。

属性速查 / Property Reference

属性 / Property中文解释保留英文语义
inputs:height 局部 Y axis 上的矩形高度，类型为 float，fallback 为 1.0。它定义发光面高度，不定义朝向。 Height of the rectangle in the local Y axis.
inputs:width 局部 X axis 上的矩形宽度，类型为 float，fallback 为 1.0。它与 inputs:height 一起决定面光源尺寸。 Width of the rectangle in the local X axis.
inputs:texture:file 作为矩形发光面 color map 的 asset。官方坐标约定为 min 在 (+X, +Y)，max 在 (-X, -Y)。 Texture used as the color map on the rectangle.
light:shaderId 用于 Sdr light shader node 注册和查找的 token，fallback 为 RectLight，不能翻译或改写 token 值。 Shader node identifier for RectLight, source type USD.
extent 从 Boundable 继承，用于范围和包围盒相关计算。 Inherited property from Boundable.
xformOpOrder 从 Xformable 继承，决定 xformOp:translate、xformOp:rotateXYZ 等变换操作的应用顺序。 Inherited transform operation order.
proxyPrim, purpose, visibility 从 Imageable 继承，分别用于代理 prim 关系、用途分类和可见性继承。它们会影响浏览、过滤和调试路径。 Inherited Imageable properties.

官方示例核对 / Official Example Check

下面的精简片段保留官方示例中的关键 token、属性名和文件名，用于说明 RectLight 的尺寸、颜色贴图和 transform 关系。真实页面还包含 `Sphere`、`Cube`、材质和完整 scene 结构，中文说明只解释阅读路径，不重命名任何 USD 标识符。

#usda 1.0
(
    upAxis = "Y"
)

def Scope "Lights"
{
    def RectLight "Light1"
    {
        float inputs:width = 5
        float inputs:height = 5
        asset inputs:texture:file = @checkerboard.png@
        color3f inputs:color = (1, 1, 1)
        float inputs:intensity = 5.0
        float3 xformOp:rotateXYZ = (0, 90, 0)
        double3 xformOp:translate = (3.5, 0, -1)
        uniform token[] xformOpOrder = ["xformOp:translate", "xformOp:rotateXYZ"]
    }
}
  • 如果灯光方向不对，先检查局部 -Z axis 是否被 transform 转到目标方向。Check the local -Z emission direction after transforms.
  • 如果贴图方向不对，检查 RectLight 的局部 X/Y axis 和 texture coordinate orientation，而不是翻译或改名 inputs:texture:file。Keep the property name intact and inspect local axes plus texture coordinate orientation.
  • 如果阴影或采样行为不符合预期，再结合 ShadowAPI、renderer support 和 light shaping 参数排查。Use ShadowAPI and renderer documentation for shadow or shaping behavior.
