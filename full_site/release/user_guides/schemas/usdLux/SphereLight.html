中文主阅读路径 / Chinese Main Reading Path

`SphereLight` 是 usdLux 中的 intrinsic light schema，用来描述从球体表面向外发光的灯。官方说明的关键点有两个：第一，它是 one-sided light，所以不会向球体“内部”发光；第二，如果 `treatAsPoint=true` 且当前 renderer 支持 zero-area lights，这个灯可以被当作 zero-radius point light。中文阅读时应把它理解成“球形或点状光源的具体 light schema”，而不是 `ShapingAPI` 那样的发射方向控制层，也不是 `ShadowAPI` 那样的阴影行为控制层。

`SphereLight` is an intrinsic light that emits outward from a sphere and can optionally be treated as a zero-radius point light.

典型用途是模拟灯泡、车灯、头灯、裸露小光源、发光球或需要近似各向发散的局部光。和 `RectLight`、`DiskLight`、`CylinderLight` 这类面积灯相比，`SphereLight` 的直觉是“从球面向外”而不是从平面、圆盘或柱体表面发光；和 `DistantLight` 相比，它不是无限远方向光，而是在场景中有位置、有半径、会随距离和遮挡关系参与照明。官方示例把 `SphereLight "Light1"` 放在一个 `Sphere` 和 `Cube` 附近，并设置 `inputs:radius = 0.8`、`inputs:color = (1, 1, 1)`、`inputs:intensity = 20.0`、`xformOp:translate = (4, 0, 1)`，说明它既有 light 属性，也仍然通过 Xformable 变换放置在场景中。

Use it for point-like or spherical light sources such as bulbs and headlamps; it remains a positioned scene prim.

本页自有的 `Properties` 很短，但语义很集中：`inputs:radius` 决定发光球的半径，fallback 是 `0.5`；`light:shaderId` 的 fallback 是 `SphereLight`，并会让 USD 注册一个 identifier 为 `SphereLight`、source type 为 `USD` 的 `Sdr` shader node，用来对应这个灯的 inputs；`treatAsPoint` 是传给 renderer 的提示，fallback 是 `False`，表示在支持 non-area lighting 的渲染器中可以把它视作 point light，而只支持 area lights 的渲染器会忽略这个属性。这三项分别对应形状尺寸、shader 识别和渲染器优化/兼容边界。

The local properties cover shape radius, shader identification, and the renderer hint for point-light treatment.

读者最容易误解的是 `treatAsPoint`。它不是把 prim 类型改成别的 schema，也不是强制所有渲染器都使用点光模型；它只是一个 hint。若 renderer 支持 zero-area lights，它可以把球光当成零半径点光来处理；若 renderer 只支持 area lights，它可能直接忽略该值。因此调试时不能只看 USD 文件里有没有 `treatAsPoint=true`，还要查当前 render delegate 的支持范围。类似地，`inputs:radius` 不是 xform scale，半径属性和 transform 缩放都可能影响最终视觉和包围盒理解，应检查 authored radius、local transform、`extent` 与 renderer 的实现。

`treatAsPoint` is a renderer hint, not a schema conversion or a guaranteed visual behavior across all renderers.

继承属性同样影响实际阅读路径。`extent` 来自 `Boundable`，用于边界数据；`xformOpOrder` 来自 `Xformable`，决定变换操作顺序，官方示例使用 `xformOp:translate` 把灯移动到 `(4, 0, 1)`；`proxyPrim`、`purpose`、`visibility` 来自 `Imageable`，分别关系到代理显示、渲染/代理/引导用途和可见性继承。也就是说，即使 SphereLight 的本体属性只有三个，实际场景排错仍要同时看位置、可见性、purpose、extent 和父层级的强意见。

Inherited `Boundable`, `Xformable`, and `Imageable` properties affect bounds, placement, render purpose, and visibility.

如果你看到球光没有照亮场景，建议按这个顺序排查：先确认 prim 类型确实是 `SphereLight`，再确认 `inputs:intensity`、`inputs:color` 和 `inputs:radius` 是否 authored；接着检查 `xformOpOrder` 和 transform 是否把灯放在预期位置；然后检查 `visibility`、`purpose`、collection/linking、payload/variant 和 layer strength；最后才检查 renderer 是否支持 `treatAsPoint` 或是否把 light units、exposure、shaping、shadow 参数按预期处理。这样能避免把位置、可见性、半径、点光提示和阴影问题混在一起。

A practical debugging path is schema type, light inputs, transform, visibility/purpose, linking/composition, then renderer support.

与相邻页面的分工也很重要：`DiskLight` 官方示例和本页配置相近，适合比较球面发光和圆盘发光的视觉差异；`RectLight` 更像面板、软箱或荧光灯；`DistantLight` 是方向光；`DomeLight` 处理环境光；`ShapingAPI` 给光加 cone、focus、IES 等方向塑形；`ShadowAPI` 管理阴影艺术控制；`LightAPI` 管理通用 light 行为和 linking。把 `SphereLight` 放在这些页面之间阅读，能明确它只负责“球形/点状具体光源”的 schema 语义。

Adjacent pages help separate spherical light geometry from disk/rect/distant/dome lights, shaping controls, shadows, and linking.

在实际制作里，`SphereLight` 常用来表达“有位置但形状很小”的光源。半径较大时，它更接近一个可产生柔和近场照明的球面面积灯；半径很小或启用 `treatAsPoint` 时，它更接近点光源。这里的关键不是只看视觉大小，而是看 renderer 如何解释面积、光强、距离衰减和阴影。若一个镜头里需要灯泡外壳本身可见，还要另建几何体或材质来表现灯泡；`SphereLight` prim 负责照明，不等于自动生成可见灯泡模型。

In production, `SphereLight` expresses a positioned spherical or point-like emitter, not a visible bulb model by itself.

`inputs:radius` 的阅读重点是“光源半径”，不是“照亮范围半径”。增大它通常会改变光源的几何尺度、近场外观和阴影柔和度，但不会简单等价于让光照传播得更远。传播强度仍要结合 `inputs:intensity`、曝光、单位约定和 renderer 的光照模型判断。若画面太暗，先不要直接把 `inputs:radius` 当亮度参数调大；应该分别检查 intensity/color/exposure、灯到物体距离、scene units，以及是否有 shadow/linking 或 visibility 限制。

Read `inputs:radius` as emitter size, not as a distance or brightness control.

`light:shaderId` 的重点是 schema 与渲染节点之间的识别关系。官方写明 fallback 为 `SphereLight`，并注册对应的 `Sdr` shader node；这让下游工具和 renderer 能以统一 identifier 找到该灯的 inputs。翻译时必须保留 `SphereLight` 这个 token，不应把它改成中文，也不应把它和页面标题、prim 名称或文件名混为一谈。prim 可以叫 `"Light1"`，但 shader ID 仍然是 `SphereLight`。

`light:shaderId` is the renderer-facing shader identifier, independent of the prim's chosen name.

`treatAsPoint` 的价值在于兼容不同 renderer 的 light model：有些渲染器能处理 zero-area/non-area lights，点光模型更便宜也更符合小灯泡或头灯的美术意图；有些渲染器只支持 area lights，那么该属性可能无效。跨 DCC 或跨 renderer 交付时，应把它当作“可用则采用”的 hint，并在验证报告或资产说明中记录当前目标 renderer 是否支持，而不是依赖它保证所有平台一致。

Treat `treatAsPoint` as a portability-sensitive hint, not a cross-renderer guarantee.

继承属性会决定本地页面是否能被顺畅使用。`extent` 影响选择、包围盒和某些视图裁剪；`xformOpOrder` 影响灯的位置和变换组合；`purpose` 与 `visibility` 会受父级和 stronger opinion 影响；`proxyPrim` 可能参与代理显示流程。一个常见误读是只看 `SphereLight` 自有属性，忽略父层级把灯隐藏、payload 未加载或 transform 被覆盖。中文主阅读路径必须把这些排错入口串起来，否则读者仍会回到英文 Doxygen 表猜测。

Inherited properties are part of the practical reading path because they affect visibility, placement, bounds, and composition.

和 `ShapingAPI` 结合时，建议先确定 `SphereLight` 的基础发光语义，再用 shaping 属性做方向约束。球光本身默认是从球体向外发光；如果需要类似聚光灯的锥形效果，应看 `inputs:shaping:cone:angle` 与 `inputs:shaping:cone:softness`，而不是试图用 `inputs:radius` 做方向控制。和 `ShadowAPI` 结合时，半径可能影响阴影外观，但阴影 enable、distance、falloff 和 color 仍属于 shadow 控制层。

Use `SphereLight` for the base spherical emitter, `ShapingAPI` for directional shaping, and `ShadowAPI` for shadow controls.

官方示例理解 / Official USDA Example

官方示例的核心结构是：在 `Scope "Lights"` 下定义 `SphereLight "Light1"`，设置 `inputs:radius = 0.8`、白色 `inputs:color`、`inputs:intensity = 20.0`，并用 `xformOp:translate = (4, 0, 1)` 放置它；随后在 `Xform "TestGeom"` 下放置一个 `Sphere` 和一个 `Cube`，二者 `purpose = "render"`，用于观察球光照明效果。该示例还提示可以和 `DiskLight` 示例比较，因为两者配置相近但发光几何不同。

The official USDA example places a `SphereLight` near a sphere and cube, with radius 0.8 and a simple transform.

#usda 1.0
(
    upAxis = "Y"
)

def Scope "Lights"
{
    def SphereLight "Light1"
    {
        float inputs:radius = 0.8
        color3f inputs:color = (1, 1, 1)
        float inputs:intensity = 20.0
        double3 xformOp:translate = (4, 0, 1)
        uniform token[] xformOpOrder = ["xformOp:translate"]
    }
}

调试这个示例时，`inputs:radius` 解释光源尺度，`inputs:intensity` 和 `inputs:color` 解释亮度与颜色，`xformOp:translate` 解释光源位置，`xformOpOrder` 解释变换顺序。不要把 `SphereLight` 与几何 `Sphere` 混淆：前者是光源 schema，后者是可渲染几何体。

Do not confuse `SphereLight` as a light schema with a geometric `Sphere` prim used in the scene.

调试顺序 / Debugging Order

  1. 确认 prim type 是 `SphereLight`，而不是普通 `Sphere` 几何体或 `GeometryLight`。Confirm the prim type is `SphereLight`.
  2. 检查 `inputs:radius`、`inputs:intensity`、`inputs:color` 是否 authored，并区分 radius 与 transform scale。Check radius, intensity, and color separately from transform scale.
  3. 检查 `xformOpOrder` 和 transform，确认灯确实位于目标物体附近。Check `xformOpOrder` and transform placement.
  4. 检查 `visibility`、`purpose`、父级可见性、payload、variant 和 stronger layer opinion。Check visibility, purpose, payloads, variants, and stronger opinions.
  5. 如果设置了 `treatAsPoint=true`，确认 renderer 支持 zero-area/non-area lighting；不支持时预期它可能被忽略。Verify renderer support before relying on `treatAsPoint`.
  6. 若形状或阴影不是预期，再转到 `ShapingAPI` 和 `ShadowAPI`，不要把所有问题都归因于 `SphereLight` 本体属性。Use `ShapingAPI` and `ShadowAPI` for shaping and shadow-specific behavior.

边界与相邻页面 / Boundaries and Adjacent Pages

  • DiskLight：官方建议可与本页示例比较；它是圆盘发光，和球面向外发光不同。Compare with `DiskLight` for a similar setup but different emitting geometry.
  • RectLight：用于面板、软箱或矩形发光面；不要用 `SphereLight` 模拟大面积矩形灯。Use `RectLight` for rectangular area emitters.
  • DistantLight 与 DomeLight：分别适合无限远方向光和环境光，不是局部球形光源。Use distant or dome lights for directional or environment lighting.
  • ShapingAPI：控制 cone、focus、IES 等发射塑形；它可以和球光一起使用，但不是球光的半径属性。Use `ShapingAPI` for emission shaping on top of the light schema.
  • ShadowAPI：控制阴影艺术行为；不要把阴影距离、falloff 或 shadow color 写到 `SphereLight` 本体属性里。Use `ShadowAPI` for shadow controls.
  • LightAPI：用于通用 light 行为、linking 和 collection 语义；`SphereLight` 只定义具体光源形状。Use `LightAPI` for common light behavior and linking semantics.
