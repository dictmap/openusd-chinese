逐段双语理解 / Paragraph-Level Bilingual Coverage

DomeLight 是 usdLux 中用于环境照明的 intrinsic light schema。它不是从场景里某个近处几何面或点发光，而是把天空、HDR environment map 或其他非常远的外部环境当成光源，从外部向场景内部发光。官方把它放在 High Dynamic Range (HDR) 图像和 Image Based Lighting (IBL) 的语境里理解：纹理本身记录环境的亮度和颜色分布，渲染器用这张环境图为整个场景提供背景式、全局式的照明。中文阅读时应先把它定位为“环境光入口”，而不是 SphereLight、DiskLight 或 RectLight 那类局部灯具。

DomeLight represents distant environment lighting, often driven by HDR images for Image Based Lighting.

官方说明还强调：DomeLight 可用于模拟 environment lighting。这意味着它特别适合天空光、摄影棚 HDRI、室外全局氛围、产品展示中的环境反射，以及需要从整个外部球面或穹顶采样光照的场景。它不负责表达“窗口里的一块发光面”或“舞台上一盏局部灯”的近场几何；如果目标是局部发光面，应优先看 RectLight、DiskLight 或 CylinderLight；如果目标是把环境光引导到室内开口，应结合 PortalLight 和本页的 portals relationship 阅读。

Use it for scene-wide environment illumination, not as a near-field local light shape.

默认朝向是本页最容易误读的部分。官方说明指出，dome 的 top pole 默认和 world +Y axis 对齐，并且遵循 OpenEXR latitude-longitude map 约定。也就是说，纹理不是任意贴到一个球壳上，而是按经纬度投影解释：图像的横向坐标对应 longitude，纵向坐标对应 latitude；在三维空间里，latitude 的两端对应 -Y 与 +Y，latitude 0 与 longitude 0 指向 +Z，latitude 0 与 longitude pi/2 指向 +X。如果 HDRI 看起来旋转了 90 度、天空倒置或太阳方向不对，第一步应检查 OpenEXR 经纬度约定、stage up axis、prim transform 和后续 DomeLight_1 中的 poleAxis 选项，而不是只调强度。

The default pole aligns to world +Y and follows the OpenEXR latitude-longitude map convention.

官方页链接到另一个 DomeLight_1 版本，用 poleAxis 提供额外的 dome orientation 控制。阅读路径上应把这两页分开：本页讲默认 DomeLight 的环境贴图、纹理格式、guide geometry 和 portals；DomeLight_1 更关注极轴方向的显式控制。若资产在多个 DCC 或 renderer 之间交换，保留 DomeLight、DomeLight_1、poleAxis 这些英文标识非常重要，因为它们是 schema、页面和属性级别的技术锚点，不能翻成普通中文标签。

DomeLight_1 is an adjacent variant that adds orientation control through poleAxis.

官方示例使用一个 def DomeLight "Dome"，并设置 asset inputs:texture:file = @orientationLatLong.tex@。这说明环境图以 USD asset 路径形式进入灯光 schema。示例后半段创建 Sphere、应用 MaterialBindingAPI，再用 PxrSurface 材质展示该 dome 环境图如何照亮一个带基本材质的球体。中文页保留代码原样，是为了让读者能直接核对 inputs:texture:file、MaterialBindingAPI、rel material:binding、PxrSurface 等关键语法；中文解释只负责说明每段代码的意图，不替换 token 或属性名。

The example authors a DomeLight with inputs:texture:file and illuminates a material-bound sphere.

guideRadius 是可视化辅助属性，不是光照真实半径。它的 USD type 是 float，fallback value 是 100000.0，用于设置 dome light guide geometry 的半径，单位是 USD units。官方说明把 fallback 1.0e5 换算为 1 km，前提是场景 metersPerUnit 使用 USD 默认 0.01，也就是 1 unit 等于 1 cm。调试时不要把 guideRadius 当作环境光衰减距离或世界真实天空半径；它主要影响视口里辅助几何的大小和读者对 dome 范围的可视化判断。

guideRadius controls guide geometry radius in USD units, not near-field light falloff.

inputs:texture:file 是本页最核心的输入之一，USD type 是 asset。它指向 DomeLight 使用的 color texture，通常是用于 IBL 的 HDR 图像。若画面黑、反射不对或天空颜色异常，应检查 asset path 是否能解析、纹理是否真的为 HDR/环境图、文件是否被打包或引用层重定向，以及 renderer 是否支持该格式。该属性名必须保留英文完整形式，尤其是 inputs: namespace 和 texture:file 结构，因为它直接对应 USD authored property。

inputs:texture:file points to the environment texture asset, typically an HDR image for IBL.

inputs:texture:format 用于声明 color texture 的 parameterization。它的 USD type 是 token，fallback value 是 automatic。有效值包括 automatic、latlong、mirroredBall、angular 和 cubeMapVerticalCross。这些 token 不应翻译。automatic 表示 renderer 尝试从文件判断布局，例如 RenderMan texture files 可以嵌入明确的 parameterization；latlong 走经纬度布局；mirroredBall 表示环境在球体上的反射图；angular 类似 mirrored ball，但半径维度线性映射到角度，边缘采样更好；cubeMapVerticalCross 表示 faces 按 vertical cross 排布的 cube map。

inputs:texture:format declares how the environment texture is parameterized.

light:shaderId 是面向 renderer 的 shader 标识，USD type 是 token，fallback value 是 DomeLight。官方说明中，USD 还会注册一个 identifier 为 DomeLight、source type 为 USD 的 Sdr shader node，用来对应该灯光的 inputs。这个 token 不能翻译，也不能和 prim name "Dome" 混淆。跨 DCC 或 renderer 排查时，如果灯没有按 dome 环境光识别，应检查 prim type、light:shaderId 是否被更强 layer 覆盖，以及目标 renderer 是否正确解析 usdLux Sdr 节点。

light:shaderId preserves the DomeLight shader identity and USD Sdr mapping.

portals 是 relationship 类型，用于声明可选 portals，以引导 light sampling。它和 PortalLight 的用途相邻，但不要把 relationship 本身理解成另一盏普通灯。典型场景是室内空间通过窗户、门洞或其他开口接收 dome 环境光；没有 portal 时，renderer 可能很难有效采样来自开口方向的环境贡献；添加 portals 后，采样可以更集中地穿过这些区域。中文阅读时应把 portals 理解为采样引导关系，而不是替代 inputs:texture:file 或局部灯光强度的参数。

portals is an optional relationship that guides environment light sampling.

继承属性同样会影响最终阅读和调试。xformOpOrder 来自 Xformable，决定 transform op 的应用顺序，可能影响 dome 的最终朝向。proxyPrim、purpose 和 visibility 来自 Imageable，分别影响代理 prim、渲染用途分类和可见性继承。若环境光不出现，不能只检查 inputs:texture:file；还要看 visibility 是否被父层级或 variant 覆盖，purpose 是否被管线过滤，引用层或 payload 是否使灯光 prim 没有进入最终 composed stage。

Inherited Xformable and Imageable properties still matter for orientation, purpose, proxying, and visibility.

推荐调试顺序是：先确认 prim type 确实为 DomeLight，再确认 inputs:texture:file 能解析到正确 HDR/environment map；接着检查 inputs:texture:format 是否匹配贴图布局，确认默认 +Y pole 与 OpenEXR 经纬度约定是否符合资产预期；然后检查 transform、DomeLight_1 或 poleAxis 相关差异；再看 guideRadius 是否只是视口辅助几何问题；最后检查 portals、composition、visibility、renderer 支持、color management 和 exposure。这样可以避免把纹理方向、采样效率、资产路径、可见性和曝光问题混成一个问题。

A practical debugging path is schema type, texture file, texture format, orientation, guide geometry, portals, composition, and renderer support.

与相邻 usdLux 页面一起读时，DomeLight 的职责边界很清楚：DistantLight 是远处方向光，适合太阳式的近似平行光；DomeLight 是环境图或天空式的全局照明；PortalLight 帮助环境采样进入开口区域；RectLight、DiskLight、SphereLight 和 CylinderLight 都描述更具体的局部几何灯；ShapingAPI 与 ShadowAPI 提供塑形和阴影控制。不要用 DomeLight 表达局部灯具外观，也不要用 DistantLight 表达 HDRI 环境反射。

Adjacent usdLux pages separate dome environment lighting from distant, portal, local geometric, shaping, and shadow concepts.

中文核对段落 / Plain Chinese Coverage Check

核对一：DomeLight 的核心职责是把非常远的天空、HDR 环境图或 IBL 环境作为光源向场景内部照明；它表达的是全局环境贡献，而不是近处灯具几何。

核对二：默认朝向需要按 OpenEXR latitude-longitude map 约定理解，top pole 对齐 world +Y axis，方向问题通常先查贴图布局、stage 轴向和 transform。

核对三：inputs:texture:file 是环境贴图 asset 入口，排查黑图或错误反射时应先确认路径解析、HDR 文件、打包引用和 renderer 支持。

核对四：inputs:texture:format 的 automatic、latlong、mirroredBall、angular、cubeMapVerticalCross 都是 token，必须原样保留，用来说明环境图参数化方式。

核对五：guideRadius 控制 guide geometry 的可视化半径，不是光照衰减距离，也不是场景真实天空半径；它主要影响视口辅助显示。

核对六：light:shaderId 的 fallback 是 DomeLight，USD 会用对应 Sdr shader node 映射灯光输入；这个标识不能翻译或混成 prim name。

核对七：portals 是采样引导 relationship，适合和 PortalLight 一起理解，用来让环境光更有效地穿过窗口、门洞等开口区域。

核对八：xformOpOrder、proxyPrim、purpose、visibility 这些继承属性仍可能影响朝向、代理、渲染用途和可见性，不能只检查灯光专有属性。

核对九：DomeLight、DistantLight、RectLight、DiskLight、SphereLight、CylinderLight、PortalLight、ShapingAPI 和 ShadowAPI 的职责边界要分开阅读。

核对十：本页中文主路径覆盖官方定义、OpenEXR 方向、示例代码、属性表、继承属性、调试顺序和相邻页面关系，英文只作为术语和原站核对。

官方示例代码 / Official Example Code

以下保留官方示例的结构和关键 token，用于核对 DomeLight、inputs:texture:file、MaterialBindingAPI 与 PxrSurface 的语义。中文解释不改写代码中的 schema 名、属性名、prim 名或 asset 引用。

The example keeps the USD schema names, authored property names, material binding, and shader identifiers intact.

#usda 1.0

def Scope "Lights"
{
    def DomeLight "Dome"
    {
        asset inputs:texture:file = @orientationLatLong.tex@
    }
}

def Xform "TestGeom"
{
    def Sphere "Sphere1"
    (
        prepend apiSchemas = ["MaterialBindingAPI"]
    )
    {
        rel material:binding = </Material>
    }
}

def Material "Material"
{
    token outputs:ri:surface.connect = </Material/Surface.outputs:out>

    def Shader "Surface"
    {
        uniform token info:id = "PxrSurface"
        float inputs:diffuseGain = 0.3
        color3f inputs:specularEdgeColor = (1, 1, 1)
        color3f inputs:specularFaceColor = (0.4, 0.4, 0.4)
        float inputs:specularRoughness = 0.02
        token outputs:out
    }
}
