BoundableLightBase 是本轮 usdLux 短页小批量冲刺中的一个完成页。中文主阅读路径覆盖页面职责、官方 section、属性/API 分组、边界、误读点、调试路径和相邻 usdLux 类型关系；英文术语用于和官方页核对，不作为主要阅读路径。

The base class for intrinsic lights that are boundable, such as RectLight and SphereLight.

官方源页核心说明：The base class for intrinsic lights that are boundable.

BoundableLightBase 是可 bound 的 intrinsic lights 的基类，例如 RectLight、SphereLight、DiskLight、CylinderLight 和 PortalLight。它不是读者通常直接 author 的最终灯类型，而是具体灯 schema 共享的基类语义。

官方说明它为 concrete derived lights 提供直接 API，能力来源对应 LightAPI。也就是说，派生灯既有自身形状属性，例如 radius、width、height、length，也继承了通用灯行为；BoundableLightBase 帮读者理解这些能力如何在可边界几何灯上组织。

“boundable” 的关键在于这些灯有可计算或可表达的空间范围，因此继承 Boundable 的 extent。这和 DistantLight、DomeLight 这类 non-boundable 灯不同，后者通常不依赖场景中的局部位置和范围来计算照明。

本页直接 Properties section 很短，甚至没有新的直接属性；它的价值在于 inherited properties。中文阅读时应明确：空的 direct properties 不代表页面没有意义，而是基类通过继承把 Boundable、Xformable 和 Imageable 的语义交给派生灯。

extent 是 float3[]，来自 Boundable。它用于表示边界范围，可被渲染、选择、裁剪或可视化逻辑使用；但它不是灯强度、曝光或 shading 的直接控制项。

调试路径是先确认具体派生灯类型，再看其形状属性和 extent 是否匹配；接着检查 xformOpOrder、visibility、purpose，最后再排查 LightAPI 的颜色、强度、normalize、linking、shadow 或 renderer 支持。

常见误读是把基类当成可直接替代 RectLight 或 SphereLight 的灯。正确理解是：BoundableLightBase 解释可边界 intrinsic lights 的共同结构；真正 author 时通常选择具体 schema。

与相邻类型关系：NonboundableLightBase 解释没有 scene bounds 的远距或环境灯；LightAPI 是通用灯 API；GeometryLight 是 deprecated 的几何灯；MeshLightAPI 是当前 mesh 发光推荐路径。

本轮已核对官方页和本地 source snapshot：base class 定义、RectLight/SphereLight 示例、指向 Representing Boundable Lights 的语义、extent、xformOpOrder、proxyPrim、purpose、visibility 都已覆盖。
