逐段双语理解 / Paragraph-Level Bilingual Coverage

DomeLight_1 和无后缀 DomeLight 一样，属于 usdLux 中用于环境照明的 intrinsic light schema。它从非常远的外部环境向场景内部发光，典型来源是 sky、High Dynamic Range (HDR) image 或用于 Image Based Lighting (IBL) 的环境贴图。中文阅读时，先把它理解为“环境图驱动的全局照明入口”，而不是近场点光、面光或几何灯。它的页面重点不只是说明 dome light 会发光，还强调这一版本通过 poleAxis 控制穹顶极轴起始方向。

DomeLight_1 is an environment-lighting schema whose orientation is controlled by poleAxis.

官方用途段说明 DomeLights 用于模拟 environment lighting。也就是说，本页面向的实际问题是：如何把一张 HDRI 或其他环境图作为光照分布应用到整个 stage，如何决定该环境图的上方向和经纬度对应关系，以及如何让 renderer 按正确的坐标约定采样。若读者想表达太阳式远处方向光，应看 DistantLight；若想表达矩形灯板、圆形灯、管灯或球灯，应看 RectLight、DiskLight、CylinderLight 或 SphereLight。

The page is about scene-wide environment illumination rather than a local geometric light shape.

poleAxis 是 DomeLight_1 相比 DomeLight 更需要单独阅读的属性。官方说明中，它决定 dome top pole 的 starting alignment，USD type 是 token，fallback value 是 scene。当值为 scene 时，dome 的 top pole 会和 stage 的 up axis 对齐；当值为 Y 时，top pole 对齐 +Y axis；当值为 Z 时，top pole 对齐 +Z axis。这个属性不是强度、颜色或纹理路径参数，而是控制环境图如何套到 dome 上的方向语义。

poleAxis controls the starting alignment of the dome top pole.

官方还特别提醒：为了让 dome light 对齐 poleAxis 所需的旋转，应由 renderer 只应用到 dome 本身，而不应继承到 dome light prim 的 USD namespace children。这个细节很重要，因为很多读者会把 prim transform、children transform 和 renderer 内部对 dome 采样空间的旋转混在一起。中文主路径应明确：poleAxis 影响环境 dome 的解释方向，不表示要把该 prim 下的所有子对象一起旋转，也不表示 USD namespace 中的 children 会自动继承这个渲染器内部旋转。

The alignment rotation is intended for the dome itself, not for namespace children of the dome light prim.

当 poleAxis 设为 Y，或者设为 scene 且 stage up axis 也是 Y 时，官方说明该默认朝向遵循 OpenEXR latitude-longitude map 约定。该约定把图像横向坐标映射为 longitude，纵向坐标映射为 latitude；三维空间中，latitude 两端对应 -Y 和 +Y，latitude 0 与 longitude 0 指向 +Z，latitude 0 与 longitude pi/2 指向 +X。排查 HDRI 倒置、太阳方位不对或环境反射旋转错误时，应同时检查 poleAxis、stage up axis、贴图格式和 prim transform。

For Y alignment, the page follows the OpenEXR latitude-longitude map convention.

本页还链接到无后缀的 DomeLight 版本。阅读顺序上，可以把 DomeLight 当作默认 dome 环境照明页，把 DomeLight_1 当作带 poleAxis 显式方向控制的相邻版本。两页都使用 guideRadius、inputs:texture:file、inputs:texture:format、light:shaderId 和 portals，但本页需要更突出地解释 poleAxis 与 OpenEXR 经纬度约定、stage up axis 和 renderer 采样空间之间的关系。

This page should be read next to the alternative DomeLight page.

guideRadius 是可视化辅助属性，USD type 是 float，fallback value 是 100000.0。它设置用于显示 dome light 的 guide geometry 半径，单位是 USD units。官方把 fallback 1.0e5 换算为 1 km，前提是场景使用 USD 默认 metersPerUnit=0.01，即 1 unit 等于 1 cm。它不控制物理天空半径，也不控制环境光衰减范围；如果视口中 dome guide 太大或太小，可以检查该值，但渲染照明问题通常应先查纹理、格式、方向和 renderer 支持。

guideRadius controls guide geometry size in USD units.

inputs:texture:file 是环境贴图入口，USD type 是 asset。它指向 DomeLight_1 使用的 color texture，通常是用于 IBL 的 HDR image。排查时先确认 asset path 是否能通过 resolver 找到，纹理是否是正确的 HDR 或环境图，路径是否被 layer、payload、package 或 asset resolver 改写，以及 renderer 是否支持该文件格式。这个属性名必须原样保留，尤其是 inputs: namespace 和 texture:file 结构。

inputs:texture:file points to the environment texture asset.

inputs:texture:format 指定 color texture 的 parameterization，USD type 是 token，fallback value 是 automatic。有效值包括 automatic、latlong、mirroredBall、angular 和 cubeMapVerticalCross。这些 token 不能翻译。automatic 让 renderer 尝试从文件中判断布局；latlong 表示经纬度图；mirroredBall 表示环境在球面反射图中的布局；angular 类似 mirrored ball，但 radial dimension 线性映射到 angle；cubeMapVerticalCross 表示 cube map faces 以 vertical cross 形式排列。

inputs:texture:format declares the environment texture parameterization.

light:shaderId 是 renderer-facing shader 标识，USD type 是 token，fallback value 是 DomeLight。虽然页面标题是 DomeLight_1，官方仍说明 USD 会注册 identifier 为 DomeLight、source type 为 USD 的 Sdr shader node，对应该灯光的 inputs。中文解释必须保留这个差异：页面名、schema 版本语境和 shader identifier 不一定逐字相同，不能把 DomeLight 翻译成中文，也不能把它误改成 DomeLight_1。

light:shaderId keeps the DomeLight shader identity and USD Sdr mapping.

portals 是 rel relationship，用于声明可选 portals 来引导 light sampling。它通常和 PortalLight 一起理解：当室内场景需要从窗口或门洞接收 dome 环境光时，portal 可以帮助 renderer 更有效地采样环境贡献。portals 不是另一张环境贴图，也不是局部灯具强度；它是采样路径和关联关系层面的指导。

portals is an optional relationship for guiding environment light sampling.

继承属性也需要纳入阅读路径。xformOpOrder 来自 Xformable，决定 transform op 的应用顺序，可能与 stage up axis 和 poleAxis 一起影响最终朝向判断。proxyPrim、purpose 和 visibility 来自 Imageable，分别影响代理 prim、渲染用途分类和可见性继承。如果 dome light 没有效果，不要只检查 inputs:texture:file；还要检查 visibility 是否被父层级覆盖，purpose 是否被渲染管线过滤，composition 是否让 authored value 被 stronger layer 改写。

Inherited Xformable and Imageable properties remain relevant for orientation and visibility debugging.

推荐调试顺序是：先确认 prim type 和页面语境确实对应 DomeLight_1，再确认 inputs:texture:file 能解析到正确 HDR/environment map；接着检查 inputs:texture:format 是否匹配贴图布局；然后检查 poleAxis、stage up axis、OpenEXR 经纬度约定和 transform；再看 guideRadius 是否只是辅助几何显示问题；随后检查 portals、composition、visibility、renderer 支持、color management 和 exposure。这样可以把方向错误、路径错误、采样噪声、可见性问题和曝光问题分开处理。

A practical debugging path is texture file, texture format, pole axis, stage axis, transform, portals, composition, and renderer support.

与相邻页面一起读时，DomeLight_1 的职责边界是“带极轴方向控制的环境 dome”。DomeLight 讲默认 dome 环境照明；DistantLight 讲远处方向光；PortalLight 讲环境采样开口引导；GeometryLight 进入几何发光语境；ShapingAPI 和 ShadowAPI 则处理塑形和阴影控制。读者如果只想旋转 HDRI 的上方向，应优先检查 poleAxis 和 texture format，而不是改用局部灯 schema。

Adjacent usdLux pages separate dome orientation control from distant, portal, geometry, shaping, and shadow concepts.

本轮已核对官方页和本地 source snapshot：标题 DomeLight_1、环境光定义、poleAxis 默认值 scene、renderer 只对 dome 本身应用对齐旋转的说明、OpenEXR 经纬度约定、DomeLight 相邻链接、properties、inherited properties、previous/next 链接均纳入中文解释。页面保留 DomeLight_1、DomeLight、poleAxis、guideRadius、inputs:texture:file、inputs:texture:format、light:shaderId、portals、xformOpOrder、proxyPrim、purpose、visibility、scene、Y、Z、automatic、latlong、mirroredBall、angular、cubeMapVerticalCross 等技术标识原样，中文负责建立独立可读的主路径。

中文核对段落 / Plain Chinese Coverage Check

核对一：DomeLight_1 的核心职责是使用 sky、HDR image 或 IBL 环境贴图向场景内部提供全局环境照明。

核对二：poleAxis 是本页与无后缀 DomeLight 的关键差异，它控制 dome top pole 的起始对齐方向。

核对三：poleAxis 的 fallback 是 scene，表示 top pole 默认对齐 stage up axis；Y 和 Z 分别表示对齐 +Y axis 和 +Z axis。

核对四：renderer 为对齐 poleAxis 而应用的旋转只针对 dome 本身，不应继承到该 dome light prim 的 USD namespace children。

核对五：当 poleAxis 为 Y，或 poleAxis 为 scene 且 stage up axis 为 Y 时，默认朝向遵循 OpenEXR latitude-longitude map 约定。

核对六：inputs:texture:file 是环境贴图 asset 入口，inputs:texture:format 决定 automatic、latlong、mirroredBall、angular、cubeMapVerticalCross 等布局语义。

核对七：guideRadius 控制 guide geometry 可视化半径，不是环境光的真实照明距离，也不是物理天空半径。

核对八：light:shaderId 的 fallback 是 DomeLight，Sdr shader node identifier 也保持 DomeLight，不应因为页面名是 DomeLight_1 而改写。

核对九：portals 是用于引导 light sampling 的 relationship，适合和 PortalLight 一起解释，不是局部灯具或额外纹理。

核对十：xformOpOrder、proxyPrim、purpose、visibility 可能影响方向、代理、用途和可见性，应纳入调试路径。

核对十一：DomeLight_1 与 DistantLight、PortalLight、GeometryLight、ShapingAPI、ShadowAPI 的职责不同，不能互相替代。

核对十二：本页中文主路径覆盖官方定义、方向约定、属性表、继承属性、调试顺序和相邻页面关系，英文只作为术语和原站核对。
