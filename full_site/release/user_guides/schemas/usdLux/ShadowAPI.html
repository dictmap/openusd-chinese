逐段双语理解 / Paragraph-Level Bilingual Coverage

`ShadowAPI` 的核心职责是给灯光增加阴影外观控制。官方页把它描述为用来控制 shadow behavior 的 API，例如 shadow color、distance 和 falloff，并明确提醒这些控制是 `non-physical controls`。这句话是阅读本页的边界：`ShadowAPI` 不是新的真实光传输模型，不承诺能量守恒，也不替代渲染器的物理阴影算法；它提供的是给内容作者和 lighting artist 调整阴影显示方式的 schema 入口。

`ShadowAPI` controls shadow behavior such as color, distance, and falloff, and the official page frames these controls as non-physical artistic controls.

所谓 `non-physical controls`，意思是这些属性可以有意绕过严格物理规律，以满足镜头、美术风格、可读性或合成需求。比如阴影在真实世界里通常不会被任意染成指定颜色，但 `inputs:shadow:color` 允许作者调节灯光投射阴影的颜色；真实世界里遮挡物和接收面之间的距离不会用一个简单阈值决定阴影是否存在，但 `inputs:shadow:distance`、`inputs:shadow:falloff` 和 `inputs:shadow:falloffGamma` 可以给渲染结果增加距离限制和渐隐曲线。

The controls are intentionally artistic: they may tint shadows or fade them by distance in ways that should not be overinterpreted as physically exact lighting.

本页与 `LightAPI` 的 shadow-linking 必须分开理解。`LightAPI` 中的 light-linking / shadow-linking 更关注“哪些几何体被这盏灯照亮或参与这盏灯的阴影关系”；`ShadowAPI` 更关注“已经参与该灯阴影计算的结果如何显示、是否启用、在距离上如何淡出”。换句话说，linking 决定集合成员和关系，`ShadowAPI` 决定该灯自身阴影输出的美术控制参数。把二者合并成一个笼统的“阴影设置”，会让排查路径变混乱。

Separate membership-oriented shadow-linking from `ShadowAPI`'s per-light shadow appearance controls.

官方页的阅读结构很短，但属性含义需要逐个建立。页面先给出 `ShadowAPI` 总述，然后指向 `Shadows` 示例页，接着在 `Properties` section 中列出 `inputs:shadow:color`、`inputs:shadow:distance`、`inputs:shadow:enable`、`inputs:shadow:falloff` 和 `inputs:shadow:falloffGamma`。本地中文页保留这个结构，并补充每个属性的使用语义、默认值含义、相邻概念边界和调试顺序，使读者不用依赖英文正文也能知道该从哪一个属性开始排查。

The local page follows the official structure: summary, `Shadows` reference, and the five `inputs:shadow:*` properties.

`inputs:shadow:color` 的 USD type 是 `color3f`，fallback value 是 `(0, 0, 0)`。官方语义是设置该灯投射阴影的颜色。中文阅读时应把它理解为阴影 tint 或 stylization 参数：当需要让阴影偏冷、偏暖、偏品牌色或与合成背景匹配时，它提供了一个直接入口。它不改变物体材质的 `UsdShade` 网络，不等价于接收面的 base color，也不表示遮挡物吸收了某种真实光谱能量。

`inputs:shadow:color` is a `color3f` with fallback `(0, 0, 0)` and sets the color of shadows cast by the light.

`inputs:shadow:distance` 的 USD type 是 `float`，fallback value 是 `-1.0`。官方语义是设置 occluder 与 shadow-receiving surface 之间仍会投射阴影的最大距离；fallback `-1.0` 表示没有距离限制。调试时，如果远处遮挡仍然投影，先检查这个值是否仍为 `-1.0`；如果阴影突然消失或范围过短，检查 scene units、灯光空间尺度和遮挡物到接收面的实际距离。它控制的是距离上限，不是阴影软硬、透明度或灯光亮度。

`inputs:shadow:distance` is a `float` with fallback `-1.0`; `-1.0` means no maximum distance limit.

`inputs:shadow:enable` 的 USD type 是 `bool`，fallback value 是 `True`。它控制应用了该 schema 的 prim 是否在 lighting calculations 中投射阴影。这个属性适合做灯光级别的阴影开关：需要保留灯光照明但临时关闭该灯造成的阴影时，可以设置它；需要更细粒度地控制“哪些几何体对这盏灯有阴影”，则应回到 light-linking / shadow-linking，而不是滥用一个总开关。它不删除灯光、不改变集合关系，也不保证所有 renderer 以完全相同方式呈现禁用结果。

`inputs:shadow:enable` is a `bool` with fallback `True`; use linking controls for finer control of which geometry participates.

`inputs:shadow:falloff` 的 USD type 是 `float`，fallback value 是 `-1.0`。它设置 shadow falloff 的大小，用来隐藏或调整到达最大 `shadow:distance` 附近时的硬截断。可以把它看成从 full shadow 逐渐过渡到 no shadow 的 fade zone 宽度：falloff zone 的起点仍有完整阴影，走到最大 shadow distance 时淡到没有阴影。官方还强调 falloff zone 不能超过最大 shadow distance；当取值小于或等于 `0.0` 时，表示没有 falloff zone。

`inputs:shadow:falloff` is a `float` with fallback `-1.0`; it defines the fade zone near the maximum shadow distance.

`inputs:shadow:falloffGamma` 的 USD type 是 `float`，fallback value 是 `1.0`。它控制 falloff zone 内阴影强度随线性距离变化的速率，并被当作 gamma/exponential 控制使用。正确使用 `falloffGamma` 依赖 `shadow:distance` 和 `shadow:falloff` 已经合理设置：如果没有最大距离或没有有效 falloff zone，gamma 曲线没有明确的发挥空间；如果 falloff 太宽或太窄，gamma 会放大这种错误的视觉观感。调试顺序应是先设距离，再设 falloff zone，最后微调 `falloffGamma`。

`inputs:shadow:falloffGamma` is a `float` with fallback `1.0`; it shapes the exponential falloff rate inside the falloff zone.

`Shadows` 链接在官方页中承担示例入口作用，而不是替代本页的属性定义。本地站点把该入口路由到同域本地页面，读者可以先在本页理解 `ShadowAPI` 的属性表，再跳到 `Shadows` 示例理解实际 authoring 场景。只有“打开官方原页 / Open official page”是明确外跳；普通阅读路径应继续留在本地中文站，避免在未提示的情况下跳回官方英文站。

The `Shadows` reference is an example path; local reading links should remain local except for the explicit official-page jump.

与 `ShapingAPI` 对照时，`ShadowAPI` 管阴影；`ShapingAPI` 管 light shaping，例如 cone、focus、IES profile 等发光分布或聚光控制。与 `RectLight`、`DiskLight`、`SphereLight`、`DistantLight` 等具体 light schema 对照时，后者描述灯光类型、形状或方向，`ShadowAPI` 作为 API schema 可以附加或参与这些灯光的阴影行为解释。与 `PortalLight` 对照时，`PortalLight` 引导 dome-light sampling；`ShadowAPI` 不描述 portal 采样入口，而描述阴影显示控制。

Use adjacent usdLux pages to distinguish shadow appearance controls from light shape, light type, and dome-light portal sampling.

生产排查可以按四层进行。第一层检查 schema 和属性名：确认目标灯光确实带有 `ShadowAPI` 相关属性，且 `inputs:shadow:*` 命名空间没有拼写错误。第二层检查 renderer 支持：这些是非物理艺术控制，具体实现和支持程度可能依赖 render delegate。第三层检查数值关系：`inputs:shadow:distance`、`inputs:shadow:falloff`、`inputs:shadow:falloffGamma` 必须形成合理组合，falloff zone 不能超过最大距离。第四层再检查 linking、visibility、material opacity、几何遮挡和 scene scale，避免把其他系统问题误判成 `ShadowAPI` 失效。

Debug schema data, renderer support, distance/falloff value relationships, and adjacent visibility/linking/material issues separately.

完成阅读后，读者应能独立回答这些问题：`ShadowAPI` 为什么是艺术控制而不是物理阴影模型；`inputs:shadow:color` 改的是阴影颜色而不是材质颜色；`inputs:shadow:distance=-1.0` 为什么表示无限制；`inputs:shadow:enable=True` 为什么只是灯光级阴影开关；`inputs:shadow:falloff` 和 `inputs:shadow:falloffGamma` 为什么必须配合距离使用；以及什么时候应该转到 `LightAPI` 的 shadow-linking、`ShapingAPI` 的 light shaping 或具体 light schema 页面继续排查。

A complete reading should make the role, properties, boundaries, and debugging path clear without relying on English prose as the main path.

在实际资产交付中，还要把 `ShadowAPI` 看成一种可分层覆盖的 USD opinion。某个上游 layer 可能只提供默认灯光，shot 或 lookdev layer 再覆盖 `inputs:shadow:color`、`inputs:shadow:distance` 或 `inputs:shadow:enable`；variant、reference、payload、sublayer 强度都会影响最终取值。因此排查“为什么阴影设置没生效”时，不要只看 HTML 属性表，还要在 USD composition 中确认当前 prim 上最强 opinion 来自哪里，属性是否被更强 layer 清空、覆盖或 block。

Treat `ShadowAPI` values as USD opinions that can be overridden by stronger layers, variants, references, payloads, or blocks.

另一个常见误区是把阴影显示异常全部归因给 `ShadowAPI`。如果物体本身通过 `visibility`、`purpose`、material opacity、render visibility、holdout、collection membership 或 renderer-specific setting 改变了可见性，阴影结果也可能变化。`ShadowAPI` 只描述这盏灯的阴影控制输入；它不替代几何可见性、材质透明度、实例代理、渲染设置或 delegate 专属选项。中文主阅读路径必须先把这些边界讲清楚，再让读者按属性表逐项核对。

Do not use `ShadowAPI` as a catch-all explanation for visibility, material, collection, or renderer-specific shadow behavior.

使用与调试顺序 / Usage and Debugging Order

  1. 先确认目标灯光或相关 prim 上的 `ShadowAPI` 属性存在，且属性名完整保留为 `inputs:shadow:*`。Confirm that the expected `inputs:shadow:*` properties are authored on the relevant light or prim.
  2. 用 `inputs:shadow:enable` 判断该灯是否整体参与阴影计算；如果只想排除某些对象，转到 `LightAPI` shadow-linking。Use `inputs:shadow:enable` for a light-level switch, and use linking for per-object participation.
  3. 用 `inputs:shadow:distance` 决定阴影是否有最大距离限制，并核对 scene units 与遮挡物到接收面的距离。Use `inputs:shadow:distance` to reason about maximum occluder-to-receiver distance.
  4. 用 `inputs:shadow:falloff` 设计最大距离附近的淡出区域；该值不应超过最大距离，且小于等于 `0.0` 时没有有效淡出区。Use `inputs:shadow:falloff` to avoid a hard cutoff near the maximum distance.
  5. 最后用 `inputs:shadow:falloffGamma` 调整曲线速度；如果前两个距离参数不合理，gamma 调整只会掩盖问题。Tune `inputs:shadow:falloffGamma` only after distance and falloff have sensible values.
  6. 如果视觉结果没有变化，检查 render delegate 是否支持这些 non-physical shadow controls，再检查 layer strength、variant、payload 和 stronger opinion 是否覆盖了属性值。If nothing changes, check renderer support and USD composition strength before changing unrelated settings.

  • ShapingAPI：用于理解 cone、focus、IES profile 等 light shaping；不要把 shaping 参数误认为阴影距离或阴影颜色。Use `ShapingAPI` for light-shape controls.

代码与配置片段 / Code and Configuration Excerpts

def RectLight "KeyLight"
{
    color3f inputs:shadow:color = (0.02, 0.03, 0.05)
    bool inputs:shadow:enable = true
    float inputs:shadow:distance = 8.0
    float inputs:shadow:falloff = 2.0
    float inputs:shadow:falloffGamma = 1.4
}

这个片段只演示属性命名和数值关系：`inputs:shadow:distance` 给最大距离，`inputs:shadow:falloff` 给淡出区宽度，`inputs:shadow:falloffGamma` 调整曲线速率，`inputs:shadow:color` 调整阴影颜色，`inputs:shadow:enable` 控制该灯阴影是否启用。实际 visual result 取决于 renderer 支持、场景单位、灯光类型、遮挡物与接收面距离，以及 USD composition 中哪个 layer 提供最强 opinion。

The snippet illustrates naming and value relationships; renderer support and composition strength determine the final result.
