逐段双语理解 / Paragraph-Level Bilingual Coverage

这页要先区分三个概念：文件名扩展、底层编码格式和 USD layer 内容。`.usda` 通常表示文本格式，`.usdc` 表示二进制 crate 格式，`.usd` 是通用扩展名，可以包装文本或二进制底层格式。USD 打开 `.usd` 文件时会检测真实底层格式，而不是只看扩展名。因此，某些场景只需要重命名，某些场景必须用工具重新写出目标格式。

The important distinction is between filename extension, underlying file format, and layer content. A `.usd` file may contain either text or binary USD data, and USD detects the actual underlying format when opening it.

官方说明 `usdcat` 既可以检查 layer 文件，也可以通过 `-o` 写出新文件。输出文件的扩展名决定了常规转换目标：把 `.usda` 写成 `.usdc` 会得到二进制文件，把 `.usdc` 写成 `.usda` 会得到文本文件。这里的转换不是改变 scene 描述语义，而是把同一套 layer 内容重新编码成另一个存储格式。

`usdcat` can inspect layer files and can also write converted files with `-o`. The output filename extension usually selects the target storage format.

第一类情况是 `.usda/.usdc` 与 `.usd` 之间的重命名。给定 `Sphere.usda`，可以直接重命名为 `Sphere.usd`；USD 打开 `Sphere.usd` 时会识别它仍是文本格式。同理，底层是 `.usdc` 的文件也可以重命名为 `.usd`。反过来，如果某个 `.usd` 文件的底层格式是文本，可以重命名为 `.usda`；如果底层格式是二进制，可以重命名为 `.usdc`。这一步不重写内容，只改变文件名。

The first case is renaming between `.usda` or `.usdc` and `.usd` without changing the underlying storage. USD detects the actual file format when reading `.usd` files.

第二类情况是 `.usda` 与 `.usdc` 之间的真实转换。官方示例从 `USD/extras/usd/tutorials/convertingLayerFormats` 中的 `Sphere.usda` 开始，它是文本文件，可以用任意文本编辑器或 `usdcat Sphere.usda` 查看。要生成二进制文件，使用 `usdcat -o NewSphere.usdc Sphere.usda`；要再转回文本，使用 `usdcat -o NewSphere.usda NewSphere.usdc`。这里发生了实际编码转换。

The second case is real conversion between `.usda` and `.usdc`. `usdcat -o NewSphere.usdc Sphere.usda` writes binary output, and `usdcat -o NewSphere.usda NewSphere.usdc` writes text output.

转换后必须验证语义一致性。官方用 `usddiff Sphere.usda NewSphere.usdc` 和 `usddiff NewSphere.usdc NewSphere.usda` 检查两个文件是否描述同一层内容。这里不要用普通文本 diff 去比较二进制和文本，也不要只看文件大小；`usddiff` 理解 USD layer 语义，适合判断转换是否保持 scene 描述一致。

After conversion, use `usddiff` to verify that two files describe equivalent USD layer content instead of relying on byte size or a plain text diff.

第三类情况是 `.usd` 文件之间的底层格式转换。因为 `.usd` 扩展名本身不说明是文本还是二进制，所以如果想让输出仍叫 `.usd`，但底层格式换成指定类型，就要用 `usdcat --usdFormat`。例如从文本底层的 `Sphere.usd` 生成二进制底层的 `NewSphere_binary.usd`，命令是 `usdcat -o NewSphere_binary.usd --usdFormat usdc Sphere.usd`。

The third case is converting between `.usd` files with different underlying formats. Use `--usdFormat` when the output extension stays `.usd` but the storage format should change.

反向操作同理：如果要把二进制底层的 `.usd` 转为文本底层的 `.usd`，使用 `usdcat -o NewSphere_text.usd --usdFormat usda NewSphere_binary.usd`，再用 `usddiff NewSphere_binary.usd NewSphere_text.usd` 验证。这个流程特别适合需要保持 `.usd` 统一扩展名、但在调试或发布阶段切换文本/二进制存储的团队。

The reverse uses `--usdFormat usda` to write a text-backed `.usd` file, followed by `usddiff` to confirm semantic equivalence.

实践中选择格式要看目的。文本 `.usda` 可读性高，适合教学、代码审查、调试和小型示例；二进制 `.usdc` 通常更适合发布、缓存和较大数据；`.usd` 则适合隐藏底层选择并让工具自动检测。转换工具不会替你解决资产路径、相对路径、外部引用或 package 组织问题，所以格式转换前后仍要检查 layer 内容和依赖路径。

Format choice depends on purpose: `.usda` is readable, `.usdc` is usually better for binary storage, and `.usd` lets tools detect the underlying format automatically.

最常见误读是把扩展名当成格式本身。`.usd` 只是一个通用入口，不等于“某种第三格式”。如果你只想把 `Sphere.usda` 交给要求 `.usd` 扩展名的工具，可以重命名；如果你想把文本编码变成二进制编码，需要 `usdcat -o`；如果你想输出仍为 `.usd` 但底层指定为 usda 或 usdc，需要 `--usdFormat`。这三种操作的意图不同，不能混用。

A common mistake is treating `.usd` as a distinct storage format. It is a generic extension; renaming, converting, and forcing an underlying format are different operations.

本页的验证标准也应具体：命令成功退出只说明文件写出成功，不说明转换语义正确。稳妥流程是先用 `usdcat` 查看输入，执行转换，再用 `usddiff` 比较输入与输出，必要时再用 usdview 打开结果做人工检查。对于进入版本库或发布管线的文件，还应记录转换命令和源文件，避免以后无法解释某个 `.usd` 文件底层到底是文本还是二进制。

A robust workflow inspects the input, performs the conversion, compares semantic equivalence with `usddiff`, and optionally opens the result in usdview for manual inspection.

在团队流程中，格式选择通常是约定而不是个人偏好。源资产、教学示例和代码评审样本常用 `.usda`，因为文本内容能直接看出 prim、attribute、relationship 和 metadata；生产缓存、几何较大或频繁读取的 layer 常用 `.usdc`，因为二进制更适合工具处理；`.usd` 常用于对外暴露统一扩展名，让读者不必关心底层格式。转换时要遵守团队约定，不要让同一目录里混入含义不清的同名变体。

In production, format choice is usually a pipeline convention. Text is useful for review, binary for storage and performance, and `.usd` for a neutral extension that hides the underlying choice.

CI 或发布脚本中应把 `usddiff` 当作转换验收门槛，而不是只检查目标文件是否存在。比如从 `.usda` 生成 `.usdc` 后，脚本可以运行 `usddiff source.usda target.usdc`；从 `.usd` 文本底层转成 `.usd` 二进制底层后，也应比较两个 `.usd` 的语义。这样可以防止路径写错、源文件拿错、空文件覆盖或转换脚本误用旧产物。

CI or publishing scripts should use `usddiff` as a semantic acceptance check, not just check that an output file was created.

格式转换通常不会改变 layer 中的 asset path、reference、payload、subLayer 或 metadata 值。也就是说，如果源文件里有相对路径或外部依赖，转换成 `.usdc` 或 `.usd` 并不会自动修复这些路径。发布前仍要确认输出文件放置位置是否改变了相对路径解释，必要时结合 `usdresolve`、`usdchecker` 或人工检查来验证依赖能被解析。

Format conversion does not automatically rewrite asset paths, references, payloads, sublayers, or metadata values. Dependency resolution still needs separate validation.

还有一种常见需求是“给人读”和“给机器读”之间切换。调试时可以把二进制 `.usdc` 或二进制底层 `.usd` 转成文本 `.usda` 或文本底层 `.usd`，便于检查属性和 composition arc；发布时再转回二进制。只要每次转换后 `usddiff` 通过，这种往返转换就是可控的；如果 `usddiff` 不通过，就必须先排查差异，而不是继续把文件交给下游。

A common workflow converts binary files to text for review and back to binary for publishing, with `usddiff` guarding each round trip.

因此，本页的中文阅读顺序是：先认清 `.usd` 是通用扩展名；再判断当前操作是重命名还是重编码；然后用 `usdcat -o` 或 `usdcat --usdFormat` 写出目标；最后用 `usddiff` 验证 layer 语义。只要抓住这条线，就不会把“文件名看起来变了”误认为“底层格式已经变了”，也不会把“二进制不可读”误认为“内容无法检查”。

The reading order is: understand `.usd` as a neutral extension, decide whether renaming or re-encoding is needed, run the appropriate `usdcat` command, and verify with `usddiff`.

也要知道什么时候不要转换。如果只是为了让工具接受 `.usd` 扩展名，且底层格式无需改变，直接重命名更简单，也不会引入重新写文件的风险。如果文件受版本控制且需要保留可读历史，贸然改成 `.usdc` 会让普通 diff 失效；如果文件很大且只用于运行时读取，长期保留 `.usda` 又可能增加存储和加载成本。格式转换应服务于审查、发布或工具兼容目标，而不是成为无意义的批处理步骤。

It is also important to know when not to convert. Renaming may be sufficient for extension compatibility, while real re-encoding should serve review, publishing, or tool compatibility goals.

执行顺序 / Execution Path

  1. 先判断目标：只是改成 `.usd` 扩展名，还是要在文本和二进制底层格式之间真正转换。
  2. 如果只是 `.usda/.usdc` 与 `.usd` 之间换扩展名，且底层格式保持不变，可以直接重命名。
  3. 如果要把 `.usda` 转成 `.usdc`，使用 `usdcat -o NewSphere.usdc Sphere.usda`。
  4. 如果要把 `.usdc` 转成 `.usda`，使用 `usdcat -o NewSphere.usda NewSphere.usdc`。
  5. 如果输出文件仍是 `.usd`，但要指定底层格式，使用 `usdcat -o target.usd --usdFormat usda/usdc source.usd`。
  6. 每次转换后用 `usddiff` 验证输入和输出是否语义一致。
  7. 进入发布或版本库前，记录转换命令、源文件和目标格式，方便后续排查。

常见误读与调试 / Common Misreadings And Debugging

  • 如果只是把 `.usda` 改名为 `.usd`，底层仍是文本格式；不要把这一步误认为二进制转换。
  • 如果需要真正生成二进制，输出扩展名用 `.usdc`，或在 `.usd` 输出时加 `--usdFormat usdc`。
  • 如果需要人工审查或教学示例，转成 `.usda` 或 `--usdFormat usda` 更容易阅读。
  • 如果 `usddiff` 报告差异，先检查转换前后是否真的使用了同一个源文件、是否引入了外部依赖路径变化。
  • 如果工具链只接受 `.usd` 扩展名，仍要记录底层格式，避免后续使用者误判文件可读性或存储特性。
  • 如果普通文本 diff 看不懂 `.usdc`，这是预期；应使用 `usddiff` 或 `usdcat` 这类理解 USD layer 的工具。
