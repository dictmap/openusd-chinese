逐段双语理解 / Paragraph-Level Bilingual Coverage

本教程是前一页 Hello World 的变体。它不改变最终生成的 scene description，而是把创作方式从 `UsdGeom.Xform.Define` / `UsdGeom.Sphere.Define` 这类 schema-specific wrapper，换成 `stage.DefinePrim(path, typeName)` 这种 generic `UsdPrim` 创作入口。读者应把它理解为一次 API 层面的对照实验：同样的 `/hello` 和 `/hello/world` 层级，可以由强类型 schema API 创建，也可以通过类型名字符串 `Xform` 和 `Sphere` 创建。

The tutorial revisits Hello World and uses hard-coded type names, `Xform` and `Sphere`, that match the type names following `def` in the `usda` representation.

官方开头强调 hard-coded typenames，是为了让读者看清 USD 文件文本和 Python authoring API 之间的直接对应关系。`def Xform "hello"` 中的 `Xform` 与 `stage.DefinePrim('/hello', 'Xform')` 的第二个参数对应；`def Sphere "world"` 中的 `Sphere` 与 `stage.DefinePrim('/hello/world', 'Sphere')` 的第二个参数对应。这里的 `Xform`、`Sphere`、`/hello`、`/hello/world` 都是可复现的技术字面量，不应翻译或改写。

The hard-coded type names correspond to the type names that appear after `def` in the `usda` scene description.

上一教程使用 `UsdGeom` API，是因为 `UsdGeom` 是 USD 内建几何 schema 的一部分。`UsdGeom.Xform` 和 `UsdGeom.Sphere` 这类 schema object 提供领域化接口：它们能帮助代码创建、操作、检查并 author 该类型上的属性。换句话说，typed schema API 不只是换了一个函数名，它还把“这是几何变换 prim”或“这是球体 prim”这种领域知识封装进了类型对象。

`UsdGeom` is part of USD's built-in geometry schema, and its prim types have C++ and Python APIs with domain-specific interfaces.

本页的关键边界是 `UsdGeomXform::Define` 和 `UsdStage::DefinePrim` 的返回值不同。前者返回 schema object，读者可以继续调用完整的 schema-specific API；后者返回 generic `UsdPrim`，它代表 scenegraph 中持久存在的 prim 本体。所有 schema classes 都可以通过 `GetPrim()` 取回底层 `UsdPrim`，这说明 schema object 是类型化访问层，而 `UsdPrim` 是通用存在层。

`UsdGeomXform::Define` returns a schema object, while `UsdStage::DefinePrim` returns a generic `UsdPrim`.

如果用工具链视角理解，`UsdPrim` 是更底层、更通用的句柄：它可以保存 path、typeName、metadata、attributes、relationships 和 scenegraph 位置。schema object 则像是给某个 schema 域加上的高层接口，能提供更明确的函数和类型检查。转换器、检查器、批处理脚本或插件兼容层经常需要 generic `UsdPrim`，因为它们不一定预先知道所有 schema wrapper；面向某类数据的业务代码通常更喜欢 schema object，因为可读性和安全性更好。

Think of `UsdPrim` as the generic persistent presence in the scenegraph, and the schema object as the first-class way to access domain-specific data and functionality.

代码第一行只导入 `Usd`，而不是同时导入 `UsdGeom`。这不是遗漏，而是本教程的核心：它故意只使用 stage 和 generic prim 的 API。`Usd.Stage.CreateNew('HelloWorldRedux.usda')` 创建新的 stage，并把 root layer 的保存目标设为 `HelloWorldRedux.usda`。这个文件名用于和前一教程的 `HelloWorld.usda` 对照，说明生成的场景结构相同，但脚本使用的 authoring API 不同。

The script imports `Usd`, creates `HelloWorldRedux.usda`, defines an `Xform` prim and a `Sphere` prim, then saves the root layer.

`xform = stage.DefinePrim('/hello', 'Xform')` 在 `/hello` 定义一个类型名为 `Xform` 的 prim，并返回 generic `UsdPrim`。这里返回变量名写作 `xform`，但它不是 `UsdGeom.Xform` schema object；如果后续代码想使用 `UsdGeom.Xform` 的强类型接口，需要再通过 schema wrapper 包装或调用对应 schema API。这个差异是本教程最容易被忽略的点。

`stage.DefinePrim('/hello', 'Xform')` creates the parent prim and returns a generic prim handle.

`sphere = stage.DefinePrim('/hello/world', 'Sphere')` 在 `/hello` 下定义子 prim `world`，类型名为 `Sphere`。它展示了 `DefinePrim` 会按给定 path 和 typeName 写入 scene namespace，而不是要求代码直接调用 `UsdGeom.Sphere.Define`。如果 path 写错，层级就会不同；如果 typeName 写错，生成的 `def` 类型也会不同；如果忘记 `Save()`，内存里的修改不会落到 `HelloWorldRedux.usda`。

`stage.DefinePrim('/hello/world', 'Sphere')` creates the child sphere prim under `/hello`.

官方还说明，这段代码生成的 scene description 与前一教程相同，可以通过 `usdcat` 查看。预期输出是一个很小的 ASCII USD layer：顶层 `def Xform "hello"`，其内部有 `def Sphere "world"`。因此，本教程的验收不是看画面是否复杂，而是看 API 调用是否能精确映射到相同的 `.usda` 文本结构。

The code produces identical scene description to the previous tutorial, and the output can be inspected with `usdcat`.

调试时应按层次定位：如果 `from pxr import Usd` 失败，先检查 OpenUSD Python 环境；如果 `HelloWorldRedux.usda` 没生成，检查 `stage.GetRootLayer().Save()` 和当前工作目录；如果文件存在但类型不是 `Xform` 或 `Sphere`，逐字核对 `DefinePrim` 的第二个参数；如果层级不是 `/hello/world`，核对 prim path 字面量；如果后续想设置 radius、extent 或其他几何属性，应转到下一教程或使用 schema-specific API，而不是误以为本页已经讲完了属性 authoring。

Debugging should separate Python environment issues, save location issues, type-name typos, prim-path typos, and schema-specific property authoring.

与下一教程的关系也很明确：本页只证明“可以用 generic prim 创建相同结构”，下一页才会继续看 `UsdGeom` API 如何作用在 prim properties 上。因此不要把 `DefinePrim` 理解成替代所有 schema API 的推荐写法；它是通用工具和兼容场景的重要能力，而 typed schema API 仍然是处理具体几何、材质、光照和其他领域数据时更直接的高层入口。

The next tutorial looks at the `UsdGeom` API in action on prim properties.

从学习顺序看，本页夹在 Hello World 和属性教程之间，作用是补上“类型名”和“对象包装”这层概念。前一页让你知道如何用 `UsdGeom` 快速生成一个最小 stage；本页让你看到同一个 stage 也可以只靠 `Usd` 层的通用接口生成；下一页再解释当你已经有 prim 后，如何检查和编写属性。这个顺序能避免把 `UsdPrim`、schema object、attribute authoring 混成一个概念。

This page separates type-name authoring and object wrapping before the following property tutorial.

这里还要区分“文件中有什么”和“Python 变量是什么”。`HelloWorldRedux.usda` 中最终有的只是 scene description：`def Xform "hello"` 和子级 `def Sphere "world"`。Python 变量 `xform`、`sphere` 是脚本运行时拿到的 generic prim handles，不会作为变量名写入 USD 文件。读者检查输出时应看 prim type 和 prim name，而不是寻找变量名 `xform` 或 `sphere`。

The saved file contains USD scene description, not the Python variable names.

generic `DefinePrim` 对插件化和跨版本工具尤其重要。一个资产检查器可能只需要确认某个 prim 的 path、typeName 和属性是否存在，并不需要链接每个 schema 的 Python wrapper；一个转换器可能需要保留未知扩展 schema 的 typeName，避免因为本地没有对应 wrapper 就丢失结构。此时 `UsdPrim` 的通用性比 schema object 的便利性更关键。

Generic prim access is useful for tools that must preserve or inspect types without relying on every schema wrapper.

但 generic API 也有代价。用 `DefinePrim('/hello/world', 'Sphere')` 只是写入类型名，并不会在代码层给你 `UsdGeom.Sphere` 的专用方法提示；要设置球体半径、读取 schema fallback、调用几何相关 convenience API，仍应使用 `UsdGeom.Sphere` 或把 `UsdPrim` 包装回对应 schema object。因此，本页的结论不是“typed schema API 不需要”，而是“USD 的同一个 prim 同时支持通用视图和类型化视图”。

Generic authoring does not remove the value of typed schema APIs; both views exist for the same prim.

如果在真实项目中选择 API 风格，可以用一个简单规则：当你明确知道要创建的是 `UsdGeom.Sphere`、`UsdLux.DiskLight` 或某个具体 schema，并且马上要读写该 schema 的属性时，优先用 typed schema API；当你写的是遍历器、验证器、迁移脚本、格式转换器或需要处理未知 typeName 的工具时，优先从 `UsdPrim` 和 `DefinePrim` 这类通用接口开始。

Use typed schema APIs for known domain objects, and generic APIs for broad traversal, validation, migration, and conversion tools.

本页的最小验收检查点是：运行脚本后能找到 `HelloWorldRedux.usda`；文件中出现 `def Xform "hello"` 与嵌套的 `def Sphere "world"`；读者能解释这两个 `def` 的 typeName 来自 `DefinePrim` 的第二个参数；读者也能说明为什么返回值是 generic `UsdPrim`，而不是 `UsdGeom.Xform` 或 `UsdGeom.Sphere` 的 schema object。四项都成立时，中文主阅读路径才真正闭环。

A complete reading of this page can explain the generated layer, the `def` type names, and the generic return type.

还要注意，generic prim 不是“无类型 prim”。它依然可以拥有 `Xform` 或 `Sphere` 这样的 typeName，只是当前 Python 变量没有采用对应 schema wrapper。这个区别能帮助读者避免把“通用访问方式”误解成“场景数据缺少 schema 类型”。

A generic prim can still have a schema type name; only the access path is generic.

完成本页后，读者应能用中文独立回答四个问题：为什么只导入 `Usd`；`DefinePrim` 的第二个参数如何变成 `usda` 中的 `def` 类型；generic `UsdPrim` 与 schema object 的差别是什么；为什么相同输出可以由不同 API 风格创作。能回答这四点，才算真正读懂 Hello World Redux，而不是只复制了脚本。

After this page, the reader should understand the `Usd`-only script, type-name authoring, generic prim handles, and why two authoring styles can produce the same layer.

教程步骤主阅读路径 / Tutorial Step Reading Path

  1. 确认上下文：先完成 `Hello World - Creating Your First USD Stage`，理解 `UsdGeom.Xform.Define` 和 `UsdGeom.Sphere.Define` 生成的基本层级。Start from the previous Hello World tutorial.
  2. 创建 stage：调用 `Usd.Stage.CreateNew('HelloWorldRedux.usda')`，新建 root layer 输出目标。Create a stage with `Usd.Stage.CreateNew('HelloWorldRedux.usda')`.
  3. 定义父 prim：调用 `stage.DefinePrim('/hello', 'Xform')`，通过 typeName 字符串写入 `Xform`。Define the parent `Xform` prim with `stage.DefinePrim('/hello', 'Xform')`.
  4. 定义子 prim：调用 `stage.DefinePrim('/hello/world', 'Sphere')`，通过 typeName 字符串写入 `Sphere`。Define the child `Sphere` prim with `stage.DefinePrim('/hello/world', 'Sphere')`.
  5. 保存 layer：调用 `stage.GetRootLayer().Save()`，让内存中的 scene description 写入磁盘。Save the root layer.
  6. 检查输出：使用 `usdcat` 或文本编辑器确认 `def Xform "hello"` 内包含 `def Sphere "world"`。Inspect the output with `usdcat` or a text editor.

关键概念边界 / Concept Boundaries

  • `UsdPrim` 是 scenegraph 中 prim 的通用持久表示；schema object 是在其上提供领域化访问的类型化包装。`UsdPrim` is the generic persistent prim representation; a schema object is a typed domain-specific wrapper.
  • `DefinePrim` 的 typeName 参数必须保留为 `Xform`、`Sphere` 这类 USD 类型名，不要翻译成中文。The `DefinePrim` type-name argument must remain the USD type name.
  • `/hello/world` 是 USD prim path，不是文件系统目录；它表达的是 scene namespace 层级。`/hello/world` is a USD prim path, not a filesystem path.
  • generic API 更适合通用工具、未知 schema、批处理和兼容层；typed schema API 更适合明确处理某个领域对象。Generic APIs suit broad tooling; typed schema APIs suit domain-specific authoring.
  • 本页不讲属性值 authoring；`radius`、`extent` 等 Sphere 属性会在后续属性教程中更自然地出现。This page does not focus on property authoring; later tutorials cover prim properties.
