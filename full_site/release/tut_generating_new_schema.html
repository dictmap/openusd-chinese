逐段双语理解 / Paragraph-Level Bilingual Coverage

阅读这页时，先把 schema class 理解为“围绕 UsdPrim 的强类型访问层”。USD scenegraph 中真正存在的是 prim、attribute、relationship、metadata 和 composition；schema class 则把这些底层数据包装成稳定的 C++/Python API，让调用者通过 Get、Create、Apply、HasAPI 等接口访问约定好的属性和行为。官方提到的 UsdModelAPI、UsdGeomImageable、UsdGeomMesh 等，都是通过生成器或同类机制得到的 schema class。

A schema class is an API layer around a UsdPrim. It gives strongly named C++ and Python access to authored properties, relationships, metadata, and schema-defined fallbacks.

生成前提有两个层面。第一个是工具前提：usdGenSchema 依赖 Python 环境中的 jinja2 和 argparse，并且 USD Python 模块必须能被找到。官方用 USD_INSTALL_ROOT 表示 USD 安装前缀，PYTHONPATH 需要包含 USD_INSTALL_ROOT/lib/python。第二个是输入前提：教程示例文件位于 USD/extras/usd/examples/usdSchemaExamples/，本页的目标是让读者理解不同 schema 类型，并能生成、编译和使用这些示例 schema。

The prerequisites are both runtime and input oriented: usdGenSchema needs Python dependencies and USD Python modules, and the tutorial examples live under USD/extras/usd/examples/usdSchemaExamples/.

官方把 schema class 分为 IsA schema 和 API schema。IsA schema 可以给 prim 一个 typeName，并且必须从 UsdTyped 这条 typed schema 基础链路派生。它又分 concrete 和 non-concrete：concrete IsA schema 可以在 stage 上实例化，non-concrete IsA schema 更像抽象基类，用来给相关 concrete schema 共享属性和接口。理解这点很重要，因为 typeName 决定 prim 的内建类型身份，而不是普通属性。

IsA schemas can impart a typeName to a prim and derive from UsdTyped. Concrete IsA schemas are instantiable, while non-concrete IsA schemas are base classes for related typed schemas.

API schema 不给 prim 指定 typeName，而是给已有 prim 增加一组可复用接口或内建属性。非应用 API schema 通常不需要在 prim 上记录“已应用”状态，适合表达核心 metadata 或通用行为；applied API schema 则需要记录和查询是否应用到某个 prim，生成器会提供 Apply 方法，应用后 HasAPI 会返回 true。applied API schema 又有 singleApply 和 multipleApply：前者每个 prim 只能应用一次，后者可以带不同实例名多次应用，例如 collection 类场景。

API schemas do not define a typeName. Non-applied APIs provide interfaces without recording application, while applied APIs are recorded on prims and may be single-apply or multiple-apply.

schema.usda 是 usdGenSchema 的核心输入层。它必须声明 libraryName，layer stack 中必须能看到 usd/schema.usda，schema typename 必须在所有库中唯一，attribute 名和 token 通常要是 camelCase 的合法标识。教程示例还通过 GLOBAL customData 声明 libraryName、libraryPath 和 libraryPrefix。中文读者应把这些 metadata 当作“代码生成和运行时注册契约”，而不是普通文档注释。

schema.usda is the main input layer for usdGenSchema. It carries library metadata, imports base schema definitions, and declares schema classes, properties, tokens, and generation hints.

示例中的 SimplePrim 是 typed 但 non-concrete 的 IsA schema：它继承 /Typed，有 intAttr 和 target relationship，但没有 typeName，所以不能直接在 UsdStage 上作为可实例化 prim type 使用。示例中的 ComplexPrim 则从 SimplePrim 派生，并声明 concrete typeName ComplexPrim，还添加 complexString。这个对比说明：抽象基类负责共享接口，concrete 类型负责成为场景中真正可声明的类型。

SimplePrim is a non-concrete typed IsA schema, while ComplexPrim is a concrete IsA schema with a typeName. The contrast shows the difference between a base schema and an instantiable prim type.

ParamsAPI 示例展示 applied API schema。它从 APISchemaBase 派生，customData 中设置 apiSchemaType 为 singleApply，并声明 params:mass、params:velocity、params:volume 三个 double attribute。属性名带 params namespace，但通过 apiName 可以让生成的接口变成 GetMassAttr、GetVelocityAttr、GetVolumeAttr，而不是暴露完整 namespaced 名称。这说明 schema.usda 不只是定义属性，也控制生成 API 的可读性。

ParamsAPI demonstrates a single-apply API schema. It derives from APISchemaBase, declares namespaced attributes, and uses apiName metadata to shape generated accessor names.

运行 usdGenSchema schema.usda . 后，生成器会处理 SimplePrim、ComplexPrim、ParamsAPI，写入 tokens.h、tokens.cpp、wrapTokens.cpp，生成 simple、complex、paramsAPI 的 C++ 文件和 Python wrapper，并写出 plugInfo.json 与 generatedSchema.usda。读者不要只看“生成了哪些文件”，还要理解文件分工：C++/wrapper 提供编程接口，tokens 固化属性和标识，plugInfo.json 与 generatedSchema.usda 让运行时能发现 schema 定义和 fallback。

usdGenSchema produces C++ classes, Python wrappers, schema tokens, plugInfo.json, and generatedSchema.usda. These outputs serve both developer-facing APIs and runtime schema registration.

生成之后还要编译和注册。官方示例从 USD build 目录运行 cmake --build . --target install --config Release 来重新构建插件。使用时，USD 必须知道插件资源目录在哪里：可以设置 PXR_PLUGINPATH_NAME 指向 resources 目录，也可以把库文件和资源目录复制到安装前缀下的 plugin/usd。Windows 上还可能遇到 Python 3.8+ 的 DLL trusted directory 问题，需要把资源目录加入 os.add_dll_directory。

Generated schemas must be built and made discoverable. USD must see the plugin resources through PXR_PLUGINPATH_NAME or installation under the USD plugin location; Windows may also require adding DLL directories for Python.

验证使用分成三层。第一层是写一个 Test.usda，声明 def ComplexPrim "Complex" 并在另一个 Xform 上 prepend apiSchemas = ["ParamsAPI"]。第二层是用 usdview 打开文件，确认没有 warning 或 error。第三层是在 C++ 或 Python 中打开 stage，构造 UsdSchemaExamplesSimple、UsdSchemaExamplesComplex、UsdSchemaExamplesParamsAPI，读取属性，并用 HasAPI 或 Apply 确认 applied API schema 的行为。只有三层都通过，才说明生成、编译、注册和使用链路闭合。

Validation includes authoring Test.usda, opening it in usdview without warnings, and using generated C++ or Python APIs to access the schema classes and applied API schema.

最后的 codeless schema 是一个重要边界。设置 skipCodeGeneration 为 True 时，可以不生成 C++/Python 代码，只生成 generatedSchema.usda 和 plugInfo.json，用于运行时 schema registration。这样做的价值是动态更新和部署更轻，但代价是没有类型安全的代码接口。它适合只需要内建属性、fallback 和发现能力的场景，不适合需要复杂计算、协调写入或强类型方法封装的 schema。

Codeless schemas skip C++ and Python code generation and rely on generatedSchema.usda plus plugInfo.json for runtime registration. They are dynamic, but do not provide generated typed APIs.

这页最容易被误读成“运行一个代码生成命令即可完成 schema”。实际闭环更长：schema.usda 是设计来源，usdGenSchema 是把设计转换成代码和注册资源的工具，CMake 构建负责把插件产物放进可加载位置，PXR_PLUGINPATH_NAME 或安装目录负责让运行时发现插件，最后 usdview、C++ 和 Python 用例负责验证类型信息、fallback、Apply 和 accessor 是否真的可用。任何一环断开，页面中的示例都可能看似生成成功但在运行时不可见。

The workflow is longer than running a generator. Design, generation, build, plugin discovery, and runtime validation all have to succeed.

IsA schema 与 API schema 的选择应从数据模型出发。若你的 prim 需要一个明确的类型身份，并且场景作者会写 def ComplexPrim 这样的 typed prim，就应考虑 concrete IsA schema。若你的目标是给许多不同类型的 prim 增加一组可选能力，例如物理参数、集合、模型信息或渲染器扩展，则 API schema 更合适。不要因为想要一个类名就滥用 IsA schema；typeName 是场景语义的一部分，会影响 schema registry、prim definition 和工具行为。

Choose IsA versus API schema from the data model. A prim type belongs in an IsA schema; optional capabilities shared across prim types usually belong in an API schema.

singleApply 与 multipleApply 的差别也不是命名细节。singleApply 表示这组内建属性在同一个 prim 上只有一套语义；multipleApply 表示同一个 prim 可以拥有多个命名实例，每个实例有自己的 namespace prefix。设计 multipleApply 时要格外注意属性命名、实例名和 property namespace，否则生成出来的接口虽然可用，却会给资产作者和工具解析带来混乱。

Single-apply and multiple-apply are semantic choices. Multiple-apply schemas create per-instance namespaces and need especially careful property design.

schema 生成后的文件还承担版本化职责。tokens.h、tokens.cpp 和 wrapper 文件是源码层面的 API；generatedSchema.usda 和 plugInfo.json 是运行时发现层面的 API；Test.usda 是使用方对 schema 的最小合约测试。维护 schema 时，应把这三类产物一起纳入评审：源码是否表达了预期接口，注册资源是否暴露了预期定义，测试场景是否证明工具和客户端能按预期使用它。

Generated sources, runtime registration resources, and test scenes should be reviewed together because they validate different parts of the schema contract.

如果后续需要自定义方法，官方提示这些方法可以在重新运行 usdGenSchema 时被保留。中文读者要把自定义方法看作 schema API 的补充层：适合封装需要同时写多个属性的 authoring 操作、计算派生值或提供更符合业务语义的 helper；但底层属性、fallback 和注册信息仍应由 schema.usda 驱动，避免让手写代码和 schema 声明分叉。

Custom methods can supplement generated schemas, but the schema declaration should remain the source of truth for properties, fallbacks, and registration.

调试时也要区分“生成失败”“编译失败”和“运行时不可见”。生成失败通常指向 schema.usda 语法、metadata 或命名规则；编译失败通常指向生成出的 C++ 名称、依赖或构建配置；运行时不可见则更多指向 plugInfo.json、resources 目录、PXR_PLUGINPATH_NAME、DLL 搜索路径或安装位置。把这三类错误拆开，能显著减少盲目修改 schema 声明的风险，并保持定位路径清晰，也便于复查。

Debug generation errors, build errors, and runtime discovery errors separately; they usually point to different parts of the schema workflow.

执行顺序 / Execution Path

  1. 确认 Python 环境能找到 jinja2、argparse 和 USD Python 模块，并设置好 USD_INSTALL_ROOT 与 PYTHONPATH。
  2. 在 schema.usda 中声明 libraryName、libraryPath、libraryPrefix，并确保 layer stack 中包含 usd/schema.usda。
  3. 决定每个 schema 是 concrete IsA、non-concrete IsA、non-applied API、singleApply API 还是 multipleApply API。
  4. 用 class 声明属性、relationship、doc、customData、apiSchemaType、apiName、className 等 metadata。
  5. 运行 usdGenSchema schema.usda .，检查 tokens、C++、wrapper、plugInfo.json、generatedSchema.usda 是否生成或更新。
  6. 重新构建插件，并通过 PXR_PLUGINPATH_NAME 或安装目录让 USD 运行时能发现它。
  7. 用 Test.usda、usdview、C++ 和 Python 访问链路验证 schema 是否真的可用。

关键命令与片段 / Commands And Fragments

命令、文件名、环境变量和 schema 标识保持英文原样，便于和官方文档、源码目录及构建日志对照。

usdGenSchema schema.usda .
cmake --build . --target install --config Release
PXR_PLUGINPATH_NAME=/path/to/plugin/resources

schema.usda 的最小读法是：通过 subLayers 接入基础 schema，通过 GLOBAL customData 声明库元数据，通过 class 声明 IsA 或 API schema，并用 customData 控制生成器行为。

# Conceptual excerpt for reading only.
subLayers = [
    @usd/schema.usda@
]

over "GLOBAL" (
    customData = {
        string libraryName = "usdSchemaExamples"
        string libraryPrefix = "UsdSchemaExamples"
    }
) {}

class ComplexPrim "ComplexPrim" (
    inherits = </SimplePrim>
) {
    string complexString = "somethingComplex"
}

常见误读与调试 / Common Misreadings And Debugging

  • 如果 usdGenSchema 不存在，先检查 jinja2、argparse、USD Python 模块和安装时是否启用了相关工具，而不是先改 schema.usda。
  • 如果 generated code 编译失败，先检查 schema typename 唯一性、identifier 命名、libraryPrefix 和 className 是否生成出冲突的 C++ 名称。
  • 如果 usdview 不认识新 schema，先检查 plugInfo.json 和 generatedSchema.usda 是否在插件资源目录中，再检查 PXR_PLUGINPATH_NAME。
  • 如果 HasAPI 返回 false，确认 Test.usda 是否真的写入 prepend apiSchemas = ["ParamsAPI"]，以及 schema 是否是 applied API。
  • 如果只需要运行时发现和 fallback，不需要强类型接口，可以评估 codeless schema；如果需要自定义方法或协调写入，仍应生成代码。

术语与保留英文 / Terms Retained

usdGenSchema

从 schema.usda 生成 C++、Python wrapper、tokens、plugInfo.json 和 generatedSchema.usda 的工具。

schema.usda

schema 生成的输入层，包含库元数据、schema class、属性、relationship 和生成器 metadata。

IsA schema

可给 prim 指定 typeName 的 typed schema，分 concrete 和 non-concrete。

API schema

不给 prim 指定 typeName，而是在已有 prim 上提供接口或内建属性的 schema。

plugInfo.json

插件发现和 schema 注册的关键资源文件，运行时找不到它时 schema 通常不可见。

generatedSchema.usda

描述生成后 schema 定义、fallback 和注册信息的 USD layer。
