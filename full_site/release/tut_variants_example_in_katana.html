范围说明 / Scope Note

本页是 OpenUSD release 教程中 “Variants Example in Katana” 的完整双语参考页。它不是当前持续测试的现代 Katana 教程，而是一个历史 proof-of-concept，用来说明 DCC 工具如何把 USD asset 内的 variant set 暴露为 UI 中可切换的选项。中文主线会保留 `Katana`、`PxrUsdInVariantSelect`、`modelingVariant`、`/ExampleModelingVariants`、`ExampleModelingVariants.usd` 等原始标识符，解释它们在 variant switching 工作流中的位置。

This local bilingual page explains the historical OpenUSD Katana variants example in Chinese first while retaining the official identifiers, file names, node names, prim paths, and links for comparison.

历史状态提醒 / Historical Status

官方页明确提醒：Katana USD plugin 已在 USD 20.05 中从 USD distribution 移除，改由 Foundry 支持的 GitHub 插件维护；本页保留是历史参考。阅读时应把它当成“怎样把 USD variant selection 映射到 DCC 节点参数”的设计示例，而不是当成当前 OpenUSD 自带插件的安装或运行说明。

The official page marks this as historical documentation: the Katana USD plugin was removed from the USD distribution in version 20.05 and the page remains for reference.

逐段双语理解 / Paragraph-Level Bilingual Coverage

本页接在 `Authoring Variants` 之后阅读最合适。前一页讲的是如何在 USD layer 内 author `variantSet`、`variantSelection` 和 variant edit context；本页换到 DCC 集成视角，展示一个 host application 如何读取资产里已经存在的 variant set，并把它们暴露成用户可以点击或选择的参数。换句话说，本页不是再教你用 Python 写 variant，而是说明 Katana 这类应用如何把 USD composition 维度接到 UI 和节点图里。

This page follows the variant-authoring tutorial and shifts the focus from authoring variants to exposing variant selection inside a DCC host.

示例资产位于 `USD/extras/usd/tutorials/exampleModelingVariantsKatana`。目录中包含 Katana 场景文件 `exampleModelingVariants.katana`，以及带有 modeling variants 的 USD asset：`ExampleModelingVariants.usd` 和相关 `.usd` 文件。这里的核心 prim 是 `/ExampleModelingVariants`，它携带可切换的 shape variants。文件名、目录名和 prim path 都是复现锚点，不能翻译；中文说明只解释这些锚点各自承担的角色。

The example scene lives under `USD/extras/usd/tutorials/exampleModelingVariantsKatana` and includes a Katana file plus USD assets with modeling variants.

官方流程先用 `usdview` 打开 USD asset。选择 `/ExampleModelingVariants` prim 后，右下角的 `Meta Data` pane 会显示该 prim 上 author 的 variant set。用户可以在这个区域切换 variant selection，例如把当前 shape 选为 `Torus`。这一步的意义是确认 USD asset 本身已经正确声明了 variant set，并且 OpenUSD 的标准工具能够通过 metadata 看到和切换它。若 `usdview` 中看不到选择器，问题通常不在 Katana，而在 asset 是否把 variant set author 到了预期 prim 上。

In `usdview`, selecting `/ExampleModelingVariants` exposes the authored variant set in the metadata pane so the user can choose variants such as `Torus`.

接下来打开 `exampleModelingVariants.katana`。这个 Katana 场景通过 `PxrUsdInVariantSelect` node 暴露同一个模型的 variant switching。节点需要 target model scenegraph location，也就是要指向哪个 USD 模型位置；当目标位置设置正确后，节点会自动填充该位置上已 author 的 variant set names。示例中选择 `variantSetName` 参数为 `modelingVariant`，随后 `variantSelection` 列表会显示可用的 shape variants。这个流程把 USD 的 prim metadata、variant set 名称、variant selection 值和 Katana 节点参数连接起来。

In Katana, the `PxrUsdInVariantSelect` node targets the model scenegraph location, auto-populates variant set names, and lets the user choose `modelingVariant` and its shape variants.

从 composition 角度看，`PxrUsdInVariantSelect` 并不是“修改几何”的特殊节点；它的关键职责是 author 或传递一个 variant selection opinion，使 Katana 看到的 USD scenegraph 按所选分支组合。实际 mesh、sphere、torus 或其他 shape 的差异仍然来自 USD asset 内部的 variant 内容。节点只是把选择接口放到 Katana node graph 中，让 layout、lighting 或 lookdev 用户不需要手写 USD layer 就能切换模型形态。

Conceptually, the Katana node provides a host-application interface for authoring or passing a variant selection opinion; the variant contents remain in the USD asset.

本页需要和现代工作流区分开。由于官方插件已从 USD distribution 移除，今天的 Katana 集成应参考 Foundry-supported `KatanaUsdPlugins` 或项目自身使用的插件版本。本地中文页不能把历史截图和旧节点当成当前安装指南。它的长期价值在于说明 DCC 集成设计模式：读取目标 scenegraph location，枚举该 prim 上的 variant sets，把 `variantSetName` 和 `variantSelection` 暴露为 UI 参数，再把选择结果作为 USD composition opinion 参与下游求值。

The page is historical, but it remains useful as an integration pattern for exposing USD variant sets through DCC UI parameters.

调试这类工作流时可以按四层排查。第一，在 `usdview` 中确认 `/ExampleModelingVariants` 是否存在，以及 metadata pane 是否显示 `modelingVariant`。第二，确认 Katana 节点 target 的 scenegraph location 是否正好指向携带 variant set 的模型 prim，而不是指向子 mesh 或错误父级。第三，确认 `variantSetName` 是否与 USD 中 author 的名称完全一致；`modelingVariant` 这样的 token 名称大小写和拼写都应保持原样。第四，如果 `variantSelection` 列表为空，先检查插件能否加载 USD asset 和 metadata，再检查 asset 是否被 flatten 成已经应用 selection 的结果而丢失 variant set 结构。

Debugging starts by checking the USD prim and its metadata in `usdview`, then verifying Katana node targeting, exact variant-set names, plugin loading, and whether the asset was flattened.

本页还展示了一个和 `Authoring Variants` 相反的读法：作者页强调如何创建 `shadingVariant` 并写入 `red`、`blue`、`green` 内容；Katana 示例强调应用如何发现一个已存在的 `modelingVariant` 并把选择暴露给用户。两个页面合起来说明 variant 的完整生命周期：asset author 负责定义分支，DCC 集成负责发现和选择分支，composition 系统负责根据当前 selection 生成最终 scenegraph。

Together with the authoring tutorial, this page shows both ends of the variant lifecycle: asset authors define variants, and DCC integrations expose variant selection.

如果要把这个模式迁移到其他 host，例如 Houdini、Maya、Omniverse Kit 或内部工具，核心不是复制 `PxrUsdInVariantSelect` 的界面，而是保留同样的数据流：定位 prim，读取 `GetVariantSets()` 可见的 set，列出每个 set 的 allowed selections，将用户选择写入合适的 edit target 或 session layer，并明确该 opinion 的强度。只要这条数据流清楚，UI 可以是节点、属性面板、右键菜单或脚本参数。

The transferable pattern is to locate the prim, inspect variant sets, list selections, and author the chosen selection into an appropriate edit target.

本页也提醒读者区分“示例文件可打开”和“生产插件可用”这两件事。`exampleModelingVariants.katana` 展示的是当时的节点图和参数命名，现代 Katana 插件、USD 版本和 studio pipeline 可能有不同安装路径、节点名称或 UI 布局；但是 `/ExampleModelingVariants` 上的 variant metadata、`variantSetName` 到 `variantSelection` 的两级选择关系，以及 selection opinion 参与 composition 的语义不会因此改变。翻译时保留这个边界，可以避免把历史教程误读成过期安装手册，也能保留它作为 DCC 集成设计案例的价值。

The historical files and UI may differ from modern deployments, but the USD variant-selection semantics are the durable part.

在实际 pipeline 中，variant selection 的 authoring 位置需要明确约定。交互式 DCC 会话通常把临时选择写入 session layer 或 host 自己的 scene state；资产发布层则可能把默认 selection 写回 asset layer；shot 或 layout layer 可能在引用资产之后提供更强的 selection override。本页虽然只展示 Katana 节点选择 shape，但读者应理解同一个 `modelingVariant` 可以在不同 layer 上被选择，最终哪一个生效由 USD 的 layer stack 强度决定。若 Katana 里看到的形态和 `usdview` 打开 asset 时不同，先比较两边的 edit target、session layer 和引用链，而不是直接认定 variant 内容损坏。

A production pipeline must define where variant selections are authored, because asset defaults, shot overrides, and session-layer choices can all participate in composition.

另一个容易忽略的边界是“枚举 variant sets”与“消费 variant 内容”不是同一件事。`PxrUsdInVariantSelect` 这类节点只需要读取目标 prim 的 variant set 名称和可选 selection 值，就能构建 UI；真正加载哪套几何、材质或子树，仍由 USD composition 在选定分支后完成。因此性能和缓存问题通常发生在 selection 改变后重新求值 scenegraph 的阶段，而不是单纯生成下拉列表的阶段。把这两层分开，有助于排查 UI 已显示选项但 viewport 结果没有更新、或结果更新但列表没有自动填充的不同问题。

Discovering variant names for UI and consuming the selected variant contents are separate phases with different debugging implications.

在跨 DCC 迁移时，还要把 variant selection 当作可审计的 scene description opinion，而不是只存在于某个应用 UI 里的临时状态。一个稳健的工具应能说明 selection 写到了哪个 layer、是否会随场景保存、是否会污染资产默认值、以及 downstream 引用该资产时能否覆盖它。Katana 示例的价值就在于把这些问题压缩成一个可观察的节点参数：目标位置决定枚举范围，`variantSetName` 决定维度，`variantSelection` 决定当前组合分支。

A robust DCC integration should make clear where the variant-selection opinion is authored and whether downstream layers can override it.

因此，本页的中文阅读目标不是复刻旧 Katana UI，而是让读者能用同一套概念检查任意 host：目标 prim 是否正确、variant set 是否可枚举、selection opinion 是否写在预期强度位置、以及最终 composed scenegraph 是否反映当前选择。这些检查项比旧截图更重要，也更适合自动化审计，并能解释跨工具结果差异，方便复核。

The reading goal is to apply the same variant-selection checks to any host application.

维护本页时必须保留这些复现锚点：`USD/extras/usd/tutorials/exampleModelingVariantsKatana`、`exampleModelingVariants.katana`、`ExampleModelingVariants.usd`、`/ExampleModelingVariants`、`Meta Data`、`Torus`、`PxrUsdInVariantSelect`、`variantSetName`、`modelingVariant`、`variantSelection`、`KatanaUsdPlugins`。这些名称用于定位官方示例、插件节点、scenegraph location 和 UI 参数，不应翻译。中文内容应解释每个标识符的职责，而不是把它们替换成中文名称。

The file names, prim path, Katana node name, parameter names, and variant-selection identifiers are reproducibility anchors and should remain unchanged.

工作流步骤 / Workflow Steps

  1. 打开 `ExampleModelingVariants.usd` 或相关 USD asset，先在 `usdview` 中确认 `/ExampleModelingVariants` prim 存在。Open the USD asset and confirm `/ExampleModelingVariants` in `usdview`.
  2. 在 `Meta Data` pane 中查看可用 variant set，并尝试选择 `Torus` 等 shape variant，验证 USD asset 自身可切换。Use the metadata pane to inspect and switch authored shape variants such as `Torus`.
  3. 打开 `exampleModelingVariants.katana`，找到用于 variant selection 的 `PxrUsdInVariantSelect` node。Open the Katana scene and locate the `PxrUsdInVariantSelect` node.
  4. 把节点 target 指向模型 scenegraph location，使节点能读取该位置上的 variant set names。Target the model scenegraph location so the node can discover variant set names.
  5. 在 `variantSetName` 中选择 `modelingVariant`，再从 `variantSelection` 列表中选择需要的 shape。Choose `modelingVariant` and then select the desired shape variant.
  6. 把结果当作 composition selection 检查，而不是把它误解成 Katana 直接改写了 USD mesh。Interpret the result as USD composition selection, not direct mesh rewriting by Katana.

关键边界 / Key Boundaries

  • 本页是历史 reference；不要把旧 Katana USD plugin 当作当前 OpenUSD distribution 的组成部分。This is historical reference documentation, not a current bundled OpenUSD plugin guide.
  • `usdview` metadata pane 用于验证 asset 内是否真的有 variant set；Katana 节点用于在 host workflow 中暴露同一选择。`usdview` validates the asset metadata; the Katana node exposes the same selection in a host workflow.
  • `PxrUsdInVariantSelect` 的核心输入是模型 scenegraph location；目标路径不对时，variant set names 不会正确自动填充。`PxrUsdInVariantSelect` depends on the correct model scenegraph location.
  • `variantSetName` 与 `variantSelection` 是两个层级：先选 set，再选该 set 内的具体分支。`variantSetName` chooses the set; `variantSelection` chooses a branch inside that set.
  • 如果资产已经被 flatten 成当前选择后的结果，host 可能无法再枚举原始 variant set 结构。A flattened asset may no longer expose the authored variant set structure to the host.

常见误读与调试 / Common Misreadings and Debugging

  • 如果 `usdview` 中看不到 `modelingVariant`，先检查 USD asset 是否保留了 variant set，而不是直接怀疑 Katana。If `modelingVariant` is missing in `usdview`, inspect the USD asset before debugging Katana.
  • 如果 Katana 节点列表为空，检查 target scenegraph location 是否指向 `/ExampleModelingVariants`。If the Katana list is empty, verify that the node targets `/ExampleModelingVariants`.
  • 如果选择后形态不变，确认 selection opinion 的 edit target 或强度是否被更强 layer 覆盖。If selection has no effect, check whether a stronger layer overrides the selection opinion.
  • 如果迁移到现代 Foundry 插件，不能假设旧 `PxrUsdInVariantSelect` UI 完全相同；应保留 USD variant semantics，而不是保留旧截图布局。When migrating to modern Foundry-supported plugins, preserve USD variant semantics rather than assuming the old UI is identical.
  • 如果导出的 USD 不再显示 variants，检查是否在 upstream 阶段执行了 flattening，导致 host 只能看到已应用的 composed result。If variants disappear, check whether an upstream flattening step removed the authored variant structure.
