逐段双语理解 / Paragraph-Level Bilingual Coverage

本页解释 USD schema 版本化为什么困难，以及如何用 per-schema versioning、versioned schema names、Schema Registry 和 UsdPrim schema-related API 处理演进。中文阅读重点是区分 prim 的组成结果、API schema 应用、schema class、registry 查询和自动应用规则。它不是简单的版本号说明，而是对 composition、authoring API 和兼容性风险的系统讨论。Chinese is the main reading path; English identifiers, API names, schema names, tokens, property names, code, commands, and links are retained for verification.

中文主阅读路线 / Chinese Main Reading Path

  1. 遇到 schema 行为差异时，先确认 prim 应用了哪些 concrete schema 和 API schema，再通过 registry 或 UsdPrim API 查看版本相关信息。Operational reading and debugging path.
  2. 设计新 schema 版本时，评估是否破坏既有资产、是否需要 versioned schema name、是否影响 C++ class 和自动应用规则。Operational reading and debugging path.
  3. 迁移资产时，保留旧层的 composition 证据，逐步更新 schema 应用和属性，而不是一次性假设所有 prim 都能采用新版本。Operational reading and debugging path.

边界、误读点与调试路径 / Boundaries, Misreads, Debugging

  • schema 版本不是 prim 的单一全局版本；一个 prim 可以同时涉及多个 schema 和 API schema。Boundary or common misread to avoid.
  • 版本化不能替代清晰迁移策略，旧资产、工具和 plugin 仍需要兼容处理。Boundary or common misread to avoid.
  • Schema Registry、UsdPrim、multiple-apply API schema 等术语必须原样保留，中文只解释语义。Boundary or common misread to avoid.
