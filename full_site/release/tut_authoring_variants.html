范围说明 / Scope Note

本页是 OpenUSD release 教程中 “Authoring Variants” 的完整双语参考页。中文主线解释如何在 `HelloWorld.usda` 上创建 `shadingVariant`，如何把颜色 opinion 写入不同 variant，为什么要先清除本地 `primvars:displayColor` opinion，以及 `stage.ExportToString()`、`GetRootLayer().Export()`、`usdview` metadata 面板和 session layer 中 variant selection 的差异。英文 API 名称、文件名、prim path、token、代码和官方链接保留，用于和官方教程核对。

This local bilingual page explains the OpenUSD tutorial for authoring variants in Chinese first while retaining official API names, file names, prim paths, tokens, code, and links for comparison.

逐段双语理解 / Paragraph-Level Bilingual Coverage

本教程接在 `Inspecting and Authoring Properties` 之后，继续使用 `USD/extras/usd/tutorials/authoringVariants/HelloWorld.usda` 和同目录下的 `authorVariants.py`。前一页已经让读者理解怎样读取和 author 属性；本页进一步说明怎样把同一个 prim 的可选表达放进 variant set。这里的目标不是复制多个资产文件，也不是用时间采样表示颜色变化，而是在同一个 `hello` prim 下建立可切换的组合分支，让 downstream layer 或 `usdview` 可以选择 `red`、`blue`、`green` 等不同结果。

The tutorial continues from the property-authoring tutorial and uses the `HelloWorld.usda` layer plus `authorVariants.py` to author a variant set.

第一步是打开 stage 并清除 sphere 上已经 author 的 `primvars:displayColor`。教程保留这个动作是为了展示 USD composition 的强度边界：local opinion 比 variant opinion 更强。如果不调用 `Clear()`，后面即使正确创建了 `shadingVariant` 并在 `red`、`blue`、`green` variant 内写入颜色，最终 composed result 仍可能显示原来的本地蓝色，因为更强的 local opinion 会压过 variant 内部的 opinion。这个细节把本页和 `LIVERPS` strength ordering 直接连接起来。

The local `primvars:displayColor` opinion is cleared first because local opinions are stronger than variant opinions.

示例代码用 `Usd.Stage.Open('HelloWorld.usda')` 打开 layer，再通过 `UsdGeom.Gprim.Get(stage, '/hello/world').GetDisplayColorAttr()` 找到 sphere 的 display color 属性。`/hello/world` 是一个 `Sphere`，也是 `Gprim` 的 schema subclass，因此可以用 `UsdGeom.Gprim` API 访问 `displayColor`。清除属性值后，scene description 中仍然可以保留 `color3f[] primvars:displayColor` 的属性声明；清除的是具体 authored value，不等于把 schema property 从 prim 上删除。

The sphere at `/hello/world` is accessed through `UsdGeom.Gprim`, and `Clear()` removes the authored value while the property declaration may remain.

创建 variant set 的核心对象是 root prim `/hello`。代码先取得 `rootPrim = stage.GetPrimAtPath('/hello')`，然后调用 `rootPrim.GetVariantSets().AddVariantSet('shadingVariant')`。`GetVariantSets()` 返回的是 prim 上所有 variant sets 的容器；`AddVariantSet('shadingVariant')` 会在 prim metadata 中声明一个名为 `shadingVariant` 的 set，并把它加入 `prepend variantSets`。此时还没有任何具体颜色被写入，只是告诉 composition 系统：这个 prim 将拥有一个可选择的变体维度。

`GetVariantSets().AddVariantSet('shadingVariant')` declares a variant set on the `/hello` prim.

接着用 `vset.AddVariant('red')`、`vset.AddVariant('blue')`、`vset.AddVariant('green')` 声明三个 variant。这里的 variant 名称是选择键，不是 shader 名称，也不是 `token` 属性值；它们定义了 `variantSet "shadingVariant"` 下面的三个分支。刚声明完时这些分支可以是空的，真正的颜色 opinion 要在后面的 variant edit context 中写入。把“声明候选”和“写入候选内容”分开理解，是读懂本页代码顺序的关键。

`AddVariant()` creates named alternatives inside the `shadingVariant` set before any opinions are authored into them.

向某个 variant 写入内容时，必须先调用 `vset.SetVariantSelection('red')`，再进入 `with vset.GetVariantEditContext():`。`SetVariantSelection()` 决定当前选择哪个分支；`GetVariantEditContext()` 则把随后 author 的属性值重定向到被选择的 variant 内。若省略 edit context，`colorAttr.Set([(1,0,0)])` 会写成普通 local opinion，正好又回到本教程开头特意清除的那类强 opinion。换句话说，variant selection 只说明“当前看哪个分支”，variant edit context 才说明“当前写到哪个分支”。

`SetVariantSelection()` chooses the branch, and `GetVariantEditContext()` redirects authoring into that branch.

在 `red` variant 中，示例把 `/hello/world` 的 `primvars:displayColor` 设为 `[(1,0,0)]`；在 `blue` variant 中设为 `[(0,0,1)]`；在 `green` variant 中设为 `[(0,1,0)]`。写完后，root layer 中会同时包含 `variants = { string shadingVariant = "green" }` 这样的当前选择，以及 `variantSet "shadingVariant"` 下每个分支的具体 `over "world"` opinion。当前选择显示最后一次 `SetVariantSelection()` 的结果，而三个分支的 authoring 内容都保留在 layer 内。

The red, blue, and green variants each author a different `primvars:displayColor` opinion for `/hello/world`.

`stage.ExportToString(addSourceFileComment=False)` 展示的是 composed stage。composition 已经把 variant selection 解析掉，因此输出里通常只看到当前选中的颜色，而不是完整的 `variantSet` 结构。教程用这个步骤说明 flattening：导出 `UsdStage` 会写出组合后结果，像 variants 这样的 composition operators 已经被求值。读者不要把“composed result 里没有 variantSet”误解成 variant 没有写成功；它只是已经被当前 selection 应用了。

Exporting the `UsdStage` shows the composed result, where the selected variant has already been evaluated.

如果要保留 layer 中的 variant 结构，应该导出 root layer，例如 `stage.GetRootLayer().Export('HelloWorldWithVariants.usda')`。导出单个 layer 与导出 composed stage 不同：layer export 按原样写出该 layer 内的 scene description，不做 flattening；stage export 则写出组合后的可消费视图。这是本教程最容易被混淆的边界之一，尤其在调试“我的 variant 去哪里了”时，先确认自己看的到底是 layer 内容还是 stage flatten 结果。

Exporting the root layer preserves the authored variant structure, while exporting the stage writes the composed view.

把 `HelloWorldWithVariants.usda` 放进 `usdview` 后，选择左侧树里的 `hello` prim，再查看 metadata，可以看到 `shadingVariant` 对应的选择控件。这个控件改变的是 variant selection，而不是重写所有 variant 内容。用户在 `usdview` 中切换 `red`、`blue`、`green` 时，session layer 会记录一个稀疏的 override，例如在 `over "hello"` 上写入 `variants = { string shadingVariant = "red" }`。这和在另一个 referencing layer 里选择 variant 是同一种 composition 思路。

In `usdview`, changing the metadata combo box authors a sparse variant-selection opinion in the session layer.

从资产工作流角度看，variant set 适合表达同一资产的离散可选状态：材质方案、LOD、造型、开关配置、平台特定资源、渲染质量级别等。它不适合替代 `timeSamples`，因为 variant selection 代表组合分支选择，而不是随时间连续变化的值；它也不应被当作普通属性 override，因为 variant 内可以包含多个 prim 和属性 opinion，甚至可以在更复杂的资产中改变子树结构。variant 是 composition 维度，不只是属性枚举。

Variants represent discrete composition alternatives, not time samples or a simple enum-valued property.

本页的调试路径可以概括为四个检查点。第一，若颜色没有随 selection 改变，检查是否还有更强 local opinion，尤其是未清除的 `primvars:displayColor`。第二，若 layer 中没有 variant 内容，检查是否忘记 `GetVariantEditContext()`。第三，若 `usdview` 中没有可选项，检查 `AddVariantSet()` 与 `AddVariant()` 是否作用在 `/hello` root prim 上，而不是错误地作用在 `/hello/world` 或临时 prim 上。第四，若保存后丢失 variant 结构，检查是否错误使用 stage flatten export 而不是 root layer export。

Common debugging checks include local opinion strength, missing edit context, wrong prim path, and confusing stage export with layer export.

和前几篇教程放在一起读时，职责边界会更清楚：`Inspecting and Authoring Properties` 教会读者对单个属性 author opinion；`Referencing Layers` 教会读者把一个 layer 作为资产引用进另一个 layer；`Traversing a Stage` 教会读者观察 composed stage；`Authoring Variants` 则说明如何在同一个 prim 上组织可切换的 opinion 分支。它们共同构成 USD 非破坏性资产编辑的最小路径：先 author 数据，再组合 layer，再遍历 composed result，最后用 variant selection 切换资产状态。

This tutorial connects property authoring, references, traversal, and variant selection into one non-destructive asset-authoring path.

还要注意 variant 的“所在 prim”决定了它能影响的 namespace 范围。本教程把 `shadingVariant` 放在 `/hello` 上，因此 variant 内可以用 `over "world"` 修改子 prim `/hello/world` 的颜色。如果把 variant set 放在 `/hello/world` 上，selection 的表达位置、metadata 面板显示位置和 downstream override 路径都会不同。资产设计时通常把会一起切换的子树放在同一个较高层 prim 下，再把 variant set 放在那个聚合 prim 上；这样材质、LOD、几何替换和附加属性能作为一个选择整体被切换。

The prim that owns a variant set determines where the selection is expressed and what namespace subtree the variant can naturally organize.

本教程虽然只 author 颜色，但同一模式可以扩展到更复杂内容：variant 内可以定义新的 child prim，可以对已有 prim 写 `over`，也可以组合材质绑定、可见性、purpose、payload 或 reference 等 opinion。工程上要避免把互相独立的选择全部塞进同一个 set；材质颜色可放在 `shadingVariant`，几何精度可放在 `lodVariant`，平台资源可放在另一个 set。多个 variant sets 可以同时存在，composition 会按每个 set 的当前 selection 组合出最终结果。这样拆分后，下游 layer 可以只改一个维度，而不必重新 author 整个资产状态。

The same pattern can organize more than color; multiple variant sets can represent independent dimensions such as shading, LOD, or platform-specific resources.

读者复现时还应区分“authoring layer 中的 default selection”和“更强 layer 或 session layer 中的选择”。`SetVariantSelection('green')` 会在当前 edit target 中写入 selection；`usdview` metadata combo box 也会写入 selection，但通常落在 session layer。两个 selection 都是 composition opinion，谁生效取决于 layer stack 的强度。若在工具里看到选择值和文件里默认值不同，不一定是文件损坏，很可能是 session layer 或上游引用层提供了更强的 sparse selection。调试时先看当前 edit target。

Variant selections are composition opinions, so a stronger layer or the session layer can override the default selection authored in the asset layer.

教程步骤 / Tutorial Steps

  1. 复制 `USD/extras/usd/tutorials/authoringVariants/HelloWorld.usda` 和 `authorVariants.py` 到可写工作目录。Copy the tutorial files to a writable working directory.
  2. 用 `Usd.Stage.Open('HelloWorld.usda')` 打开 stage，并清除 `/hello/world` 上已有的 `primvars:displayColor` authored value。Open the stage and clear the authored display-color value.
  3. 在 `/hello` 上调用 `GetVariantSets().AddVariantSet('shadingVariant')`，声明变体维度。Declare the `shadingVariant` variant set on `/hello`.
  4. 用 `AddVariant('red')`、`AddVariant('blue')`、`AddVariant('green')` 创建三个候选分支。Create the red, blue, and green variants.
  5. 对每个候选分支先 `SetVariantSelection()`，再进入 `GetVariantEditContext()` 写入颜色 opinion。Select each variant and author opinions inside its edit context.
  6. 用 `stage.ExportToString()` 检查 composed result，用 `GetRootLayer().Export()` 保存保留 variants 的 layer。Compare composed-stage export with root-layer export.
  7. 在 `usdview` 中打开 `HelloWorldWithVariants.usda`，通过 metadata 面板切换 variant，并检查 session layer 中的 sparse selection opinion。Use `usdview` to toggle the selection and inspect the session-layer opinion.

关键 API 边界 / Key API Boundaries

  • `GetVariantSets()` 返回 prim 上的 variant-set 容器；`AddVariantSet()` 声明一个 set，不等于已经写入某个具体分支。`GetVariantSets()` returns the variant-set container; `AddVariantSet()` declares a set.
  • `AddVariant()` 只创建命名候选；候选内容需要在 edit context 内 author。`AddVariant()` creates a named alternative; its contents are authored later.
  • `SetVariantSelection()` 是 composition selection；`GetVariantEditContext()` 是 authoring target。`SetVariantSelection()` chooses the branch; `GetVariantEditContext()` chooses where authoring goes.
  • `Clear()` 清的是已有 authored value，主要用于避免更强 local opinion 压过 variant opinion。`Clear()` removes the stronger local authored value so variant opinions can compose visibly.
  • 导出 `UsdStage` 会得到 flatten 后的组合结果；导出 root layer 才保留 variant set 结构。Stage export writes a composed result; root-layer export preserves authored variants.

代码摘录 / Code Excerpts

from pxr import Usd, UsdGeom

stage = Usd.Stage.Open('HelloWorld.usda')
colorAttr = UsdGeom.Gprim.Get(stage, '/hello/world').GetDisplayColorAttr()
colorAttr.Clear()
rootPrim = stage.GetPrimAtPath('/hello')
vset = rootPrim.GetVariantSets().AddVariantSet('shadingVariant')

vset.AddVariant('red')
vset.AddVariant('blue')
vset.AddVariant('green')
vset.SetVariantSelection('red')
with vset.GetVariantEditContext():
    colorAttr.Set([(1, 0, 0)])

vset.SetVariantSelection('blue')
with vset.GetVariantEditContext():
    colorAttr.Set([(0, 0, 1)])

vset.SetVariantSelection('green')
with vset.GetVariantEditContext():
    colorAttr.Set([(0, 1, 0)])
print(stage.ExportToString(addSourceFileComment=False))
stage.GetRootLayer().Export('HelloWorldWithVariants.usda')

Composition 与导出差异 / Composition and Export Differences

如果目标是把当前选择后的结果交给只消费普通 scene description 的下游工具，flatten 后的 `UsdStage` export 可能足够；如果目标是保留可切换资产，必须保存 root layer 或 authoring layer。前者会把当前 `shadingVariant` selection 的结果应用到输出中，后者会保留 `variantSet "shadingVariant"` 的三个分支和当前 selection metadata。

A composed-stage export applies the selected variant, while a layer export preserves the authored variant structure.

over "hello" (
    variants = {
        string shadingVariant = "red"
    }
)
{
}

上面的 session-layer 片段表达的是“当前会话选择 red”，不是把 red 分支的所有属性复制到 session layer。它是 sparse opinion，只改变 selection。这个行为与在更强 layer 中重新选择 variant 的方式一致，也是 USD 变体适合资产发布和下游覆盖的原因。

The session layer can author a sparse variant-selection opinion without copying all variant contents.

常见误读与调试 / Common Misreadings and Debugging

  • 如果切换 variant 后颜色不变，先检查是否仍有更强的 local `primvars:displayColor` opinion。If color does not change, check for a stronger local display-color opinion.
  • 如果 variant 分支为空，检查是否只调用了 `SetVariantSelection()`，但忘记进入 `GetVariantEditContext()`。If the branches are empty, verify that authoring occurred inside `GetVariantEditContext()`.
  • 如果 `usdview` 看不到 combo box，确认 `shadingVariant` 是声明在 `/hello` 上，而不是错误 prim 上。If the combo box is missing, confirm the variant set is on `/hello`.
  • 如果保存文件后 variants 消失，检查是否用了 `stage.ExportToString()` 或 stage export 的 flatten 视图。If variants disappear after export, check whether a flattened stage export was used.
  • 不要把 variant 当作动画帧；需要连续变化时使用 `timeSamples`，需要离散组合分支时使用 variant selection。Do not confuse variants with animation frames or `timeSamples`.
