逐段双语理解 / Paragraph-Level Bilingual Coverage

这页要读成“usdview 扩展点教程”，不是一般 Python 脚本教程。usdview 插件运行在 usdview 进程里，能访问当前 stage、选择状态、视图设置和 Qt 主窗口，因此它适合做检查、辅助可视化、截图、批量状态输出或内部工具入口。它不应该被误解为修改 USD 核心格式的机制；插件只是给 usdview 加命令和 UI，并通过 usdviewApi 与当前会话交互。

The page is a usdview extension tutorial. A usdview plugin adds commands and UI to the running usdview application and interacts with the current session through usdviewApi.

第一步是创建插件目录，例如某个 usdviewPlugins 目录下的 tutorialPlugin。这个两层结构有意区分“插件集合目录”和“单个插件模块目录”：PYTHONPATH 应该指向可以 import tutorialPlugin 的上层目录，而 PXR_PLUGINPATH_NAME 应该指向包含 plugInfo.json 的插件目录。很多加载失败都来自这两个路径混淆，或者文件名没有严格写成 __init__.py 与 plugInfo.json。

The tutorial creates a tutorialPlugin package inside a plugins directory. PYTHONPATH should expose the Python package, while PXR_PLUGINPATH_NAME should point libplug at the plugin directory containing plugInfo.json.

PluginContainer 是插件的组织单元。它不是一个命令本身，而是一个负责注册命令、再把命令挂到 usdview UI 上的容器。官方示例里的 TutorialPluginContainer 派生自 pxr.Usdviewq.plugin.PluginContainer，并通过 Tf.Type.Define(TutorialPluginContainer) 注册为 Tf.Type。这个注册步骤很关键，因为 libplug 需要通过类型系统找到容器，而不是靠用户手工 import 后调用。

PluginContainer is the organizing object for usdview plugins. It registers command plugins, exposes them in the UI, and is declared as a Tf.Type so libplug can discover it.

registerPlugins 负责把命令加入 plugin registry。每个 command plugin 都需要一个全局唯一的 identifier、一个显示名和一个 callback。官方示例使用 TutorialPluginContainer.printMessage 作为 identifier，用 Print Message 作为菜单文字，并让 callback 接收 usdviewApi 后打印 Hello, World!。唯一 identifier 最好带容器名前缀，避免多个生产插件之间发生命名冲突。

registerPlugins adds command plugins to the registry. Each command has a globally unique identifier, display name, and callback that receives a usdviewApi object.

configureView 负责把已经注册的命令暴露给用户。官方示例通过 plugUIBuilder.findOrCreateMenu("Tutorial") 创建或找到 Tutorial 菜单，然后把 _printMessage 命令加入菜单。这里要注意 UI 能力边界：教程说明插件当前可以创建简单的菜单，也可以打开新的 Qt 窗口；复杂 UI 仍应尊重 usdview 的主窗口生命周期和交互模型。

configureView exposes registered commands in the usdview UI. The tutorial creates a Tutorial menu and adds the Print Message command to it.

plugInfo.json 是发现链路的核心文件。它声明插件 Type 为 python，Name 为 tutorialPlugin，并在 Info.Types 中把 tutorialPlugin.TutorialPluginContainer 标记为基于 pxr.Usdviewq.plugin.PluginContainer 的类型，还提供 displayName。修改示例时，必须让 Name 匹配 Python 模块名，让 Types key 匹配容器类型路径，让 displayName 表达给用户看的插件名称。

plugInfo.json is the discovery resource. It declares a Python plugin, the Python module name, the PluginContainer type, its base class, and a display name.

环境变量是第二个常见断点。libplug 会直接 import Python 模块，所以 PYTHONPATH 必须包含能导入 tutorialPlugin 的父目录；libplug 还要扫描插件资源，所以 PXR_PLUGINPATH_NAME 必须包含插件目录本身。验证成功的最小信号是：打开 usdview 后出现 Tutorial 菜单，点击 Print Message 后控制台输出 Hello, World!。若菜单不出现，先用绝对路径检查两个环境变量，再检查文件名是否准确。

Environment setup is a common failure point. The plugin package must be importable through PYTHONPATH, and libplug must find plugInfo.json through PXR_PLUGINPATH_NAME.

usdviewApi 是插件与当前 usdview 会话交互的入口。官方列出的 dataModel 包含 stage、currentFrame、viewSettings 和 selection；qMainWindow 可作为 Qt dialog 的父窗口；PrintStatus 可以在 usdview 底部打印状态；GrabViewportShot 和 GrabWindowShot 可以抓取 viewport 或整个窗口截图。插件作者应优先通过这些受支持入口操作 usdview，不要绕过 API 直接操纵内部对象。

usdviewApi exposes supported integration points such as dataModel, qMainWindow, PrintStatus, GrabViewportShot, and GrabWindowShot.

dataModel 的读取边界也要清楚。stage 是当前 Usd.Stage，selection 表示当前 prim 和 property 选择，viewSettings 影响 viewport 表现，例如 complexity、freeCamera、renderMode。插件可以读取或调整这些状态，但应把修改范围限制在用户明确触发的命令中，避免插件加载时就改变视图、选择或 stage 内容。加载阶段越轻，usdview 启动体验越稳定。

dataModel gives access to the current stage, frame, viewport settings, and selection state. Plugins should avoid changing user-visible state during load.

deferredImport 是为了保护 usdview 启动时间。官方建议把实际插件逻辑放到 printer.py 等独立文件里，再在 PluginContainer 中用 self.deferredImport(".printer") 延迟导入。这样 usdview 启动时不会立即 import 重模块；第一次执行命令时才真正加载目标模块。要注意 deferredImport 在真正导入前不知道模块里有什么对象，它假设访问的是函数；如果目标函数不存在，调用时会抛出 ImportError。

deferredImport delays importing plugin logic until a command is first invoked. It keeps usdview launch fast, but missing target functions become ImportError at call time.

sendMail.py 示例说明插件可以做比 Hello, World! 更实用的事：它打开一个 dialog，让用户发送包含 usdview 截图的邮件，并展示 GrabWindowShot、GrabViewportShot、qMainWindow 以及从 usdviewApi 读取上下文数据的用法。阅读这个示例时，重点不是邮件功能本身，而是“如何把 usdview 当前状态、截图和 Qt UI 组合成一个可复用工具”。

The sendMail.py example shows a practical plugin that opens a dialog, captures usdview screenshots, and uses usdviewApi context in the email body.

生产环境组织方式也有明确建议。虽然 PluginContainer 可以发现和执行任意数量的插件模块，但官方说明 Pixar 倾向把插件组织在一个由维护者设置好的模块中。这样普通用户新增命令时不必理解 plugInfo.json，也更容易把所有菜单组织成连贯、有序的结构。换句话说，面向团队的插件平台应把发现和注册细节集中维护，把命令实现留给工具作者。

For production organization, the tutorial recommends centralizing plugin setup so users can add commands without editing plugInfo.json and so menus remain coherent.

插件生命周期也要按阶段理解。发现阶段由 libplug、plugInfo.json、PXR_PLUGINPATH_NAME 和 PYTHONPATH 决定；注册阶段由 registerPlugins 把命令加入 registry；展示阶段由 configureView 决定菜单或窗口入口；执行阶段才真正调用 callback 并传入 usdviewApi。把这四个阶段分清楚，调试时就能快速定位是“没有被发现”“命令没有注册”“菜单没有挂上”还是“命令执行失败”。

It is useful to separate discovery, registration, UI exposure, and command execution. Each stage has different failure modes and debugging signals.

usdview 插件的设计边界是“辅助当前会话”，不是“替代完整应用”。如果命令需要修改 stage，应清楚区分只读检查、UI 状态变化和实际 authoring 操作；如果命令只是截图或状态输出，也应避免意外改变 selection、currentFrame 或 viewSettings。插件越贴近用户当前可见状态，就越应该让副作用显式、可撤销或至少可解释。

A usdview plugin is best treated as a helper for the current usdview session. Side effects on stage, selection, frame, or viewport settings should be deliberate and visible.

registerCommandPlugin 的 callback 签名是另一个关键契约。所有 command plugin callback 都以 usdviewApi 作为唯一参数，因此业务逻辑应该从 usdviewApi.dataModel 读取 stage、selection 或 viewSettings，而不是依赖全局变量保存旧状态。这样插件在不同文件、不同 frame、不同选择状态下被触发时，能基于当前会话做出正确行为。

Command callbacks receive usdviewApi as their only argument. They should query the current session through that object rather than relying on stale global state.

菜单组织也不是小事。一个团队可能有检查、发布、截图、调试、可视化辅助等很多命令。如果每个插件都随意创建顶层菜单，usdview 菜单栏会很快混乱。官方提到集中模块的优势，实质上是在建议把工具入口设计成稳定的信息架构：维护者控制菜单顺序和分组，工具作者只贡献命令实现。

Menu organization matters in production. Centralized plugin setup helps keep commands grouped and ordered instead of scattering many independent top-level menus.

验证一个插件是否合格，不应只看“Hello, World!” 是否打印。完整检查至少包括：usdview 启动时没有明显变慢；插件能被发现；菜单名称和命令名称符合预期；点击命令后 callback 能拿到当前 usdviewApi；命令对 stage、selection、viewSettings 或截图的访问符合预期；关闭和重新打开 usdview 后仍能稳定加载。这些检查能覆盖发现、注册、UI 和执行四个层面。

A useful plugin validation path checks launch cost, discovery, menu exposure, callback execution, access to current usdview state, and repeatability across usdview launches.

执行顺序 / Execution Path

  1. 创建插件集合目录和 tutorialPlugin 模块目录，确保模块目录中有 __init__.py。
  2. 在 __init__.py 中定义 TutorialPluginContainer，派生 PluginContainer，并用 Tf.Type.Define 注册类型。
  3. 在 registerPlugins 中注册 command plugin，给它唯一 identifier、显示名和接收 usdviewApi 的 callback。
  4. 在 configureView 中用 plugUIBuilder.findOrCreateMenu 创建菜单，并把命令加入菜单。
  5. 创建 plugInfo.json，声明 Type、Name、Types、bases 和 displayName。
  6. 设置 PYTHONPATH 指向可 import 插件模块的父目录，设置 PXR_PLUGINPATH_NAME 指向包含 plugInfo.json 的插件目录。
  7. 启动 usdview，确认 Tutorial 菜单出现，点击 Print Message 后控制台输出 Hello, World!。
  8. 将较重逻辑移到独立模块，并用 deferredImport 延迟导入，避免拖慢 usdview 启动。

关键片段 / Key Fragments

以下片段是阅读导向的摘要，保留官方标识以便与源码和文档对照。

mkdir -p <some path>/usdviewPlugins/tutorialPlugin/

from pxr import Tf
from pxr.Usdviewq.plugin import PluginContainer

class TutorialPluginContainer(PluginContainer):
    def registerPlugins(self, plugRegistry, usdviewApi):
        self._printMessage = plugRegistry.registerCommandPlugin(
            "TutorialPluginContainer.printMessage",
            "Print Message",
            printMessage)

    def configureView(self, plugRegistry, plugUIBuilder):
        tutMenu = plugUIBuilder.findOrCreateMenu("Tutorial")
        tutMenu.addItem(self._printMessage)

Tf.Type.Define(TutorialPluginContainer)
{
    "Plugins": [
        {
            "Type": "python",
            "Name": "tutorialPlugin",
            "Info": {
                "Types": {
                    "tutorialPlugin.TutorialPluginContainer": {
                        "bases": ["pxr.Usdviewq.plugin.PluginContainer"],
                        "displayName": "Usdview Tutorial Plugin"
                    }
                }
            }
        }
    ]
}

常见误读与调试 / Common Misreadings And Debugging

  • 如果 Tutorial 菜单不出现，先检查 PYTHONPATH 是否指向 tutorialPlugin 的父目录，PXR_PLUGINPATH_NAME 是否指向含 plugInfo.json 的目录。
  • 如果模块无法 import，检查目录是否有 __init__.py，模块名是否与 plugInfo.json 的 Name 完全一致。
  • 如果插件被发现但命令不可用，检查 registerCommandPlugin 的 identifier 是否唯一，callback 是否接受 usdviewApi 参数。
  • 如果菜单出现但点击后异常，检查 callback 内部逻辑，以及 deferredImport 指向的函数名是否真实存在。
  • 如果 usdview 启动变慢，把重依赖移动到独立文件，并通过 deferredImport 延迟到用户第一次触发命令时加载。
  • 如果要打开 Qt dialog，用 usdviewApi.qMainWindow 作为父窗口；不要把 qMainWindow 当作任意内部状态的入口。

术语与保留英文 / Terms Retained

PluginContainer

usdview 插件容器，负责注册命令并把命令暴露到 usdview UI。

registerPlugins

注册 command plugin 的方法，创建 identifier、display name 和 callback 的绑定。

configureView

把已注册命令放入菜单或窗口的 UI 配置方法。

plugInfo.json

libplug 发现 Python 插件和 PluginContainer 类型的资源文件。

usdviewApi

插件访问当前 stage、selection、viewSettings、qMainWindow 和截图功能的受支持入口。

deferredImport

延迟导入插件逻辑的机制，用于避免重模块拖慢 usdview 启动。
